# Production Database Hybrid Architecture — Design

**Date:** 2026-07-06
**Status:** Draft for user review
**Decision:** Supabase PostgreSQL is the primary production database. MongoDB is reserved for a later content/draft/generated-data layer. Phase 1 migrates user data first and keeps learning content static.

## 1. Overview & Goals

### Goal

Move the parts of the AI learning platform that should persist per user into a production database while preserving the current static-content strengths of the repo.

The platform currently works well as a static/local-first Next.js app: lessons are MDX/JSON in the repo, search is generated by Pagefind, and progress is stored in `localStorage`. For production, user-owned state should become durable, queryable, and syncable across devices. Supabase PostgreSQL should become the source of truth for that state.

### Primary direction

Use **Minimal production DB + hybrid-ready architecture**:

```txt
Static content layer
  content/**/*.mdx
  quiz/challenge JSON
  Pagefind build-time index
        │
        ▼
Next.js app
        │
        ├── Anonymous users → localStorage adapter
        │
        └── Authenticated users → Supabase adapter
                                  │
                                  ▼
                         Supabase Auth + PostgreSQL

Future optional layer:
MongoDB for drafts, generated content, AI sessions, or flexible content workflows
```

### Why this direction

- The existing repo is content-heavy and static-friendly. Moving all lessons/quizzes/challenges into a database immediately would force CMS, import/export, cache invalidation, revalidation, and search-index changes.
- User data has the clearest production value: progress, attempts, bookmarks, notes, and snippets should sync across devices.
- Supabase PostgreSQL gives auth integration, relational constraints, JSONB, indexes, and Row Level Security. It is enough for Phase 1 user data.
- MongoDB remains useful later for flexible drafts or AI-generated data, but adding it to the request path now would increase operational complexity without immediate benefit.

## 2. Scope

### In scope for Phase 1

- Add Supabase project configuration via environment variables.
- Use Supabase Auth for authenticated user identity.
- Add PostgreSQL tables for user-owned data:
  - profiles
  - lesson progress
  - topic progress or derived progress cache
  - quiz attempts
  - challenge attempts
  - bookmarks
  - notes
  - saved snippets
- Add RLS policies so users can only access their own records.
- Keep anonymous/local-first behavior using `localStorage`.
- Add a storage abstraction so app code can use local or remote storage without duplicating business logic.
- Add local-to-remote sync after login.
- Update production configuration such as `NEXT_PUBLIC_SITE_URL` and remove hardcoded `https://ai-roadmap.local` usage.
- Add `.env.example` and deployment notes for Supabase-backed production.

### Out of scope for Phase 1

- Moving MDX lesson body content into a database.
- Moving all quiz/challenge definitions into a database.
- Building an admin CMS.
- Making MongoDB part of the runtime request path.
- Real-time collaboration.
- Paid subscriptions/billing.
- Advanced analytics/event warehouse.
- Fully sandboxing playground execution. This remains a separate production-hardening track.

### Future scope

- MongoDB for drafts, AI-generated content, AI learning plans, chat sessions, or flexible payloads.
- Admin workflow for content publishing.
- Search-index regeneration from database-backed content.
- More robust analytics and recommendation systems.

## 3. Data Ownership Model

### Static source of truth

These remain in the repo for Phase 1:

- Roadmap structure
- Phase/topic metadata
- Lesson MDX body
- Quiz definitions
- Challenge definitions
- Project pages
- Pagefind search index generated at build time

The app continues to identify static content by stable slugs such as `lesson_slug`, `topic_slug`, `quiz_slug`, and `challenge_slug`.

### Supabase PostgreSQL source of truth

Supabase stores user-owned state:

- Who the user is
- What the user completed
- Quiz/challenge attempt history
- Bookmarks
- Personal notes
- Saved code snippets
- Optional dashboard/activity data

### MongoDB future role

MongoDB should not be required for Phase 1. It can be introduced later for data that benefits from document flexibility:

- Draft lessons before publication
- AI-generated study plans
- AI chat/session transcripts
- Generated quiz variants
- Raw event payloads if a future analytics pipeline needs document storage

MongoDB data should publish into the app through a deliberate workflow rather than being mixed into every user-facing read path from day one.

## 4. PostgreSQL Schema Design

The schema below assumes Supabase Auth is used. `auth.users.id` is the user identity. All app-owned tables live in `public` unless the implementation later chooses a separate schema.

### 4.1 `profiles`

```sql
create table public.profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  display_name text,
  avatar_url text,
  role text not null default 'learner' check (role in ('learner', 'admin')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

Purpose:

- Public app profile linked one-to-one with Supabase Auth user.
- Role is intentionally minimal: learner/admin.
- More profile fields can be added later without affecting progress tables.

### 4.2 `lesson_progress`

```sql
create table public.lesson_progress (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  lesson_slug text not null,
  status text not null default 'not_started'
    check (status in ('not_started', 'in_progress', 'completed')),
  progress_percent integer not null default 0 check (progress_percent between 0 and 100),
  last_seen_at timestamptz,
  completed_at timestamptz,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (user_id, lesson_slug)
);

create index lesson_progress_user_status_idx
  on public.lesson_progress (user_id, status);
```

Purpose:

- One row per user per lesson.
- Tracks durable completion and recency.
- `lesson_slug` references static content by convention, not by FK, because lessons remain in the repo.

### 4.3 `topic_progress`

```sql
create table public.topic_progress (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  topic_slug text not null,
  status text not null default 'not_started'
    check (status in ('not_started', 'in_progress', 'completed')),
  completed_lessons integer not null default 0 check (completed_lessons >= 0),
  total_lessons integer not null default 0 check (total_lessons >= 0),
  updated_at timestamptz not null default now(),
  unique (user_id, topic_slug)
);
```

Purpose:

- Provides fast dashboard reads.
- Can be derived from lesson progress, but a cached table is useful if the dashboard expects quick aggregate queries.
- If implementation finds existing topic progress is simple enough to derive on read, this table can be skipped initially. The preferred production path is to keep it if dashboard performance matters.

### 4.4 `quiz_attempts`

```sql
create table public.quiz_attempts (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  quiz_slug text not null,
  score integer not null check (score >= 0),
  total integer not null check (total >= 0),
  answers jsonb not null default '{}'::jsonb,
  duration_seconds integer check (duration_seconds is null or duration_seconds >= 0),
  completed_at timestamptz not null default now(),
  created_at timestamptz not null default now()
);

create index quiz_attempts_user_quiz_idx
  on public.quiz_attempts (user_id, quiz_slug, completed_at desc);
```

Purpose:

- Keeps attempt history, not only latest state.
- `answers` stores selected answer IDs, correctness, and optional per-question timing without over-normalizing too early.

### 4.5 `challenge_attempts`

```sql
create table public.challenge_attempts (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  challenge_slug text not null,
  status text not null check (status in ('started', 'passed', 'failed')),
  language text not null,
  code text,
  test_results jsonb not null default '{}'::jsonb,
  duration_seconds integer check (duration_seconds is null or duration_seconds >= 0),
  submitted_at timestamptz not null default now(),
  created_at timestamptz not null default now()
);

create index challenge_attempts_user_challenge_idx
  on public.challenge_attempts (user_id, challenge_slug, submitted_at desc);
```

Purpose:

- Stores challenge submissions and result payloads.
- `code` is user-owned and should have length limits in the application layer.
- Because current playground execution is client-side, server-side test trust should be treated carefully. This table records user activity; it should not be treated as authoritative certification unless challenge execution becomes server-verified later.

### 4.6 `bookmarks`

```sql
create table public.bookmarks (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  target_type text not null check (target_type in ('lesson', 'topic', 'quiz', 'challenge', 'project')),
  target_slug text not null,
  created_at timestamptz not null default now(),
  unique (user_id, target_type, target_slug)
);
```

Purpose:

- Simple saved-items model across static content types.

### 4.7 `notes`

```sql
create table public.notes (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  lesson_slug text not null,
  content text not null,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index notes_user_lesson_idx
  on public.notes (user_id, lesson_slug, updated_at desc);
```

Purpose:

- Personal lesson notes.
- Notes are private by default.
- Application should enforce reasonable content length limits.

### 4.8 `saved_snippets`

```sql
create table public.saved_snippets (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  title text not null,
  language text not null,
  code text not null,
  lesson_slug text,
  challenge_slug text,
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);

create index saved_snippets_user_updated_idx
  on public.saved_snippets (user_id, updated_at desc);
```

Purpose:

- Lets users save playground/challenge code.
- Records optional context so snippets can be shown on a lesson or challenge page.

## 5. Row Level Security Baseline

All user-owned tables must have RLS enabled. The baseline policy is: authenticated users can select/insert/update/delete only rows where `user_id = auth.uid()`.

Example policy shape:

```sql
alter table public.lesson_progress enable row level security;

create policy "Users can read own lesson progress"
  on public.lesson_progress
  for select
  using (auth.uid() = user_id);

create policy "Users can insert own lesson progress"
  on public.lesson_progress
  for insert
  with check (auth.uid() = user_id);

create policy "Users can update own lesson progress"
  on public.lesson_progress
  for update
  using (auth.uid() = user_id)
  with check (auth.uid() = user_id);

create policy "Users can delete own lesson progress"
  on public.lesson_progress
  for delete
  using (auth.uid() = user_id);
```

Apply the same pattern to:

- `topic_progress`
- `quiz_attempts`
- `challenge_attempts`
- `bookmarks`
- `notes`
- `saved_snippets`

For `profiles`:

- Users can read and update their own profile.
- Public profile reads should remain disabled unless a feature requires public user pages.
- Admin policies should not be added until admin features exist.

Service-role keys must never be exposed to the browser. The browser should use the anonymous key plus RLS.

## 6. Auth & Session Model

### Recommended Supabase integration

Use Supabase's Next.js-compatible packages:

- `@supabase/supabase-js`
- `@supabase/ssr`

Use server/client helpers along these lines:

```txt
lib/supabase/client.ts      browser client
lib/supabase/server.ts      server component / server action client
lib/supabase/middleware.ts  optional session refresh middleware
```

### Auth UX

Phase 1 should support:

- Sign in
- Sign out
- Basic profile creation/update
- Optional account button in the existing navigation
- Anonymous mode remains available

### Anonymous-to-authenticated flow

When a user logs in:

1. Read local progress/attempt/bookmark/note/snippet data.
2. Fetch existing remote data from Supabase.
3. Merge with deterministic rules.
4. Upsert merged state into Supabase.
5. Mark local data as synced or keep it as a fallback cache.

The app should not delete local data immediately after sync. Keeping a local cache helps with temporary offline/network failures.

## 7. Storage Adapter Design

Introduce explicit data-access boundaries instead of scattering Supabase calls across components.

### Core interfaces

```ts
interface ProgressStore {
  getLessonProgress(lessonSlug: string): Promise<LessonProgress | null>;
  listLessonProgress(): Promise<LessonProgress[]>;
  markLessonStarted(lessonSlug: string): Promise<void>;
  markLessonCompleted(lessonSlug: string): Promise<void>;
  updateLessonProgress(lessonSlug: string, progressPercent: number): Promise<void>;
}

interface AttemptStore {
  recordQuizAttempt(input: QuizAttemptInput): Promise<void>;
  listQuizAttempts(quizSlug?: string): Promise<QuizAttempt[]>;
  recordChallengeAttempt(input: ChallengeAttemptInput): Promise<void>;
  listChallengeAttempts(challengeSlug?: string): Promise<ChallengeAttempt[]>;
}

interface UserLibraryStore {
  listBookmarks(): Promise<Bookmark[]>;
  toggleBookmark(input: BookmarkTarget): Promise<void>;
  listNotes(lessonSlug?: string): Promise<Note[]>;
  saveNote(input: SaveNoteInput): Promise<Note>;
  deleteNote(noteId: string): Promise<void>;
  listSnippets(): Promise<SavedSnippet[]>;
  saveSnippet(input: SaveSnippetInput): Promise<SavedSnippet>;
  deleteSnippet(snippetId: string): Promise<void>;
}
```

### Implementations

```txt
lib/stores/local-progress-store.ts
lib/stores/supabase-progress-store.ts
lib/stores/local-attempt-store.ts
lib/stores/supabase-attempt-store.ts
lib/stores/local-user-library-store.ts
lib/stores/supabase-user-library-store.ts
lib/stores/store-provider.ts
lib/stores/sync-local-to-remote.ts
```

The provider chooses the implementation based on auth state:

```txt
No Supabase session → local stores
Supabase session exists → Supabase stores + local cache/sync
```

### Why this boundary matters

- Components stay mostly unchanged.
- Local anonymous mode remains a supported feature.
- Future Mongo/content changes do not leak into progress and attempt logic.
- Testing becomes easier because stores can be mocked.

## 8. Merge Rules for Local-to-Remote Sync

Use deterministic merge rules so login does not surprise users.

### Lesson progress

- `completed` wins over `in_progress` and `not_started`.
- Higher `progress_percent` wins.
- Latest `last_seen_at` wins.
- Preserve earliest `completed_at` if both local and remote are completed.

### Quiz attempts

- Append attempt history.
- Avoid duplicate insert if a local attempt has the same `quiz_slug`, `completed_at`, `score`, and `total` as a remote attempt.
- Dashboard can show best score and latest attempt derived from history.

### Challenge attempts

- Append attempt history.
- `passed` should be reflected in dashboard if any attempt passed.
- Do not treat client-side pass as a secure certificate.

### Bookmarks

- Union by `(target_type, target_slug)`.

### Notes

- If local note has no remote counterpart, insert it.
- If both exist for the same note ID or local sync key, latest `updated_at` wins.
- If conflict cannot be matched safely, keep both notes rather than overwriting.

### Saved snippets

- If the same local sync key exists remotely, latest `updated_at` wins.
- Otherwise insert as a new snippet.
- Keep application-level limits for title/code length.

## 10. Prisma vs Supabase SQL Decision

The repo already contains `prisma/schema.prisma`, but Phase 1 should prefer **Supabase SQL migrations + Supabase client** for user data.

Reasoning:

- Supabase Auth and RLS are central to this design.
- RLS policies are SQL-native.
- The browser can safely call Supabase with the anon key when RLS is correct.
- Adding Prisma immediately would create two ways to access the same data and may bypass RLS if used with privileged server credentials.

Recommended decision:

- Keep `prisma/schema.prisma` untouched initially or mark it as future/experimental in docs.
- Add Supabase SQL migrations or a documented SQL setup file.
- Only introduce Prisma later if server-side relational querying becomes complex enough to justify it.

## 11. MongoDB Decision

MongoDB should be configured as a future optional integration, not as part of Phase 1 runtime.

### Phase 1

- Add `MONGODB_URI` and `MONGODB_DB_NAME` to `.env.example` only if useful for planning.
- Do not install MongoDB client yet unless a concrete feature uses it.
- Do not read lessons/quizzes/challenges from MongoDB in production routes.

### Phase 2 candidates

```txt
mongodb collections:
  content_drafts
  generated_learning_plans
  ai_chat_sessions
  generated_quiz_variants
  content_import_jobs
```

### Publish model

Prefer a controlled publish pipeline:

```txt
MongoDB draft/generated content
  → validation
  → review/admin approval
  → export to static content or publish snapshot
  → rebuild Pagefind index
  → deploy/revalidate
```

This avoids mixing unreviewed dynamic content into SEO pages and keeps the current static content quality.

## 12. App Areas to Update

### Progress/dashboard

- Replace direct `localStorage` usage with `ProgressStore`.
- Use Supabase store after login.
- Dashboard should read remote data for authenticated users and local data for anonymous users.

### Quiz

- Record attempts through `AttemptStore`.
- Keep immediate quiz feedback client-side.
- Save completed attempts when available.

### Challenge/practice

- Record challenge submissions through `AttemptStore`.
- Save code/results as user-owned activity.
- Clearly separate client-side practice completion from trusted server validation.

### Bookmarks/notes/snippets

- Add or route existing UI actions through `UserLibraryStore`.
- These features should be login-aware but can keep anonymous local mode.

### Layout/auth navigation

- Show login/account state.
- Keep reading content available without login.
- Trigger local-to-remote sync after successful login.

## 13. Error Handling

### Supabase unavailable

- Do not block lesson reading.
- Show non-intrusive save/sync error for user-owned writes.
- Keep local fallback data so progress is not lost.

### Auth expired

- Refresh session via Supabase helpers/middleware.
- If refresh fails, switch to local mode and prompt user to sign in again.

### RLS rejection

- Treat as a production bug or auth mismatch.
- Surface a friendly error to the user.
- Log enough context in development to debug table/policy issues.

### Sync conflict

- Use merge rules from Section 8.
- Prefer preserving data over overwriting data.

## 14. Security & Privacy

- Enable RLS on every user-owned table before exposing writes from the browser.
- Never expose `SUPABASE_SERVICE_ROLE_KEY` to client code.
- Validate content slugs against known static content where practical.
- Limit text/code payload sizes for notes and snippets.
- Treat saved code as private user data.
- Add privacy copy explaining that authenticated progress is stored in Supabase and anonymous progress is stored locally.
- Do not mark client-side challenge results as secure certifications.
- Keep CSP/playground hardening as a separate production-hardening task because it has different risk boundaries.

- Sitemap/robots metadata use production site URL.

## 17. Success Criteria

Phase 1 is production-ready when:

- Users can authenticate with Supabase.
- Authenticated progress syncs across devices.
- Anonymous users can still use the app with local progress.
- Local progress can merge into remote progress after login.
- Quiz/challenge attempts persist for authenticated users.
- User bookmarks/notes/snippets are private and durable.
- RLS prevents cross-user data access.
- Static content, Pagefind search, and build behavior remain stable.
- MongoDB is not required for the production user-data path.

## 18. Explicit Non-Goals

- Do not rewrite the app into a fully dynamic CMS in Phase 1.
- Do not make MongoDB a dependency for reading normal lessons.
- Do not remove static MDX/JSON content until there is an approved CMS/content migration design.
- Do not rely on service-role database access from browser paths.
- Do not treat client-side challenge execution as secure validation.
