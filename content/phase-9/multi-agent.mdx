Multi-Agent Systems không tự động thông minh hơn vì có nhiều agent. Đây là **coordination architecture**: đổi single-agent simplicity lấy specialization, parallelism, critique, và governance complexity. Một hệ thống nhiều agent có thể tốt hơn, nhưng cũng có thể tốn hơn, chậm hơn và mâu thuẫn hơn nếu thiết kế sai.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: 3 agents research cùng đọc 1 source, tạo báo cáo mâu thuẫn, tốn 450 USD">
**Bối cảnh:** Một team build "research agent" với 3 specialist agents: Planner, Searcher, Writer. Mỗi agent có budget riêng, gọi tool riêng, không có shared state. Mục tiêu: tạo báo cáo thị trường.

**Sự cố:** Planner chia task, 3 Searcher chạy song song cùng tìm "thị trường X". Mỗi Searcher tìm 1 source, đọc 1 source, nhưng không có shared state. 3 Searcher đọc cùng 1 source, nhưng không biết nhau. Writer tổng hợp, nhưng 3 Searcher đưa ra 3 claims khác nhau (cùng source, khác cách diễn giải).

1. Không có shared state — agents không biết nhau đã làm gì.
2. Không có communication contract — agents chat free-form, không có structured handoff.
3. Không có budget per role — 3 agents × 10 steps × tool calls = 450 USD.
4. Không có deduplication — agents duplicate work.
5. Không có verifier — Writer tổng hợp mâu thuẫn không ai phát hiện.

**Hậu quả:** Báo cáo có 3 claims mâu thuẫn, 450 USD cho 1 task, và team phải tắt multi-agent để thiết kế lại shared state, communication contract và budget per role.
</Callout>

Bài học rút ra: multi-agent không tự động thông minh hơn. Nó thêm specialization và parallelism, nhưng cũng thêm coordination overhead, duplicated work, contradiction, latency và cost. More agents does not mean more intelligence.

## 2. Architecture Blueprint (Multi-Agent Coordination)

```text
[ Goal ]
        │
        ▼
[ Coordinator / graph ]
        │
        ▼
[ Specialist agents (roles, tools, budgets) ]
        │
        ▼
[ Artifacts / shared state ]
        │
        ▼
[ Synthesizer / verifier ]
        │
        ▼
[ Final output ]
```

Câu hỏi thiết kế:

1. Cần mấy agents?
2. Role nào có tools/permissions gì?
3. Agents giao tiếp theo contract nào?
4. Ai tổng hợp/chốt final output?
5. Khi nào cần human?
6. Budget/cost chia thế nào?

Output của multi-agent design không phải là "nhiều agents". Output là một tập roles, contracts, state, và budgets có kiểm soát.

## 3. Pattern Taxonomy

### Supervisor → worker

```text
Supervisor
 ├── Researcher
 ├── Analyst
 ├── Writer
 └── Verifier
```

Pros: central control, dễ audit, clear final decision. Cons: supervisor bottleneck, bad routing hại toàn bộ work.

### Pipeline

```text
Planner → Researcher → Writer → Reviewer → Finalizer
```

Pros: predictable, dễ eval stage-by-stage. Cons: error propagation, ít adaptive.

### Peer-to-peer

```text
Agent A ↔ Agent B ↔ Agent C
```

Pros: linh hoạt. Cons: khó kiểm soát, communication overhead, loop risk.

### Debate / adversarial

```text
Proposer → Critic → Judge
```

Pros: tốt cho review, surfacing weaknesses. Cons: có thể optimize rhetoric thay vì truth, judge model có thể bias.

### Blackboard / shared workspace

Agents đọc/ghi shared state. Pros: tốt cho research/code, persistent artifact. Cons: state conflicts, cần ownership rules.

<Callout type="warning" title="State graph tốt hơn free-form multi-agent">
Workflow có branches, validation gates và human checkpoints rõ thường dễ test/debug hơn multi-agent tự do chat với nhau. Đừng dùng peer-to-peer khi pipeline đủ.
</Callout>

## 4. Role Design

Mỗi agent cần role schema rõ:

```json
{
  "agent": "Reviewer",
  "objective": "Find correctness, security, and maintainability issues in code changes.",
  "tools": ["read_file", "run_tests"],
  "cannot": ["modify_code", "merge_pr"],
  "input_schema": {"diff": "string"},
  "output_schema": {
    "issues": "array",
    "risk_level": "low|medium|high",
    "requires_human": "boolean"
  },
  "budget_usd": 1.0,
  "escalation_rules": ["security_issue", "low_confidence"]
}
```

Thiết kế role tốt nghĩa là agent biết:

- việc gì làm,
- việc gì không được làm,
- output contract gì,
- khi nào escalate.

Role không có `cannot` không phải role — đó là agent có thể làm mọi thứ.

## 5. Communication Contracts

Tránh free-form chatter. Dùng structured messages:

```json
{
  "from": "Researcher",
  "to": "Writer",
  "artifact_type": "research_notes",
  "claims": [
    {
      "claim": "Hybrid search improves recall in mixed corpora.",
      "source": "doc_12",
      "confidence": 0.82
    }
  ],
  "open_questions": []
}
```

Structured contracts giúp:

- validate handoff,
- trace artifacts,
- avoid ambiguity,
- test role compliance.

Free-form chat giữa agents gây ambiguity, khó trace và khó test role compliance. Đừng để agents chat tự do — dùng contract.

## 6. LangGraph-style Stateful Graphs

Concepts:

- nodes = agents/tools/functions,
- edges = transitions,
- conditional edges = routing logic,
- parallel branches,
- checkpoints = persist state,
- shared state object,
- human approval nodes,
- retry/error edges.

Example graph:

```text
START
  ↓
Planner
  ↓
Researcher A ─┐
Researcher B ─┼→ Synthesizer → Critic → Human approval? → Final
Researcher C ─┘
```

Shared state:

```json
{
  "task": "Write market research report",
  "plan": [],
  "research_notes": [],
  "draft": null,
  "critic_findings": [],
  "approval_status": "pending"
}
```

State graph giúp control flow rõ, test được, và có checkpoint để resume. Đó là lý do LangGraph-style graphs tốt hơn free-form multi-agent cho production.

## 7. Human-in-the-Loop Checkpoints

Yêu cầu human khi:

- external side effects,
- financial actions,
- legal/medical claims,
- low confidence,
- conflicting evidence,
- high cost continuation,
- policy violation,
- customer-facing communication,
- security-sensitive changes.

Checkpoint schema:

```json
{
  "checkpoint_id": "cp_123",
  "reason": "High-impact email send",
  "proposed_action": {},
  "agent_rationale": "...",
  "evidence": [],
  "options": ["approve", "reject", "edit", "escalate"]
}
```

Human checkpoint không phải là "model hỏi user". Nó là runtime gate: agent đề xuất, human approve, runtime execute. Đừng để model tự quyết high-risk actions.

## 8. Use Cases: Code Review và Research

### Code Review Agent

Agents:

1. Reviewer — tìm issues.
2. Security critic — tìm vulnerabilities.
3. Improver — đề xuất patch.
4. Tester — chạy tests trong sandbox.
5. Judge — tổng hợp, cite evidence.

```text
Input PR/diff
  ↓
Reviewer finds issues
  ↓
Security critic finds vulnerabilities
  ↓
Improver proposes patch
  ↓
Tester runs tests in sandbox
  ↓
Judge summarizes with evidence
  ↓
Human approves
```

Important: Improver không merge trực tiếp, Tester chạy trong sandbox, Security critic dùng explicit checklist, Judge cite test output và diffs.

### Research Agent

Agents: Planner, Searcher, Paper reader, Evidence extractor, Synthesizer, Skeptic/verifier, Citation checker.

Failure risks: citation laundering, source duplication, low-quality sources, unsupported synthesis, conflicting evidence bị ignore.

## 9. Multi-Agent Metrics

| Metric | Meaning |
|---|---|
| End task success | final result quality |
| Role compliance | agents stayed in role |
| Handoff validity | outputs match next input contract |
| Duplicate work rate | wasted parallel effort |
| Contradiction rate | inconsistent agent outputs |
| Unsupported claim rate | bad synthesis |
| Human override rate | trust signal |
| Cost per successful task | economics |
| Latency per workflow | UX |
| Loop/deadlock rate | orchestration quality |

Multi-agent metrics khác single-agent: thêm role compliance, handoff validity, duplicate work rate, contradiction rate. Đừng chỉ đo final task success.

## 10. Production Concerns

- shared state locking,
- parallel branch timeouts,
- cost multiplication,
- trace visualization,
- deterministic graph versioning,
- agent role versioning,
- tool permission per role,
- human approval SLA,
- checkpoint persistence,
- rollback for side effects,
- secure sandbox cho code agents,
- confidentiality giữa agents nếu cần.

<Callout type="warning" title="Multi-agent cost multiply nhanh">
3 agents × 10 steps × tool calls có thể tốn nhiều hơn single-agent 5x. Cần budget per role và per workflow. Nếu không có budget per role, multi-agent sẽ tiêu tiền nhanh hơn nhiều single-agent.
</Callout>

## Implementation Playbook: Code Review Multi-Agent Graph

```python
from dataclasses import dataclass, field


@dataclass
class ReviewState:
    code: str
    issues: list[dict] = field(default_factory=list)
    patch: str | None = None
    test_results: dict | None = None
    final_report: str | None = None
    requires_human: bool = False


def reviewer_node(state: ReviewState) -> ReviewState:
    state.issues = find_issues(state.code)
    return state


def improver_node(state: ReviewState) -> ReviewState:
    state.patch = propose_patch(state.code, state.issues)
    return state


def tester_node(state: ReviewState) -> ReviewState:
    state.test_results = run_tests_in_sandbox(state.patch)
    return state


def judge_node(state: ReviewState) -> ReviewState:
    state.final_report = summarize(state.issues, state.patch, state.test_results)
    state.requires_human = any(i["severity"] == "high" for i in state.issues)
    return state
```

Production checklist:

1. Mỗi role có objective/tools/output contract không?
2. Communication có structured contract không?
3. Có shared state schema không?
4. Có conditional routing rõ không?
5. Human checkpoints ở đúng chỗ không?
6. Có budget per role/workflow không?
7. Có trace artifact per agent không?
8. Có eval role compliance không?
9. Có deadlock/loop detection không?
10. Tool permissions per role không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Over-agentify simple workflow">
**Triệu chứng:** 3 bước deterministic được tạo thành 5 agents, cost tăng, latency tăng, không thêm value.

**Fix:** Nếu deterministic 3 bước giải quyết được, đừng tạo 5 agents. Multi-agent không tự động tốt hơn. Dùng least autonomy principle.
</Callout>

<Callout type="warning" title="Pitfall 2: Không có communication contract">
**Triệu chứng:** Agents chat free-form, handoff mơ hồ, khó trace, contradiction rate cao.

**Fix:** Dùng structured messages. Mỗi handoff có artifact_type, claims, source, confidence, open_questions. Validate handoff, trace artifacts, test role compliance.
</Callout>

<Callout type="danger" title="Pitfall 3: Không budget per role">
**Triệu chứng:** Multi-agent tiêu tiền nhanh hơn nhiều single-agent, không có budget per role, cost失控.

**Fix:** Mỗi role có budget_usd. Mỗi workflow có budget. Cần budget per role và per workflow. Nếu không, multi-agent sẽ tốn nhiều hơn single-agent 5x.
</Callout>

## Cost / Scalability / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Multi-agent thay vì single-agent | specialization, parallelism | coordination overhead, cost |
| Parallel branches | latency giảm | cost multiply, deadlock |
| Debate/critic | surface weaknesses | optimize rhetoric, judge bias |
| Human checkpoints | safety | latency, SLA |
| Shared state | dedup, trace | state conflicts, locking |
| No budget per role | đơn giản | cost失控 |

Multi-agent là coordination architecture, không phải magic intelligence. Mỗi agent thêm cost, nhưng cũng thêm control. Đừng thêm agents — thêm roles, contracts, và budgets.

## Answer key

1. Khi work phân rã được, cần specialists, parallel research, critique hoặc human checkpoints rõ.
2. Supervisor bottleneck và bad routing hại toàn bộ work.
3. Để validate handoff, trace artifacts, tránh ambiguity và test role compliance.
4. Nodes, edges, conditional edges, parallel branches, checkpoints, shared state, human approval nodes.
5. Side effects, financial actions, low confidence, conflicting evidence, customer-facing outputs, security changes.
6. Thêm role compliance, handoff validity, duplicate work rate, contradiction rate.
7. Nhiều agents × steps × tool calls, không có budget per role sẽ失控.
