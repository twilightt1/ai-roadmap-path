AI Agent không đơn giản là "LLM + tools". Một agent production là **stateful control loop**: nó nhận goal, quan sát state, chọn action, gọi tools, nhận observation, cập nhật state/memory, rồi quyết định tiếp tục, trả lời, hỏi user, hoặc escalate. Điểm nguy hiểm của agent không phải vì nó "suy nghĩ"; nguy hiểm là nó có thể **hành động lặp lại** dưới uncertainty — và mỗi hành động có thể tốn tiền, thay đổi dữ liệu thật, hoặc mở rộng quyền truy cập.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Agent research tốn 1,800 USD trong một đêm vì không có termination">
**Bối cảnh:** Một team build "research agent" tự động tìm kiếm thông tin đối thủ. Architecture ban đầu rất đơn giản: user gửi câu hỏi, LLM gọi tool `search_web` trong vòng lặp, nhận kết quả, gọi tiếp nếu chưa "đủ", rồi tổng hợp báo cáo.

**Sự cố:** Một user thử câu hỏi mơ hồ: "Tìm hiểu mọi thứ về thị trường X". Agent gọi search, nhận kết quả, nhưng không có done condition rõ. Nó reasoning: "vẫn chưa đủ thông tin", gọi search tiếp, đổi query, search lại, đổi query, search lại. 1.2 triệu token và 4,000 lần search tool sau đó, nó vẫn chưa hài lòng.

1. Không có max steps, max tokens, max cost hay max wall-clock.
2. Stop condition duy nhất là "model tự quyết đã xong".
3. Không có progress detector, không nhận ra mình đang lặp.
4. Tool budget không được enforce — model có thể gọi tool vô hạn.
5. Không có alert khi cost spike.

**Hậu quả:** Hóa đơn API tăng 1,800 USD trong một đêm. Agent phải tắt khẩn cấp, và team phải thiết kế lại toàn bộ budget và termination controls.
</Callout>

Bài học rút ra: agent không production-ready khi nó "chạy đúng". Agent production-ready khi bạn **giới hạn được autonomy, đo được trajectory, và dừng được khi nó sai**. Một agent không có budget là một incident đang chờ xảy ra.

## 2. Architecture Blueprint (Agent Control Loop)

Một agent production không phải là một LLM call. Nó là một control loop có runtime controls:

```text
[ Goal / User Task ]
        │
        ▼
[ State store ]
        │
        ▼
[ LLM / controller decides next action ]
        │
        ▼
[ Validate + authorize action ]   ← guardrails
        │
        ▼
[ Tool call / response / ask user / final answer ]
        │
        ▼
[ Observation ]
        │
        ▼
[ Update state + memory ]
        │
        ▼
[ Continue or terminate ]   ← budget, terminator
```

LLM không phải toàn bộ agent. Runtime quyết định:

- tools nào available,
- state nào visible,
- action nào được phép,
- max steps/cost/time,
- khi nào cần human approval,
- log những gì,
- điều kiện success là gì.

Output của agent design không phải là một prompt đẹp; output là một tập quyết định kiểm soát: state, action space, policy, termination và trace.

## 3. Agent as a Control Loop

Mỗi bước agent có thể fail. Đó là lý do control loop phải có **feedback**.

```text
observe → decide → act → observe → decide → act → ...
```

Mỗi bước có thể fail ở:

- **observe**: tool result mơ hồ, không có thông tin, hoặc prompt injection.
- **decide**: model chọn sai tool, sai argument, hoặc hallucinated tool.
- **act**: tool timeout, permission denied, hoặc side effect ngoài ý muốn.
- **terminate**: model không biết khi nào dừng, hoặc dừng quá sớm.

Vì agent loop có thể fail ở nhiều điểm, production agent phải có budget, trace, guardrails và terminator. Nếu không, bạn chỉ đang chạy một demo và hy vọng nó ổn.

## 4. Formal Model: state, action, policy, observation

Một cách nhìn có tính hệ thống:

```text
s_t = current state at step t
a_t = action chosen at step t
o_t = observation after action
π(a_t | s_t) = policy choosing action given state
```

Trong agent LLM:

- `π` thường là LLM + prompt + tool schema + runtime policy.
- `a_t` có thể là `call_tool`, `ask_user`, `write_memory`, `final_answer`, `escalate`.
- `o_t` là tool result, user response, error, observation.
- `s_t` gồm goal, plan, history, memory, budget, errors.

Một định nghĩa chặt chẽ:

```text
Agent failure = bad state, bad action space, bad policy, bad observation, or bad termination.
```

Khi agent fail, bạn không chỉ "sửa prompt". Bạn phải biết lỗi nằm ở đâu trong 5 thành phần này. Đó là lý do trace và state schema quan trọng.

## 5. Agent vs Prompt vs Chain vs Workflow

Không phải mọi LLM app có tool là agent. Đừng nhầm lẫn.

| System | Control flow | Decision maker | Best for | Risk |
|---|---|---|---|---|
| Prompt | single call | developer | simple generation | no multi-step control |
| Chain | fixed sequence | developer | predictable tasks | brittle to ambiguity |
| Workflow | conditional deterministic | code/rules | business automation | rule complexity |
| Tool-using chatbot | model calls tools in one turn | model within narrow scope | assisted tasks | tool misuse |
| Agent | dynamic multi-step loop | model + runtime policy | uncertain multi-step tasks | cost/safety/loop risk |

Key idea:

```text
Chains encode the plan in code.
Agents infer part of the plan at runtime.
```

Đây là ranh giới quan trọng. Nếu bạn biết chính xác 5 bước cần chạy, viết workflow. Nếu task mơ hồ, cần tool selection động, có partial failures và cần planning, agent mới đáng cân nhắc.

---

## 6. Least Autonomy Principle

Dùng hệ thống ít autonomous nhất mà giải quyết được bài toán.

```text
prompt < chain < workflow < agent < multi-agent autonomous system
```

<Callout type="warning" title="Đừng dùng agent khi workflow đủ">
Nếu bạn biết chính xác 5 bước cần chạy, viết workflow deterministic sẽ dễ test, rẻ, nhanh và an toàn hơn agent tự quyết. Agent là công cụ cho task không biết trước, không phải "nâng cấp" cho mọi task.
</Callout>

Autonomy càng cao, cost càng cao, debugging càng khó, và rủi ro càng lớn. Agent không tự động tốt hơn workflow; nó là công cụ cho một loại bài toán khác.

## 7. Types of Agents

### ReAct agent

ReAct dùng vòng lặp:

```text
Reason → Act → Observe → Reason → Act → Observe → Final
```

Strength: đơn giản, dynamic tool use, dễ học.

Weakness: myopic, dễ loop, planning dài hạn kém, phụ thuộc tool descriptions.

### Plan-and-Execute

Planner tạo plan trước, executor chạy từng bước.

```text
Planner → steps → Executor → observations → maybe Replanner
```

Tốt cho task dài, nhưng plan có thể stale khi observations thay đổi. Cần replanning khi môi trường động.

### MRKL / modular routing

Agent route subproblems tới modules chuyên biệt: calculator, search, database, code interpreter, retriever. Tốt vì tools deterministic hơn model cho nhiều subtask.

### Reflection / critique agent

Agent tự kiểm tra output hoặc dùng critic model. Tốt cho drafts/code/reports, nhưng self-critique có thể chỉ là "performative reasoning" nếu không có external evidence/tests.

### Multi-agent systems

Nhiều agents có roles khác nhau. Có thể hữu ích nhưng tăng coordination cost. Chương 4 sẽ đi sâu.

## 8. Levels of Autonomy

| Level | Description | Example | Required control |
|---|---|---|---|
| L0 | No tool use | chat answer | prompt validation |
| L1 | Suggested actions only | draft email | user approval |
| L2 | Read-only tools | search docs/web | access control |
| L3 | Low-risk write tools | create ticket | audit log |
| L4 | High-impact actions with approval | refund/email customer | human checkpoint |
| L5 | Autonomous high-impact actions | trading/security changes | usually unacceptable |

Production design nên assign autonomy **per action**, không per agent. Một agent có thể vừa có L2 (read-only) và vừa có L4 (high-impact) tools — nhưng mỗi action có permission riêng.

---

## 9. Agent Failure Taxonomy

| Failure | Cause | Symptom | Mitigation |
|---|---|---|---|
| Goal ambiguity | vague user task | wrong objective | clarify before acting |
| Bad decomposition | poor plan | irrelevant steps | planner eval |
| Tool hallucination | model invents tool | invalid action | schema-constrained tools |
| Argument hallucination | wrong args | failed/unsafe call | validation |
| Observation misread | model ignores result | repeats errors | structured observations |
| Looping | no termination | cost spike | max steps + loop detector |
| Over-planning | too many steps | latency/cost explosion | budget-aware planner |
| Memory pollution | stores wrong facts | persistent future errors | memory validation |
| Tool side effect | bad external action | data changed incorrectly | approval/sandbox |
| Prompt injection | untrusted content controls agent | unsafe action | instruction isolation |
| Premature termination | stops too early | incomplete output | verifier/done criteria |

Khi agent fail, không đủ để nói "model sai". Bạn phải biết lỗi nằm ở đâu trong taxonomy này. Trace là cách duy nhất để phân loại.

---

## 10. When NOT to Use an Agent

Không dùng agent nếu:

- task deterministic,
- steps cố định,
- side effects high-risk,
- latency/cost rất tight,
- eval không rõ,
- tool permissions chưa có,
- bạn không thể log/audit trajectory,
- failure không chấp nhận được.

Agent là công cụ cho task không biết trước, có partial failures, cần planning động. Đừng dùng agent khi workflow đủ — agent thêm cost, risk, và debugging difficulty mà không thêm value.

## Implementation Playbook: Minimal Framework-Free Agent Loop

```python
from dataclasses import dataclass, field


@dataclass
class AgentState:
    goal: str
    observations: list[dict] = field(default_factory=list)
    steps: list[dict] = field(default_factory=list)
    max_steps: int = 8
    cost_usd: float = 0.0


def run_agent(goal, policy, tools, guard, terminator):
    state = AgentState(goal=goal)

    for step in range(state.max_steps):
        action = policy.decide(state=state, tools=tools)
        state.steps.append({"step": step, "action": action})

        if action["type"] == "final":
            return {"status": "done", "answer": action["answer"], "trace": state.steps}

        if not guard.allowed(action, state):
            return {"status": "blocked", "reason": "action_not_allowed", "trace": state.steps}

        tool = tools[action["tool_name"]]
        observation = tool.execute(action["arguments"])
        state.observations.append(observation)

        if terminator.should_stop(state):
            return {"status": "stopped", "reason": "terminator", "trace": state.steps}

    return {"status": "escalate", "reason": "max_steps_exceeded", "trace": state.steps}
```

Điểm quan trọng trong skeleton:

- model chỉ **đề xuất** action, không execute.
- guard kiểm soát action có được phép hay không.
- tool execute có contract rõ.
- state lưu trace từng step.
- terminator có quyền dừng ngay cả khi model muốn tiếp.

Agent trace tối thiểu:

```json
{
  "run_id": "run_123",
  "goal": "Find current refund policy",
  "steps": [
    {
      "step": 1,
      "action": "search_docs",
      "arguments": {"query": "refund policy current"},
      "observation_summary": "Found policy v7",
      "latency_ms": 420,
      "cost_usd": 0.001
    }
  ],
  "status": "completed",
  "final_answer": "..."
}
```

Without trace, agent debugging is storytelling. Bạn không biết agent đã làm gì, chỉ biết nó đã làm xong.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Định nghĩa agent quá rộng">
**Triệu chứng:** Mọi LLM app có tool được gọi "agent", dẫn đến over-engineering và bối rối.

**Fix:** Agent cần dynamic multi-step control flow, state, action selection và termination. Một tool-using chatbot trong một turn không phải agent.
</Callout>

<Callout type="warning" title="Pitfall 2: Không có termination conditions">
**Triệu chứng:** Agent chạy dài không có cách dừng, cost spike, hoặc loop vô hạn.

**Fix:** Max steps, max tokens, max cost, max wall-clock. Stop condition duy nhất không được là "model tự quyết đã xong".
</Callout>

<Callout type="danger" title="Pitfall 3: Để model tự quyết side effects">
**Triệu chứng:** Model đề xuất action có side effect (email, refund, deploy) và runtime execute ngay.

**Fix:** Model đề xuất action không đồng nghĩa được phép execute. Runtime phải validate, authorize, audit và (cho high-risk) yêu cầu human approval.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Agent thay vì workflow | adaptive cho task mơ hồ | cost spike, loop risk, harder debugging |
| Nhiều tool calls | thông tin phong phú | cost tăng, latency cao, tool failure |
| Long context | memory đầy đủ | input tokens tăng, context overflow |
| Autonomous high-impact | không cần human | rủi ro side effect, compliance |
| No trace | đơn giản | debugging không thể, audit không thể |
| No budget | chạy đến khi "xong" | incident cost |

Agent design là nghệ thuật chuyển "tự động" thành "có kiểm soát". Mỗi autonomy tăng thêm phải có budget, trace và guardrail tương ứng.

## Điểm cần nhớ

- Agent là stateful control loop, không chỉ LLM + tools.
- LLM proposes actions; runtime controls execution.
- Least autonomy principle: dùng hệ thống ít autonomous nhất đủ giải quyết task.
- Agents cần budgets, termination, validation, authorization và trace.
- Debugging agent phải dựa trên trajectory, không chỉ final answer.
