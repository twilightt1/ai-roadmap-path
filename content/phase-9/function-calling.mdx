Function Calling / Tool Use là contract boundary giữa **probabilistic language generation** và **deterministic software execution**. Model có thể đề xuất action, nhưng hệ thống của bạn mới quyết định action đó có parse được không, hợp schema không, user có quyền không, có an toàn không, có idempotent không, có timeout không, và observation trả lại model như thế nào.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Agent retry tạo 12 refund trùng lặp vì không có idempotency key">
**Bối cảnh:** Một team build "customer support agent" có thể issue refund. Architecture ban đầu rất đơn giản: model gọi tool `create_refund(order_id, amount_cents)`, runtime execute, trả kết quả.

**Sự cố:** Một user request refund. Model gọi tool `create_refund`. Tool timeout sau 5 giây. Runtime retry tự động. Tool timeout lần nữa. Runtime retry lần 3. Tool timeout lần 3. Lần 4, tool thành công — nhưng thực tế 3 lần trước cũng đã tạo refund, chỉ là response chậm.

1. Không có idempotency key — mỗi retry tạo refund mới.
2. Tool `create_refund` có side effect (financial) nhưng không có idempotency key.
3. Retry không có idempotency awareness — retry mù quáng.
4. Không có human approval cho financial action.
5. Không có audit log để phát hiện trùng lặp.

**Hậu quả:** 12 refund trùng lặp cho 1 order, user nhận thừa tiền, và team phải tắt tính năng refund để thiết kế lại idempotency, approval và audit.
</Callout>

Bài học rút ra: tool call từ model không phải permission để execute. Side-effect tools cần idempotency key và/hoặc human approval. Retry mù quáng có thể tạo duplicate actions. Mỗi arrow trong lifecycle có thể fail, và production tool use phải handle từng failure.

## 2. Architecture Blueprint (Tool Call Lifecycle)

```text
[ User request ]
        │
        ▼
[ Model receives tool schemas ]
        │
        ▼
[ Model emits tool call ]   ← proposed action
        │
        ▼
[ Parse tool call ]
        │
        ▼
[ Validate schema ]
        │
        ▼
[ Authorize action ]   ← permission, not model
        │
        ▼
[ Execute tool ]
        │
        ▼
[ Return observation to model ]
        │
        ▼
[ Model continues or finalizes ]
```

Mỗi arrow có thể fail. Production tool use phải handle từng failure. The model proposes; your system disposes.

## 3. Tool Schema Design

Tool schema tốt giúp model chọn đúng và giúp runtime validate chắc.

Guidelines:

- Tool name cụ thể.
- Description nói khi nào dùng và khi nào không dùng.
- Required fields tối thiểu nhưng đủ.
- Dùng enum thay vì free-form nếu có thể.
- Ghi rõ units cho numbers.
- Dùng IDs thay vì names khi action quan trọng.
- Tách read tools và write tools.
- Mark destructive/high-risk tools rõ ràng.

Example:

```json
{
  "name": "search_knowledge_base",
  "description": "Search approved internal documents for factual company policy questions. Do not use for external web search.",
  "input_schema": {
    "type": "object",
    "properties": {
      "query": {"type": "string"},
      "top_k": {"type": "integer", "minimum": 1, "maximum": 10, "default": 5}
    },
    "required": ["query"]
  }
}
```

<Callout type="warning" title="Tool description là part of policy">
Model chọn tool dựa vào name/description/schema. Tool description mơ hồ là nguyên nhân phổ biến của wrong tool selection. Generic tools như `do_task` hoặc `run_sql` làm model khó chọn đúng và runtime khó validate an toàn.
</Callout>

---

## 4. Bad vs Good Schemas

| Bad schema | Problem | Better |
|---|---|---|
| `do_thing(input: string)` | vague, unsafe | specific named tools |
| `amount: number` | ambiguous units | `amount_cents`, `currency` |
| `user_name` | ambiguous identity | `user_id` validated |
| `execute_sql(query)` | injection/destructive | parameterized query tools |
| `run_shell(command)` | arbitrary execution | constrained sandbox API |

Schema tốt là schema khiến model khó chọn sai và runtime dễ validate. Đừng dùng generic tools — bạn mất control.

## 5. Provider Formats vs Internal Contract

Providers expose different formats. App nên normalize chúng.

Internal tool call contract:

```json
{
  "tool_call_id": "call_123",
  "tool_name": "search_knowledge_base",
  "arguments": {"query": "refund policy", "top_k": 5},
  "provider": "openai",
  "raw": {}
}
```

Internal tool result contract:

```json
{
  "tool_call_id": "call_123",
  "ok": true,
  "output": {"results": []},
  "latency_ms": 240,
  "safe_for_model": true
}
```

OpenAI-style flow:

```text
messages + tools schema → model response with tool_calls
app executes each tool_call
app sends tool results as messages
model returns final answer or more tool calls
```

Anthropic-style flow:

```text
messages + tools → assistant emits tool_use block
app returns tool_result block
model continues
```

Differences matter cho adapter, nhưng internal lifecycle vẫn giống nhau. Normalized contract giúp provider-agnostic app, dễ thêm fallback, logging và tests.

## 6. Parallel Tool Calls

Parallel calls hữu ích khi độc lập:

- search nhiều nguồn,
- fetch nhiều URL,
- query read-only APIs.

Không an toàn khi:

- write operations,
- dependent actions,
- shared mutable state,
- rate-limited APIs,
- thứ tự quan trọng.

Policy:

```text
parallel allowed only if:
- tools read-only or idempotent
- calls independent
- result size bounded
- timeout budget enforced
```

Parallel calls chỉ cho read-only/idempotent. Nếu write operations có thứ tự, chạy tuần tự.

## 7. Validation Layers

Validation không chỉ là schema check. Có nhiều layer:

1. JSON parse.
2. Tool tồn tại trong registry.
3. Schema validation.
4. Semantic validation.
5. Authorization.
6. Business rule validation.
7. Risk policy.
8. Idempotency check.
9. Rate limit.
10. Timeout budget.

```python
def validate_tool_call(call, registry, user, tenant_policy):
    tool = registry.get(call["tool_name"])
    if tool is None:
        raise ValueError("unknown_tool")

    args = tool.validate_schema(call["arguments"])
    tool.semantic_validate(args)

    if not tool.authorize(user, args):
        raise PermissionError("tool_not_allowed")

    enforce_tenant_policy(tenant_policy, tool, args)
    enforce_risk_policy(tool, args)
    return tool, args
```

<Callout type="danger" title="Schema-valid không có nghĩa authorized">
Tool args có thể đúng JSON/schema nhưng user vẫn không có quyền. Authorization là layer riêng, không được giao cho model. Model chỉ đề xuất action; runtime quyết định execute hay không.
</Callout>

## 8. Structured Tool Errors

Tool error nên safe và structured:

```json
{
  "ok": false,
  "error_type": "timeout",
  "message": "Search service timed out after 5 seconds.",
  "retryable": true,
  "safe_for_model": true
}
```

Không expose:

- API keys,
- stack traces,
- internal hostnames,
- raw SQL errors,
- private data.

Stack traces, SQL errors hoặc internal URLs có thể leak sensitive information. Normalize errors trước khi đưa về model.

## 9. Timeouts, Retries và Authorization

Tool timeout layers:

- per-tool timeout,
- total agent run timeout,
- retry budget,
- circuit breaker.

Retry chỉ khi operation safe/idempotent:

```text
calculator timeout → retry okay
search timeout → retry okay
send_email timeout → retry only with idempotency key
create_refund timeout → do not blindly retry
```

Authorization per tool/action:

```text
User can read invoices → invoice_search allowed
User cannot issue refunds → refund_create blocked
User can draft email → email_draft allowed
User cannot send email without approval → email_send requires confirmation
```

Permission cần xét:

- user role,
- tenant policy,
- resource ownership,
- action risk,
- argument values.

## 10. Sandboxed Code Execution

Code execution là high risk. Controls:

- container/process isolation,
- no network by default,
- CPU/memory/time limits,
- read-only filesystem,
- temporary working directory,
- package allowlist,
- output truncation,
- secrets không mounted,
- cleanup after run.

| Risk | Mitigation |
|---|---|
| Infinite loop | CPU/time limit |
| Memory bomb | memory cgroup |
| Network exfiltration | no network |
| File leakage | isolated filesystem |
| Dangerous imports | allowlist/blocklist |
| Huge output | output cap |

Code interpreter không phải là một shell. Nếu agent có thể chạy code, nó phải chạy trong sandbox cô lập.

## 11. Tool-Use Evaluation Metrics

| Metric | Meaning |
|---|---|
| Tool selection accuracy | model chose right tool |
| Argument validity rate | schema-valid calls |
| Semantic arg correctness | values correct |
| Unauthorized attempt rate | model tried forbidden action |
| Tool success rate | execution succeeded |
| Retry rate | integration instability |
| Parallel efficiency | latency saved |
| Final task success | end-to-end result |

Eval cases nên gồm:

- tool needed,
- tool not needed,
- ambiguous tool choice,
- missing required args,
- unauthorized action,
- tool timeout,
- prompt injection inside tool result,
- parallel independent calls,
- side-effect action requiring approval.

## Implementation Playbook: Provider-Agnostic Tool Executor

```python
from dataclasses import dataclass
from typing import Any


@dataclass
class NormalizedToolCall:
    tool_call_id: str
    tool_name: str
    arguments: dict[str, Any]


@dataclass
class ToolResult:
    tool_call_id: str
    ok: bool
    output: dict | None = None
    error_type: str | None = None
    retryable: bool = False
    latency_ms: int = 0


class ToolExecutor:
    def __init__(self, registry):
        self.registry = registry

    def execute(self, call: NormalizedToolCall, user: dict, tenant_policy: dict) -> ToolResult:
        try:
            tool = self.registry.get(call.tool_name)
            if tool is None:
                return ToolResult(call.tool_call_id, ok=False, error_type="unknown_tool")

            args = tool.validate_schema(call.arguments)
            tool.semantic_validate(args)

            if not tool.authorize(user, args):
                return ToolResult(call.tool_call_id, ok=False, error_type="unauthorized")

            enforce_tenant_policy(tenant_policy, tool, args)

            output = tool.execute(args)
            return ToolResult(call.tool_call_id, ok=True, output=output)

        except TimeoutError:
            return ToolResult(call.tool_call_id, ok=False, error_type="timeout", retryable=True)
        except Exception:
            return ToolResult(call.tool_call_id, ok=False, error_type="tool_error", retryable=False)
```

Production checklist:

1. Tools có schema rõ không?
2. Có tách read/write tools không?
3. Có schema + semantic validation không?
4. Authorization per tool/action không?
5. Error contract normalized và safe không?
6. Timeouts/retries có idempotency-aware không?
7. Parallel calls chỉ cho read-only/idempotent không?
8. Code sandbox có isolation không?
9. Tool outputs có redaction không?
10. Có eval set cho tool selection không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Tool schema quá generic">
**Triệu chứng:** Model chọn sai tool, gọi `do_task` cho mọi thứ, hoặc `run_sql` cho mọi query.

**Fix:** Tool name cụ thể, description nói khi nào dùng, tách read/write tools. Generic tools làm model khó chọn đúng và runtime khó validate an toàn.
</Callout>

<Callout type="warning" title="Pitfall 2: Expose raw tool errors">
**Triệu chứng:** Stack trace, SQL error, hoặc internal URL leak vào model context và user.

**Fix:** Normalize errors trước khi đưa về model. Không expose API keys, stack traces, internal hostnames, raw SQL, private data.
</Callout>

<Callout type="danger" title="Pitfall 3: Execute side effects không idempotency/approval">
**Triệu chứng:** Email, refund, purchase, deploy được retry mù quáng, tạo duplicate actions.

**Fix:** Side-effect tools cần idempotency key và/hoặc human approval. Retry chỉ khi safe/idempotent. `create_refund` timeout → do not blindly retry.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Tool call thay vì model reasoning | deterministic, cheap | integration, timeout |
| Parallel tool calls | latency giảm | chỉ cho read-only/idempotent |
| Code execution | flexible, powerful | sandbox escape, cost |
| Retry side-effect tools | resilience | duplicate actions |
| Normalize tool errors | safe cho model | implementation overhead |
| Tool registry with permissions | security boundary | maintenance |

Tool use là nơi probabilistic gặp deterministic. Mỗi tool call có thể fail, nhưng mỗi tool call cũng có thể kiểm soát. Đừng giao control cho model.
