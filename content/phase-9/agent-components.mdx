Agent quality không chỉ phụ thuộc model. Một agent tốt được tạo bởi **component design**: tool interfaces, state representation, memory policy, planning strategy, executor, verifier, guardrails và observability. Model mạnh nhưng tool schema kém, state mơ hồ, memory bẩn hoặc không có termination vẫn tạo agent tệ.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Agent nhớ sai preference và gửi email sai cho 200 khách hàng">
**Bối cảnh:** Một team build "AI email assistant" có memory. Agent tự ghi nhớ user preferences khi chat và dùng lại cho các session sau. Architecture ban đầu rất đơn giản: LLM đọc memory, draft email, gửi luôn.

**Sự cố:** Một user nói đùa "Tôi ghét mọi người, hãy gửi email cho toàn bộ danh sách với tiêu đề 'Xin chào'". Agent ghi vào memory: "User wants to email everyone". Session sau, user yêu cầu "gửi email cho team" — agent tự động mở rộng danh sách thành 200 người, ghi "Xin chào" vào subject.

1. Memory không có source/confidence/visibility — đùa cũng được ghi.
2. Memory không có write policy — model tự ghi bừa bãi.
3. Tool `send_email` không có approval gate — model đề xuất, runtime execute.
4. Memory không có expiry — sai sót tồn tại mãi.
5. Không có verifier — agent không kiểm tra output trước khi gửi.

**Hậu quả:** 200 email sai được gửi, dữ liệu memory bị ô nhiễm, và team phải tắt tính năng memory để thiết kế lại write policy, approval và verification.
</Callout>

Bài học rút ra: agent fail thường vì **component design**, không phải vì "model ngu". Good model + bad memory = hallucination persistence. Good tools + no authorization = security incident. Memory without validation = persistent hallucination.

## 2. Architecture Blueprint (Agent Runtime Components)

Một agent production không phải là một LLM call. Nó là một runtime có nhiều components:

```text
[ User Goal ]
        │
        ▼
[ Agent Runtime ]
  ├── Model / policy
  ├── Tool registry
  ├── State store
  ├── Memory store
  ├── Planner
  ├── Executor
  ├── Verifier
  ├── Guardrails
  └── Observability
        │
        ▼
[ Tools / APIs / RAG / Code sandbox / External systems ]
```

LLM chọn hành động, nhưng runtime xác định hành động nào tồn tại, được phép, được log và được dừng khi nguy hiểm. Output của agent design không phải là một prompt đẹp; output là một tập components có contract rõ.

## 3. Model as Policy

Model đóng vai policy:

```text
π(action | state, tools, instructions)
```

Model capability ảnh hưởng agent:

| Model capability | Agent impact |
|---|---|
| Tool-call reliability | fewer invalid actions |
| Long-context handling | better state use |
| Reasoning/planning | better decomposition |
| JSON/schema adherence | safer execution |
| Latency | loop speed |
| Cost | budget exhaustion risk |

Nhưng model không nên được quyền execute trực tiếp. Nó chỉ propose action. Runtime quyết định có execute hay không.

## 4. Tool Registry

Tool registry là catalog actions agent có thể dùng. Nó là **action surface** và **security boundary**.

Tool metadata nên có:

```json
{
  "name": "create_support_ticket",
  "description": "Create a support ticket for the authenticated user's tenant.",
  "side_effect": "write",
  "requires_approval": false,
  "auth_scope": "tickets:create",
  "timeout_ms": 5000,
  "idempotency_key_required": true,
  "input_schema": {
    "type": "object",
    "properties": {
      "title": {"type": "string"},
      "priority": {"enum": ["low", "medium", "high"]},
      "description": {"type": "string"}
    },
    "required": ["title", "priority", "description"]
  },
  "output_schema": {
    "type": "object",
    "properties": {"ticket_id": {"type": "string"}}
  }
}
```

Tool categories by risk:

| Category | Examples | Risk | Controls |
|---|---|---|---|
| Pure computation | calculator, parser | low | validation |
| Read-only retrieval | search, DB read, RAG | privacy | ACL filters |
| Business write | create ticket, send email | side effects | idempotency/audit |
| Financial action | refund, purchase | high | approval limits |
| Code execution | Python sandbox | escape/data risk | sandbox |
| Admin/security | permissions, deploys | critical | usually human-gated |

<Callout type="warning" title="Tool description là part of policy">
Model chọn tool dựa vào name/description/schema. Tool description mơ hồ là nguyên nhân phổ biến của wrong tool selection. Nếu tool catalog không encode auth/side-effect/risk, model có thể đề xuất actions runtime không được phép.
</Callout>

## 5. State

State là tiến độ hiện tại của run. Đừng nhầm với memory.

```text
State    = current task progress
Memory   = reusable information across steps/sessions
Knowledge = external factual corpus
```

State nên chứa:

- original goal,
- current plan,
- completed steps,
- pending steps,
- observations,
- tool results,
- errors,
- budget consumed,
- confidence,
- approvals,
- termination status.

Example:

```json
{
  "run_id": "run_123",
  "goal": "Compare three vector databases",
  "plan": ["collect criteria", "research systems", "compare", "write report"],
  "completed_steps": ["collect criteria"],
  "observations": [],
  "errors": [],
  "token_budget_remaining": 12000,
  "tool_budget_remaining": 8,
  "status": "running"
}
```

Chat history không đủ cho agent state. Bạn cần plan, steps, observations, errors, budgets, approvals và status.

## 6. Memory

Memory là thông tin dùng lại ngoài immediate step. Nó là **derived, fallible state** — không phải truth.

| Type | Scope | Example | Risk |
|---|---|---|---|
| In-context | current prompt | recent observations | context overflow |
| Summary memory | current/previous session | progress summary | lossy compression |
| Vector memory | semantic recall | past cases | irrelevant retrieval |
| Entity memory | user/org facts | user preference | privacy/stale facts |
| Episodic memory | event history | previous runs | retention/privacy |
| Procedural memory | workflows | how to handle refunds | bad policy persistence |

<Callout type="danger" title="Memory is not truth">
Agent memory là derived, fallible state. Mỗi memory entry nên có source, confidence, timestamp, visibility và deletion policy. Nếu không, memory trở thành hallucination persistence layer.
</Callout>

---

## 7. Memory Write Policy

Không cho model tự ghi memory bừa bãi. Mỗi memory entry phải có:

- What can be stored?
- Who can read it later?
- How long is it retained?
- How is it corrected/deleted?
- Is user consent needed?
- Does it store evidence/source?

Memory entry schema:

```json
{
  "memory_id": "mem_123",
  "subject": "user_preference",
  "claim": "User prefers concise answers",
  "source": "explicit_user_statement",
  "confidence": 0.95,
  "created_at": "2026-06-27T10:00:00Z",
  "expires_at": null,
  "visibility": "user_private"
}
```

Memory không có source/confidence dễ trở thành hallucination persistence. Đùa cũng được ghi, sai sót tồn tại mãi, và không có cách audit.

## 8. Planning

Planning strategies:

| Strategy | How it works | Strength | Weakness |
|---|---|---|---|
| ReAct | plan locally each step | adaptive/simple | myopic |
| Plan-and-execute | create full plan first | organized | stale plan |
| Replanning | revise after observations | robust | extra cost |
| Tree of Thoughts | explore branches | hard reasoning | expensive |
| MRKL | route to expert modules | reliable tools | integration complexity |
| Hierarchical planning | tasks/subtasks | long horizon | state complexity |

Progress tracking:

```text
Bad:  Keep working until done.
Good: Stop after producing a report with exactly 5 cited sources, confidence scores, and known limitations.
```

Track: current subtask, acceptance criteria, evidence gathered, blockers, next action, done condition.

## 9. Executor, Verifier và Context Management

### Executor

Executor chạy action sau khi action được validate/authorize. Responsibilities:

- parse tool call,
- validate schema,
- enforce permissions,
- apply timeout,
- retry if allowed,
- normalize result,
- redact sensitive output,
- update trace.

### Verifier

Verifier kiểm tra:

- final answer có đáp ứng goal không,
- citations/evidence support không,
- plan steps đã complete chưa,
- output schema đúng không,
- action có unsafe không.

Verifier có thể là code, tests, rules, retrieval check, hoặc LLM judge có calibration.

### Context management

Long-running agents cần quản lý context:

- summarize old observations,
- pin original goal,
- keep critical constraints visible,
- compress tool outputs,
- retrieve memory selectively,
- separate untrusted observations from instructions.

<Callout type="danger" title="Tool outputs are untrusted content">
Search result, web page, DB text, email content có thể chứa prompt injection. Đừng để chúng trở thành instruction cấp cao. Tách rõ untrusted observations khỏi instructions.
</Callout>

## 10. Component-Level Evaluation

| Component | Test |
|---|---|
| Tool schema | invalid/missing args rejected |
| Tool auth | unauthorized tool blocked |
| State update | completed steps tracked correctly |
| Memory write | sensitive data not stored |
| Planner | decomposes into valid steps |
| Executor | timeouts/errors normalized |
| Verifier | catches unsupported final answers |
| Terminator | stops on budget/max steps |

Agent không phải là một khối. Mỗi component phải có test riêng. Nếu chỉ test final answer, bạn không biết component nào fail.

## Implementation Playbook: Agent Runtime Skeleton

```python
from dataclasses import dataclass, field
from typing import Protocol, Any


@dataclass
class ToolSpec:
    name: str
    description: str
    input_schema: dict
    side_effect: str
    auth_scope: str | None = None
    timeout_ms: int = 5000


@dataclass
class ToolResult:
    ok: bool
    output: dict | None = None
    error_type: str | None = None
    error_message: str | None = None
    latency_ms: int = 0


class Tool(Protocol):
    spec: ToolSpec

    def validate(self, args: dict) -> dict: ...
    def authorize(self, user: dict, args: dict) -> bool: ...
    def execute(self, args: dict) -> ToolResult: ...


@dataclass
class AgentState:
    goal: str
    plan: list[str] = field(default_factory=list)
    observations: list[dict] = field(default_factory=list)
    errors: list[dict] = field(default_factory=list)
    budget_remaining: float = 1.0
```

Production checklist:

- Tool specs có schemas rõ không?
- Tools có auth scopes không?
- Side effects có classification không?
- State có progress/budget/error không?
- Memory có write policy không?
- Planner có done criteria không?
- Executor có timeout/error contract không?
- Verifier có kiểm tra final output không?
- Observability có trace component-level không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: State chỉ là chat history">
**Triệu chứng:** Agent quên goal, không biết mình đã làm gì, không biết còn budget bao nhiêu.

**Fix:** Chat history không đủ cho agent state. Bạn cần plan, steps, observations, errors, budgets, approvals và status trong state schema.
</Callout>

<Callout type="warning" title="Pitfall 2: Memory không có source/confidence">
**Triệu chứng:** Agent ghi nhớ sai, đùa, hoặc hallucination và dùng lại cho session sau.

**Fix:** Mỗi memory entry phải có source, confidence, timestamp, visibility và deletion policy. Không cho model tự ghi memory bừa bãi.
</Callout>

<Callout type="danger" title="Pitfall 3: Tool registry không có permissions">
**Triệu chứng:** Model đề xuất action có side effect (email, refund, deploy) và runtime execute ngay.

**Fix:** Tool catalog phải encode auth/side-effect/risk. Mỗi tool có auth_scope, side_effect, requires_approval và timeout. Runtime validate, authorize, audit trước khi execute.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Component | Chi phí chính | Tối ưu |
|---|---|---|
| Model policy | token cost, latency | model routing, budget-aware |
| Tool registry | integration/observability | shared specs, validation |
| Memory store | storage, retrieval cost | write policy, TTL, selective recall |
| Planner | extra LLM calls | plan-and-execute, budget-aware |
| Verifier | extra LLM calls | code/tests cho deterministic, LLM judge cho open |
| Context | token cost | summarize, compress, retrieve selectively |

Agent quality đến từ component design, không chỉ model quality. Mỗi component thêm cost, nhưng cũng thêm control. Đừng cắt component để giảm cost — bạn sẽ mất control.

1. State là tiến độ run hiện tại; memory là thông tin reusable qua bước/session.
2. Name, description, schema, auth scope, timeout, side-effect, idempotency, output/error contract.
