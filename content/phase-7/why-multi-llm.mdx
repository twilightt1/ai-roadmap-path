Multi-LLM Architecture không phải là "gọi nhiều model cho vui". Đây là chiến lược production để cân bằng **quality, latency, cost, reliability, context length, compliance, modality và capability**. Ở quy mô sản phẩm thật, model choice không nên là hằng số hard-code; nó phải là một **runtime policy** được đo, log, kiểm thử và govern.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Startup lỗ 18,000 USD/tháng vì hard-code frontier model cho mọi request">
**Bối cảnh:** Một team build "AI chatbot support" cho SaaS. Architecture ban đầu rất đơn giản: `model = "gpt-4o"`, hard-code cho mọi request. App chạy ngon, user happy, nhưng đội ngũ không để ý đến chi phí.

**Sự cố:** 1.2 triệu request/tháng, 90% là simple classification (phân loại câu hỏi, routing intent). 10% là reasoning thực sự. Mỗi request trung bình 1,800 token, 2 USD/1M token. Hóa đơn API tăng 18,000 USD/tháng.

1. Không có routing — model mạnh nhất cho mọi request.
2. Không có task classification — 90% là simple task, không cần frontier model.
3. Không có cost tracking — không biết ai tiêu tiền, không biết ai đang lãng phí.
4. Không có capability matrix — không biết model nào phù hợp cho task nào.
5. Không có fallback policy — chỉ phụ thuộc vào một provider.

**Hậu quả:** Startup lỗ tiền. Đội ngũ phải tắt chatbot, thiết kế lại toàn bộ routing, cost tracking, và model selection. Hóa đơn giảm 18,000 USD/tháng xuống 4,200 USD/tháng sau khi 90% simple task được chuyển sang model rẻ.
</Callout>

Bài học rút ra: model choice là một quyết định production, không phải hard-code. Nếu không có routing, cost tracking, và capability matrix, bạn đang lãng phí tiền và không biết ai tiêu. Một model không tối ưu cho mọi task.

## 2. Architecture Blueprint (Model Choice as Runtime Policy)

Một request production không phải là một LLM call. Nó là một **decision layer**:

```text
[ User request ]
        │
        ▼
[ Task / risk / modality classifier ]
        │
        ▼
[ Candidate model set ]
        │
        ▼
[ Constraints: cost, latency, compliance, context, tools ]
        │
        ▼
[ Selected provider/model ]
        │
        ▼
[ Validation + fallback + trace ]
```

Thay vì:

```python
model = "gpt-4o"  # hard-coded forever
```

Bạn muốn:

```python
model = router.select(request, tenant_policy, budget, eval_scores)
```

Runtime quyết định: model nào, budget nào, fallback nào, trace nào, và policy nào. Một model không tối ưu cho mọi task, cost, latency, context và compliance.

## 3. Single-model vs Fallback-only vs True Multi-LLM

Không phải mọi "gọi nhiều model" là multi-LLM. Đừng nhầm lẫn.

| Architecture | Ý tưởng | Ưu điểm | Hạn chế |
|---|---|---|---|
| Single-model | mọi request dùng một model | đơn giản | cost/latency/availability phụ thuộc một model |
| Fallback-only | model B khi model A lỗi | tăng reliability | không tối ưu cost/task quality |
| True multi-LLM | route theo task/policy/metrics | tối ưu quality-cost-latency | cần router, eval, observability |

<Callout type="warning" title="Multi-LLM không phải redundancy đơn giản">
Nếu bạn chỉ gọi provider B khi provider A lỗi, đó mới là fallback. Multi-LLM architecture thật sự cần routing policy, quality measurement, cost accounting, observability và governance.
</Callout>

---

## 4. Why One Model is Never Optimal for All Tasks

Các task khác nhau có requirements khác nhau. Một model frontier có thể làm mọi thứ "đủ tốt", nhưng unit economics có thể chết nếu bạn dùng nó cho hàng triệu tasks đơn giản.

| Task | Điều quan trọng | Model phù hợp |
|---|---|---|
| Simple classification | rẻ, nhanh, stable JSON | small/cheap model |
| Legal reasoning | accuracy, citations, long context | strong reasoning/long-context model |
| Code generation | code benchmarks, tool use | code-strong model |
| Vision QA | image understanding | multimodal model |
| Sensitive internal docs | privacy/data residency | private/local/compliant provider |
| High-volume extraction | cost, schema reliability | structured-output reliable model |

Một model không tối ưu cho mọi task. Đó là lý do bạn cần capability matrix.

---

## 5. Capability Matrix Design

Capability matrix là registry mô tả model/provider theo dimensions đo được. Nó là nền của routing và fallback.

| Dimension | What to measure | Example |
|---|---|---|
| Quality | task eval score | RAG correctness 88% |
| Latency | TTFT, p95 total latency | p95 < 4s |
| Cost | input/output price | $/1M tokens |
| Context | max + effective context | 128k max, lost-middle eval |
| Tools | tool-call validity rate | 97% valid args |
| JSON | schema pass rate | 99% valid JSON |
| Modality | text/image/audio/video | supports vision |
| Data policy | retention/training/region | EU allowed |
| Reliability | error/rate-limit rate | 99.9% success |

Example registry row:

```json
{
  "provider": "anthropic",
  "model": "claude-sonnet",
  "context_window": 200000,
  "supports_tools": true,
  "supports_vision": true,
  "json_score": 0.94,
  "rag_quality_score": 0.89,
  "p95_latency_ms": 6200,
  "cost_tier": "high",
  "allowed_regions": ["us", "eu"]
}
```

<Callout type="warning" title="Capability matrix là dữ liệu, không phải marketing">
Benchmark marketing của provider không phải eval score. Eval score phải đo trên data nội bộ của bạn. Nếu không có eval, routing chỉ là opinion.
</Callout>

## 6. Hidden Complexity of Multi-Provider Systems

Multi-provider không chỉ là đổi endpoint. Mỗi provider có khác biệt:

| Difference | Production impact |
|---|---|
| Tokenizer khác | token cost/context estimate khác |
| Message schema khác | prompt adapter cần normalize |
| Tool format khác | tool call parser/validator khác |
| Streaming event khác | frontend/gateway phải chuẩn hóa |
| Safety filter khác | answer/refusal behavior khác |
| Pricing khác | router phải cost-aware |
| Model versioning khác | regression monitoring cần thiết |
| Data policy khác | tenant routing phải enforce |

Đó là lý do multi-LLM cần provider abstraction layer. App không nên phụ thuộc provider-specific SDK.

---

## 7. Prompt Portability Problem

Prompt tối ưu cho model A có thể kém trên model B:

- model B cần JSON instruction rõ hơn,
- model C không hiểu cùng tool schema,
- model D over-refuse với safety wording,
- model E bỏ qua examples dài.

Vì vậy multi-LLM cần eval prompt trên từng target model. Prompt không portable là một rủi ro production.

## 8. Multi-LLM Architecture Patterns

| Pattern | Ý tưởng | Tradeoff |
|---|---|---|
| Static model selection | mỗi feature dùng một model cố định | đơn giản nhưng kém linh hoạt |
| User-tier routing | free → cheap, pro → stronger, enterprise → compliant | cost control nhưng quality gap |
| Task-based routing | router phân loại task rồi chọn model | tối ưu quality-cost |
| Verifier/judge model | generator → answer, judge → pass/fail/score | tăng cost/latency cho high-risk |
| Specialist models | embedding model cho retrieval, reranker cho ranking, code model cho code | chuyên biệt nhưng cần orchestration |

<Callout type="warning" title="Verifier/judge model đổi quality lấy gì">
Tăng cost/latency nhưng có thể tăng correctness/safety. Chỉ dùng cho high-risk workflows. Không phải mọi request cần judge.
</Callout>

---

## 9. Production Decision Framework

Trước khi thêm model mới, hỏi:

1. Model giải quyết gap nào?
2. Task nào dùng model này?
3. Eval score trên data nội bộ ra sao?
4. Cost/request và p95 latency là bao nhiêu?
5. Provider có được phép xử lý dữ liệu tenant không?
6. Model có support schema/tool/vision/context cần thiết không?
7. Fallback path là gì?
8. Prompt có portable không?
9. Observability có log model decision không?

## 10. Implementation Playbook: Model Capability Scorecard

```python
from dataclasses import dataclass


@dataclass
class ModelScore:
    name: str
    quality: float      # 0-1 from eval
    latency: float      # 0-1, higher is faster
    cost: float         # 0-1, higher is cheaper
    compliance: float   # 0-1
    json_reliability: float

    def weighted_score(self, weights):
        return (
            self.quality * weights["quality"] +
            self.latency * weights["latency"] +
            self.cost * weights["cost"] +
            self.compliance * weights["compliance"] +
            self.json_reliability * weights["json"]
        )


models = [
    ModelScore("frontier", 0.95, 0.55, 0.20, 0.80, 0.96),
    ModelScore("cheap-fast", 0.72, 0.95, 0.95, 0.85, 0.88),
    ModelScore("private-local", 0.78, 0.70, 0.70, 1.00, 0.82),
]

weights = {"quality": 0.4, "latency": 0.2, "cost": 0.2, "compliance": 0.1, "json": 0.1}
for m in sorted(models, key=lambda x: x.weighted_score(weights), reverse=True):
    print(m.name, round(m.weighted_score(weights), 3))
```

**Production checklist:**

- [ ] Có capability matrix cho từng model không?
- [ ] Có eval score theo task không?
- [ ] Có cost/latency thực tế không?
- [ ] Có data policy per tenant không?
- [ ] Có prompt tests cross-model không?
- [ ] Có log route reason không?
- [ ] Có fallback policy không?
- [ ] Có monitor model regression không?
- [ ] Có hard constraints cho compliance không?
- [ ] Có provider abstraction layer không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: thêm nhiều model nhưng không có eval">
**Triệu chứng:** Routing chỉ là opinion. Mỗi request có thể đến model sai, không biết ai tiêu tiền, không biết ai đang lãng phí.

**Fix:** Multi-LLM cần metrics theo task, không chỉ benchmark marketing. Eval score phải đo trên data nội bộ của bạn.
</Callout>

<Callout type="warning" title="Pitfall 2: quên prompt portability">
**Triệu chứng:** Prompt chạy tốt trên model A có thể fail trên model B. Routing đổi model, output thay đổi đột ngột, không biết tại sao.

**Fix:** Mỗi model route cần prompt compatibility tests. Prompt phải được eval trên từng target model.
</Callout>

<Callout type="danger" title="Pitfall 3: routing bỏ qua data policy">
**Triệu chứng:** Model tốt nhất về quality có thể không được phép nhận dữ liệu tenant. Routing chỉ tối ưu quality, không có hard constraints.

**Fix:** Compliance constraints phải là hard constraints, không phải soft score. Nếu tenant không cho external provider, route đó phải bị block tuyệt đối.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Frontier model cho mọi request | quality cao nhất | cost chết, không tối ưu |
| Cheap model cho mọi request | cost thấp | quality thấp, hallucination |
| Task-based routing | cost/quality cân bằng | cần router, eval, observability |
| Verifier/judge model | correctness cao | cost/latency tăng |
| No capability matrix | đơn giản | routing mù, không audit được |
| No fallback | đơn giản | provider outage chết toàn app |
| No route reason logging | đơn giản | debug không thể, audit không thể |

Multi-LLM không miễn phí. Mỗi model thêm phải có eval, cost, fallback, và trace. Không có gì miễn phí trong production.
