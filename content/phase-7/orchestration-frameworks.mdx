LLM Orchestration Frameworks như LangChain, LangGraph, LlamaIndex, CrewAI và AutoGen không phải "AI brain". Chúng là abstraction cho prompt templates, model adapters, tools, memory/state, retrieval, workflow graphs, tracing và multi-agent coordination. Một AI Engineer giỏi phải biết **khi nào dùng framework, khi nào không, và làm sao không để framework che mất architecture**.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Agent chạy 3,000 USD trong 2 giờ vì không có termination">
**Bối cảnh:** Một team build "AI research agent" cho SaaS. Agent dùng LangChain với `AgentExecutor` tự động. Agent có 3 tools: `search_web`, `read_doc`, `summarize`. Workflow: user hỏi → agent tự loop → agent trả answer.

**Sự cố:** Một user hỏi: "Tìm hiểu mọi thứ về thị trường X". Agent gọi search, nhận kết quả, nhưng không có done condition rõ. Nó reasoning: "vẫn chưa đủ thông tin", gọi search tiếp, đổi query, search lại. LangChain `AgentExecutor` không có max iterations mặc định (hoặc set quá cao), không có cost cap, không có progress detector.

1. Không có max steps, max tokens, max cost hay max wall-clock.
2. Stop condition duy nhất là "model tự quyết đã xong".
3. Không có progress detector, không nhận ra mình đang lặp.
4. Tool budget không được enforce — model có thể gọi tool vô hạn.
5. Hidden prompts trong framework làm agent reasoning khó debug.

**Hậu quả:** Hóa đơn API tăng 3,000 USD trong 2 giờ. Agent phải tắt khẩn cấp, và team phải thiết kế lại toàn bộ budget, termination controls, và trace resolved prompts.
</Callout>

Bài học rút ra: framework không thay thế architecture. Nếu bạn không hiểu state, retries, tool permissions, validation và tracing, LangChain hay LangGraph chỉ giúp bạn tạo bug phức tạp nhanh hơn. Framework là abstraction cho control/context flow, không phải AI brain.

## 2. Architecture Blueprint (Control Flow and Context Flow)

Frameworks thường gom nhiều thành phần:

```text
[ Prompt templates ]
        + [ Model adapters ]
        + [ Tools ]
        + [ Memory/state ]
        + [ Retrieval ]
        + [ Workflow graph ]
        + [ Tracing/evals ]
= Orchestration framework
```

Câu hỏi không phải "framework nào hot nhất", mà là:

```text
Problem của bạn cần abstraction nào?
```

## 3. Framework Comparison Matrix

| Framework | Strong at | Weakness/risk | Use when |
|---|---|---|---|
| LangChain | components, chains, tools, integrations | abstraction churn, hidden behavior | fast LLM app assembly |

| LlamaIndex | data connectors, indexes, query engines | ít general app orchestration hơn | RAG-heavy systems |
| CrewAI | role-based multi-agent tasks | dễ over-agentify | task decomposition demos/internal workflows |

## 4. LangChain

LangChain cung cấp:

- prompt templates,
- chains,
- tools,
- agents,
- retrievers,
- memory abstractions,
- LCEL,
- integrations,
- LangSmith tracing.

**Strength:**

- prototype nhanh,
- nhiều integrations,
- cộng đồng lớn,
- useful components.

**Risk:**

- abstraction thay đổi nhanh,
- hidden prompts/callbacks,
- khó kiểm soát khi agent phức tạp,
- app logic dễ phụ thuộc sâu vào framework objects.

**Production recommendation:**

```text
Use LangChain components behind your own interface.
Do not let framework types spread everywhere in business logic.
```

---

## 5. LangGraph

LangGraph phù hợp workflow có state và control flow rõ.

**Concepts:**

- state object,
- nodes,
- edges,
- conditional edges,
- cycles,
- checkpoints,
- human-in-the-loop.

**Graph example:**

```text
START
  ↓
classify_task
  ├── simple → answer_node
  ├── needs_tool → tool_node → validate_node
  └── uncertain → human_review
  ↓
END
```

LangGraph tốt khi bạn muốn agent không chạy tự do mà theo graph kiểm soát được.

<Callout type="tip" title="State graph tốt hơn agent loop mơ hồ">
Nếu workflow có steps rõ, conditional branches và validation gates, graph explicit thường dễ test/debug hơn agent tự quyết mọi thứ.
</Callout>

---

## 6. LlamaIndex

LlamaIndex mạnh ở data/RAG layer:

- data connectors/readers,
- document parsing,
- indexes,
- retrievers,
- query engines,
- response synthesizers,
- metadata filters.

Dùng khi core problem là:

```text
ingest data → index → retrieve → synthesize answer
```

**Production considerations:**

- kiểm soát chunking,
- metadata ACL filters,
- eval retrieval recall,
- citation correctness,
- avoid black-box query engines nếu cần audit sâu.

---

## 7. CrewAI

CrewAI tổ chức workflows bằng agents có roles/goals/tasks.

**Example mental model:**

```text
Researcher agent → gather facts
Writer agent → draft answer
Reviewer agent → critique
```

**Useful for:**

- internal automation prototypes,
- role-based task decomposition,
- demos/training.

**Risks:**

- role prompting theater,
- nhiều agents tăng cost/latency,
- unclear accountability,
- agent outputs amplify errors.

<Callout type="warning" title="Đừng over-agentify">
Nếu một deterministic workflow 3 bước giải quyết được task, đừng tạo 5 agents đóng vai chuyên gia. Multi-agent không tự động tốt hơn.
</Callout>

---

## 8. AutoGen

AutoGen tập trung vào multi-agent conversations:

- AssistantAgent,
- UserProxy,
- group chat,
- code execution,
- conversational collaboration.

**Strong for:**

- research experiments,
- multi-agent discussion,
- coding assistants with execution loops.

**Need controls:**

- max turns,
- termination conditions,
- cost caps,
- sandboxed code execution,
- tool permissions,
- transcript logging.

## 9. Framework Selection Decision Tree

```text
Core problem là retrieval/data indexing?
  yes → consider LlamaIndex

Need stateful workflow with cycles/branches/human review?
  yes → consider LangGraph

Need quick chain/tool prototype with many integrations?
  yes → consider LangChain

Need role-based decomposition for internal task automation?
  maybe CrewAI, but validate necessity

Need multi-agent conversation experiments?
  consider AutoGen

Need simple deterministic flow?
  use plain code first
```

## 10. Plain Code vs Framework

**Plain code tốt khi:**

- workflow đơn giản,
- cần full control,
- team chưa cần nhiều integrations,
- muốn dễ debug/test.

**Framework tốt khi:**

- nhiều integrations,
- graph/state phức tạp,
- RAG data connectors nhiều,
- need tracing/evals ecosystem,
- prototype speed quan trọng.

**Rule:**

```text
Start with architecture; choose framework to implement architecture, not to define it.
```

## 11. Production Risks

| Risk | Symptom | Mitigation |
|---|---|---|
| Hidden prompts | output thay đổi khó hiểu | log resolved prompts |
| Hidden retries | cost tăng bất ngờ | centralize retry policy |
| Agent loop | chạy mãi/tốn tiền | max turns, termination |
| Memory leak | context phình to | explicit state limits |
| Tool permission leak | agent gọi tool không nên | permission layer outside framework |
| Version churn | upgrade phá behavior | pin versions, tests |
| Lock-in | khó đổi framework | internal interfaces |

## 13. Implementation Playbook: Raw Code vs Graph Workflow

**Workflow:**

```text
classify request
  ↓
route model
  ↓
call tool if needed
  ↓
validate output
  ↓
fallback on failure
  ↓
return answer
```

**Plain code sketch:**

```python
def run_workflow(req):
    task = classify(req)
    model = route(task, req)

    if task.needs_tool:
        observation = call_tool(task.tool, req)
    else:
        observation = None

    answer = call_model(model, req, observation)
    if not validate(answer):
        answer = fallback(req)

    return answer
```

**Graph mental model:**

```text
START → classify → route → maybe_tool → generate → validate
                              ↑                  ↓
                              └──── fallback ← fail
```

**Compare:**

| Criteria | Plain code | Graph framework |
|---|---|---|
| Simplicity | tốt cho flow nhỏ | overhead |
| Visualization | manual | built-in graph |
| State checkpoint | manual | framework support |
| Debug | direct | needs tracing |
| Complex cycles | messy | stronger |

**Production checklist:**

- [ ] Có workflow diagram không?
- [ ] Có state schema rõ không?
- [ ] Có max turns/cost cap không?
- [ ] Có log resolved prompts không?
- [ ] Tool permissions nằm ngoài framework không?
- [ ] Có validation gates không?
- [ ] Có trace node/chain/tool không?
- [ ] Framework versions có pin không?
- [ ] Có tests không gọi provider thật không?
- [ ] Có exit criteria cho agents không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: chọn framework trước khi hiểu workflow">
**Triệu chứng:** Nếu chưa vẽ được state/data flow bằng text, framework sẽ che giấu vấn đề thay vì giải quyết. Output thay đổi khó debug, không biết tại sao.

**Fix:** Bắt đầu với architecture; chọn framework để implement architecture, không để framework define architecture. Vẽ workflow diagram trước.
</Callout>

<Callout type="warning" title="Pitfall 2: để framework quyết retry/tool permissions">
**Triệu chứng:** Retry, fallback và permissions là policy của hệ thống. Không nên rải rác trong framework defaults. Cost tăng bất ngờ, tool permission leak.

**Fix:** Centralize retry policy. Tool permissions nằm ở policy/security layer của app/gateway, không chỉ trong framework default.
</Callout>

<Callout type="danger" title="Pitfall 3: uncontrolled agent loops">
**Triệu chứng:** Agent không có max turns, termination criteria và cost cap có thể chạy rất lâu, gọi tool liên tục và tạo bill lớn. Framework defaults không đủ.

**Fix:** Max turns, termination criteria, cost cap, tool permissions, sandboxing, traces. Agent loop phải có budget, terminator và exit criteria.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Framework | prototype nhanh, nhiều integrations | hidden behavior, version churn |
| Plain code | full control, dễ debug | ít integrations, nhiều code |
| LangGraph | stateful controlled workflows | overhead, cần design rõ |
| LlamaIndex | RAG data layer mạnh | ít general app orchestration |
| CrewAI | role-based decomposition | over-agentify, cost/latency |
| AutoGen | multi-agent experiments | loop/cost control phức tạp |
| No framework | simple, deterministic | nhiều code, ít ecosystem |

Framework selection là nghệ thuật chuyển "chọn framework" thành "chọn framework đúng problem shape". Mỗi framework phải có architecture, policy, và observability rõ.

## Điểm cần nhớ

- Framework là abstraction cho control/context flow, không phải AI brain.
- Chọn framework theo problem shape: RAG, graph workflow, integrations, multi-agent experiments.
- LangGraph phù hợp stateful controlled workflows; LlamaIndex mạnh về RAG/data layer.
- CrewAI/AutoGen hữu ích nhưng dễ tạo uncontrolled multi-agent complexity.
- Architecture và policy phải rõ trước khi chọn framework.
- Observability, validation, permissions và cost caps vẫn là trách nhiệm của bạn.
