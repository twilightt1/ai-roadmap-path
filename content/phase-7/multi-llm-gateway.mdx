Multi-LLM Gateway là control plane cho AI calls trong production. Nó không chỉ forward request sang OpenAI/Claude/Gemini; nó enforce policy: ai được gọi model nào, với budget nào, prompt version nào, fallback nào, cache nào, validation nào và trace nào. Đây là chương capstone của Phase 7 vì nó gom routing, fallback, provider abstraction, cost tracking, cache, security và observability thành một kiến trúc thống nhất.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Không có gateway, mỗi app tự gọi provider, 50,000 USD/tháng không biết ai tiêu">
**Bối cảnh:** Một công ty SaaS có 5 product teams. Mỗi team tự gọi OpenAI/Anthropic trực tiếp. Mỗi team tự implement API keys, retries, fallback, token counting, logging, cost tracking. Không có central control plane.

**Sự cố:** Hóa đơn API tăng 50,000 USD/tháng. Finance team không biết ai tiêu tiền. Không có cost attribution per tenant/user/feature. Một team dùng frontier model cho 90% simple classification tasks. Một team leak API key trong code. Một team không có fallback, provider outage chết toàn app.

1. Mỗi app tự implement API keys, retries, fallback, cost tracking.
2. Không có central model registry, không có capability matrix.
3. Không có prompt versioning, không có audit log.
4. Không có tenant policy, không có data residency enforcement.
5. API key leak, không có key rotation, không có audit.

**Hậu quả:** 50,000 USD/tháng không biết ai tiêu. 5 product teams phải tắt khẩn cấp, thiết kế lại toàn bộ central gateway. Hóa đơn giảm 50,000 USD/tháng xuống 18,000 USD/tháng sau khi 90% simple task được chuyển sang model rẻ, cost tracking được centralize, và tenant policy được enforce.
</Callout>

Bài học rút ra: gateway không chỉ là proxy. Nếu gateway không enforce policy, route, validate, log cost và manage fallback, nó chưa giải quyết multi-LLM production problems. Gateway là control plane, không chỉ forward request.

## 4. Provider Abstraction Layer

```python
from typing import Protocol, AsyncIterator
from dataclasses import dataclass


@dataclass
class ChatRequest:
    messages: list[dict]
    model: str
    temperature: float = 0.2
    max_tokens: int = 1000


@dataclass
class Usage:
    input_tokens: int
    output_tokens: int


@dataclass
class ChatResponse:
    text: str
    provider: str
    model: str
    finish_reason: str
    usage: Usage
    latency_ms: float


class LLMProvider(Protocol):
    async def chat(self, request: ChatRequest) -> ChatResponse:
        ...

    async def stream(self, request: ChatRequest) -> AsyncIterator[dict]:
        ...

    async def embed(self, texts: list[str], model: str) -> list[list[float]]:
        ...
```

Normalized response giúp app không phụ thuộc provider-specific SDK.

## 6. Request Lifecycle

```text
1. Authenticate tenant/user
2. Load tenant AI policy
3. Resolve prompt version
4. Normalize request
5. Estimate tokens and cost
6. Check quota/rate limit
7. Cache lookup
8. Route model
9. Execute provider call
10. Retry/fallback if needed
11. Validate response
12. Store usage event/trace
13. Return normalized response
```

<Callout type="warning" title="Lifecycle order là policy">
Cache trước hay sau quota? Token count trước hay sau prompt resolution? Validation trước hay sau logging? Những quyết định này ảnh hưởng cost, security và correctness.
</Callout>

## 11. Cache Manager

### Exact cache

Key:

```text
sha256(tenant_id + prompt_version + model + normalized_input + settings + data_version)
```

### Semantic cache

Uses embedding similarity. More dangerous; requires:

- tenant isolation,
- similarity threshold,
- answer freshness,
- no high-risk personalized outputs,
- audit cache hits.

### Embedding cache

Key includes:

```text
embedding_model + text_hash + preprocessing_version
```

## 17. Implementation Playbook: Gateway Skeleton

```python
from dataclasses import dataclass


@dataclass
class GatewayRequest:
    tenant_id: str
    user_id: str
    feature: str
    prompt_name: str
    input_text: str
    requires_json: bool = False


class Gateway:
    def __init__(self, registry, router, cache, providers, validator, usage_store):
        self.registry = registry
        self.router = router
        self.cache = cache
        self.providers = providers
        self.validator = validator
        self.usage_store = usage_store

    async def chat(self, req: GatewayRequest):
        policy = load_tenant_policy(req.tenant_id)
        prompt = resolve_prompt(req.prompt_name, policy)
        payload = build_payload(prompt, req)

        token_estimate = estimate_tokens(payload)
        check_quota(policy, token_estimate)

        cache_key = build_cache_key(req, payload)
        if cached := self.cache.get(cache_key):
            return cached

        route = self.router.select(req, policy, self.registry)
        provider = self.providers[route.provider]

        response = await provider.chat(route.to_chat_request(payload))
        validated = self.validator.validate(response, req)

        self.usage_store.record(req, route, response, cache_hit=False)
        self.cache.set(cache_key, validated)
        return validated
```

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: gateway chỉ là thin proxy">
**Triệu chứng:** Nếu gateway không enforce policy, route, validate, log cost và manage fallback, nó chưa giải quyết multi-LLM production problems. Mỗi app tự gọi provider, cost không biết ai tiêu.

**Fix:** Gateway phải enforce policy: model, budget, tenant, prompt, routing, fallback, validation, cost và observability. Gateway là control plane, không chỉ forward request.
</Callout>

<Callout type="warning" title="Pitfall 2: không có model registry">
**Triệu chứng:** Hard-code model metadata trong code làm routing/fallback/cost updates chậm và dễ sai. Model registry không có, không có capability matrix.

**Fix:** Model registry là nguồn sự thật cho router/fallback/cost estimator. Registry phải có: provider, model, context, price, modalities, tools, JSON support, region, data policy, health, eval scores.
</Callout>

<Callout type="danger" title="Pitfall 3: semantic cache shared giữa tenants">
**Triệu chứng:** Semantic cache không tenant-isolated có thể leak dữ liệu. Cache phải tôn trọng tenant, user permissions và data version. Nếu không, tenant A có thể thấy answer của tenant B.

**Fix:** Cache phải tôn trọng tenant, user permissions và data version. Semantic cache cần tenant isolation, similarity threshold, data version, permission version, no high-risk personalized cache.
</Callout>
