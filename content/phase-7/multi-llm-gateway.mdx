Multi-LLM Gateway là control plane cho AI calls trong production. Nó không chỉ forward request sang OpenAI/Claude/Gemini; nó enforce policy: ai được gọi model nào, với budget nào, prompt version nào, fallback nào, cache nào, validation nào và trace nào. Đây là chương capstone của Phase 7 vì nó gom routing, fallback, provider abstraction, cost tracking, cache, security và observability thành một kiến trúc thống nhất.

## Why this matters

Nếu không có gateway, mỗi app/team sẽ tự implement:

- API keys,
- retries,
- fallback,
- token counting,
- logging,
- prompt versions,
- cost tracking,
- provider adapters,
- schema validation,
- rate limits.

Kết quả thường là:

- duplicated bugs,
- inconsistent security,
- không biết ai tiêu tiền,
- khó đổi provider,
- prompt/version phân tán,
- không có global observability.

<Callout type="danger" title="Gateway là control plane, không chỉ proxy">
Nếu gateway chỉ forward request sang provider, nó chưa phải multi-LLM gateway. Gateway phải enforce policy: model, budget, tenant, prompt, routing, fallback, validation, cost và observability.
</Callout>

## Provider abstraction layer

Interface:

```python
from typing import Protocol, AsyncIterator
from dataclasses import dataclass


@dataclass
class ChatRequest:
    messages: list[dict]
    model: str
    temperature: float = 0.2
    max_tokens: int = 1000


@dataclass
class Usage:
    input_tokens: int
    output_tokens: int


@dataclass
class ChatResponse:
    text: str
    provider: str
    model: str
    finish_reason: str
    usage: Usage
    latency_ms: float


class LLMProvider(Protocol):
    async def chat(self, request: ChatRequest) -> ChatResponse:
        ...

    async def stream(self, request: ChatRequest) -> AsyncIterator[dict]:
        ...

    async def embed(self, texts: list[str], model: str) -> list[list[float]]:
        ...
```

Normalized response giúp app không phụ thuộc provider-specific SDK.

## Request lifecycle

```text
1. Authenticate tenant/user
2. Load tenant AI policy
3. Resolve prompt version
4. Normalize request
5. Estimate tokens and cost
6. Check quota/rate limit
7. Cache lookup
8. Route model
9. Execute provider call
10. Retry/fallback if needed
11. Validate response
12. Store usage event/trace
13. Return normalized response
```

<Callout type="warning" title="Lifecycle order là policy">
Cache trước hay sau quota? Token count trước hay sau prompt resolution? Validation trước hay sau logging? Những quyết định này ảnh hưởng cost, security và correctness.
</Callout>

## Cache manager

### Exact cache

Key:

```text
sha256(tenant_id + prompt_version + model + normalized_input + settings + data_version)
```

### Semantic cache

Uses embedding similarity. More dangerous; requires:

- tenant isolation,
- similarity threshold,
- answer freshness,
- no high-risk personalized outputs,
- audit cache hits.

### Embedding cache

Key includes:

```text
embedding_model + text_hash + preprocessing_version
```

## Scaling architecture

```text
Load Balancer
   ↓
Stateless Gateway Pods
   ├── Redis: cache, rate limit, circuit state
   ├── PostgreSQL: prompt registry, usage events, policies
   ├── Kafka/Queue: async logs, batch jobs, eval events
   ├── Vector DB: semantic cache / RAG if needed
   └── Provider APIs / vLLM / MCP servers
```

Scaling concerns:

- gateway pods stateless,
- provider concurrency limits,
- Redis hot keys,
- Postgres write volume for usage events,
- streaming connections hold workers,
- backpressure during provider outage,
- per-tenant quotas distributed correctly.

## Implementation lab: gateway skeleton

```python
from dataclasses import dataclass


@dataclass
class GatewayRequest:
    tenant_id: str
    user_id: str
    feature: str
    prompt_name: str
    input_text: str
    requires_json: bool = False


class Gateway:
    def __init__(self, registry, router, cache, providers, validator, usage_store):
        self.registry = registry
        self.router = router
        self.cache = cache
        self.providers = providers
        self.validator = validator
        self.usage_store = usage_store

    async def chat(self, req: GatewayRequest):
        policy = load_tenant_policy(req.tenant_id)
        prompt = resolve_prompt(req.prompt_name, policy)
        payload = build_payload(prompt, req)

        token_estimate = estimate_tokens(payload)
        check_quota(policy, token_estimate)

        cache_key = build_cache_key(req, payload)
        if cached := self.cache.get(cache_key):
            return cached

        route = self.router.select(req, policy, self.registry)
        provider = self.providers[route.provider]

        response = await provider.chat(route.to_chat_request(payload))
        validated = self.validator.validate(response, req)

        self.usage_store.record(req, route, response, cache_hit=False)
        self.cache.set(cache_key, validated)
        return validated
```

## Common mistakes & debugging checklist

<Callout type="warning" title="Mistake 1: gateway chỉ là thin proxy">
Nếu gateway không enforce policy, route, validate, log cost và manage fallback, nó chưa giải quyết multi-LLM production problems.
</Callout>

<Callout type="warning" title="Mistake 2: không có model registry">
Hard-code model metadata trong code làm routing/fallback/cost updates chậm và dễ sai.
</Callout>

<Callout type="danger" title="Mistake 3: semantic cache shared giữa tenants">
Semantic cache không tenant-isolated có thể leak dữ liệu. Cache phải tôn trọng tenant, user permissions và data version.
</Callout>

Checklist:

1. Có provider abstraction không?
2. Có model/capability registry không?
3. Có tenant AI policy không?
4. Có prompt registry/versioning không?
5. Có token/cost tracking không?
6. Có router reason logs không?
7. Có fallback manager không?
8. Có response validation layers không?
9. Có exact cache tenant-safe không?
10. Có streaming normalization không?
11. Có metrics/traces/logs không?
12. Có security controls cho gateway keys/logs không?

## Answer key

1. Vì nó centralize/enforce AI policy: model, prompt, routing, budget, validation, logging.
2. Thin proxy chỉ forward; gateway route, validate, cache, fallback, track cost, enforce tenant policy.
3. Provider, model, context, price, modalities, tools, JSON support, region, data policy, health, eval scores.
4. Auth, policy, prompt, normalize, token/cost estimate, quota, cache, route, provider call, fallback, validate, log, return.
5. Tenant, user, feature, provider, model, prompt version, route policy, cache/fallback.
6. Tenant isolation, similarity threshold, data version, permission version, no high-risk personalized cache.
7. Transport, finish reason, syntax, schema, business, groundedness/citation, safety/policy.
8. Normalized events, cancellation, partial errors, usage metadata, client disconnect, buffering if needed.
9. Stateless pods; shared state in Redis/Postgres/queues/vector DB depending component.
10. It holds provider keys, prompts, logs, policies and can route sensitive data; needs strict security/audit.
