Routing là bộ não của multi-LLM system. Router biến một user request cộng với business constraints thành quyết định: dùng model nào, provider nào, prompt version nào, max tokens bao nhiêu, có được gửi dữ liệu này ra ngoài không, và nếu uncertain thì fallback thế nào. Ở mức giáo sư AI engineering, routing là **constrained optimization under uncertainty**, không phải vài câu `if/else` cảm tính.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Router route task khó sang model rẻ, tạo 12 refund sai cho khách hàng">
**Bối cảnh:** Một team build "AI refund agent" cho SaaS. Router có rule: `task == "simple_classification" → cheap model`. Mọi request đi qua classifier trước, rồi route.

**Sự cố:** Một user gửi câu: "Tôi muốn refund 3 đơn hàng vì giao hàng chậm". Classifier misclassify: "simple_classification" → cheap model. Cheap model hiểu "refund 3 đơn hàng" = "refund 3 đơn bất kỳ". Nó gọi tool `issue_refund` 3 lần cho 3 đơn hàng gần nhất — không phải 3 đơn hàng user muốn.

1. Router chỉ dựa vào task label, không có confidence threshold.
2. Không có hard constraint cho side-effect actions.
3. Không có human approval cho refund.
4. Tool `issue_refund` không có idempotency key.
5. Không có validation gate trước khi execute side effect.

**Hậu quả:** 12 refund sai (3 đơn × 4 user bị ảnh hưởng do retry). Khách hàng khiếu nại, team phải refund tay, mất 2 ngày làm việc. Router phải được thiết kế lại với confidence threshold, hard constraints, và approval gate.
</Callout>

Bài học rút ra: router sai thì toàn bộ hệ thống sai. Một router chọn model rẻ cho task cần reasoning có thể tạo answer tự tin nhưng sai. Một router chọn model mạnh cho mọi request có thể làm sản phẩm lỗ tiền. Routing phải được đo bằng quality, cost và latency cùng lúc.

## 2. Architecture Blueprint (Constrained Optimization Router)

Một router production không phải là một `if/else`. Nó là một **constrained optimizer**:

```text
[ User request ]
        │
        ▼
[ Task / modality / tokens classifier ]
        │
        ▼
[ Candidate model set from registry ]
        │
        ▼
[ Hard constraints: context, schema, compliance, cost, latency ]
        │
        ▼
[ Soft optimization: quality, cost, latency ]
        │
        ▼
[ Selected model + fallback candidates + route reason ]
        │
        ▼
[ Validation + fallback + trace ]
```

Router chọn model `m` cho request `r`:

```text
choose model m
maximize expected_quality(task, m)
subject to:
  estimated_cost(m, tokens) <= budget
  expected_latency(m) <= SLA
  provider_allowed(m, tenant_data) == true
  required_capabilities ⊆ capabilities(m)
  context_window(m) >= estimated_input + output_budget
```

Đây là bài toán nhiều mục tiêu. Bạn không tối ưu quality một mình; bạn tối ưu quality trong ràng buộc production.

## 3. Routing Inputs

Một router tốt cần nhiều signals:

| Input | Example | Dùng để |
|---|---|---|
| task type | classify, RAG, code, vision | chọn model family |
| modality | text/image/audio | require multimodal |
| token estimate | 8k input | context/cost |
| output schema | strict JSON | schema-capable model |
| user tier | free/pro/enterprise | budget/model tier |
| tenant policy | EU-only, no external | compliance |
| latency SLA | p95 &lt; 3s | fast model/streaming |
| remaining budget | monthly quota | cost routing |
| eval scores | per-task model score | quality routing |
| cache availability | exact/semantic hit | avoid provider call |

Task là signal mạnh nhất, nhưng không đứng một mình. Context length, schema, latency, cost và compliance có thể override task route.

## 4. Task-Based Routing

Task là signal mạnh nhất.

| Task | Candidate route | Vì sao |
|---|---|---|
| Simple classification | small cheap model | output ngắn, deterministic |
| JSON extraction | schema reliable model | parser downstream cần ổn định |
| Hard reasoning | frontier reasoning model | quality quan trọng |
| Code generation | code-strong model | syntax/tests quan trọng |
| Vision QA | multimodal model | cần image understanding |
| Long document QA | RAG + long-context fallback | context lớn |
| Sensitive data | private/local/compliant model | data policy |

<Callout type="warning" title="Task label chưa đủ">
Task label quan trọng nhưng chưa đủ. Context length, schema, latency, cost và compliance có thể override task route. Nếu chỉ dựa vào task label, bạn có thể route sai.
</Callout>

---

## 5. Cost-Based Routing

Cost estimate:

```text
estimated_cost = input_tokens * input_price + output_budget * output_price
```

Cost routing strategies:

- free tier dùng cheap model,
- pro tier dùng stronger model,
- enterprise dùng compliant model,
- compress context trước khi gọi model đắt,
- cap `max_output_tokens`,
- route batch/offline sang cheaper slower model.

<Callout type="warning" title="Cost routing không được làm mù quality">
Nếu model rẻ làm sai và cần human correction hoặc retry nhiều lần, tổng cost có thể cao hơn model mạnh ngay từ đầu. Cost routing phải đo được quality, không chỉ price.
</Callout>

---

## 6. Latency-Based Routing

Latency dimensions:

- TTFT,
- total latency,
- provider region,
- streaming support,
- queue/load,
- model size.

Real-time chat khác batch job:

```text
chat user waiting → optimize TTFT + streaming
nightly summarization → optimize cost/throughput
```

---

## 7. Context-Window Routing

Nếu prompt estimate vượt model context:

```text
input_tokens + output_budget > context_window
```

Options:

1. compress/summarize,
2. reduce retrieved chunks,
3. multi-step retrieval,
4. route to long-context model,
5. ask user to narrow query.

Long-context route nên là last resort nếu retrieval/compression không đủ hoặc task thật sự cần full context.

---

## 8. Compliance and Privacy Routing

Compliance constraints là hard constraints, không phải soft score:

```text
if tenant.policy.external_providers_allowed == false:
    candidates = local_or_private_models_only
```

Examples:

- EU tenant → EU region only.
- Healthcare data → approved provider only.
- Source code → no training/retention provider.
- Secrets detected → block or private model.

<Callout type="danger" title="Compliance là hard constraint">
Nếu tenant không cho external provider, route đó phải bị block tuyệt đối. Không được để quality/cost bù compliance violation.
</Callout>

## 9. Rule-Based vs Classifier-Based Router

### Rule-based router

Rule-based router dễ hiểu, dễ audit, phù hợp giai đoạn đầu.

```python
from dataclasses import dataclass


@dataclass
class Request:
    task: str
    modality: str
    estimated_tokens: int
    requires_json: bool
    contains_pii: bool
    user_tier: str
    tenant_requires_private: bool


def route(req: Request) -> str:
    if req.contains_pii and req.tenant_requires_private:
        return "local-private-model"
    if req.modality == "image":
        return "vision-model"
    if req.estimated_tokens > 100_000:
        return "long-context-model"
    if req.requires_json:
        return "json-reliable-model"
    if req.task == "simple_classification":
        return "cheap-fast-model"
    if req.user_tier == "enterprise":
        return "high-quality-model"
    return "default-balanced-model"
```

Giới hạn:

- rules phình to,
- conflict khó quản lý,
- không tự học từ performance,
- brittle khi tasks đa dạng.

### Classifier-based router

Dùng classifier/LLM nhỏ để dự đoán task hoặc difficulty.

```text
query → router model → {task, difficulty, confidence}
```

Nếu confidence thấp:

- route tới default safe model,
- ask clarification,
- use stronger model,
- run multiple candidates.

```python
def route_with_confidence(router_output):
    if router_output.confidence < 0.65:
        return "safe-default-model"
    if router_output.task == "code":
        return "code-model"
    if router_output.difficulty == "hard":
        return "frontier-model"
    return "cheap-model"
```

<Callout type="warning" title="Router model cũng có lỗi">
Nếu router là LLM, nó cũng hallucinate hoặc misclassify. Router cần eval, calibration và fallback khi uncertain. Confidence threshold là bắt buộc.
</Callout>

## 10. A/B, Shadow, Canary và Bandit Routing

### A/B routing

Một phần traffic thật dùng route A, phần khác route B. Metrics: task success, user satisfaction, cost/request, latency, invalid JSON rate, escalation/human correction.

### Shadow routing

Gửi request tới model B nhưng không trả response cho user. Dùng để đo model mới an toàn hơn.

### Canary routing

Rollout model mới cho 1–5% traffic trước, monitor errors/cost/quality.

### Bandit routing

Bandit routing cân bằng exploration và exploitation.

```text
exploit: dùng model đang tốt nhất
explore: thử model khác để học performance
```

Phù hợp: high-volume low-risk tasks, online feedback rõ, cost per request chấp nhận được.

Không phù hợp: safety-critical, legal/medical high stakes, feedback noisy, compliance constraints phức tạp.

## 12. Implementation Playbook: Intelligent Router

```python
from dataclasses import dataclass


@dataclass
class ModelInfo:
    name: str
    context_window: int
    cost_per_1k: float
    p95_latency_ms: int
    supports_json: bool
    supports_vision: bool
    allowed_for_private: bool
    quality_by_task: dict[str, float]


@dataclass
class RouteRequest:
    task: str
    estimated_tokens: int
    output_budget: int
    requires_json: bool
    modality: str
    private_data: bool
    max_cost: float
    max_latency_ms: int


def estimate_cost(model: ModelInfo, req: RouteRequest) -> float:
    return ((req.estimated_tokens + req.output_budget) / 1000) * model.cost_per_1k


def route(req: RouteRequest, models: list[ModelInfo]):
    candidates = []
    reasons = {}

    for m in models:
        reject = []
        if req.estimated_tokens + req.output_budget > m.context_window:
            reject.append("context_window_too_small")
        if req.requires_json and not m.supports_json:
            reject.append("json_not_supported")
        if req.modality == "image" and not m.supports_vision:
            reject.append("vision_not_supported")
        if req.private_data and not m.allowed_for_private:
            reject.append("data_policy_blocked")
        if estimate_cost(m, req) > req.max_cost:
            reject.append("cost_too_high")
        if m.p95_latency_ms > req.max_latency_ms:
            reject.append("latency_too_high")

        if reject:
            reasons[m.name] = reject
            continue
        candidates.append(m)

    if not candidates:
        return None, reasons

    best = max(candidates, key=lambda m: m.quality_by_task.get(req.task, 0.0))
    return best.name, reasons
```

**Production checklist:**

- [ ] Router có hard constraints không?
- [ ] Có estimate tokens trước routing không?
- [ ] Có model capability registry không?
- [ ] Có eval scores per task không?
- [ ] Có confidence threshold không?
- [ ] Có route reason logs không?
- [ ] Có A/B hoặc shadow testing không?
- [ ] Có monitor cost/latency by route không?
- [ ] Có fallback khi router uncertain không?
- [ ] Có log rejected candidates không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: router chỉ dựa vào task label">
**Triệu chứng:** Task label quan trọng nhưng chưa đủ. Context length, schema, latency, cost và compliance có thể override task route.

**Fix:** Router phải xét nhiều signals: task, modality, tokens, schema, user tier, tenant policy, latency SLA, budget, eval scores.
</Callout>

<Callout type="warning" title="Pitfall 2: không log rejected candidates">
**Triệu chứng:** Khi route sai, bạn cần biết model nào bị loại và vì sao. Nếu chỉ log selected model, debug rất khó.

**Fix:** Route reason log phải chứa selected model, reasons, rejected candidates, constraints, estimated cost, confidence, policy version.
</Callout>

<Callout type="danger" title="Pitfall 3: coi compliance là score mềm">
**Triệu chứng:** Nếu tenant không cho external provider, route đó phải bị block tuyệt đối. Không được để quality/cost bù compliance violation.

**Fix:** Compliance constraints phải là hard constraints. Nếu `tenant.policy.external_providers_allowed == false`, route đó phải bị block tuyệt đối.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Rule-based router | đơn giản, dễ audit | brittle khi tasks đa dạng |
| Classifier-based router | adaptive | router model cũng có lỗi |
| Bandit routing | tự học performance | không phù hợp high-risk |
| No confidence threshold | đơn giản | route sai khi uncertain |
| No route reason logs | đơn giản | debug không thể, audit không thể |
| No A/B/shadow | đơn giản | rollout mù, regression không biết |
| No fallback candidates | đơn giản | router uncertain thì chết |

Routing là nghệ thuật chuyển "chọn model" thành "chọn model có kiểm soát". Mỗi decision phải có reason, fallback, và trace.

## Answer key

1. Chọn model tối đa quality kỳ vọng dưới ràng buộc cost, latency, compliance, context và capabilities.
2. Task, modality, tokens, schema, user tier, tenant policy, budget, latency SLA, eval scores.

4. Để xử lý uncertainty; confidence thấp nên route safe/default hoặc ask clarification.
5. Shadow không trả output model mới cho user; A/B trả output thật cho một phần traffic.

7. Selected model, reasons, rejected candidates, constraints, estimated cost, confidence, policy version.

## Điểm cần nhớ

- Router là constrained optimizer, không chỉ if/else task label.
- Compliance, context window và required capabilities là hard constraints.
- Cost/latency/quality là tradeoff cần đo.
- Classifier router cần confidence threshold và calibration.
- A/B, shadow, canary giúp rollout routing policy an toàn.
- Route reason logs là bắt buộc để debug và audit.
