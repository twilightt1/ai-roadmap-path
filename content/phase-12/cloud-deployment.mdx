Bạn đã có Docker image và hiểu Kubernetes. Nhưng production AI thường không chạy trong phòng máy tự dựng. Team cần GPU quota, object storage, managed databases, IAM, secret manager, logging, autoscaling và region compliance. Đây là lúc cloud trở thành một phần của kiến trúc AI.

**Cloud Deployment** không chỉ là chọn AWS/GCP/Azure theo sở thích. Với AI systems, cloud decision ảnh hưởng trực tiếp đến latency, data residency, GPU availability, egress cost, security boundary và khả năng scale khi sản phẩm tăng trưởng.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Dữ liệu khách hàng EU bị gửi sang region Mỹ">
**Bối cảnh:** Một công ty fintech châu Âu deploy chatbot RAG trên cloud. Backend chạy ở `us-central1`, object storage chứa hợp đồng khách hàng cũng ở Mỹ vì engineer chọn default region. LLM provider endpoint cũng route qua US.

**Sự cố:** Audit GDPR phát hiện dữ liệu PII của công dân EU rời khỏi khu vực cho phép. Không chỉ file storage sai region, logs và traces cũng chứa prompt có PII và được lưu ở vendor SaaS ngoài EU.

1. Không có region policy trước khi deploy.
2. Không phân loại dữ liệu PII vs non-PII.
3. Không kiểm tra data residency của LLM provider/observability vendor.
4. Infrastructure được tạo thủ công, không có review compliance.

**Hậu quả:** Hệ thống bị dừng, phải migrate sang EU region, xóa logs nhạy cảm và trải qua security review kéo dài.
</Callout>

Bài học rút ra: cloud deployment là quyết định kỹ thuật và pháp lý. AI request thường chứa dữ liệu nhạy cảm; region, logging và provider routing phải được thiết kế ngay từ đầu.

## 2. Architecture Blueprint (Cloud AI Landing Zone)

Một cloud deployment production cần landing zone có mạng, IAM, storage, compute và observability.

```text
[ Users ]
   │
   ▼
[ CDN / Load Balancer / API Gateway ]
   - TLS
   - WAF
   - auth
   - rate limit
   │
   ▼
[ App runtime ]
   - Cloud Run / ECS / EKS / GKE / AKS
   - FastAPI gateway
   - workers
   │
   ├────────► [ Managed AI / LLM APIs ]
   │           OpenAI / Azure OpenAI / Bedrock / Vertex AI
   │
   ├────────► [ GPU inference endpoints ]
   │           SageMaker / Vertex Endpoint / self-host vLLM
   │
   ├────────► [ Data layer ]
   │           S3/GCS/Blob, Postgres, Redis, Vector DB
   │
   └────────► [ Observability + Security ]
               logs, traces, metrics, secrets, IAM, audit
```

Mục tiêu: không deploy app đơn lẻ; deploy một hệ sinh thái có boundary rõ ràng, có least privilege và có cost visibility.

## 3. AWS vs GCP vs Azure for AI Workloads

| Cloud | Điểm mạnh | Khi phù hợp |
|---|---|---|
| AWS | SageMaker, Bedrock, EKS, S3, IAM mạnh, GPU instance đa dạng | doanh nghiệp lớn, hạ tầng phức tạp, cần nhiều dịch vụ |
| GCP | Vertex AI, GKE, BigQuery, Cloud Run, TPU/JAX ecosystem | data/analytics mạnh, Kubernetes mượt, serverless tốt |
| Azure | Azure OpenAI, enterprise compliance, Microsoft ecosystem | công ty dùng Microsoft, cần private Azure OpenAI/data residency |
| Multi-cloud | tránh lock-in, chọn provider theo model | team đủ mạnh để chịu complexity |

Lựa chọn thực dụng:

- Nếu công ty đã chuẩn hóa cloud nào, bắt đầu ở cloud đó.
- Nếu cần OpenAI với enterprise/data controls, Azure OpenAI thường là lựa chọn phổ biến.
- Nếu team nhỏ, serverless/container managed có thể tốt hơn tự vận hành EKS/GKE.

## 4. Deployment Options: Serverless, Managed Endpoints, Kubernetes

| Option | Ví dụ | Ưu điểm | Giới hạn |
|---|---|---|---|
| Serverless containers | Cloud Run, AWS App Runner, Azure Container Apps | đơn giản, scale to zero, ít DevOps | cold start, GPU hạn chế tùy platform |
| Managed ML endpoint | SageMaker Endpoint, Vertex AI Endpoint, Azure ML | tích hợp model registry, autoscaling | cost cao, platform-specific |
| Kubernetes | EKS/GKE/AKS | linh hoạt, GPU scheduling, ecosystem mạnh | vận hành phức tạp |
| API provider | OpenAI/Anthropic/Gemini | không quản GPU, model mạnh | cost/token, rate limit, data controls |
| Hybrid | gateway self-host + provider fallback | cân bằng cost/reliability | routing và observability khó hơn |

Không cần Kubernetes cho mọi app. Một internal chatbot nhỏ có thể chạy tốt trên Cloud Run + managed API. Một platform nhiều model/tenant có thể cần K8s.

## 5. IAM, Secrets và Network Boundaries

Cloud AI system thường hỏng vì quyền quá rộng.

Bad pattern:

```text
App service account:
  s3:* on *
  secrets:* on *
  cloudwatch:* on *
```

Better pattern:

```text
ai-api-prod service account:
  read secret: openai_api_key_prod
  read/write bucket: s3://ai-prod-logs/tenant-scoped/*
  read bucket: s3://ai-prod-models/customer-support/v12/*
  no permission to raw PII bucket
```

Network boundary cần:

- private subnets cho database/vector DB,
- VPC endpoint/private link đến storage nếu có,
- egress allowlist đến LLM providers,
- no public bucket,
- secrets trong Secret Manager, không trong env file commit,
- audit logs cho IAM changes.

## Implementation Playbook: Deploy AI API to Cloud Run

Ví dụ này phù hợp cho API nhỏ/medium dùng managed LLM provider hoặc model nhẹ CPU.

### Bước 1: Build và push image

```bash
PROJECT_ID="my-ai-platform"
REGION="asia-southeast1"
IMAGE="${REGION}-docker.pkg.dev/${PROJECT_ID}/ai/ai-api:1.0.0"

gcloud auth configure-docker ${REGION}-docker.pkg.dev

docker build -t "$IMAGE" .
docker push "$IMAGE"
```

### Bước 2: Tạo secret

```bash
echo -n "$OPENAI_API_KEY" | gcloud secrets create openai-api-key \
  --data-file=- \
  --replication-policy="automatic"
```

### Bước 3: Deploy service

```bash
gcloud run deploy ai-api \
  --image "$IMAGE" \
  --region "$REGION" \
  --service-account ai-api-prod@${PROJECT_ID}.iam.gserviceaccount.com \
  --set-secrets OPENAI_API_KEY=openai-api-key:latest \
  --set-env-vars MODEL_NAME=gpt-4o-mini,APP_ENV=prod \
  --memory 2Gi \
  --cpu 2 \
  --min-instances 1 \
  --max-instances 20 \
  --allow-unauthenticated=false
```

### Bước 4: Production checklist

- `--allow-unauthenticated=false` nếu không phải public API.
- Region đúng policy dữ liệu.
- Logs không chứa raw PII.
- Service account least privilege.
- Alert cho 5xx, latency, cost spikes.
- Budget alert ở project/account level.

## 6. Data Residency and Compliance Checklist

Trước khi deploy AI workload, trả lời:

| Câu hỏi | Vì sao quan trọng |
|---|---|
| User data đến từ quốc gia nào? | GDPR, PDPA, HIPAA, local laws |
| Prompt/log có chứa PII không? | observability vendor có thể thành nơi rò rỉ |
| LLM provider xử lý request ở region nào? | data transfer và compliance |
| Storage bucket có public không? | tránh breach cơ bản |
| Backup và replicas nằm region nào? | dữ liệu có thể rời khỏi boundary |
| Ai có quyền đọc raw data? | least privilege |
| Retention policy bao lâu? | quyền xóa dữ liệu và cost |

AI logging đặc biệt nguy hiểm vì prompt có thể chứa nội dung người dùng gõ tự do. Mặc định mask hoặc không log raw payload cho domain nhạy cảm.

## 7. Cloud Cost Controls

Cloud bill cho AI thường tăng từ các nguồn không rõ ràng:

- GPU endpoint chạy 24/7,
- egress giữa regions/clouds,
- object storage API calls cho nhiều file nhỏ,
- logs/traces lưu raw payload quá lớn,
- vector DB overprovision,
- autoscaling không có max cap,
- retry storm gọi LLM API.

Thiết lập tối thiểu:

```text
Budget alert:
  50% monthly budget -> Slack
  80% -> email + ticket
  100% -> page owner

Quota:
  max instances per service
  max tokens per tenant/day
  max concurrent jobs

Labels/tags:
  project=ai-platform
  env=prod
  owner=ml-platform
  feature=rag-chat
```

Không có labels/tags, bạn sẽ không biết tính năng nào đang đốt tiền.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Cold start làm request đầu tiên timeout">
**Triệu chứng:** Service serverless scale to zero, request đầu tiên mất 20-60 giây.

**Fix:** `min-instances=1`, giảm image size, lazy load phần không critical, hoặc dùng managed endpoint luôn warm cho latency-sensitive API.
</Callout>

<Callout type="warning" title="Pitfall 2: Cloud quota GPU mặc định bằng 0">
**Triệu chứng:** Tạo GPU endpoint fail dù tài khoản có billing.

**Fix:** Request quota sớm, chuẩn bị justification, có fallback provider/region và capacity plan trước launch.
</Callout>

<Callout type="danger" title="Pitfall 3: Egress cost do app và data ở khác region">
**Triệu chứng:** Bill tăng dù compute không tăng.

**Fix:** Co-locate app, storage, vector DB và model endpoint cùng region; dùng private networking; tránh loop dữ liệu qua nhiều cloud.
</Callout>

## Cost / Region / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Scale to zero | tiết kiệm idle cost | cold start |
| Multi-region active-active | latency thấp, HA cao | replication, consistency, cost cao |
| Managed endpoints | ít vận hành | giá cao và lock-in |
| Kubernetes tự quản | linh hoạt | cần SRE và security hardening |
| Same-region architecture | giảm latency/egress | phụ thuộc capacity region |
| Cross-cloud fallback | resilience vendor | egress, routing, observability phức tạp |

Cloud architecture tốt tối ưu tổng chi phí sở hữu, không chỉ giá mỗi GPU/giờ.

## Answer key

1. Dữ liệu người dùng có thể bị luật như GDPR/PDPA giới hạn nơi lưu và xử lý.
2. Khi workload đơn giản, team nhỏ, traffic vừa phải và không cần GPU orchestration phức tạp.
3. Service chỉ có quyền đúng tài nguyên cần thiết, không quyền wildcard rộng.
4. Chi phí chuyển dữ liệu ra khỏi cloud/region/network boundary.
5. Prompt/output có thể chứa PII, secrets hoặc dữ liệu nhạy cảm do user nhập tự do.

## Điểm cần nhớ

- Cloud deployment cho AI phải xét region, IAM, network, observability và cost cùng lúc.
- Managed services giảm DevOps nhưng có trade-off về cost và lock-in.
- Serverless tốt cho đơn giản/scale-to-zero, Kubernetes tốt cho platform nhiều workload.
- Prompt/log có thể là dữ liệu nhạy cảm; data residency áp dụng cả logs và traces.
- Budget alerts, quotas và labels là guardrails tài chính bắt buộc.
