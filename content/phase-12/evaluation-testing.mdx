Bạn đã chạy training job ổn định và có checkpoint mới. Nhưng trong AI production, câu hỏi quan trọng không phải là “loss có giảm không?” mà là “model mới có thật sự tốt hơn cho người dùng, không phá use case cũ, không tăng cost quá mức và không vi phạm safety không?”.

**Model Evaluation & Testing** biến cảm tính thành release gate. Một AI Engineer chuyên nghiệp không deploy model sau khi chat thử vài câu; họ xây bộ eval nhiều tầng: offline benchmark, golden dataset, regression tests, LLM-as-a-judge, red teaming, shadow traffic và A/B testing.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Accuracy 99% nhưng production trả lời sai hàng loạt">
**Bối cảnh:** Team customer support fine-tune model v2.0. Báo cáo nội bộ cho thấy accuracy tăng từ 85% lên 99%. Pipeline CI xanh, model được deploy.

**Sự cố:** Người dùng bắt đầu phàn nàn model trả lời sai về chính sách đổi trả mới. Điều tra cho thấy file `test_set.csv` đã vô tình bị đưa vào training dataset. Model không học hiểu; nó học thuộc bài kiểm tra. Đồng thời bộ eval không có case chính sách mới, không có regression suite cho câu hỏi nhạy cảm và không kiểm tra format JSON.

1. Test set contamination làm score ảo.
2. Không có hold-out golden dataset được bảo vệ.
3. Eval chỉ đo accuracy, không đo groundedness, safety, format validity hoặc latency.
4. Không có shadow/canary trước khi bật 100% traffic.

**Hậu quả:** Model bị rollback, đội hỗ trợ phải xử lý thủ công hàng nghìn ticket và toàn bộ quy trình release model bị đóng băng để audit.
</Callout>

Bài học rút ra: eval không phải báo cáo trang trí. Eval là **hàng rào deploy**. Nếu eval bị ô nhiễm hoặc đo sai mục tiêu, production sẽ là nơi phát hiện lỗi đắt nhất.

## 2. Architecture Blueprint (Multi-Tier AI Evaluation Gate)

Một hệ thống evaluation production nên tách rõ offline và online.

```text
[ Candidate model / prompt / RAG config ]
        │
        ▼
[ Offline evaluation ]
  - schema / format tests
  - golden dataset regression
  - task metrics
  - LLM-as-a-judge rubric
  - safety and jailbreak suite
  - cost / latency simulation
        │
        ▼
[ Release decision ]
  - pass / fail gates
  - signed eval report
  - registry promotion
        │
        ▼
[ Online evaluation ]
  - shadow mode
  - canary release
  - A/B test
  - user feedback
  - monitoring and rollback
```

Mục tiêu: không có model/prompt/index nào đi vào production nếu không có bằng chứng chất lượng và đường rollback.

---

## 3. Offline Eval vs Online Eval

| Loại eval | Chạy khi nào | Đo gì | Rủi ro |
|---|---|---|---|
| Offline golden set | trước deploy | quality trên case đã biết | có thể không đại diện production |
| Regression tests | mỗi PR/release | use case cũ còn hoạt động không | bộ test thiếu case mới |
| LLM-as-a-judge | trước deploy hoặc async | semantic correctness, groundedness | judge bias, cost |
| Red teaming | định kỳ | jailbreak, unsafe output | không bao phủ hết attack mới |
| Shadow mode | production ngầm | output trên traffic thật | không có user feedback trực tiếp |
| A/B testing | rollout một phần | business metric và UX | cần đủ sample, rủi ro user impact |

Offline eval giúp chặn lỗi rõ ràng. Online eval giúp phát hiện behavior thật trong môi trường thật. Bạn cần cả hai.

## 4. Golden Dataset và Regression Suite

Golden dataset là bộ case nội bộ có chất lượng cao, được bảo vệ khỏi leakage.

Một case tốt nên có:

```json
{
  "id": "refund-policy-017",
  "input": "Tôi mua gói Pro hôm qua, có hoàn tiền được không?",
  "context": "Chính sách 2026: Gói Pro hoàn tiền 100% trong 7 ngày đầu nếu chưa dùng quá 100 credits.",
  "expected_behavior": "Nêu điều kiện 7 ngày và 100 credits; không hứa hoàn tiền nếu thiếu thông tin.",
  "risk_level": "medium",
  "tags": ["billing", "refund", "policy-2026"]
}
```

Regression suite nên bao gồm:

- câu hỏi thường gặp,
- case từng gây incident,
- case high-risk: y tế, tài chính, pháp lý,
- prompt injection examples,
- format/JSON schema edge cases,
- multilingual/user typo cases,
- long context và missing context.

<Callout type="warning" title="Golden dataset phải được version và khóa quyền sửa">
Nếu ai cũng có thể chỉnh expected answer sau khi model fail, eval sẽ mất ý nghĩa. Golden set cần owner, review process, version hash và audit log.
</Callout>

## 5. LLM-as-a-Judge: Useful but Dangerous

LLM-as-a-judge dùng một model mạnh hơn để chấm output theo rubric. Đây là cách thực dụng cho tác vụ mở như chatbot/RAG.

Rubric tốt cần:

- tiêu chí rõ ràng,
- thang điểm cụ thể,
- yêu cầu trả JSON,
- context và expected behavior,
- hướng dẫn phạt hallucination,
- calibration với human labels.

Ví dụ rubric ngắn:

```text
Score 5: Answer is correct, grounded in context, complete, safe, and follows format.
Score 3: Mostly correct but incomplete or slightly vague.
Score 1: Incorrect, hallucinated, unsafe, or ignores the user's request.
Return JSON only: {"score": number, "reason": string, "failure_type": string | null}
```

Các bias cần xử lý:

| Bias | Mô tả | Cách giảm |
|---|---|---|
| Position bias | judge ưu tiên answer A hơn B | swap order A/B |
| Verbosity bias | câu dài được đánh giá cao hơn | rubric phạt dài dòng |
| Self-preference | model cùng family thiên vị nhau | dùng judge khác provider |
| Prompt sensitivity | rubric wording đổi score | freeze prompt version |
| Non-determinism | chấm lúc cao lúc thấp | temperature 0, majority vote cho case critical |

## Implementation Playbook: LLM-as-Judge Eval Gate

### Bước 1: Viết eval dataset

```json
[
  {
    "id": "billing-001",
    "query": "Tôi muốn hủy gói Pro thì có được hoàn tiền không?",
    "context": "Hoàn tiền trong 7 ngày đầu nếu chưa dùng quá 100 credits.",
    "expected_behavior": "Trả lời có điều kiện, hỏi thêm ngày mua và credits đã dùng."
  }
]
```

### Bước 2: Tạo judge prompt có rubric

```python
JUDGE_SYSTEM = """
You are a strict AI quality evaluator.
Grade the model answer using the provided query, context, and expected behavior.
Return JSON only with keys: score, reason, failure_type.
Score 5 means fully correct and grounded.
Score 3 means partially correct.
Score 1 means wrong, hallucinated, unsafe, or not grounded.
"""
```

### Bước 3: Chạy eval và fail pipeline nếu dưới ngưỡng

```python
import json
import statistics
from openai import OpenAI

client = OpenAI()

cases = json.load(open("eval/golden_set.json", encoding="utf-8"))
scores = []
failures = []

for case in cases:
    model_answer = generate_answer(case["query"], case["context"])
    judge_input = {
        "query": case["query"],
        "context": case["context"],
        "expected_behavior": case["expected_behavior"],
        "model_answer": model_answer,
    }

    response = client.chat.completions.create(
        model="gpt-4o",
        temperature=0,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": JUDGE_SYSTEM},
            {"role": "user", "content": json.dumps(judge_input, ensure_ascii=False)},
        ],
    )

    result = json.loads(response.choices[0].message.content)
    scores.append(result["score"])
    if result["score"] < 4:
        failures.append({"case_id": case["id"], **result})

avg = statistics.mean(scores)
report = {"average_score": avg, "failures": failures}
json.dump(report, open("eval_report.json", "w", encoding="utf-8"), indent=2, ensure_ascii=False)

if avg < 4.2 or failures:
    raise SystemExit("Evaluation failed: quality gate not met")
```

### Bước 4: Lưu eval report vào artifact/registry

Eval report cần được lưu cùng model version:

```text
model: customer-support-v12
artifact: s3://models/customer-support/v12
code_commit: 7c9f2ab
dataset_hash: sha256:...
eval_report: s3://reports/customer-support/v12/eval_report.json
status: staging
```

## 6. RAG Evaluation Metrics

Nếu hệ thống là RAG, đánh giá final answer thôi chưa đủ. Cần tách retrieval và generation.

| Metric | Câu hỏi |
|---|---|
| Context precision | retrieved chunks có liên quan không? |
| Context recall | có lấy đủ tài liệu cần thiết không? |
| Faithfulness | answer có bịa ngoài context không? |
| Answer relevance | answer có trả lời đúng query không? |
| Citation accuracy | citation có trỏ đúng nguồn không? |
| Refusal correctness | khi không có context, model có biết từ chối không? |

Một lỗi RAG có thể đến từ retriever, reranker, chunking, prompt, hoặc model. Eval phải giúp bạn biết lỗi nằm ở đâu.

---

## 7. Release Gates and Rollback

Một release gate thực dụng:

```text
Gate 1: Unit tests pass
Gate 2: Prompt/schema tests pass
Gate 3: Golden eval average >= 4.2
Gate 4: No critical safety failures
Gate 5: p95 latency increase <= 15%
Gate 6: cost/request increase <= 20%
Gate 7: canary 5% traffic for 30 minutes without alert
```

Nếu model fail Gate 5 hoặc Gate 6, có thể nó “thông minh hơn” nhưng không phù hợp sản phẩm. Production AI là tối ưu đa mục tiêu, không chỉ quality.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Judge chọn câu trả lời dài hơn dù không đúng hơn">
**Triệu chứng:** Model verbose thắng mọi comparison.

**Fix:** Rubric phải phạt thông tin thừa, yêu cầu groundedness và có giới hạn độ dài nếu sản phẩm cần ngắn gọn.
</Callout>

<Callout type="warning" title="Pitfall 2: Eval flaky làm CI lúc đỏ lúc xanh">
**Triệu chứng:** Không đổi code nhưng rerun CI cho kết quả khác.

**Fix:** Đặt temperature 0, freeze judge prompt/model version, dùng threshold có margin và rerun/majority vote cho case quan trọng.
</Callout>

<Callout type="danger" title="Pitfall 3: Golden set bị học thuộc hoặc bị chỉnh để model pass">
**Triệu chứng:** Score tăng đều nhưng production không cải thiện.

**Fix:** Tách hidden hold-out set, giới hạn quyền sửa, version eval data và audit mọi thay đổi expected behavior.
</Callout>

## Cost / Quality / Risk Unit Economics

| Hoạt động | Chi phí | Tối ưu |
|---|---|---|
| Judge GPT-4o trên 10,000 cases | API cost và thời gian cao | chạy sanity set trên PR, full eval nightly bằng Batch API |
| Human eval | đắt nhưng chất lượng cao | dùng cho calibration và high-risk cases |
| Shadow traffic | chạy hai model song song | chỉ shadow subset có giá trị, sample theo risk |
| A/B testing | tác động user thật | canary nhỏ, rollback nhanh, guardrail metrics |
| Red teaming | cần chuyên môn | ưu tiên attack từng gây incident, cập nhật suite định kỳ |

Eval rẻ nhất là eval không chạy; nhưng incident production thường đắt hơn nhiều.
