Bạn đã có monitoring để biết hệ thống đang làm gì. Nhưng production không đứng yên: prompt thay đổi, code thay đổi, dataset thay đổi, model thay đổi và infrastructure thay đổi. Nếu mỗi thay đổi đều deploy thủ công, sớm muộn gì một prompt “nhỏ” cũng phá production.

**CI/CD for AI** mở rộng CI/CD truyền thống bằng eval gates, dataset/model artifact checks, prompt regression tests, security scans, canary rollout và automated rollback. Mục tiêu không phải deploy nhanh bằng mọi giá; mục tiêu là deploy có kiểm soát, đo được và quay lui được.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Một dòng prompt merge chiều thứ Sáu gây incident tài chính">
**Bối cảnh:** Chatbot hỗ trợ ngân hàng có system prompt: “Hãy hướng dẫn người dùng thao tác tài khoản, không đưa lời khuyên đầu tư.” Một prompt engineer sửa thành “Hãy đưa ra lời khuyên quyết đoán hơn” để làm chatbot tự tin hơn. PR chỉ chạy unit test Python và pass.

**Sự cố:** Cuối tuần, chatbot bắt đầu đưa khuyến nghị mua/bán cổ phiếu khi user hỏi về tài khoản đầu tư. Không có prompt regression test, không có compliance eval và deploy bật 100% traffic ngay sau merge.

1. CI chỉ kiểm tra syntax, không kiểm tra semantic behavior.
2. Prompt không được version và review như code.
3. Không có golden dataset cho câu hỏi regulated domain.
4. Không có canary hoặc feature flag để rollback nhanh.

**Hậu quả:** Tính năng bị tắt, compliance mở điều tra và team bắt buộc đưa AI eval vào release pipeline.
</Callout>

Bài học rút ra: Lỗi code làm app crash; lỗi prompt/model có thể làm app vẫn chạy nhưng trả lời nguy hiểm. CI/CD cho AI phải test **hành vi** chứ không chỉ test cú pháp.

## 2. Architecture Blueprint (CT/CI/CD Pipeline for AI)

AI pipeline có ba dòng thay đổi: code, data và model/prompt.

```text
[ Pull request ]
  - app code
  - prompts
  - eval cases
  - infra manifests
        │
        ▼
[ CI: checks ]
  - lint/unit tests
  - security scan
  - prompt regression tests
  - small golden eval
  - schema/data checks
        │
        ▼
[ Build ]
  - Docker image
  - SBOM / vulnerability scan
  - tag with git SHA
        │
        ▼
[ Release gates ]
  - eval score threshold
  - cost/latency budget
  - model registry status
  - approval for high-risk changes
        │
        ▼
[ CD: progressive delivery ]
  - deploy staging
  - smoke tests
  - canary 5%
  - monitor metrics
  - promote 100% or rollback
```

CT (Continuous Training) có thể được thêm vào khi dataset mới kích hoạt training/eval/model registry. Nhưng không phải team nào cũng cần continuous training ngay từ đầu.

## 3. What Makes AI CI/CD Different

| Truyền thống | AI/LLM |
|---|---|
| Unit test deterministic | output stochastic và semantic |
| Code là artifact chính | code + prompt + data + model + index |
| Build pass thì deploy được | eval gate mới quyết định deploy |
| Rollback image là đủ | có thể cần rollback prompt/model/index |
| Monitoring là error/latency | thêm hallucination, cost, toxicity, tool loops |

Vì vậy pipeline AI phải biết artifact nào thay đổi:

```text
prompt changed -> run prompt regression
retriever changed -> run RAG eval
model changed -> run full eval + latency test
data changed -> run data validation + train/eval
infra changed -> run security/resource checks
```

## 4. Prompt, Dataset and Model Versioning in CI/CD

Một release AI nên có version bundle:

```json
{
  "release_id": "support-ai-2026.07.02",
  "app_image": "registry/ai-api:7c9f2ab",
  "prompt_version": "support-v14",
  "model_version": "customer-support-sft-v12",
  "retriever_index": "kb-index-2026-07-01",
  "dataset_hash": "sha256:...",
  "eval_report": "s3://eval-reports/support-ai-2026.07.02.json"
}
```

Rollback cần quay về bundle trước đó, không chỉ Docker image. Nếu bạn rollback app nhưng vẫn dùng prompt/index/model mới, incident có thể tiếp tục.

## 5. Test Pyramid for AI Systems

```text
                 [ Human review / red team ]
               [ Online canary / A/B tests ]
             [ Full golden eval / nightly ]
          [ PR semantic sanity eval ]
       [ Unit tests / schema tests / lint ]
```

| Tầng | Chạy khi nào | Mục tiêu |
|---|---|---|
| Unit/schema | mọi PR | nhanh, deterministic |
| Semantic sanity | mọi PR prompt/model logic | phát hiện regression rõ ràng |
| Full eval | nightly hoặc release candidate | chất lượng rộng hơn |
| Canary | sau staging | kiểm tra traffic thật |
| Human review | high-risk domain | compliance và nuance |

Không nên chạy 10,000 eval cases trong mọi PR nếu làm developer chờ 5 giờ. Dùng tiered testing.

### Bước 1: Script eval tối giản

```python
# scripts/ci_eval.py
import json
import sys
from pathlib import Path

THRESHOLD = 4.0


def generate_answer(query: str) -> str:
    # In real CI: call staging model or local mocked pipeline.
    # Keep PR sanity set small and deterministic where possible.
    return "Bạn có thể đổi mật khẩu trong Cài đặt > Bảo mật."


def judge_answer(case: dict, answer: str) -> float:
    # In real CI: call LLM-as-a-judge or semantic metric.
    # Here we use a placeholder to show pipeline shape.
    required = case["expected_keyword"].lower()
    return 5.0 if required in answer.lower() else 1.0


def main():
    cases = json.loads(Path("eval/sanity_set.json").read_text(encoding="utf-8"))
    scores = []
    failures = []

    for case in cases:
        answer = generate_answer(case["query"])
        score = judge_answer(case, answer)
        scores.append(score)
        if score < THRESHOLD:
            failures.append({"id": case["id"], "score": score, "answer": answer})

    avg = sum(scores) / len(scores)
    report = {"average_score": avg, "failures": failures}
    Path("eval_report.json").write_text(json.dumps(report, indent=2), encoding="utf-8")

    if failures:
        print(json.dumps(report, indent=2, ensure_ascii=False))
        sys.exit(1)

    print("AI eval passed", report)


if __name__ == "__main__":
    main()
```

### Bước 2: GitHub Actions workflow

```yaml
name: AI CI

on:
  pull_request:
    branches: ["main"]

jobs:
  test-and-eval:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"

      - name: Install dependencies
        run: pip install -r requirements.txt

      - name: Unit tests
        run: pytest

      - name: AI semantic sanity eval
        env:
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
        run: python scripts/ci_eval.py

      - name: Upload eval report
        uses: actions/upload-artifact@v4
        with:
          name: eval-report
          path: eval_report.json
```

### Bước 3: Build image with immutable tag

```yaml
  build:
    needs: test-and-eval
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Build image
        run: docker build -t registry.example.com/ai-api:${{ github.sha }} .
```

## Project Guide: Production AI Platform on Kubernetes (Phase 12 Hard)

Mục tiêu project: xây platform AI production nhỏ nhưng đầy đủ vòng đời: gateway, router, LLM proxy, storage, monitoring, CI/CD và Kubernetes deployment.

### Services

```text
api-gateway
  - auth, rate limit, request ids

llm-router
  - choose provider/model
  - fallback policy
  - cost/latency budget

llm-proxy
  - OpenAI-compatible client
  - retries, timeouts, circuit breaker

observability
  - Langfuse/Helicone traces
  - Prometheus metrics
  - Grafana dashboard

data layer
  - Postgres for app metadata
  - Redis for cache/queue
```

### Kubernetes requirements

- Docker image cho từng service.
- Helm chart hoặc Kustomize overlays cho staging/prod.
- ConfigMap/Secret tách khỏi image.
- HPA cho gateway/router theo RPS/CPU.
- Optional GPU node pool cho vLLM.
- Prometheus + Grafana dashboard.
- Readiness/liveness probes.
- Network policy tối thiểu.

### CI/CD requirements

- PR: lint, unit tests, prompt sanity eval.
- Main: build images, scan vulnerabilities, push registry.
- Staging deploy tự động.
- Smoke tests staging.
- Manual approval hoặc eval gate để production.
- Canary rollout 5% rồi promote.
- Rollback bằng image + prompt/model bundle version.

### Acceptance criteria

- Một request có request id xuyên qua gateway/router/proxy.
- Dashboard thấy latency, cost, tokens và error rate.
- Pipeline chặn PR nếu eval sanity fail.
- Kubernetes có thể rolling update không downtime.
- Có runbook rollback trong README.

Stretch goal: dùng ArgoCD hoặc Argo Rollouts để canary tự động dựa trên Prometheus metrics.

## 6. Progressive Delivery: Canary, Blue/Green, Feature Flags

| Pattern | Cách hoạt động | Khi dùng |
|---|---|---|
| Canary | 1-5% traffic dùng version mới | kiểm tra production thật với rủi ro thấp |
| Blue/Green | chạy song song hai environment, switch traffic | rollback nhanh, release lớn |
| Feature flag | bật/tắt logic theo user/tenant | prompt/model changes, A/B testing |
| Shadow | version mới nhận traffic nhưng không trả cho user | đánh giá model trước exposure |

AI release nên ưu tiên feature flags cho prompt/model/index, vì rollback image có thể chậm hơn tắt flag.

## 7. Security in CI/CD

CI/CD giữ secrets và quyền deploy, nên là mục tiêu tấn công.

Checklist:

- secrets chỉ trong GitHub Secrets/secret manager,
- không in secrets ra logs,
- OIDC federation thay vì long-lived cloud keys nếu có thể,
- scan dependencies và Docker image,
- protect main branch,
- require eval report artifact,
- sign images/SBOM cho môi trường enterprise,
- giới hạn ai được approve production deploy.

Prompt/dataset cũng cần review nếu thuộc domain regulated. Đừng cho mọi PR tự động deploy prompt tài chính/y tế/pháp lý.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Pipeline chạy 5 giờ làm developer né CI">
**Triệu chứng:** PR nhỏ phải chờ full eval quá lâu.

**Fix:** Tiered testing: sanity set trên PR, full eval nightly/release, cache artifacts và chạy parallel/batch.
</Callout>

<Callout type="warning" title="Pitfall 2: Flaky LLM tests đỏ xanh ngẫu nhiên">
**Triệu chứng:** Rerun cùng commit cho kết quả khác.

**Fix:** Freeze model/judge version, temperature 0, semantic thresholds có margin, retry/majority vote cho case critical.
</Callout>

<Callout type="danger" title="Pitfall 3: Rollback image nhưng prompt mới vẫn active">
**Triệu chứng:** Sau rollback Docker image, behavior nguy hiểm vẫn còn.

**Nguyên nhân:** Prompt được lưu trong database/config service và không rollback theo release bundle.

**Fix:** Version bundle gồm image + prompt + model + index; rollback bundle nguyên khối.
</Callout>

## Cost / Velocity / Safety Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Full eval mọi PR | an toàn cao | chậm, đắt, developer friction |
| Sanity eval PR | nhanh | miss lỗi hiếm |
| Nightly full eval | bao phủ rộng | phát hiện muộn hơn |
| Canary rollout | giảm blast radius | vận hành phức tạp |
| Manual approval | hợp regulated domain | chậm release |
| Auto rollback | giảm downtime | cần metrics đáng tin |

CI/CD tốt tối ưu cả tốc độ phát triển và độ an toàn. Pipeline quá chậm sẽ bị bỏ qua; pipeline quá lỏng sẽ gây incident.

---

## Answer key

1. Nó phải kiểm tra hành vi semantic của prompt/model/data, không chỉ code pass unit tests.
2. Behavior còn phụ thuộc prompt/model/index/config bên ngoài image.
3. Cân bằng tốc độ và độ bao phủ: test nhỏ trên PR, test lớn định kỳ/release.
4. Chỉ expose version mới cho một phần nhỏ traffic và rollback nếu metrics xấu.
5. CI/CD có quyền deploy và truy cập cloud/provider; lộ secret có thể dẫn đến takeover hoặc bill shock.
