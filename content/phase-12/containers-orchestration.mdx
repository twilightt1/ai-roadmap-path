Ở bài trước, bạn đã chạy được inference engine. Nhưng một service chạy được trên một máy chưa phải production platform. Khi traffic tăng, instance chết, image cần rollback hoặc GPU node cần scale, bạn không thể SSH vào từng máy rồi `docker run` thủ công.

**Containerization & Orchestration** biến AI service thành đơn vị triển khai chuẩn hóa. Docker giúp môi trường chạy giống nhau; Kubernetes giúp hệ thống tự phục hồi, rollout, scale và quản lý tài nguyên GPU ở quy mô cụm.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Chạy bằng tmux trên GPU server và downtime 3 ngày">
**Bối cảnh:** Team xây RAG nội bộ. Prototype chạy tốt trên máy Mac của engineer. Khi deploy, họ SSH vào GPU server Ubuntu, `pip install -r requirements.txt`, tải model từ Hugging Face và chạy FastAPI trong `tmux`.

**Sự cố:** Driver NVIDIA tự cập nhật, CUDA/PyTorch lệch version. Process memory leak và sập lúc nửa đêm. Engineer mới không dựng lại được môi trường vì requirements không pin version, model cache nằm trong home directory của người cũ và không có health check.

1. Không có Docker image bất biến.
2. Không có process supervisor hoặc orchestration.
3. Không có resource requests/limits cho GPU.
4. Không có rolling update, rollback hoặc readiness probe.
5. Model weights tải runtime làm restart mất 20 phút.

**Hậu quả:** Dịch vụ dừng 3 ngày, stakeholder mất niềm tin và team phải rebuild deployment bằng Docker + Kubernetes.
</Callout>

Bài học rút ra: nếu AI service phụ thuộc vào “máy này đang cài đúng thư viện”, nó chưa production-ready. Môi trường phải được đóng gói, version, health-check và orchestration.

## 2. Architecture Blueprint (Docker Image to Kubernetes GPU Deployment)

Một vòng đời triển khai cloud-native:

```text
[ Source code + Dockerfile ]
        │
        ▼
[ Build immutable image ]
  - pinned dependencies
  - non-root user
  - health endpoint
  - model loading strategy
        │
        ▼
[ Container registry ]
  - image tags
  - vulnerability scan
  - signed image
        │
        ▼
[ Kubernetes cluster ]
  - Deployment / StatefulSet
  - Service / Ingress
  - GPU node pool
  - HPA / KEDA
  - ConfigMap / Secret
        │
        ▼
[ Runtime operations ]
  - rolling update
  - readiness/liveness probes
  - logs/metrics/traces
  - rollback
```

Mục tiêu: mỗi release là một image có tag rõ ràng, deploy qua manifest/Helm, có health check và rollback được.

## 3. Docker Fundamentals for AI Services

Docker image cho AI khác web app thường vì:

- phụ thuộc CUDA/PyTorch lớn,
- model weights có thể hàng GB,
- cần NVIDIA Container Toolkit,
- startup có thể lâu,
- image size dễ phình trên 10GB,
- secrets/API keys không được bake vào image.

Một Dockerfile production cần:

| Yếu tố | Lý do |
|---|---|
| Pin version | tránh drift giữa build và runtime |
| Multi-stage build | giảm image size |
| Non-root user | giảm rủi ro bảo mật |
| `.dockerignore` | tránh copy dataset/cache vào image |
| Health endpoint | Kubernetes biết pod đã sẵn sàng chưa |
| Env config | cùng image chạy được staging/prod |
| Runtime CUDA image | tránh kéo dev tools thừa |

<Callout type="warning" title="Đừng bake secrets vào Docker image">
Nếu `OPENAI_API_KEY` hoặc HF token nằm trong image layer, xóa khỏi code chưa đủ. Secret vẫn có thể tồn tại trong image history. Dùng Kubernetes Secret hoặc secret manager.
</Callout>

## 4. Kubernetes Concepts for AI Engineers

| K8s object | Vai trò trong AI service |
|---|---|
| Pod | một hoặc nhiều container chạy cùng nhau |
| Deployment | quản lý replicas và rolling update |
| Service | endpoint nội bộ ổn định cho pods |
| Ingress/Gateway | public routing, TLS, host/path routing |
| ConfigMap | config không nhạy cảm: model name, thresholds |
| Secret | API keys, DB passwords |

Kubernetes không tự hiểu “VRAM còn bao nhiêu”. Bạn phải khai báo GPU resource và cài NVIDIA device plugin/DCGM exporter nếu muốn metrics GPU.

## 5. GPU Scheduling and Runtime

Kubernetes quản lý GPU như tài nguyên đặc biệt.

```yaml
resources:
  limits:
    nvidia.com/gpu: "1"
```

Điều này thường cấp nguyên một GPU vật lý cho pod. Nếu muốn chia GPU:

- dùng NVIDIA MIG trên A100/H100,
- dùng time-slicing qua NVIDIA device plugin,
- hoặc chạy nhiều model nhỏ trong một inference server có scheduler riêng.

| Chiến lược | Khi dùng | Rủi ro |
|---|---|---|
| 1 pod / 1 GPU | model lớn, isolation tốt | có thể lãng phí VRAM |
| MIG | nhiều workload nhỏ, isolation phần cứng | setup phức tạp, chỉ GPU hỗ trợ |
| Time-slicing | dev/test, model nhẹ | noisy neighbor |
| Shared vLLM server | nhiều tenant/model routing | cần quota và isolation logic |

## Implementation Playbook: Production Dockerfile + K8s Deployment

### Bước 1: Dockerfile tối thiểu cho FastAPI AI API

```dockerfile
FROM python:3.11-slim AS builder

WORKDIR /app
RUN apt-get update && apt-get install -y --no-install-recommends build-essential \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

FROM python:3.11-slim AS runner

WORKDIR /app
RUN useradd -m -u 10001 appuser

COPY --from=builder /root/.local /home/appuser/.local
COPY ./src ./src

ENV PATH=/home/appuser/.local/bin:$PATH
ENV PYTHONUNBUFFERED=1

USER appuser
EXPOSE 8000

CMD ["uvicorn", "src.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Bước 2: Health endpoints trong FastAPI

```python
from fastapi import FastAPI

app = FastAPI()
model_loaded = False

@app.on_event("startup")
def load_model():
    global model_loaded
    # load tokenizer/model/client here
    model_loaded = True

@app.get("/healthz")
def healthz():
    return {"status": "ok"}

@app.get("/readyz")
def readyz():
    return {"ready": model_loaded}
```

### Bước 3: Kubernetes Deployment có probes và GPU limit

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ai-api
spec:
  replicas: 2
  selector:
    matchLabels:
      app: ai-api
  template:
    metadata:
      labels:
        app: ai-api
    spec:
      containers:
        - name: api
          image: registry.example.com/ai-api:1.0.0
          ports:
            - containerPort: 8000
          envFrom:
            - secretRef:
                name: ai-api-secrets
            - configMapRef:
                name: ai-api-config
          resources:
            requests:
              cpu: "2"
              memory: "8Gi"
            limits:
              cpu: "4"
              memory: "16Gi"
              nvidia.com/gpu: "1"
          readinessProbe:
            httpGet:
              path: /readyz
              port: 8000
            initialDelaySeconds: 30
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8000
            initialDelaySeconds: 60
            periodSeconds: 20
```

### Bước 4: Service nội bộ

```yaml
apiVersion: v1
kind: Service
metadata:
  name: ai-api
spec:
  selector:
    app: ai-api
  ports:
    - port: 80
      targetPort: 8000
```

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Pod scale lên nhưng cùng tranh một GPU">
**Triệu chứng:** HPA tăng replicas, pod mới schedule vào node đã có model chiếm gần hết VRAM, cả hai OOM.

**Fix:** Khai báo `nvidia.com/gpu: 1`, dùng node affinity/taints, hoặc MIG/time-slicing có kiểm soát.
</Callout>

<Callout type="warning" title="Pitfall 2: Startup 15 phút vì tải model mỗi lần pod restart">
**Triệu chứng:** Rolling update cực chậm, readiness mãi không pass.

**Fix:** Dùng PVC/model cache, preload weights vào node, hoặc bake model nhỏ vào image. Với model lớn, tránh tải từ Internet public mỗi restart.
</Callout>

<Callout type="danger" title="Pitfall 3: Image 12GB làm rollout chậm và tốn egress">
**Triệu chứng:** Scale node mới mất nhiều phút chỉ để pull image.

**Fix:** Multi-stage build, runtime base image, `.dockerignore`, không copy dataset/cache, dùng registry gần cluster và image layer caching.
</Callout>

## Cost / Reliability / Scaling Unit Economics

| Quyết định | Tác động | Tối ưu |
|---|---|---|
| Image quá lớn | pull chậm, egress cao | slim/runtime base, layer cache |
| Min replicas cao | uptime tốt nhưng tốn GPU | autoscale theo queue, keep warm nhỏ |
| HPA theo CPU | AI service có thể GPU overload nhưng CPU thấp | custom GPU/queue metrics |
| PVC model cache | restart nhanh hơn | quản lý version và cleanup |
| 1 GPU per pod | isolation tốt | MIG/time-slicing cho model nhỏ |
| Rolling update mặc định | an toàn web app | với model load lâu, cần readiness và maxUnavailable phù hợp |

Kubernetes giúp scale, nhưng nếu metric sai hoặc image quá nặng, nó chỉ scale sự cố nhanh hơn.

## Answer key

1. Đóng gói code, dependencies và runtime thành image bất biến, giảm lỗi môi trường.
2. Để container có thể request và sử dụng GPU vật lý qua resource `nvidia.com/gpu`.
3. Readiness cho biết pod sẵn sàng nhận traffic; liveness cho biết pod còn sống hay cần restart.
4. Làm startup chậm, phụ thuộc Internet, tốn băng thông và gây downtime khi restart/scale.
5. Khi model nhỏ hoặc workload nhẹ không cần nguyên GPU, và bạn chấp nhận quản lý isolation phức tạp hơn.
