Bạn đã có model hoặc prompt vượt qua eval gates. Nhưng một model tốt trong notebook vẫn có thể thất bại trong production nếu serving layer không xử lý được concurrency, streaming, KV cache, rate limit, timeout và cost/token.

**Model Deployment & Serving** là nơi AI Engineer phải suy nghĩ như backend engineer và systems engineer. Mục tiêu không chỉ là “model trả lời đúng”, mà là trả lời ổn định cho nhiều người dùng, với p95 latency chấp nhận được, throughput cao, quan sát được và rollback được.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Hugging Face pipeline làm sập launch ngày đầu">
**Bối cảnh:** Startup legal RAG launch beta cho 10,000 user. Backend deploy LLaMA-3-8B bằng `transformers.pipeline("text-generation")` trong FastAPI trên một GPU A100 80GB.

**Sự cố:** Khi traffic đạt 50 RPS, VRAM tăng lên 80GB và process bị kill. Model weights chỉ khoảng 16GB, nhưng KV cache cho nhiều request đồng thời bị cấp phát lãng phí. Request dài từ RAG làm prefill chậm, API timeout và user retry càng làm tải tăng.

1. Dùng thư viện research inference thay vì inference engine production.
2. Không có continuous batching.
3. Không giới hạn max input/output tokens.
4. Không có queue/backpressure/rate limit.
5. Không đo TTFT, TPOT và cost/token.

**Hậu quả:** Feature bị tắt trong ngày launch, team phải rebuild serving layer bằng vLLM + gateway + streaming.
</Callout>

Bài học rút ra: model serving không phải là `model.generate()` bọc bằng HTTP. Production serving cần engine tối ưu bộ nhớ, scheduler, API contract và vận hành như một service thật.

## 2. Architecture Blueprint (LLM Serving Gateway + Inference Engine)

Một serving stack production thường có nhiều lớp:

```text
[ Client / product app ]
        │
        ▼
[ API Gateway ]
  - auth
  - tenant quota
  - rate limit
  - request size limit
        │
        ▼
[ LLM Router / Policy Layer ]
  - model selection
  - fallback
  - prompt assembly
  - safety checks
        │
        ▼
[ Inference Engine: vLLM / TGI / TensorRT-LLM ]
  - OpenAI-compatible API
  - continuous batching
  - KV cache manager
  - tensor parallelism
  - streaming
        │
        ▼
[ Observability + Storage ]
  - traces
  - token usage
  - latency
  - errors
  - model version
```

Mục tiêu: tách policy/business logic khỏi inference engine. Engine tập trung tối ưu GPU; gateway tập trung bảo mật, routing và quota.

## 3. Why `transformers.generate()` Is Not Enough

Hugging Face `transformers` rất tốt cho nghiên cứu, evaluation và batch jobs. Nhưng production serving có yêu cầu khác:

| Yêu cầu production | Vì sao khó với generate thủ công |
|---|---|
| Concurrent requests | mỗi request có độ dài khác nhau, dễ lãng phí VRAM |
| Streaming | cần gửi token liên tục và xử lý disconnect |
| Dynamic batching | phải thêm request mới khi GPU đang chạy |
| KV cache memory | cần cấp phát động và tránh fragmentation |
| Backpressure | khi queue đầy phải reject thay vì OOM |
| Metrics | TTFT, TPOT, tokens/sec, queue time |
| API compatibility | client muốn OpenAI-like API |

Inference engine như vLLM/TGI giải quyết các vấn đề này ở tầng scheduler và CUDA kernels.

---

## 4. KV Cache, PagedAttention và Continuous Batching

LLM generation có hai giai đoạn:

1. **Prefill:** đọc toàn bộ prompt/input context.
2. **Decode:** sinh từng token output.

KV cache lưu attention keys/values để không tính lại toàn bộ context ở mỗi token. KV cache có thể chiếm rất nhiều VRAM khi nhiều user đồng thời.

```text
Request A: prompt 3,000 tokens + output 500 tokens
Request B: prompt 100 tokens + output 50 tokens
Request C: prompt 8,000 tokens + output 1,000 tokens
```

Nếu cấp phát tĩnh theo max length, VRAM bị lãng phí nặng. vLLM dùng PagedAttention để chia KV cache thành blocks nhỏ như memory pages.

```text
KV blocks:
[A1][A2][A3] ...
[B1]
[C1][C2][C3][C4] ...
```

Continuous batching cho phép engine thêm request mới vào batch đang chạy khi request cũ kết thúc, thay vì đợi cả batch xong.

| Kỹ thuật | Lợi ích |
|---|---|
| PagedAttention | giảm fragmentation và lãng phí KV cache |
| Continuous batching | tăng GPU utilization và throughput |
| Prefix caching | tái dùng system prompt/RAG prefix lặp lại |
| Speculative decoding | tăng tốc decode bằng draft model |
| Quantization | giảm memory bandwidth và cost |

## 5. Serving Metrics You Must Know

| Metric | Ý nghĩa | Vì sao quan trọng |
|---|---|---|
| TTFT | Time To First Token | cảm giác app có phản hồi nhanh không |
| TPOT | Time Per Output Token | tốc độ streaming có mượt không |
| E2E latency | tổng thời gian request | SLA tổng thể |
| Queue time | request chờ trước khi vào GPU | dấu hiệu overload |
| Tokens/sec | throughput engine | cost efficiency |
| GPU utilization | GPU có được dùng hiệu quả không | capacity planning |
| KV cache usage | VRAM cache đang gần đầy không | tránh OOM/reject |
| Error rate | 4xx/5xx/timeouts | reliability |

Không đo TTFT/TPOT thì bạn không biết user đang thấy chậm ở prefill hay decode.

## Implementation Playbook: vLLM OpenAI-Compatible Serving

### Bước 1: Chạy vLLM bằng Docker

```bash
docker run --runtime nvidia --gpus all \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  -p 8000:8000 \
  --ipc=host \
  vllm/vllm-openai:latest \
  --model meta-llama/Meta-Llama-3-8B-Instruct \
  --dtype auto \
  --api-key "$VLLM_API_KEY" \
  --gpu-memory-utilization 0.85 \
  --max-model-len 8192 \
  --max-num-seqs 128 \
  --enable-prefix-caching
```

### Bước 2: Gọi bằng OpenAI SDK

```python
from openai import OpenAI

client = OpenAI(
    api_key="local-secret",
    base_url="http://localhost:8000/v1",
)

stream = client.chat.completions.create(
    model="meta-llama/Meta-Llama-3-8B-Instruct",
    messages=[
        {"role": "system", "content": "You are a concise AI assistant."},
        {"role": "user", "content": "Giải thích KV cache trong 2 câu."},
    ],
    temperature=0.2,
    max_tokens=256,
    stream=True,
)

for chunk in stream:
    delta = chunk.choices[0].delta.content
    if delta:
        print(delta, end="")
```

### Bước 3: Thêm gateway guardrails tối thiểu

```python
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

MAX_INPUT_CHARS = 20_000
MAX_OUTPUT_TOKENS = 1024

app = FastAPI()

class ChatRequest(BaseModel):
    user_id: str
    message: str
    max_tokens: int = 512

@app.post("/chat")
def chat(req: ChatRequest):
    if len(req.message) > MAX_INPUT_CHARS:
        raise HTTPException(status_code=413, detail="Input too large")
    if req.max_tokens > MAX_OUTPUT_TOKENS:
        raise HTTPException(status_code=400, detail="max_tokens too high")

    # call vLLM/OpenAI-compatible endpoint here
    return {"status": "accepted"}
```

Engine mạnh vẫn cần gateway chặn input vô hạn, quota abuse và request không hợp lệ.

## 6. Routing, Fallback và Model Selection

Một product thường không dùng một model duy nhất.

```text
simple FAQ      -> small/cheap model
complex legal   -> stronger model
unsafe request  -> refusal / moderation
provider outage -> fallback provider
```

Router có thể dựa trên:

- tenant tier,
- task type,
- prompt length,
- latency SLA,
- cost budget,
- model availability,
- safety risk.

Fallback cần cẩn thận: nếu model chính bị lỗi vì prompt quá dài, fallback sang model context ngắn hơn sẽ fail tương tự. Fallback policy phải hiểu nguyên nhân lỗi.

## 7. Deployment Patterns

| Pattern | Khi dùng | Trade-off |
|---|---|---|
| Single GPU server | PoC/internal tool | dễ nhưng khó scale |
| Multi-GPU tensor parallel | model không fit một GPU | cần NVLink, phức tạp |
| Multiple replicas | tăng concurrency | cần load balancing và shared weights |
| Serverless API provider | không muốn vận hành GPU | vendor cost/lock-in |
| Hybrid provider + self-host | cân bằng cost và reliability | routing/observability phức tạp |

Serving architecture nên bắt đầu đơn giản nhưng để sẵn chỗ cho routing, metrics và rollback.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: OOM ngay khi vLLM khởi động">
**Triệu chứng:** Engine crash trước khi nhận request.

**Nguyên nhân:** `--gpu-memory-utilization` quá cao hoặc model/context length chiếm gần hết VRAM.

**Fix:** Giảm utilization, giảm `--max-model-len`, tắt process khác trên GPU hoặc dùng quantization/tensor parallel đúng hạ tầng.
</Callout>

<Callout type="warning" title="Pitfall 2: Request RAG dài làm TTFT rất cao">
**Triệu chứng:** Streaming bắt đầu sau 8-15 giây dù decode sau đó nhanh.

**Nguyên nhân:** Prefill prompt quá dài: nhiều chunks, system prompt lặp lại, citation thừa.

**Fix:** Rerank tốt hơn, giới hạn context, prefix caching, prompt compression, hoặc cache answer cho query phổ biến.
</Callout>

<Callout type="danger" title="Pitfall 3: Không có backpressure khiến retry storm làm sập GPU">
**Triệu chứng:** Queue dài, timeout tăng, client retry liên tục và OOM.

**Fix:** Đặt queue limit, trả 429/503 có `Retry-After`, circuit breaker, rate limit theo tenant và autoscaling dựa trên queue time.
</Callout>

## Cost / Latency / Throughput Unit Economics

| Quyết định | Tác động | Tối ưu |
|---|---|---|
| Context quá dài | TTFT và KV cache tăng | retrieval tốt hơn, prompt compression |
| Output quá dài | TPOT tổng và cost tăng | max tokens, concise prompt, stop sequences |
| GPU nhỏ nhiều chiếc | có thể rẻ nhưng communication chậm | ưu tiên GPU đủ VRAM hoặc NVLink |
| Quantization INT8/INT4 | giảm VRAM/cost | kiểm tra quality regression |
| Overprovision replicas | latency ổn nhưng đắt | autoscale theo queue/usage |
| API provider | không quản hạ tầng | theo dõi vendor cost/token và rate limits |

Serving tốt là bài toán cân bằng user experience và gross margin. Token rẻ nhưng latency tệ vẫn làm sản phẩm thất bại.

## Answer key

1. Nó thiếu scheduler/memory manager production như continuous batching và KV cache paging.
2. Lưu keys/values attention để không tính lại toàn bộ context khi sinh token tiếp theo.
3. Chia KV cache thành blocks/pages nhỏ, cấp phát động theo nhu cầu thay vì đặt chỗ tĩnh lớn.
4. Prefill: đọc prompt/context đầu vào trước khi sinh token đầu tiên.
5. Cơ chế giới hạn/điều tiết request khi hệ thống quá tải, ví dụ queue limit, 429, rate limit.
