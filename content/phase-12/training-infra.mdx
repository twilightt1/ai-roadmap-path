Bạn đã có dataset được version và experiment tracking để biết run nào tốt. Nhưng khi mô hình lớn hơn laptop và dataset lớn hơn ổ cứng local, câu hỏi tiếp theo là: làm sao chạy training ổn định trên nhiều GPU mà không đốt tiền vào idle time, OOM và checkpoint mất tích?

**Training Infrastructure** là lớp hạ tầng giúp bạn biến script training thành một job có thể chịu lỗi, scale được và đo được. Với AI Engineer, đây không chỉ là DDP/DeepSpeed trên slide; đây là bài toán chọn GPU, network, precision, checkpoint, storage và scheduler sao cho training thật sự hoàn thành.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Thuê 8 GPU A100 nhưng chỉ dùng được 20% công suất">
**Bối cảnh:** Team muốn full fine-tune model 70B. Họ thuê một instance 8xA100 đắt tiền và chạy script PyTorch cũ bằng `DataParallel`.

**Sự cố:** GPU 0 luôn 100% utilization, 7 GPU còn lại chỉ 15-20%. Step time rất chậm. Sau vài giờ, job OOM ở GPU 0 vì optimizer states và gradient aggregation dồn về một nơi.

1. Dùng `DataParallel` cũ thay vì DistributedDataParallel/FSDP/DeepSpeed.
2. Không tính memory footprint của weights, gradients, optimizer states và KV/activations.
3. Không kiểm tra NVLink/InfiniBand giữa GPU.
4. Checkpoint lưu local trên máy spot, không đẩy lên object storage.

**Hậu quả:** Hàng nghìn USD GPU bị lãng phí, training không hoàn thành và team phải thiết kế lại stack bằng Accelerate + DeepSpeed ZeRO-3.
</Callout>

Bài học rút ra: nhiều GPU không tự động làm training nhanh hơn. Hạ tầng training tốt phải tối ưu **memory, communication, storage và fault tolerance** cùng lúc.

## 2. Architecture Blueprint (Distributed Training Job Stack)

Một training job production không chỉ là `python train.py`:

```text
[ Dataset / object storage ]
  - sharded parquet/jsonl
  - cached locally
  - deterministic split
        │
        ▼
[ Job scheduler ]
  - Slurm / Kubernetes / SageMaker / Vertex AI
  - resource request: GPU, CPU, RAM
  - retries and preemption policy
        │
        ▼
[ Distributed launcher ]
  - torchrun / accelerate / deepspeed
  - world size, rank, master address
        │
        ▼
[ Training workers ]
  - DDP / FSDP / ZeRO
  - mixed precision
  - gradient checkpointing
  - checkpoint writer
        │
        ├────────► [ Experiment tracking ]
        │           metrics, artifacts, GPU telemetry
        │
        └────────► [ Checkpoint storage ]
                    latest, best, resumable
```

Mục tiêu: job có thể chạy trên nhiều GPU, log đúng metrics, resume sau khi bị kill và tận dụng GPU đủ cao để chi phí hợp lý.

## 3. Memory Anatomy of LLM Training

Khi train LLM, VRAM không chỉ chứa model weights.

| Thành phần | Vì sao tốn memory |
|---|---|
| Model weights | tham số model, thường FP16/BF16 |
| Gradients | gradient tương ứng với weights |
| Optimizer states | Adam thường lưu momentum/variance, rất lớn |
| Activations | tensor trung gian cho backward pass |
| Temporary buffers | communication, kernels, fragmentation |
| Dataloader batch | input ids, attention masks, labels |

Ước lượng đơn giản cho full fine-tuning với Adam có thể cần nhiều lần kích thước weights. Một model 7B FP16 khoảng 14GB weights, nhưng training có thể cần 60GB+ VRAM nếu không dùng sharding/checkpointing.

<Callout type="warning" title="Inference fit không có nghĩa training fit">
Nếu model chạy inference vừa trên 24GB VRAM, điều đó không có nghĩa bạn train được nó trên 24GB. Training cần gradients, optimizer states và activations nên memory cao hơn nhiều.
</Callout>

## 4. DDP, FSDP và DeepSpeed ZeRO

### Data Parallelism / DDP

DDP copy toàn bộ model lên mỗi GPU, mỗi GPU xử lý một mini-batch khác nhau rồi all-reduce gradients.

```text
GPU0: full model + batch A
GPU1: full model + batch B
GPU2: full model + batch C
GPU3: full model + batch D
        │
        ▼
Ring all-reduce gradients
```

Phù hợp khi model fit vào một GPU và bạn muốn tăng throughput.

### FSDP / ZeRO

FSDP và DeepSpeed ZeRO shard weights/gradients/optimizer states để giảm memory per GPU.

```text
GPU0: shard weights + shard optimizer + shard gradients
GPU1: shard weights + shard optimizer + shard gradients
GPU2: shard weights + shard optimizer + shard gradients
GPU3: shard weights + shard optimizer + shard gradients
```

| Phương pháp | Khi dùng | Trade-off |
|---|---|---|
| DDP | model fit trên mỗi GPU | đơn giản, nhanh, nhưng memory nhân bản |
| ZeRO-1 | shard optimizer states | giảm memory vừa phải |
| ZeRO-2 | shard optimizer + gradients | giảm nhiều hơn |
| ZeRO-3 / FSDP | shard cả weights | fit model lớn nhưng communication cao |
| CPU/NVMe offload | VRAM thiếu nghiêm trọng | train chậm hơn nhiều |

## 5. Precision, Checkpointing và Sequence Packing

Ba đòn bẩy quan trọng nhất để tránh OOM:

### Mixed precision

- FP32: ổn định nhưng rất tốn memory.
- FP16: tiết kiệm memory nhưng dễ NaN nếu scale không tốt.
- BF16: thường ổn định hơn FP16 trên A100/H100 và nhiều GPU mới.

### Gradient checkpointing

Thay vì lưu mọi activation cho backward, hệ thống tính lại một phần activation khi backward.

```text
less VRAM  <---- trade-off ----> more compute time
```

### Sequence packing

LLM training thường bị lãng phí vì nhiều câu ngắn bị padding đến max length. Packing ghép nhiều sample ngắn vào cùng một sequence.

```text
without packing: [short][PAD][PAD][PAD]
with packing:    [short][short][short]
```

Packing tốt giúp tăng tokens/sec và giảm chi phí mỗi token train.

## Implementation Playbook: Accelerate + DeepSpeed ZeRO-3

### Bước 1: Cấu hình Accelerate

```bash
pip install accelerate deepspeed transformers datasets
accelerate config
```

Một config mẫu:

```yaml
compute_environment: LOCAL_MACHINE
distributed_type: DEEPSPEED
num_processes: 8
mixed_precision: bf16
deepseed_config:
  zero_stage: 3
  gradient_accumulation_steps: 8
  offload_optimizer_device: none
  offload_param_device: none
```

### Bước 2: TrainingArguments an toàn hơn cho LLM

```python
from transformers import TrainingArguments

training_args = TrainingArguments(
    output_dir="s3_mount/checkpoints/customer-support-qlora",
    per_device_train_batch_size=1,
    gradient_accumulation_steps=16,
    learning_rate=2e-5,
    num_train_epochs=3,
    bf16=True,
    gradient_checkpointing=True,
    logging_steps=10,
    save_steps=500,
    save_total_limit=3,
    evaluation_strategy="steps",
    eval_steps=500,
    report_to="wandb",
    dataloader_num_workers=8,
)
```

### Bước 3: Launch distributed job

```bash
accelerate launch \
  --num_processes 8 \
  train.py \
  --config configs/qlora_8gpu.yaml \
  --resume_from_checkpoint auto
```

### Bước 4: Checkpoint vào durable storage

```python
from pathlib import Path

checkpoint_dir = Path("/mnt/checkpoints/run-2026-07-02")
checkpoint_dir.mkdir(parents=True, exist_ok=True)

# Trainer sẽ lưu checkpoint vào path này.
# Một sidecar hoặc job cuối có thể sync sang S3/GCS.
```

Nếu dùng spot/preemptible instance, checkpoint không được chỉ nằm trên local disk.

## 6. Scheduler Choices: Slurm, Kubernetes, Managed ML

| Scheduler | Điểm mạnh | Khi nên dùng |
|---|---|---|
| Slurm | HPC truyền thống, multi-node GPU mạnh | research lab, cluster riêng |
| Kubernetes | cloud-native, service + job cùng hệ sinh thái | platform team đã dùng K8s |
| SageMaker/Vertex/Azure ML | managed jobs, logs, IAM, auto cleanup | team muốn giảm DevOps |
| Ray | distributed Python, training + serving + data | workloads cần orchestration linh hoạt |

Không có lựa chọn tuyệt đối. Nếu team thiếu SRE, managed ML jobs có thể rẻ hơn tự vận hành K8s dù giá list cao hơn.

## 7. Network and Storage Bottlenecks

Training multi-GPU rất nhạy với network.

| Bottleneck | Triệu chứng | Fix |
|---|---|---|
| Không có NVLink | scale 8 GPU không nhanh hơn 2 GPU | chọn instance có NVLink/InfiniBand |
| Object storage đọc từng file nhỏ | GPU utilization răng cưa | pack data thành parquet/tar, local cache |
| Checkpoint ghi quá thường xuyên | step time spike mỗi save | tăng save interval, async upload |
| CPU preprocessing chậm | GPU idle | pre-tokenize, tăng workers, cache tokenized dataset |
| Network multi-node yếu | all-reduce chậm | placement group, RDMA/InfiniBand |

Một cụm GPU đắt nhất vẫn có thể thua một cụm nhỏ hơn nếu data loading và network kém.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: OOM chỉ xảy ra ở một vài GPU">
**Triệu chứng:** GPU 2 OOM còn GPU khác ổn.

**Nguyên nhân:** Batch chứa sequence dài hơn hoặc packing không đều.

**Fix:** bucket theo length, sequence packing, giới hạn max sequence, log max tokens per batch.
</Callout>

<Callout type="warning" title="Pitfall 2: GPU utilization thấp dù VRAM đầy">
**Triệu chứng:** VRAM 90% nhưng utilization 30%.

**Nguyên nhân:** GPU đang chờ CPU/dataloader/storage hoặc communication.

**Fix:** pre-tokenize data, tăng `dataloader_num_workers`, local NVMe cache, kiểm tra NCCL logs và network topology.
</Callout>

<Callout type="danger" title="Pitfall 3: Spot instance bị thu hồi làm mất 18 giờ training">
**Triệu chứng:** Cloud reclaim máy, job biến mất, checkpoint local mất theo.

**Fix:** checkpoint định kỳ vào durable storage, bật resume, lưu optimizer/scheduler states, và test restore trước khi chạy job dài.
</Callout>

## Cost / Throughput / Resilience Unit Economics

| Quyết định | Tác động chi phí | Cách tối ưu |
|---|---|---|
| On-demand GPU | ổn định nhưng đắt | dùng cho jobs critical hoặc không resume được |
| Spot GPU | rẻ hơn nhiều nhưng bị preempt | checkpoint thường xuyên, job idempotent |
| ZeRO offload CPU | fit model lớn hơn | có thể chậm 3-5x; tính total cost thay vì hourly cost |
| Batch size nhỏ | ít OOM | throughput thấp; dùng gradient accumulation |
| Save checkpoint quá dày | an toàn hơn | tốn storage và pause; tune theo preemption risk |
| Data streaming từ S3 | setup đơn giản | có thể chậm; cache local hoặc shard file lớn |

Câu hỏi đúng không phải “GPU nào rẻ nhất mỗi giờ?” mà là “chi phí để hoàn thành một training run reproducible là bao nhiêu?”.

## Answer key

1. Nó dồn nhiều việc về GPU chính và không tối ưu multi-process/all-reduce như DDP/FSDP.
2. Shard weights, gradients và optimizer states qua nhiều GPU thay vì nhân bản đầy đủ.
3. Giảm VRAM bằng cách tính lại activation, đổi lại tăng compute time.
4. Multi-GPU training cần đồng bộ gradient/parameters liên tục; network chậm làm GPU chờ nhau.
5. Spot có thể bị thu hồi, local disk mất; durable storage cho phép resume.

## Điểm cần nhớ

- Training infrastructure là bài toán memory + communication + storage + resilience.
- DDP phù hợp khi model fit trên mỗi GPU; FSDP/ZeRO dùng khi cần shard model lớn.
- Mixed precision, gradient checkpointing và sequence packing là ba kỹ thuật tiết kiệm VRAM quan trọng.
- GPU utilization thấp thường do data loading hoặc network, không phải model quá nhẹ.
- Checkpoint restore phải được test trước khi tin vào spot/preemptible savings.
