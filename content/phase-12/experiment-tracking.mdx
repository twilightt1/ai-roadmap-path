Ở bài trước, bạn đã học cách version dữ liệu để một model có thể reproduce. Nhưng reproducibility chưa đủ nếu bạn không biết **run nào** đã tạo ra model tốt nhất, với hyperparameters nào, trên GPU nào, dùng dataset hash nào và eval report nào.

**Experiment Tracking** là nhật ký vận hành của AI engineering. Nó thay thế file Excel, tên thư mục `final_v7_best_REAL`, screenshot loss curve và trí nhớ của con người bằng một hệ thống có thể truy vấn, so sánh, audit và promote model artifact vào production.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Run tốt nhất biến mất sau cuối tuần train GPU">
**Bối cảnh:** Team fine-tune một model medical QA. Lead engineer chạy 48 thử nghiệm trong cuối tuần trên nhiều GPU, mỗi run thay learning rate, LoRA rank, scheduler và dataset mix.

**Sự cố:** Sáng thứ Hai, engineer nói run 27 là tốt nhất: eval loss thấp, doctor review tốt. Nhưng khi chuẩn bị deploy, team không tìm được config chính xác của run 27. File `train.py` đã bị sửa, validation set đã đổi, checkpoint tốt nhất nằm trong thư mục local bị xóa khi máy spot bị thu hồi.

1. Hyperparameters không được log tự động.
2. Dataset/version hash không gắn với run.
3. Checkpoint không được lưu như artifact.
4. System metrics như GPU utilization, OOM và step time không được ghi lại.

**Hậu quả:** Team phải train lại nhiều ngày để đoán lại cấu hình, đốt thêm GPU budget và mất niềm tin vào benchmark nội bộ.
</Callout>

Bài học rút ra: training run không có tracking là một thí nghiệm không tồn tại. Nếu bạn không thể tìm lại config, data, code và artifact, bạn không thể ship nó vào production.

## 2. Architecture Blueprint (Experiment Control Tower)

Một tracking system tốt nằm ở trung tâm training pipeline, nhận telemetry từ code, data, hardware và eval.

```text
[ Git commit + config ]       [ Dataset manifest ]       [ GPU cluster ]
        │                            │                         │
        └──────────────┬─────────────┴─────────────┬───────────┘
                       ▼                           ▼
              [ Training script / Trainer / Accelerate ]
                       │
                       │ async logging
                       ▼
[ Experiment Tracking Server: W&B / MLflow / Neptune ]
  - run metadata
  - hyperparameters
  - metrics curves
  - system metrics
  - artifacts
  - eval reports
  - notes and tags
                       │
                       ▼
[ Model Registry ]
  - candidate
  - staging
  - production
  - archived
```

Một run production-ready nên có tối thiểu:

- `run_id`, owner, timestamp,
- Git commit SHA,
- dataset manifest hash,
- model/base checkpoint,
- training config,
- metrics train/eval,
- system metrics,
- artifact links,
- eval report,
- approval state.

## 3. Runs, Metrics, Artifacts và Registry

Experiment tracking không chỉ là biểu đồ loss. Nó gồm bốn lớp:

| Lớp | Nội dung | Câu hỏi trả lời |
|---|---|---|
| Run metadata | tên run, tags, owner, commit, dataset hash | Run này là gì? |
| Metrics | loss, accuracy, reward, latency, tokens/s | Run có tốt hơn không? |
| Artifacts | checkpoints, LoRA adapter, tokenizer, eval report | Lấy model ở đâu? |
| Registry | stage, approval, deployment target | Model có được deploy không? |

Nếu chỉ log `train_loss`, bạn không thể giải thích vì sao run tốt hơn. Nếu chỉ lưu checkpoint, bạn không thể biết checkpoint đó có an toàn để deploy hay không.

## 4. W&B vs MLflow vs TensorBoard

| Công cụ | Điểm mạnh | Giới hạn | Khi nên dùng |
|---|---|---|---|
| Weights & Biases | UI mạnh, cộng tác tốt, system metrics tốt, artifact tracking tiện | SaaS mặc định, cần kiểm soát dữ liệu nhạy cảm | LLM/fine-tuning team cần tốc độ và dashboard đẹp |
| MLflow | Open-source, tự host dễ hơn, registry mạnh | UI ít hiện đại hơn, setup backend store/artifact store cần DevOps | Doanh nghiệp cần on-prem/private cloud |
| TensorBoard | đơn giản, local, tích hợp PyTorch/TensorFlow | yếu về collaboration, registry và artifacts | học tập, debug local, research nhỏ |
| Neptune/ClearML | tracking đầy đủ, team workflow | thêm vendor/tooling | tổ chức đã chuẩn hóa theo vendor này |

Quy tắc thực dụng:

- Team nhỏ, muốn nhanh: dùng W&B.
- Enterprise cần kiểm soát dữ liệu: MLflow self-host hoặc W&B enterprise/on-prem.
- Debug local: TensorBoard vẫn đủ tốt.

## 5. What to Log for LLM Fine-Tuning

Với LLM, các field quan trọng hơn ML truyền thống vì chi phí lớn và lỗi khó debug.

### Config nên log

```json
{
  "base_model": "meta-llama/Meta-Llama-3-8B-Instruct",
  "method": "qlora",
  "lora_rank": 16,
  "lora_alpha": 32,
  "learning_rate": 0.00002,
  "scheduler": "cosine",
  "warmup_ratio": 0.05,
  "max_seq_length": 4096,
  "dataset_manifest_hash": "sha256:...",
  "git_commit": "7c9f2ab"
}
```

### Metrics nên log

| Metric | Vì sao quan trọng |
|---|---|
| `train_loss` | phát hiện training không hội tụ |
| `eval_loss` | phát hiện overfit tương đối |
| task metric | đo đúng bài toán: F1, exact match, pass@k |
| judge score | đo semantic quality |
| tokens/sec | tính throughput và GPU efficiency |
| GPU utilization | phát hiện bottleneck CPU/I/O |
| memory allocated | tránh OOM, tối ưu batch size |
| cost estimate | quyết định early stop hoặc scale |

<Callout type="warning" title="Loss giảm không đồng nghĩa production tốt hơn">
Một model có eval loss thấp hơn vẫn có thể trả lời dài dòng hơn, vi phạm format JSON nhiều hơn hoặc an toàn kém hơn. Vì vậy tracking phải log cả task eval và safety/regression eval.
</Callout>

## Implementation Playbook: W&B Tracking for Hugging Face Trainer

### Bước 1: Thiết lập biến môi trường

```bash
export WANDB_API_KEY="your_api_key"
export WANDB_PROJECT="customer-support-sft"
export WANDB_ENTITY="ml-platform"
export WANDB_LOG_MODEL="end"
```

Không hardcode API key trong notebook hoặc repo.

### Bước 2: Khởi tạo run với metadata đầy đủ

```python
import os
import subprocess
import wandb


def git_sha() -> str:
    return subprocess.check_output(["git", "rev-parse", "HEAD"]).decode().strip()

wandb.init(
    project=os.environ["WANDB_PROJECT"],
    name="qlora-r16-lr2e-5-seq4096",
    tags=["qlora", "customer-support", "candidate"],
    config={
        "base_model": "meta-llama/Meta-Llama-3-8B-Instruct",
        "method": "qlora",
        "lora_rank": 16,
        "learning_rate": 2e-5,
        "max_seq_length": 4096,
        "dataset_manifest_hash": "sha256:abc123",
        "git_commit": git_sha(),
    },
)
```

### Bước 3: Kết nối với Hugging Face Trainer

```python
from transformers import TrainingArguments, Trainer

training_args = TrainingArguments(
    output_dir="./outputs",
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=2e-5,
    num_train_epochs=3,
    logging_steps=10,
    eval_steps=100,
    save_steps=500,
    evaluation_strategy="steps",
    save_strategy="steps",
    report_to="wandb",
    run_name="qlora-r16-lr2e-5-seq4096",
    bf16=True,
)

# trainer = Trainer(model=model, args=training_args, train_dataset=train_ds, eval_dataset=eval_ds)
# trainer.train()
```

### Bước 4: Log eval report và artifact

```python
import json
from pathlib import Path

report = {
    "golden_set_score": 4.42,
    "json_validity": 0.997,
    "safety_failures": 0,
    "notes": "Passed sanity eval; needs longer domain review before production.",
}
Path("eval_report.json").write_text(json.dumps(report, indent=2), encoding="utf-8")

artifact = wandb.Artifact("qlora-adapter-r16", type="model")
artifact.add_dir("./outputs/best_adapter")
artifact.add_file("eval_report.json")
wandb.log_artifact(artifact)
wandb.finish()
```

## 6. Tracking Distributed Training

Khi train đa GPU, nếu mỗi process đều log cùng một metric, dashboard sẽ bị duplicate hoặc race condition.

Pattern đúng:

```python
from accelerate import Accelerator

accelerator = Accelerator()

if accelerator.is_main_process:
    wandb.init(project="llm-sft", config={"distributed": True})

for step, batch in enumerate(train_loader):
    loss = train_step(batch)
    gathered_loss = accelerator.gather(loss).mean().item()

    if accelerator.is_main_process and step % 10 == 0:
        wandb.log({"train/loss": gathered_loss, "step": step})
```

Nguyên tắc:

- chỉ main process tạo run,
- aggregate metrics trước khi log,
- log rank/GPU errors riêng nếu cần,
- checkpoint lên shared storage, không chỉ local disk.

## 7. Experiment Comparison và Decision Making

Tracking không có giá trị nếu team không có quy tắc ra quyết định.

Ví dụ scorecard để promote model:

| Gate | Điều kiện pass |
|---|---|
| Reproducibility | có Git commit và dataset manifest |
| Quality | golden eval `>= 4.2/5` |
| Regression | không fail case critical nào |
| Format | JSON validity `>= 99.5%` |
| Safety | prompt injection suite pass |
| Cost | cost/request không tăng quá 20% |
| Latency | p95 latency không tăng quá 15% |

Một run thắng loss nhưng thua latency/cost có thể không phải run tốt nhất cho sản phẩm.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Upload mọi checkpoint làm đầy disk và bill artifact">
**Triệu chứng:** Training job crash vì `No space left on device`; W&B artifact storage tăng rất nhanh.

**Nguyên nhân:** `WANDB_LOG_MODEL=true` hoặc save checkpoint quá dày cho model 7B/70B.

**Fix:** Chỉ upload best/final checkpoint, dùng retention policy, và lưu checkpoint trung gian vào object storage có lifecycle.
</Callout>

<Callout type="warning" title="Pitfall 2: Offline cluster làm script treo khi gọi SaaS tracking">
**Triệu chứng:** Training chạy trong VPC/air-gapped environment bị treo ở bước init logger.

**Fix:** Dùng offline mode hoặc MLflow self-host trong nội mạng.

```bash
export WANDB_MODE="offline"
# Sau đó sync khi có mạng:
# wandb sync ./wandb/offline-run-...
```
</Callout>

<Callout type="danger" title="Pitfall 3: So sánh run dùng validation set khác nhau">
**Triệu chứng:** Run mới có score cao hơn rõ rệt nhưng production tệ hơn.

**Nguyên nhân:** Validation hash khác, dataset dễ hơn hoặc bị leakage.

**Fix:** Dashboard phải hiển thị dataset hash; chỉ cho compare run cùng validation snapshot hoặc gắn cảnh báo rõ ràng.
</Callout>

## Cost / GPU Efficiency Unit Economics

| Chỉ số | Cách đọc | Hành động |
|---|---|---|
| GPU utilization thấp | GPU dưới 50% dù job đang chạy | tăng dataloader workers, cache dataset, tăng batch size |
| Step time tăng | một số step chậm bất thường | kiểm tra sequence length, packing, network storage |
| Eval cost cao | LLM-as-judge gọi quá nhiều | sample tiered eval, batch API, nightly full eval |
| Artifact storage phình | nhiều checkpoint lớn | retention policy, chỉ promote best model |
| Run không reproduce | thiếu commit/data hash | bắt buộc metadata gate trước khi mark candidate |

Tracking tốt biến GPU bill thành dữ liệu ra quyết định: tiếp tục train, early stop, đổi data hay đổi architecture.

## Answer key

1. Tracking lưu metadata, metrics, config, artifacts, eval report và lineage; checkpoint chỉ là trọng số model.
2. Để đảm bảo so sánh hợp lệ và reproduce đúng dữ liệu đã dùng.
3. Khi có yêu cầu bảo mật/on-prem/data residency hoặc không thể gửi metadata ra SaaS.
4. Nếu mọi process cùng log, metrics bị duplicate và dashboard sai.
5. Không tự động; cần scorecard cân bằng quality, latency, cost, safety và regression.

## Điểm cần nhớ

- Training run không có tracking là thí nghiệm không thể audit.
- Log cả code commit, dataset hash, config, metrics, artifacts và eval report.
- Loss chỉ là một tín hiệu; production decision cần quality, safety, latency và cost.
- Distributed training cần logging có rank/process discipline.
- Model registry biến run tốt thành artifact có lifecycle rõ ràng.
