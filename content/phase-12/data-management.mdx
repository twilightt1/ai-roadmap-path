Phase 11 đã đưa bạn qua generative AI beyond text: image, audio, video và multimodal. Nhưng một mô hình đẹp trong demo chưa phải là production system. Phase 12 bắt đầu phần khó nhất của nghề AI Engineer: biến model, prompt, dataset và hạ tầng thành một hệ thống có thể tái tạo, triển khai, giám sát và bảo vệ.

Bài đầu tiên là **Data Management & Versioning**. Trong MLOps/LLMOps, dữ liệu không phải là file phụ trợ; dữ liệu là một phần của sản phẩm. Nếu bạn không biết chính xác model được train bằng dataset nào, prompt nào, config nào và commit nào, bạn không thể debug, rollback hoặc chứng minh chất lượng hệ thống.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Model production không thể reproduce vì mất dataset gốc">
**Bối cảnh:** Một team fine-tune LLaMA cho tác vụ trích xuất điều khoản hợp đồng. Sau 2 tuần thử nghiệm, run tốt nhất đạt điểm rất cao trên benchmark nội bộ. Team deploy model weights lên production.

**Sự cố:** Sau deploy, model bắt đầu hallucinate điều khoản. Team muốn reproduce run cũ để kiểm tra nhưng phát hiện file `train_data_final_v3_REAL.jsonl` nằm trên laptop của một data scientist đã bị ghi đè bởi script cleaning mới. File validation cũng đã thay đổi nhưng không ai biết thay đổi ở commit nào.

1. Dataset thật không được version bằng DVC hoặc object storage có hash.
2. Training config không lưu dataset fingerprint.
3. Validation set bị sửa sau khi benchmark được báo cáo.
4. Không có data lineage nối từ raw data đến cleaned dataset đến model artifact.

**Hậu quả:** Team không thể xác định model hỏng do data drift, cleaning bug hay inference regression. Dự án bị rollback, mất thêm ngân sách GPU để train lại từ đầu.
</Callout>

Bài học rút ra: trong AI production, **code + config + data + model artifact** phải được quản lý như một đơn vị không thể tách rời. Mất một mắt xích là mất khả năng vận hành.

## 2. Architecture Blueprint (Reproducible Data & Model Lineage)

Một kiến trúc data management tốt không cố nhét file lớn vào Git. Nó dùng Git để version metadata và dùng object storage để lưu artifact thật.

```text
[ Raw data sources ]
  - uploads
  - databases
  - logs
  - vendor datasets
        │
        ▼
[ Ingestion pipeline ]
  - schema validation
  - PII scanning
  - source metadata
        │
        ▼
[ Versioned data lake / object storage ]
  - s3://bucket/raw/...
  - s3://bucket/clean/...
  - immutable snapshots
        │
        ▼
[ DVC / LakeFS / Data registry ] <------ [ Git commit ]
  - dataset hash                       - training code
  - split hash                         - config yaml
  - artifact pointer                   - prompt templates
        │
        ▼
[ Training / eval pipeline ]
  - dataset fingerprint
  - model artifact
  - eval report
        │
        ▼
[ Model registry ]
  - model version
  - linked dataset version
  - linked eval version
```

Mục tiêu: bất kỳ model nào trong production cũng phải trả lời được bốn câu hỏi:

1. Nó được train bằng data snapshot nào?
2. Data snapshot đó được tạo từ raw sources nào?
3. Code/config/prompt nào đã tạo ra model?
4. Eval report nào chứng minh model đủ điều kiện deploy?

## 3. Data as a First-Class Artifact

Trong phần mềm truyền thống, build artifact là Docker image hoặc binary. Trong AI, artifact còn bao gồm:

| Artifact | Ví dụ | Vì sao cần version? |
|---|---|---|
| Raw data | log hội thoại, PDF, ảnh, audio | Audit nguồn gốc và compliance |
| Cleaned data | JSONL instruction dataset | Reproduce training |
| Splits | train/validation/test split | Tránh benchmark bị thay đổi |
| Prompts | system prompt, eval prompt | Prompt thay đổi làm behavior đổi |
| Embeddings | vector index snapshot | RAG answer phụ thuộc index version |
| Model weights | `.safetensors`, LoRA adapter | Rollback và registry |
| Eval reports | score, judge output, failure cases | Release gate |

Nếu chỉ version model weights mà không version data, bạn chỉ lưu “kết quả” nhưng mất “nguyên nhân”.

## 4. Git vs DVC vs Object Storage

Git rất tốt cho source code nhỏ, text diff và review. Nhưng Git không phù hợp cho dataset/model nặng hàng GB.

```text
Git repo:
  train.py
  configs/sft.yaml
  data/train.jsonl.dvc     -> pointer hash only
  reports/eval-v12.json

Object storage:
  s3://ml-artifacts/dvc/files/md5/ab/cdef...
  s3://ml-artifacts/models/customer-support/v12/
```

| Công cụ | Dùng cho | Không nên dùng cho |
|---|---|---|
| Git | Code, config, small eval reports, DVC pointers | Dataset 10GB, model 20GB |
| DVC | Version file artifacts, data pipelines, remote storage | Streaming data cực lớn realtime |
| Git LFS | File lớn ít thay đổi trong repo | Dataset biến động liên tục |
| LakeFS | Data lake versioning, branch/merge trên object storage | Team nhỏ chỉ có vài file JSONL |
| Feature store | Online/offline features cho ML truyền thống | Instruction dataset đơn giản |

Quy tắc thực dụng: **Git quản lý con trỏ, object storage quản lý bytes thật**.

## 5. Dataset Lineage và Fingerprinting

Dataset fingerprint là hash đại diện cho nội dung và cách tạo dataset. Nó nên bao gồm:

- hash của raw input files,
- version của cleaning script,
- config lọc dữ liệu,
- seed khi split train/validation/test,
- schema version,
- timestamp và người tạo snapshot.

Ví dụ metadata tối thiểu:

```json
{
  "dataset_id": "contract-extraction-sft",
  "version": "2026-07-02.1",
  "raw_sources": ["s3://legal/raw/contracts/2026-06"],
  "cleaning_commit": "7c9f2ab",
  "schema_version": "chatml-v2",
  "split_seed": 42,
  "train_hash": "sha256:...",
  "validation_hash": "sha256:...",
  "pii_scan": "passed",
  "owner": "ml-platform"
}
```

Không cần metadata quá phức tạp lúc đầu. Nhưng nếu metadata không đủ để reproduce, nó chưa đạt chuẩn production.

## 6. Data Quality Gates

Data versioning chỉ đảm bảo bạn lưu đúng phiên bản. Nó không đảm bảo dữ liệu tốt. Vì vậy pipeline cần quality gates trước khi dataset được phép đi vào training.

| Gate | Kiểm tra | Ví dụ lỗi bị chặn |
|---|---|---|
| Schema | field bắt buộc, type, max length | thiếu `messages`, output null |
| Safety | PII, secrets, toxic content | lộ số thẻ, API key |
| Deduplication | duplicate exact/near-duplicate | cùng câu trả lời lặp 10,000 lần |
| Distribution | ngôn ngữ, domain, length histogram | validation toàn câu ngắn hơn train |
| Leakage | overlap train/test | test case nằm trong train |
| License | quyền sử dụng dữ liệu | web scrape không rõ license |

<Callout type="warning" title="Đừng chỉ kiểm tra schema">
Một file JSONL có thể đúng schema nhưng vẫn phá model: duplicate quá nhiều, câu trả lời sai, PII chưa mask, hoặc dữ liệu test bị trộn vào train. Data validation trong AI phải kiểm tra cả cấu trúc lẫn ngữ nghĩa.
</Callout>

## Implementation Playbook: DVC + Pandera + Dataset Manifest

Playbook này thiết lập workflow tối thiểu cho một instruction dataset.

### Bước 1: Khởi tạo DVC remote

```bash
git init
dvc init

dvc remote add -d storage s3://company-ml-artifacts/customer-support
dvc remote modify storage region ap-southeast-1

git add .dvc/config
git commit -m "Configure DVC remote"
```

### Bước 2: Version dataset bằng DVC

```bash
mkdir -p dataset
python scripts/build_dataset.py \
  --raw s3://company-raw/support-tickets/2026-06 \
  --out dataset/train_chatml.jsonl \
  --split-seed 42

dvc add dataset/train_chatml.jsonl dataset/validation_chatml.jsonl

git add dataset/*.dvc dataset/.gitignore scripts/build_dataset.py
git commit -m "Add customer support SFT dataset v1"
dvc push
```

### Bước 3: Validate schema trước training

```python
import json
from pathlib import Path

REQUIRED_ROLES = {"system", "user", "assistant"}


def validate_chatml(path: str) -> None:
    errors = []
    for line_no, line in enumerate(Path(path).read_text(encoding="utf-8").splitlines(), 1):
        item = json.loads(line)
        messages = item.get("messages")
        if not isinstance(messages, list) or len(messages) < 2:
            errors.append(f"line {line_no}: messages must be a non-empty list")
            continue

        for msg in messages:
            if msg.get("role") not in REQUIRED_ROLES:
                errors.append(f"line {line_no}: invalid role {msg.get('role')}")
            if not msg.get("content") or len(msg["content"]) > 8000:
                errors.append(f"line {line_no}: invalid content length")

    if errors:
        raise ValueError("\n".join(errors[:20]))


validate_chatml("dataset/train_chatml.jsonl")
validate_chatml("dataset/validation_chatml.jsonl")
print("Dataset schema passed")
```

### Bước 4: Tạo manifest nối data với training run

```python
import hashlib
import json
from pathlib import Path


def sha256_file(path: str) -> str:
    h = hashlib.sha256()
    with open(path, "rb") as f:
        for chunk in iter(lambda: f.read(1024 * 1024), b""):
            h.update(chunk)
    return h.hexdigest()

manifest = {
    "dataset_id": "customer-support-sft",
    "version": "v1",
    "train_hash": sha256_file("dataset/train_chatml.jsonl"),
    "validation_hash": sha256_file("dataset/validation_chatml.jsonl"),
    "split_seed": 42,
    "schema": "chatml-v1",
}

Path("dataset/manifest.json").write_text(json.dumps(manifest, indent=2), encoding="utf-8")
```

Training script nên log `manifest.json` vào W&B/MLflow cùng với hyperparameters.

## 7. Data Governance và Access Control

Production dataset thường chứa dữ liệu nhạy cảm. Data management phải trả lời được:

- Ai có quyền đọc raw data?
- Ai có quyền tạo snapshot mới?
- Ai có quyền promote dataset vào training?
- Snapshot nào chứa PII và chính sách retention là gì?
- Khi user yêu cầu xóa dữ liệu, dataset/model bị ảnh hưởng ra sao?

Thiết kế tối thiểu:

```text
raw bucket:
  access: data-platform only
  encryption: KMS
  retention: policy by domain

clean bucket:
  access: ml-team read, pipeline write
  PII: masked/tokenized

training bucket:
  access: training jobs only
  immutable snapshots

registry:
  approval required to promote dataset to production training
```

Đừng để notebook cá nhân trở thành nơi lưu bản copy duy nhất của dữ liệu nhạy cảm.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Git checkout nhưng quên dvc pull">
**Triệu chứng:** Bạn checkout code cũ nhưng training vẫn dùng dataset mới.

**Nguyên nhân:** Git chỉ đổi file pointer `.dvc`; file vật lý trong workspace chưa được đồng bộ.

**Fix:** Luôn chạy cặp lệnh:

```bash
git checkout release-v1.4
dvc pull
```

Có thể thêm Git hook `post-checkout` để nhắc hoặc tự chạy `dvc pull` trong môi trường CI.
</Callout>

<Callout type="warning" title="Pitfall 2: Validation set bị thay đổi âm thầm">
**Triệu chứng:** Eval score tăng mạnh nhưng production quality giảm.

**Nguyên nhân:** Validation set mới dễ hơn hoặc bị trộn với training data.

**Fix:** Version validation/test split riêng, khóa quyền sửa, log hash vào experiment tracking và release report.
</Callout>

<Callout type="danger" title="Pitfall 3: DVC remote public chứa dữ liệu nhạy cảm">
**Triệu chứng:** Security team phát hiện PII nằm trong bucket artifact có quyền public read.

**Fix:** Bật bucket private by default, KMS encryption, IAM least privilege, PII scanning trước `dvc add`, và lifecycle policy để xóa/retire snapshot theo compliance.
</Callout>

## Cost / Storage / Compliance Unit Economics

| Quyết định | Chi phí ẩn | Tối ưu |
|---|---|---|
| Lưu mọi snapshot mãi mãi | S3/GCS tăng đều theo tháng | Lifecycle sang cold storage, retention theo domain |
| Pull dataset lớn trong CI | Chậm pipeline, tốn egress | Self-hosted runner cùng VPC, cache artifact |
| Lưu nhiều bản model 30GB | Registry phình nhanh | Chỉ promote model có eval report, xóa checkpoint trung gian |
| Validation toàn bộ dữ liệu mỗi PR | CI chậm và đắt | Sanity validation trên PR, full validation nightly |
| Copy data về laptop | Rủi ro PII và mất kiểm soát | Workbench trong VPC, signed URL ngắn hạn |

Một data platform tốt không chỉ “lưu được file”; nó giảm thời gian debug, giảm chi phí train lại và giảm rủi ro compliance.

## Answer key

1. Git được tối ưu cho text diff nhỏ; dataset lớn làm repo nặng, clone chậm và khó quản lý lịch sử.
2. Git lưu pointer/hash `.dvc`; object storage lưu bytes thật của dataset/model.
3. Raw file hashes, cleaning code commit, config, split seed, schema version và hash của train/validation/test.
4. Leakage làm benchmark đo trí nhớ thay vì năng lực generalization, dẫn đến score production sai lệch.
5. `dvc pull` sau `git checkout` để lấy đúng file vật lý tương ứng với pointer trong commit.

## Điểm cần nhớ

- Data là artifact production, không phải file phụ trợ.
- Reproducibility cần liên kết code, config, data, prompt, model và eval report.
- Git quản lý metadata/pointers; object storage quản lý file lớn.
- Data quality gates phải kiểm tra schema, PII, duplicate, distribution và leakage.
- Dataset không có lineage sẽ biến mọi lỗi model thành điều tra thủ công tốn kém.
