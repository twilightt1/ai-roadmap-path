Bạn đã deploy AI service lên cloud. Nhưng với LLM, HTTP 200 không có nghĩa là hệ thống đúng. Model có thể trả lời sai, hallucinate, gọi tool lặp vô hạn, leak dữ liệu nhạy cảm hoặc âm thầm đốt hàng nghìn USD token mà dashboard web truyền thống không phát hiện.

**Monitoring & Logging** trong LLMOps là observability cho hành vi ngữ nghĩa, không chỉ observability cho server. Bạn cần thấy prompt, retrieval context, tool calls, token usage, latency, cost, judge scores, user feedback và model version trong cùng một trace.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Agent loop đốt 15,000 USD cuối tuần nhưng APM vẫn xanh">
**Bối cảnh:** Startup EdTech dùng LLM agent giải toán. Backend có Datadog theo dõi CPU, RAM, HTTP 500 và request count. Tất cả dashboard đều xanh.

**Sự cố:** Sáng thứ Hai, hóa đơn LLM API tăng 15,000 USD. Điều tra thủ công cho thấy một bug trong tool router khiến agent gọi sai calculator tool, nhận lỗi, thêm toàn bộ history vào prompt rồi retry vô hạn. HTTP vẫn trả 200 cho nhiều request nên APM truyền thống không báo động.

1. Không log token usage theo user/session/feature.
2. Không trace tool calls trong agent loop.
3. Không có budget alert hoặc max tool iterations.
4. Không có eval ngầm để phát hiện answer sai hoặc loop bất thường.
5. Raw prompt logs quá lớn nên team không tìm kiếm được nguyên nhân nhanh.

**Hậu quả:** Feature bị tắt, CFO yêu cầu hard budget cap và team phải triển khai LLM observability trước khi bật lại.
</Callout>

Bài học rút ra: AI systems có thể thất bại trong im lặng. Monitoring phải đo **tokens, traces, quality và cost**, không chỉ uptime.

## 2. Architecture Blueprint (LLM Observability Pipeline)

Một kiến trúc observability production gồm ba lớp: traces, metrics và evals.

```text
[ User request ]
      │
      ▼
[ API Gateway / Backend ]
      │
      ├── request metadata
      ├── tenant / user / feature
      ├── prompt version
      │
      ▼
[ RAG / Agent / LLM pipeline ]
  - retrieval spans
  - rerank spans
  - LLM generation spans
  - tool call spans
  - safety spans
      │
      ├────────► [ Metrics store ]
      │           latency, token, cost, error, queue
      │
      ├────────► [ Trace store ]
      │           prompt, context, output, tool arguments
      │
      └────────► [ Eval pipeline ]
                  groundedness, toxicity, user feedback
```

Mục tiêu: khi một user báo “AI trả lời sai”, bạn mở trace và thấy toàn bộ đường đi: query, chunks được retrieve, prompt cuối, model version, token usage, output, eval score và latency từng bước.

---

## 3. Logs vs Metrics vs Traces vs Evals

| Loại telemetry | Ví dụ | Dùng để làm gì |
|---|---|---|
| Logs | error message, request id, warning | debug sự kiện cụ thể |
| Metrics | p95 latency, token cost/day, error rate | dashboard và alert |
| Traces | chain spans, tool calls, retrieval steps | hiểu request đi qua pipeline thế nào |
| Evals | groundedness score, toxicity score | đo chất lượng semantic |

APM truyền thống thường mạnh về logs/metrics. LLMOps cần traces có cấu trúc và evals để hiểu hành vi AI.

## 4. What to Monitor in LLM/RAG/Agent Systems

### LLM generation metrics

- input tokens,
- output tokens,
- total cost,
- TTFT,
- TPOT,
- model/provider,
- finish reason,
- rate limit errors,
- context length.

### RAG metrics

- number of retrieved chunks,
- retrieval latency,
- rerank score,
- context precision/recall sample,
- citation accuracy,
- no-context refusal rate.

### Agent metrics

- number of tool calls,
- tool error rate,
- max iterations hit,
- loop detection,
- tool latency,
- state transitions.

### Product metrics

- thumbs up/down,
- conversation abandonment,
- escalation to human,
- copy/export rate,
- cost per successful task.

<Callout type="warning" title="Đừng chỉ monitor model provider">
Nếu OpenAI/Anthropic/vLLM hoạt động bình thường nhưng retriever trả sai chunks, answer vẫn sai. Observability phải bao phủ toàn pipeline: retrieval, prompt assembly, LLM, tools và UI feedback.
</Callout>

## Implementation Playbook: Langfuse Tracing for RAG

### Bước 1: Cài đặt và cấu hình

```bash
pip install langfuse openai

export LANGFUSE_PUBLIC_KEY="pk-lf-..."
export LANGFUSE_SECRET_KEY="sk-lf-..."
export LANGFUSE_HOST="https://cloud.langfuse.com"
```

### Bước 2: Tạo trace cho một request RAG

```python
from langfuse import Langfuse
from openai import OpenAI

langfuse = Langfuse()
client = OpenAI()


def answer_question(user_id: str, query: str):
    trace = langfuse.trace(
        name="rag-chat",
        user_id=user_id,
        metadata={"feature": "support_chat", "prompt_version": "v3"},
    )

    retrieval = trace.span(name="retrieve-context", input={"query": query})
    chunks = retrieve_chunks(query)
    retrieval.end(output={"chunk_ids": [c["id"] for c in chunks], "count": len(chunks)})

    prompt = build_prompt(query, chunks)
    generation = trace.generation(
        name="llm-answer",
        model="gpt-4o-mini",
        input=prompt,
    )

    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=prompt,
        temperature=0,
    )

    answer = response.choices[0].message.content
    generation.end(
        output=answer,
        usage={
            "input": response.usage.prompt_tokens,
            "output": response.usage.completion_tokens,
            "total": response.usage.total_tokens,
        },
    )

    return answer
```

### Bước 3: Ghi user feedback

```python
def record_feedback(trace_id: str, liked: bool, comment: str | None = None):
    langfuse.score(
        trace_id=trace_id,
        name="user_feedback",
        value=1 if liked else 0,
        comment=comment,
    )
```

### Bước 4: Chạy async eval sample

```python
def score_groundedness(trace_id: str, answer: str, context: str):
    score = run_groundedness_judge(answer=answer, context=context)
    langfuse.score(
        trace_id=trace_id,
        name="groundedness",
        value=score,
        comment="LLM-as-judge sampled eval",
    )
```

Không nhất thiết chấm mọi request realtime. Có thể sample theo risk hoặc chạy batch nền để tiết kiệm chi phí.

## Project Guide: LLMOps Monitoring Dashboard (Phase 12 Medium)

Mục tiêu project: instrument một AI API và xây dashboard theo dõi token, latency, cost, errors và quality.

### Requirements

- FastAPI endpoint `/chat` gọi OpenAI/vLLM.
- Trace mỗi request với `request_id`, `user_id`, `feature`, `model`, `prompt_version`.
- Log input/output token usage.
- Đo p50/p95 latency và error rate.
- Ghi user feedback endpoint `/feedback`.
- Dashboard Langfuse/Helicone hoặc Grafana hiển thị:
  - cost/day,
  - cost/user,
  - p95 latency,
  - top expensive traces,
  - error rate,
  - thumbs-down traces.
- Alert khi daily cost vượt ngưỡng hoặc p95 latency tăng mạnh.

### Suggested architecture

```text
FastAPI app
  ├── LLM client wrapper
  ├── Langfuse/Helicone instrumentation
  ├── Postgres for app data
  ├── Prometheus metrics endpoint
  └── Grafana dashboard
```

### Acceptance criteria

- Mỗi request có trace xem được prompt/model/tokens/latency.
- Có thể filter theo user hoặc feature.
- Có alert rule cho cost spike.
- Feedback của user gắn vào trace đúng.
- Raw PII được mask hoặc không log trong môi trường production.

Stretch goal: thêm async groundedness eval cho 10% requests và hiển thị score trend theo ngày.

## 6. Alert Design for AI Systems

Alert tốt phải actionable. Đừng page SRE vì một request chậm; hãy page khi có pattern.

| Alert | Điều kiện gợi ý | Người nhận |
|---|---|---|
| Cost spike | cost/hour tăng `> 3x` baseline | owner feature + FinOps |
| Latency degradation | p95 latency vượt SLA 15 phút | platform/SRE |
| Error rate | 5xx hoặc provider errors tăng | backend/platform |
| Quality drop | groundedness score giảm dưới ngưỡng | AI team |
| Agent loop | tool calls/request vượt max | AI platform |
| PII logging | detector thấy PII trong trace | security/privacy |

AI alert cần kết hợp system metrics và semantic metrics. Nếu chỉ alert error rate, bạn sẽ bỏ lỡ hallucination spike.

## 7. Privacy-Aware Logging

LLM logs rất nhạy cảm vì user có thể nhập bất cứ thứ gì.

Các chiến lược:

- mask PII trước khi gửi trace,
- log metadata thay vì raw prompt cho domain nhạy cảm,
- hash user id thay vì lưu email,
- retention ngắn cho raw traces,
- role-based access control cho dashboard,
- sampling thay vì log 100% traffic,
- redaction cho secrets/API keys.

```text
raw prompt -> PII masker -> trace payload -> observability platform
```

Nếu observability SaaS nằm ngoài region compliance, logs cũng có thể trở thành data breach.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Logging đồng bộ làm tăng latency user path">
**Triệu chứng:** LLM trả trong 1 giây nhưng API mất 4 giây.

**Nguyên nhân:** App chờ observability vendor nhận trace trước khi trả response.

**Fix:** Gửi log async/background queue. Nếu observability down, app vẫn phải phục vụ user.
</Callout>

<Callout type="warning" title="Pitfall 2: Streaming không ghi usage tokens đúng">
**Triệu chứng:** Dashboard cost thấp hơn hóa đơn provider.

**Nguyên nhân:** SDK không bắt usage ở chunk cuối hoặc tự đếm token sai.

**Fix:** Update SDK, bật `include_usage` nếu provider hỗ trợ, reconcile cost với invoice hằng ngày.
</Callout>

<Callout type="danger" title="Pitfall 3: Trace chứa PII/secrets nguyên văn">
**Triệu chứng:** Security phát hiện số thẻ, API key hoặc dữ liệu y tế trong Langfuse logs.

**Fix:** Client-side redaction trước khi log, retention policy, access control và incident response để purge traces nhạy cảm.
</Callout>

## Cost / Observability / Privacy Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Log 100% raw prompts | debug tốt | storage cao, privacy risk |
| Sampling traces | giảm cost | có thể miss incident hiếm |
| Async eval | đo quality | thêm API cost và delay |
| Self-host observability | kiểm soát data | cần vận hành DB/storage |
| SaaS observability | setup nhanh | data residency/vendor risk |
| Long retention | audit tốt | storage và privacy exposure |

Một policy thực dụng: log metadata 100%, trace raw có redaction/sampling, eval sample theo risk, retention ngắn cho payload nhạy cảm.
