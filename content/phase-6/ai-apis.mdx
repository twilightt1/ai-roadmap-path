## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Tool call chạy 2 lần, user nhận 2 email, 5,000 user bị duplicate refund">
**Bối cảnh:** Một team build chatbot có tool calling để send email và refund. Họ dùng OpenAI tool calling, deploy production, tool chạy ngon. Họ không có retry policy, không có idempotency, không có timeout.

**Sự cố:** API timeout 1 lần. Họ set retry=True. Retry gọi lại tool. Tool send_email chạy 2 lần. User nhận 2 email. Refund tool chạy 2 lần. 5,000 user bị duplicate refund.

1. Không có timeout — request treo.
2. Retry không idempotency — retry gọi tool 2 lần.
3. Không có idempotency key — tool không biết retry là request cũ.
4. Không có tool result validation — tool không kiểm tra output.
5. Không có observability — không biết tool chạy 2 lần.

**Hậu quả:** 5,000 user bị duplicate refund. Hóa đơn refund 2x. Họ phải tắt chatbot, refund users, audit log, và thiết kế lại tool calling state machine. Hóa đơn incident response 2 tuần.
</Callout>

Bài học rút ra: tool calling là state machine, không phải function call. Mỗi request phải có trace: input tokens, output tokens, prompt version, model, latency, cache hit, retry count, fallback path, validation status. LLM call không được là black box. Provider API là một dependency, không phải "hộp đen".

## 4. Provider contracts

### Model

Mỗi provider có model naming. Model quyết định:

- context window,
- capability (reasoning, tool use, vision),
- cost,
- latency,
- output format,
- safety policy.

### Parameters

| Parameter | Mục đích |
|---|---|
| `temperature` | độ ngẫu nhiên |
| `top_p` | nucleus sampling |
| `max_tokens` | giới hạn output |
| `stop` | stop sequences |
| `response_format` | JSON mode |
| `seed` | reproducibility |
| `n` | số candidates |
| `tool_choice` | tool calling behavior |

### Context window

Input tokens + output tokens phải &lt;= context window.

```text
system + developer + chat_history + retrieved_context + user_query + output_budget <= context_window
```

### Token

Token là đơn vị model xử lý. Token không phải từ. Token có thể là prefix, word, hoặc subword.

```text
1 token ≈ 0.75 English word
```

Tokenization có thể khác nhau giữa model, ngôn ngữ, và encoding.

### Cost

| Pricing | Ý nghĩa |
|---|---|
| `input_price` | USD per 1M input tokens |
| `output_price` | USD per 1M output tokens |
| `cached_input_price` | giá cho cached input |
| `tool_call_price` | tool call pricing |
| `batch_price` | giảm giá cho batch |

### Error

| Error | Retry? | Strategy |
|---|---|---|
| 400 invalid request | no | fix payload |
| 401/403 auth | no | check key |
| 408 timeout | yes | retry with timeout |
| 429 rate limit | yes | backoff + jitter |
| 500/502/503/504 | yes | retry then fallback |
| context length | no | reduce context |
| invalid JSON output | repair | schema repair loop |

## 5. Tool calling state machine

Tool calling là một **state machine**, không phải function call.

```text
[ User: "What is the refund status?" ]
        │
        ▼
[ Model decides to call tool ]
        │
        ▼
[ Tool: check_refund_status(order_id=123) ]
        │
        ▼
[ Model: tool result observation ]
        │
        ▼
[ Model: "Your refund is processing..." ]
        │
        ▼
[ Return to user ]
```

Key insight: tool calling không phải "model gọi function". Model **chỉ output tool call arguments**. App phải validate, execute, và trả result.

```text
model → tool call arguments (JSON)
        │
        ▼
[ app validates arguments ]
        │
        ▼
[ app executes tool with permission ]
        │
        ▼
[ app returns tool result ]
        │
        ▼
[ model continues with tool result ]
```

### Tool result validation

```python
def validate_tool_call(tool_name: str, arguments: dict) -> bool:
    if tool_name == "check_refund_status":
        return "order_id" in arguments and isinstance(arguments["order_id"], str)
    if tool_name == "create_refund":
        return "order_id" in arguments and "amount" in arguments
    return False


def execute_tool(tool_name: str, arguments: dict, user_id: str) -> dict:
    if not validate_tool_call(tool_name, arguments):
        return {"error": "invalid arguments"}
    if not check_permission(user_id, tool_name, arguments):
        return {"error": "permission denied"}
    return run_tool(tool_name, arguments)
```

## 7. Streaming

### Why streaming?

```text
Non-streaming: wait for all output → return
Streaming:     return token-by-token
```

Streaming giảm TTFT (time-to-first-token) và tăng UX. User thấy output ngay khi model sinh, không phải đợi toàn bộ.

### Streaming code

```python
stream = client.chat.completions.create(
    model="gpt-4o-mini",
    messages=[{"role": "user", "content": "hi"}],
    stream=True,
)
for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")
```

### Stream parsing challenges

- Tool call args có thể split across chunks.
- Need buffer.
- Need handle partial JSON.
- Need handle stop signal.
- Need handle errors.

### Stream parsing

```python
def parse_tool_call_stream(stream):
    tool_args = ""
    for chunk in stream:
        delta = chunk.choices[0].delta
        if delta.tool_calls:
            tool_args += delta.tool_calls[0].function.arguments
        if chunk.choices[0].finish_reason:
            break
    return json.loads(tool_args)
```

## 10. Provider abstraction

### Why abstraction?

Provider abstraction giúp switch provider, test với mock, fallback, cost tracking, và different model cho different task.

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass


@dataclass
class LLMResponse:
    text: str
    model: str
    input_tokens: int
    output_tokens: int


class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list[dict]) -> LLMResponse:
        pass

    @abstractmethod
    def embed(self, texts: list[str]) -> list[list[float]]:
        pass
```

### OpenAI implementation

```python
class OpenAIProvider(LLMProvider):
    def __init__(self, client):
        self.client = client

    def chat(self, messages: list[dict]) -> LLMResponse:
        resp = self.client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
        )
        return LLMResponse(
            text=resp.choices[0].message.content,
            model=resp.model,
            input_tokens=resp.usage.prompt_tokens,
            output_tokens=resp.usage.completion_tokens,
        )
```

### Benefits

- Switch provider easily.
- Test with mock provider.
- Fallback to other provider.
- Different model for different task.
- Cost tracking.

```python
def estimate_cost(input_tokens, output_tokens, input_price, output_price):
    return (input_tokens * input_price + output_tokens * output_price) / 1_000_000


def estimate_cost_per_request(input_tokens, output_tokens, model):
    prices = {
        "gpt-4o-mini": (0.15, 0.60),
        "gpt-4o": (2.50, 10.00),
        "claude-3-5-sonnet": (3.00, 15.00),
```

## 15. Error handling

### Why error handling?

```python
def call_with_retry(provider, messages, max_attempts=3):
    for attempt in range(max_attempts):
        try:
            return provider.chat(messages)
        except Exception as e:
            if attempt == max_attempts - 1:
                raise
            time.sleep(0.1 * (2 ** attempt))
```

### Error matrix

- 400: fix payload.
- 401/403: check key.
- 408: timeout.
- 429: rate limit.
- 500/502/503/504: server error.
- Context length: reduce context.
- Invalid JSON: repair.

## Implementation Playbook: Provider Abstraction with Tool Calling

```python
import json
from dataclasses import dataclass
from abc import ABC, abstractmethod
from typing import Callable


@dataclass
class LLMResponse:
    text: str
    model: str
    input_tokens: int
    output_tokens: int
    tool_calls: list[dict] | None = None


class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list[dict], tools: list[dict] | None = None) -> LLMResponse:
        pass


class MockProvider(LLMProvider):
    def __init__(self, response: str, tool_calls: list[dict] | None = None):
        self.response = response
        self.tool_calls = tool_calls
        self.call_count = 0

    def chat(self, messages: list[dict], tools: list[dict] | None = None) -> LLMResponse:
        self.call_count += 1
        return LLMResponse(
            text=self.response,
            model="mock",
            input_tokens=100,
            output_tokens=50,
            tool_calls=self.tool_calls,
        )


def run_tool_call(tool_name: str, arguments: dict, user_id: str) -> dict:
    if tool_name == "check_refund_status":
        return {"status": "processing", "order_id": arguments["order_id"]}
    if tool_name == "create_refund":
        return {"status": "created", "order_id": arguments["order_id"]}
    return {"error": "unknown tool"}


def chat_with_tools(provider, messages, tools, user_id, max_iterations=3):
    for _ in range(max_iterations):
        response = provider.chat(messages, tools)
        if response.tool_calls:
            for call in response.tool_calls:
                args = json.loads(call["function"]["arguments"])
                result = run_tool_call(call["function"]["name"], args, user_id)
                messages.append({"role": "tool", "content": json.dumps(result), "tool_call_id": call["id"]})
        else:
            return response
    return response


provider = MockProvider(
    response="Your refund is processing.",
    tool_calls=[{
        "id": "call_1",
        "function": {
            "name": "check_refund_status",
            "arguments": json.dumps({"order_id": "123"}),
        },
    }],
)
messages = [{"role": "user", "content": "What is the refund status?"}]
result = chat_with_tools(provider, messages, tools=[], user_id="user_1")
print(result)
```

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: không có timeout">
**Triệu chứng:** Request treo làm worker bị giữ, user chờ vô hạn và queue nghẽn. Luôn đặt timeout theo route/task.

**Fix:** Đặt timeout theo route/task. Dùng `timeout` parameter trong SDK.
</Callout>

<Callout type="warning" title="Pitfall 2: không có tool result validation">
**Triệu chứng:** Tool result có thể bị injection, sai format, hoặc không có permission. Validate tool result trước khi dùng.

**Fix:** Validate tool arguments, check permission, execute tool, validate result.
</Callout>

<Callout type="danger" title="Pitfall 3: retry không idempotency">
**Triệu chứng:** Retry gọi tool 2 lần, user nhận duplicate email/refund. Tool actions cần idempotency key.

**Fix:** Dùng idempotency key: `operation_id = hash(user_id + action_type + business_id)`.
</Callout>
