API Integration Patterns là bước biến LLM từ một "call" thành một **hệ thống có resilience**. Ở cấp độ giáo sư AI, bạn không chỉ "gọi provider rồi trả kết quả". Production cần middleware: token budget guard, retry policy, caching, fallback, circuit breaker, rate limit, observability. Mỗi layer là một decision point ảnh hưởng cost, latency, reliability và correctness.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Retry storm 3:00 AM, provider down, 50,000 USD/tháng wasted, 1,000 users bị duplicate refund">
**Bối cảnh:** Một team build chatbot RAG. Họ dùng OpenAI, không có retry policy, không có cache, không có fallback. Họ deploy production, app chạy ngon. Họ không có observability.

**Sự cố:** 3:00 AM, provider OpenAI bị down. App retry 3 lần, mỗi request. Họ set retry=True với exponential backoff. 3 phút, 1,000 requests, 50,000 USD. Họ không có fallback. Họ không có circuit breaker. Họ không có cache.

1. Không có retry policy — retry 3 lần, không có max attempts.
2. Không có circuit breaker — retry storm khi provider down.
3. Không có fallback — chỉ phụ thuộc OpenAI.
4. Không có cache — mỗi request gọi provider, không cache.
5. Không có observability — không biết retry storm, không biết cost.

**Hậu quả:** 50,000 USD/tháng wasted. Họ phải tắt chatbot, thiết kế lại retry policy, circuit breaker, fallback provider, cache. Hóa đơn 3x sau khi tối ưu.
</Callout>

```text
        ▼
[ Response validation ]
        │
        ▼

        │

        │
        ▼

```

Mỗi middleware là một decision point. Mỗi middleware có thể short-circuit, retry, cache, hoặc fallback.

## 5. Token budget guard

### Why token budget guard?

```text
system + developer + chat_history + retrieved_context + user_query + output_budget <= context_window
```

### Why token budget guard?

```python
def check_token_budget(prompt: str, model: str, max_tokens: int = 1000) -> bool:
    prompt_tokens = count_tokens(prompt, model)
    total = prompt_tokens + max_tokens
    return total <= model_context_window(model)
```

### Why token budget guard?

- Context overflow: model reject.
- Cost: input tokens tăng.
- Latency: input tokens tăng.
- Quality: context dài có thể lost-in-the-middle.

### Why token budget guard?

```python
def enforce_token_budget(prompt: str, model: str, max_tokens: int = 1000) -> str:
    prompt_tokens = count_tokens(prompt, model)
    context_window = model_context_window(model)
    if prompt_tokens + max_tokens > context_window:
        return truncate_prompt(prompt, context_window - max_tokens)
    return prompt
```

## 6. Retry policy

### Why retry policy?

```text
[ Provider call ]
        │
        ▼
[ Error? ]
        │
        ├── retryable → retry
        │
        ├── non-retryable → fail
        │
        ▼
[ Return or fallback ]
```

### Why retry policy?

| Error | Retry? | Strategy |
|---|---|---|
| 400 invalid request | no | fix payload |
| 401/403 auth | no | check key |
| 408 timeout | yes | retry with timeout |
| 429 rate limit | yes | backoff + jitter |
| 500/502/503/504 | yes | retry then fallback |
| context length | no | reduce context |
| invalid JSON output | repair | schema repair loop |

### Why retry policy?

```python
import time
import random


def call_with_retry(
    provider,
    messages,
    max_attempts=3,
    base_delay=0.1,
    max_delay=10.0,
):
    for attempt in range(max_attempts):
        try:
            return provider.chat(messages)
        except RetryableError:
            if attempt == max_attempts - 1:
                raise
            delay = min(base_delay * (2 ** attempt), max_delay)
            delay += random.uniform(0, 0.1)
            time.sleep(delay)
```

### Why retry policy?

- 429: backoff + jitter.
- 500/502/503/504: retry then fallback.
- 408: timeout.
- Context length: reduce context.
- Invalid JSON: repair.

## 7. Caching

### Why caching?

| Cache type | Lợi ích | Hạn chế |
|---|---|---|
| Exact match | đơn giản | không match semantically |
| Semantic | match semantically | phức tạp, cost embeddings |
| Prefix | cache prefix | không match semantically |
| Response | cache response | không match semantically |

### Why caching?

```python
import hashlib
import json


def cache_key(prompt: str, model: str, params: dict) -> str:
    data = {"prompt": prompt, "model": model, "params": params}
    return hashlib.sha256(json.dumps(data, sort_keys=True).encode()).hexdigest()


def get_cached_response(cache, key: str):
    return cache.get(key)


def set_cached_response(cache, key: str, response: dict, ttl: int = 3600):
    cache.set(key, response, ttl=ttl)
```

### Why caching?

- Exact match: hash prompt + model + params.
- Semantic: embeddings + similarity.
- Prefix: cache prefix (prompt caching).
- Response: cache response.

```python
def semantic_cache_lookup(query: str, cache: list[dict], threshold: float = 0.95):
```

## 9. Fallback

### Why fallback?

```text
[ Provider A ]
        │
        ├── fail
        │
        ▼
[ Provider B ]
        │
        ├── fail
        │
        ▼
[ Provider C ]
        │
        ├── fail
        │
        ▼
[ Default response ]
```

### Why fallback?

```python
class FallbackChain:
    def __init__(self, providers: list):
        self.providers = providers

    def chat(self, messages: list[dict]) -> LLMResponse:
        for provider in self.providers:
            try:
                return provider.chat(messages)
            except Exception:
                continue
        raise RuntimeError("all providers failed")
```

### Why fallback?

- Reliability: provider A fail, provider B.
- Cost: provider A cheap, provider B expensive.
- Quality: provider A strong, provider B weak.
- Latency: provider A fast, provider B slow.

## 10. Circuit breaker

### Why circuit breaker?

```text
[ Provider call ]
        │
        ▼
[ Error count > threshold? ]
        │
        ├── yes → open circuit
        │
        ├── no → closed circuit
        │
        ▼
[ Circuit open? ]
        │
        ├── yes → fail fast
        │
        ├── no → call provider
        │
        ▼
[ Circuit half-open? ]
        │
        ├── yes → test call
        │
        ├── no → call provider
        │
        ▼
[ Return or fallback ]
```

### Why circuit breaker?

```python
class CircuitBreaker:
    def __init__(self, threshold: int = 5, reset_timeout: int = 60):
        self.threshold = threshold
        self.reset_timeout = reset_timeout
        self.failure_count = 0
        self.state = "closed"
        self.last_failure_time = None

    def record_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.threshold:
            self.state = "open"

    def record_success(self):
        self.failure_count = 0
        self.state = "closed"

    def can_call(self) -> bool:
        if self.state == "closed":
            return True
        if self.state == "open":
            if time.time() - self.last_failure_time > self.reset_timeout:
                self.state = "half-open"
                return True
            return False
        return True
```

### Why circuit breaker?

- Prevent retry storm.
- Fail fast when provider down.
- Allow recovery.
- Reduce load on provider.

## 11. Rate limiting

### Why rate limiting?

| Limit | Mục đích |
|---|---|
| TPM (tokens per minute) | token budget |
| RPM (requests per minute) | request count |
| Concurrent | concurrency limit |
| Per-user | per-user quota |
| Per-tenant | per-tenant quota |
| Global | global quota |

### Why rate limiting?

```python
from collections import defaultdict
import time


class RateLimiter:
    def __init__(self, max_requests: int, window_seconds: int = 60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = defaultdict(list)

    def allow(self, key: str) -> bool:
        now = time.time()
        self.requests[key] = [t for t in self.requests[key] if t > now - self.window]
        if len(self.requests[key]) >= self.max_requests:
            return False
        self.requests[key].append(now)
        return True
```

### Why rate limiting?

- Per-user: 100 requests/min.
- Per-tenant: 1,000 requests/min.
- Global: 10,000 requests/min.
- TPM: 1,000,000 tokens/min.
- Concurrent: 10 requests.

## 15. Streaming

### Why streaming?

```text
Non-streaming: wait for all output → return
Streaming:     return token-by-token
```

### Why streaming?

- TTFT giảm.
- User experience tốt hơn.

- Server có thể abort sớm.

### Why streaming?

```python
def stream_chat(provider, messages):
    stream = provider.chat(messages, stream=True)
    for chunk in stream:
        if chunk.choices[0].delta.content:
            yield chunk.choices[0].delta.content
```

```python
def make_idempotent_request(
    provider, messages, idempotency_key: str, idempotency_store: dict
):
    if idempotency_key in idempotency_store:
        return idempotency_store[idempotency_key]
    response = provider.chat(messages)
    idempotency_store[idempotency_key] = response
    return response
```

### Why idempotency?

## 18. Provider abstraction

### Why provider abstraction?

```python
class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list[dict]) -> LLMResponse:
        pass


class OpenAIProvider(LLMProvider):
    def __init__(self, client):
        self.client = client

    def chat(self, messages: list[dict]) -> LLMResponse:
        resp = self.client.chat.completions.create(
            model="gpt-4o-mini",
            messages=messages,
        )
        return LLMResponse(
            text=resp.choices[0].message.content,
            model=resp.model,
            input_tokens=resp.usage.prompt_tokens,
            output_tokens=resp.usage.completion_tokens,
        )
```

### Why provider abstraction?

- Switch provider easily.
- Test with mock provider.
- Fallback to other provider.
- Different model for different task.
- Cost tracking.

## Implementation Playbook: Resilient API Pipeline

```python
import time
import hashlib
import random
from dataclasses import dataclass
from abc import ABC, abstractmethod


@dataclass
class LLMResponse:
    text: str
    model: str
    input_tokens: int
    output_tokens: int
    latency_ms: int
    cache_hit: bool = False
    retry_count: int = 0
    fallback_path: list[str] = None


class LLMProvider(ABC):
    @abstractmethod
    def chat(self, messages: list[dict]) -> LLMResponse:
        pass


class CircuitBreaker:
    def __init__(self, threshold: int = 5, reset_timeout: int = 60):
        self.threshold = threshold
        self.reset_timeout = reset_timeout
        self.failure_count = 0
        self.state = "closed"
        self.last_failure_time = 0

    def record_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.threshold:
            self.state = "open"

    def record_success(self):
        self.failure_count = 0
        self.state = "closed"

    def can_call(self) -> bool:
        if self.state == "closed":
            return True
        if self.state == "open":
            if time.time() - self.last_failure_time > self.reset_timeout:
                self.state = "half-open"
                return True
            return False
        return True


class RateLimiter:
    def __init__(self, max_requests: int, window_seconds: int = 60):
        self.max_requests = max_requests
        self.window = window_seconds
        self.requests = {}

    def allow(self, key: str) -> bool:
        now = time.time()
        self.requests[key] = [t for t in self.requests.get(key, []) if t > now - self.window]
        if len(self.requests[key]) >= self.max_requests:
            return False
        self.requests[key].append(now)
        return True


class ResilientPipeline:
    def __init__(self, providers: list[LLMProvider], cache: dict, rate_limiter: RateLimiter, circuit_breaker: CircuitBreaker):
        self.providers = providers
        self.cache = cache
        self.rate_limiter = rate_limiter
        self.circuit_breaker = circuit_breaker

    def chat(self, messages: list[dict], user_id: str, max_attempts: int = 3) -> LLMResponse:
        if not self.rate_limiter.allow(user_id):
            return LLMResponse(
                text="Rate limit exceeded",
                model="rate_limiter",
                input_tokens=0,
                output_tokens=0,
                latency_ms=0,
            )

        key = hashlib.sha256(str(messages).encode()).hexdigest()
        if key in self.cache:
            return self.cache[key]

        if not self.circuit_breaker.can_call():
            return LLMResponse(
                text="Service unavailable",
                model="circuit_breaker",
                input_tokens=0,
                output_tokens=0,
                latency_ms=0,
            )

        for provider in self.providers:
            for attempt in range(max_attempts):
                try:
                    start = time.time()
                    response = provider.chat(messages)
                    response.latency_ms = int((time.time() - start) * 1000)
                    response.retry_count = attempt
                    self.circuit_breaker.record_success()
                    self.cache[key] = response
                    return response
                except Exception:
                    if attempt == max_attempts - 1:
                        self.circuit_breaker.record_failure()
                        continue
                    delay = min(0.1 * (2 ** attempt), 10.0)
                    delay += random.uniform(0, 0.1)
                    time.sleep(delay)

        return LLMResponse(
            text="Service unavailable",
            model="fallback",
            input_tokens=0,
            output_tokens=0,
            latency_ms=0,
            fallback_path=[p.__class__.__name__ for p in self.providers],
        )
```

**Triệu chứng:** Retry không có max attempts, không có backoff, không có jitter. Retry storm khi provider down.
