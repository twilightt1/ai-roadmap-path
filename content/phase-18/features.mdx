Bạn đã có overview, component boundaries và stack decisions. Bài cuối của Capstone biến tất cả thành implementation roadmap: build feature nào trước, acceptance criteria ra sao, demo cuối cùng cần chứng minh gì, và làm sao biến project này thành portfolio-grade evidence cho năng lực AI Engineer production.

Đây là backlog chính thức cho Capstone. Nếu hoàn thành phần lớn acceptance criteria dưới đây, bạn không chỉ có một app AI; bạn có một nền tảng có thể trình bày trong interview, demo cho khách hàng hoặc dùng làm foundation cho sản phẩm thật.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Capstone có nhiều code nhưng không demo được production value">
**Bối cảnh:** Một developer dành 2 tháng xây AI platform. Repo có nhiều endpoints, UI đẹp, vài chain RAG và agent demo.

**Sự cố:** Khi trình bày, project không chứng minh được các năng lực production:

1. Không có acceptance criteria cho từng feature.
2. Không có demo script end-to-end.
3. Không có cost dashboard hoặc trace để debug response.
4. Không chứng minh được tenant isolation.
5. Không có safety/red-team regression.
6. Không có deployment hoặc runbook.

**Hậu quả:** Người xem thấy đây là “một app AI nhiều tính năng” nhưng không thấy năng lực platform engineering. Capstone thất bại ở portfolio storytelling, dù code nhiều.
</Callout>

Bài học rút ra: Capstone cần backlog, acceptance criteria và demo narrative. Production value phải được chứng minh, không chỉ được mô tả.

## 3. Feature 1: Multi-Tenant User Management

Requirements:

- Tenants table.
- Users belong to tenants.
- Roles: owner/admin/member/viewer.
- Tenant settings: allowed providers, budgets, safety policy, data retention.
- Every request resolves tenant context.
- Row-level filters or service-level tenant guards.

Acceptance criteria:

| Check | Expected |
|---|---|
| User A accesses Tenant B conversation | Denied |
| Tenant A document retrieval | Only Tenant A chunks returned |
| Admin usage dashboard | Filterable by tenant |
| Cache keys | Include tenant ID |
| Audit logs | Include tenant ID and user ID |

Test cases:

```text
- create tenant_a and tenant_b
- upload docs for both
- login user_a
- query document from tenant_b by ID
- assert 403 or no result
```

## 4. Feature 2: Per-User API Key Management

Two modes:

| Mode | Description | Use case |
|---|---|---|
| Platform keys | Platform pays provider bills | SaaS model, simpler UX |
| BYOK | Tenant/user supplies provider key | Enterprise privacy/cost control |

Requirements:

- Encrypted provider keys.
- Never expose keys to frontend.
- Key validation endpoint.
- Rotation support.
- Disable provider per tenant.
- Audit failed key usage without logging key.

Acceptance criteria:

- Provider key is encrypted at rest.
- Frontend never receives raw key.
- Tenant can revoke key and future calls fail gracefully.
- Router respects tenant provider allowlist.

## 5. Feature 3: Streaming AI Chat with Memory

Requirements:

- `/v1/chat` for non-streaming.
- `/v1/chat/stream` for SSE streaming.
- Conversation and messages stored in PostgreSQL.
- Redis for hot conversation/session state.
- Long conversation summarization or windowing.
- Token usage logged per message/model call.

Acceptance criteria:

| Check | Expected |
|---|---|
| First token | Streams before full answer completes |
| Conversation resume | Works by conversation ID |
| Usage logging | Input/output tokens stored |
| Cancellation | Client disconnect stops generation or marks run canceled |
| Trace ID | Visible in response/debug UI |

<Callout type="tip" title="Streaming là UX và observability feature">
Streaming không chỉ làm chat cảm giác nhanh hơn. Nó buộc bạn xử lý cancellation, partial output moderation, trace IDs và lifecycle events đúng cách.
</Callout>

---

## 6. Feature 4: Intelligent Model Routing with Cost Optimization

Requirements:

- Model registry with cost/context/capabilities.
- Task classifier or heuristic router.
- Tenant provider policies.
- Token/cost estimate before call.
- Fallback chains.
- Circuit breaker for provider failures.
- Cost log after call.

Routing examples:

| Task | Preferred route |
|---|---|
| Simple rewrite | Cheap fast model |
| Long document analysis | Long-context model |
| Code explanation | Code-capable model |
| High-risk answer | Safer model + stricter output checks |
| Provider outage | Fallback model/provider |

Acceptance criteria:

- Simple tasks route to cheaper model.
- Long-context tasks avoid models with insufficient context window.
- Provider outage triggers fallback.
- Every route decision is logged with reason.
- Cost per request is visible in dashboard.

## 7. Feature 5: RAG Pipeline with Multiple Document Types

Requirements:

- Upload PDF/DOCX/TXT/Markdown/web URL.
- Parse and normalize text.
- Chunk documents.
- Generate embeddings.
- Store documents/chunks/embeddings with tenant metadata.
- Retrieve with vector/hybrid search.
- Return citations.
- Refuse or ask clarification when evidence is weak.

Acceptance criteria:

| Check | Expected |
|---|---|
| Tenant isolation | No cross-tenant chunks |
| Citation output | Answer includes source IDs/titles |
| Low evidence | Model abstains or asks clarification |
| Delete document | Chunks/embeddings removed |
| RAG eval | Golden Q&A has faithfulness/citation metrics |

RAG golden eval example:

```json
{
  "question": "What is the refund window for Pro plan?",
  "expected_source": "refund_policy_v5.md#section-2.1",
  "must_contain": ["14 days", "Pro plan"],
  "must_not_contain": ["30 days"]
}
```

## 8. Feature 6: Prompt Version Management

Requirements:

- Prompt templates table.
- Prompt versions immutable.
- Changelog.
- Variables schema.
- Rollout percentage.
- Rollback without deploy.
- Every model call stores prompt version.

Acceptance criteria:

- Admin can create new prompt version.
- Requests log prompt version.
- Rollback changes behavior without code deploy.
- A/B experiment can assign traffic between prompt versions.

Prompt version schema:

```json
{
  "template_id": "rag_answer",
  "version": 7,
  "system_prompt": "Answer with citations. Abstain if evidence is weak.",
  "variables_schema": ["tenant_policy", "retrieved_context", "user_question"],
  "changelog": "Added abstention rule for low evidence.",
  "created_by": "admin_123"
}
```

## 9. Feature 7: A/B Testing for Prompts and Models

Requirements:

- Experiment config.
- Traffic split.
- Assignment persistence.
- Metrics collection:
  - latency
  - cost
  - user rating
  - task success
  - safety blocks
  - fallback rate
- Winner decision report.

Acceptance criteria:

- Same user gets stable variant during experiment.
- Dashboard compares variant metrics.
- Experiment logs model/prompt version.
- Safety regression can stop experiment.

<Callout type="warning" title="A/B test không chỉ đo thumbs-up">
Thumbs-up có thể reward verbosity hoặc sycophancy. Kết hợp user feedback với task metrics, safety, cost và latency.
</Callout>

## 10. Feature 8: Agent Workflows + MCP Tools

Requirements:

- LangGraph-style workflow.
- MCP tool registry.
- Tool schema validation.
- Tenant-specific allowlist.
- Step/time/cost budgets.
- Human approval gates.
- Agent run status endpoint.
- Tool call audit logs.

Acceptance criteria:

| Check | Expected |
|---|---|
| Unauthorized tool | Blocked server-side |
| Long-running run | Status visible by run ID |
| Cost budget exceeded | Agent stops with reason |
| Human approval required | Run pauses and resumes |
| Tool output injection | Sanitized/treated as untrusted data |

Demo workflow:

```text
User: Research competitor pricing and draft summary.
Planner → web_search MCP → document_reader MCP → critic → synthesizer → safety review → final answer
```

## 11. Feature 9: Safety, Privacy, Red-Team Regression

Requirements:

- Input moderation.
- Prompt injection detection.
- PII masking before provider/logging.
- Output moderation.
- RAG citation/evidence checks.
- Tool permission enforcement.
- Safety events table.
- Red-team test suite in CI.

Acceptance criteria:

- Indirect prompt injection in uploaded doc is detected or neutralized.
- PII is masked according to policy.
- Fallback model goes through same safety checks.
- Critical red-team regression fails CI.
- Safety event appears in admin dashboard.

Red-team test categories:

- Direct jailbreak.
- Indirect RAG injection.
- Tool abuse.
- Data exfiltration.
- PII leak.
- Over-refusal false positive.

## 12. Feature 10: Cost Tracking, Usage Limits, Billing

Requirements:

- Model call usage events.
- Per-provider pricing table.
- Cost estimate and actual cost.
- Per-tenant monthly budget.
- Soft alerts and hard limits.
- Downgrade route when budget near limit.

Acceptance criteria:

- Dashboard shows cost by tenant/user/feature/model.
- Budget exceeded blocks or downgrades requests.
- Cost estimate error is tracked.
- Heavy users are visible.

Usage event schema:

```json
{
  "tenant_id": "tenant_acme",
  "user_id": "user_123",
  "feature": "rag_chat",
  "provider": "openai",
  "model": "gpt-4o-mini",
  "input_tokens": 1800,
  "output_tokens": 420,
  "cost_usd": 0.0062,
  "latency_ms": 1420,
  "created_at": "2026-06-30T12:00:00Z"
}
```

## 18. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="danger" title="Pitfall 1: Feature breadth without depth">
**Triệu chứng:** Platform có nhiều menu nhưng mỗi feature chỉ là mock/demo.

**Nguyên nhân:** Build ngang quá rộng, không có vertical slice production-grade.

**Cách khắc phục:** Ưu tiên one complete flow: tenant → chat/RAG → router → safety → cost → trace → admin.
</Callout>

<Callout type="warning" title="Pitfall 2: Acceptance criteria không test được">
**Triệu chứng:** README nói “supports multi-tenancy” nhưng không có test chứng minh.

**Nguyên nhân:** Criteria viết mơ hồ.

**Cách khắc phục:** Mỗi critical claim cần test hoặc demo evidence: cross-tenant access denied, fallback triggered, safety event logged.
</Callout>

<Callout type="warning" title="Pitfall 3: Dashboard đẹp nhưng không có data thật">
**Triệu chứng:** Admin UI nhìn tốt nhưng numbers hard-coded hoặc không drill-down được.

**Nguyên nhân:** Frontend build trước observability data model.

**Cách khắc phục:** Instrument backend first. Dashboard phải đọc từ usage_events, model_calls, traces, safety_events.
</Callout>

<Callout type="danger" title="Pitfall 4: Không có final narrative">
**Triệu chứng:** Người xem không hiểu vì sao project này khó và đáng giá.

**Nguyên nhân:** Demo thiếu story, README thiếu architecture/decisions/trade-offs.

**Cách khắc phục:** Chuẩn bị demo script, diagrams, ADRs, screenshots và incident story như một portfolio case study.
</Callout>

## 19. Platform Unit Economics

| Feature | Cost driver | Control |
|---|---|---|
| Chat | output tokens | max tokens, cheap model route |
| RAG | embeddings, reranking, context length | batch ingestion, rerank top-k only, context compression |
| Agents | repeated model/tool calls | step/cost budgets, caching, HITL |
| Safety | classifiers, judge models, human review | risk-tiered cascade |
| Observability | trace/log storage | retention, sampling, redaction |
| Admin analytics | DB query load | rollups/materialized views |

Capstone phải chứng minh unit economics: mỗi feature có cost driver, metric và control.
