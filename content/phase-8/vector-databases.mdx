Vector Database không chỉ là nơi “lưu embeddings”. Nó là retrieval infrastructure: index type, similarity metric, metadata filtering, tenancy, latency, consistency, backup, hybrid search và operational behavior quyết định RAG có production được hay không.

## Why this matters

Local demo với 1,000 chunks khác hoàn toàn production với 10 triệu chunks:

- Exact scan quá chậm.
- ANN index có thể trade recall lấy latency.
- Metadata filter sai làm mất relevant docs hoặc leak tenant data.
- Delete không propagate có thể retrieve tài liệu đã xóa.
- Dense-only search miss exact IDs.
- Managed vector DB có cost/vendor lock-in; self-host có ops burden.

<Callout type="warning" title="Vector DB là search infrastructure">
Đừng chọn vector DB chỉ vì tutorial. Hãy chọn theo workload: scale, filter semantics, hybrid search, tenancy, latency SLA, ops capability và cost.
</Callout>

## Core intuition: nearest neighbor search at scale

```text
query text → query vector → vector index → nearest candidate chunks → filters/rerank → top-k evidence
```

Vector DB tối ưu câu hỏi:

```text
Trong hàng triệu vectors, vectors nào gần query vector nhất, dưới metadata constraints nào?
```

## What a vector DB stores

Một record production thường gồm:

```json
{
  "id": "policy_123#c007",
  "vector": [0.012, -0.044, 0.19],
  "text": "Refunds are available within 30 days...",
  "metadata": {
    "doc_id": "policy_123",
    "tenant_id": "tenant_a",
    "acl_groups": ["support"],
    "page": 12,
    "section": "Refunds",
    "document_version": "v3",
    "embedding_model": "text-embedding-3-large"
  }
}
```

Nếu DB chỉ lưu vector mà không lưu metadata, production RAG sẽ khó citation, filtering, debug và compliance.

## Exact vs approximate search

Exact search so query với mọi vector.

```text
O(N × dimensions)
```

Chính xác nhưng chậm khi N lớn.

Approximate nearest neighbor (ANN) dùng index để tìm gần đúng nhanh hơn.

Tradeoff:

```text
lower latency ↔ maybe lower recall
```

Professor-level point: vector DB tuning là bài toán recall-latency-memory, không có setting tốt nhất cho mọi workload.

## HNSW intuition

HNSW — Hierarchical Navigable Small World — xây graph liên kết vectors gần nhau. Search đi qua graph thay vì scan toàn bộ.

```text
entry point → navigate graph layers → local nearest neighbors
```

Key parameters:

| Param | Ý nghĩa | Tradeoff |
|---|---|---|
| `M` | số edges/neighbors mỗi node | cao hơn = recall/memory cao hơn |
| `efConstruction` | effort khi build index | cao hơn = build chậm hơn, recall tốt hơn |
| `efSearch` | effort khi query | cao hơn = query chậm hơn, recall tốt hơn |

<Callout type="tip" title="Tune efSearch bằng eval set">
Tăng `efSearch` có thể cải thiện recall nhưng tăng latency. Hãy vẽ curve recall@k vs p95 latency thay vì chọn theo cảm giác.
</Callout>

## IVF and product quantization overview

Một số index dùng clustering/quantization:

- IVF chia vector space thành clusters, search một số clusters gần query.
- PQ nén vectors để giảm memory.

Tradeoff:

- memory thấp hơn,
- search nhanh hơn,
- recall có thể giảm,
- tuning phức tạp hơn.

## Metadata filtering

Filtering examples:

```text
tenant_id = current_tenant
acl_groups intersects user_groups
doc_type = "policy"
updated_at >= 2026-01-01
```

### Pre-filter vs post-filter

Pre-filter: filter trước hoặc trong vector search.

- tốt cho security,
- có thể ảnh hưởng recall/latency tùy DB.

Post-filter: search rồi lọc.

- dễ implement,
- nguy hiểm nếu unauthorized candidates được retrieve vào app memory,
- có thể top-k sau filter quá ít.

<Callout type="danger" title="Tenant filter phải là hard filter">
Multi-tenant RAG không được dựa vào prompt để ngăn leak. Tenant/ACL filter phải xảy ra ở retrieval layer.
</Callout>

## Hybrid search

Dense embeddings mạnh semantic, sparse BM25 mạnh exact terms.

Hybrid:

```text
sparse_results = BM25(query)
dense_results = vector_search(embed(query))
merged = fuse(sparse_results, dense_results)
```

Reciprocal Rank Fusion (RRF) intuition:

```text
score(doc) = Σ 1 / (k + rank_i(doc))
```

Dùng hybrid khi corpus có:

- product names,
- error codes,
- API names,
- numbers,
- legal clauses,
- acronyms.

## Index lifecycle

Production index không static.

Lifecycle:

```text
create → bulk import → query → upsert → delete → rebuild → migrate → backup/restore
```

Operations:

- upsert changed chunks,
- tombstone deleted docs,
- rebuild after embedding model change,
- reindex after chunking strategy change,
- backup metadata and source references,
- verify deletes propagated.

## Operational concerns

Monitor:

- p50/p95/p99 retrieval latency,
- recall@k on eval set,
- index memory,
- build time,
- query throughput,
- filter selectivity,
- empty result rate,
- delete propagation lag,
- cost per million vectors.

## Common mistakes & debugging checklist

<Callout type="warning" title="Mistake 1: dùng vector-only cho docs có nhiều exact IDs">
Dense search có thể miss error codes/API names. Hybrid search thường cần thiết cho enterprise docs.
</Callout>

<Callout type="warning" title="Mistake 2: không test metadata filters">
Filter sai có thể làm mất relevant docs hoặc leak docs không được phép.
</Callout>

<Callout type="danger" title="Mistake 3: delete source doc nhưng vector vẫn còn">
Nếu deleted content vẫn retrieve được, đó là data retention/security incident. Delete propagation phải được test.
</Callout>

Checklist:

1. Metric config có đúng với embeddings không?
2. Có tenant/ACL filters không?
3. Có eval recall@k không?
4. Có test delete/upsert không?
5. Có hybrid search cho exact terms không?
6. Có backup/rebuild plan không?
7. Có embedding version metadata không?
8. Có latency/cost monitoring không?

## Answer key

1. Text/payload, metadata, doc/chunk IDs, tenant/ACL, version, embedding model info.
2. Giảm latency nhưng có thể giảm recall.
3. Graph neighbors nhiều tầng để navigate tới vectors gần query.
4. Filter sai gây miss relevant docs hoặc leak unauthorized docs.
5. Kết hợp exact keyword và semantic retrieval.
6. Re-embed và rebuild/update index, không trộn vector spaces.
7. Unauthorized candidates có thể được retrieve trước khi lọc và top-k sau lọc có thể thiếu kết quả.
