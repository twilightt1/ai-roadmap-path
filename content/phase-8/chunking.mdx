Chunking là information architecture của RAG. Bạn không retrieve "document"; bạn retrieve **đơn vị bằng chứng**. Chunk quá nhỏ mất ngữ cảnh, chunk quá lớn làm retrieval nhiễu và đốt context budget. Một RAG system tốt bắt đầu bằng câu hỏi: **đơn vị kiến thức nào nên được retrieve để trả lời câu hỏi?**

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Fixed-size chunking cắt giữa table, LLM trả lời sai số liệu">
**Bối cảnh:** Một team build RAG trên 200 pricing documents. Mỗi doc có bảng pricing. Họ dùng fixed-size chunking 1000 chars, overlap 100. Mục tiêu: chatbot trả lời câu hỏi pricing.

**Sự cố:** Fixed-size chunking cắt giữa table. Một chunk chứa "Plan Price Refund Free 0 No Pro 20 Yes" — không có header, không có row mapping. LLM trả lời: "Free plan có refund." (Sai. Free plan không refund.) Table bị tách sai làm số liệu vô nghĩa.

**Hậu quả:** 23 user khiếu nại về pricing sai, 1 user bị quote sai giá, 2 support ticket.

1. Chunk quá nhỏ mất context, chunk quá lớn làm retrieval nhiễu và đốt context budget.
2. Table bị tách sai làm số liệu vô nghĩa. Heading/section bị mất nên chunk mơ hồ.
3. Chunking quyết định evidence unit. Nếu evidence unit sai, retrieval sai ngay từ data design.
</Callout>

## 2. Architecture Blueprint (Chunking Pipeline)

```text
[ Source document ]
        │
        ▼
[ Parse ]  ←  extract text, tables, code, images
        │
        ▼
[ Clean ]  ←  remove headers/footers, boilerplate
        │
        ▼
[ Chunking strategy ]  ←  fixed / sentence / recursive / structure-aware / semantic
        │
        ▼
[ Add metadata ]  ←  doc_id, chunk_id, section, page, offsets, version, ACL
        │
        ▼
[ Chunk + metadata ]  ←  evidence unit
```

Chunking quyết định evidence unit. Nếu evidence unit sai, retrieval sai ngay từ data design. Đừng xem chunking là preprocessing phụ; nó là kiến trúc thông tin của RAG.

---

## 3. Core Intuition: Retrieve the Smallest Sufficient Evidence Unit

```text
good chunk = self-contained enough + specific enough + citeable enough
```

Chunk tốt phải đủ thông tin để LLM trả lời, nhưng không quá dài để chứa nhiều noise.

```text
Too small → missing context
Too large → low precision + high token cost
```

---

## 4. Chunking Objectives

| Objective | Vì sao quan trọng |
|---|---|
| Self-contained meaning | LLM hiểu chunk không cần quá nhiều context ngoài |
| Retrieval specificity | embedding match chính xác hơn |
| Local context | giữ điều kiện, heading, definitions |
| Stable citations | cite đúng source/page/section |
| Token efficiency | pack được nhiều evidence trong context window |
| ACL/version support | production security và freshness |

## 5. Fixed-Size Chunking

Cắt theo số tokens/chars cố định.

Ưu điểm:

- đơn giản,
- dễ implement,
- baseline tốt.

Nhược điểm:

- có thể cắt giữa câu/bảng/code,
- mất heading,
- không tôn trọng document structure.

```python
def fixed_chunks(text: str, size: int = 1000, overlap: int = 100):
    chunks = []
    start = 0
    while start < len(text):
        end = start + size
        chunks.append(text[start:end])
        start = end - overlap
    return chunks
```

## 6. Sentence and Paragraph Chunking

Cắt theo ranh giới tự nhiên hơn:

```text
paragraphs → group until token budget
```

Tốt cho docs prose, policy, articles.

Hạn chế:

- paragraph đơn lẻ có thể thiếu heading,
- section dài vẫn cần split,
- sentence segmentation có thể sai với abbreviations/OCR.

---

## 7. Recursive Character Splitting

Recursive splitter thử separators theo thứ tự:

```text
heading → paragraph → sentence → whitespace → character
```

Ý tưởng: giữ boundary tự nhiên nhất có thể, chỉ fallback xuống boundary nhỏ hơn khi cần.

Pseudo:

```python
SEPARATORS = ["\n## ", "\n\n", ". ", " ", ""]
```

Đây là default phổ biến trong LangChain-style workflows.

---

## 8. Structure-Aware Chunking

Document có structure. Hãy dùng nó.

Examples:

| Document | Structure signal |
|---|---|
| Markdown | headings, code blocks, tables |
| HTML | h1/h2/h3, article tags, DOM sections |
| PDF | pages, layout, headers/footers |
| DOCX | heading styles, tables |
| Legal docs | clauses, sections, definitions |
| API docs | endpoints, parameters, examples |

Structure-aware chunk giữ heading path:

```text
Refund Policy > Enterprise Plan > Exceptions
```

vào metadata hoặc chunk text.

---

## 9. Semantic Chunking

Semantic chunking tách theo topic shifts. Một cách làm:

1. Split document thành paragraphs.
2. Embed từng paragraph.
3. Tính similarity paragraph i và i+1.
4. Split khi similarity giảm mạnh.

Ưu điểm:

- topic-coherent,
- tránh split giữa ý.

Nhược điểm:

- tốn embeddings,
- ít deterministic,
- khó debug,
- cần threshold tuning.

---

## 10. Parent-Child Chunking

Ý tưởng:

```text
small child chunks for retrieval
larger parent chunks for context
```

Flow:

```text
Retrieve child chunk → return parent section to LLM
```

Tốt khi small chunks match query tốt nhưng answer cần section lớn hơn.

---

## 11. Chunk Size vs Overlap

| Chunk size | Best for | Risk |
|---:|---|---|
| 100–250 tokens | FAQs, short facts | thiếu context |
| 300–600 tokens | support docs, policy snippets | cần overlap vừa đủ |
| 800–1200 tokens | technical docs, sections | có thể nhiễu |
| 1500–2500 tokens | legal/long-form sections | tốn context, lower precision |

Overlap:

- giúp giữ context ở boundary,
- tăng số chunks/cost,
- có thể tạo duplicate top-k.

<Callout type="tip" title="Chunk size là hyperparameter cần eval">
Không có chunk size tốt nhất. Hãy benchmark recall@k và answer faithfulness theo corpus/task của bạn.
</Callout>

## 15. Special Case: Scanned PDFs/OCR

OCR có lỗi:

- ký tự sai,
- thứ tự cột sai,
- header/footer lặp,
- page number noise,
- hyphenation.

Chunking nên chạy sau cleaning/layout parsing, không chỉ raw OCR text.

## 16. Chunking Evaluation

Evaluate chunking bằng retrieval, không bằng cảm giác.

Metrics:

- recall@k theo query set,
- citation usefulness,
- duplicate rate trong top-k,
- average context tokens,
- answer faithfulness,
- human review of top failures.

Compare strategies:

```text

recursive-800-overlap100
heading-aware-v2
semantic-threshold-0.72
```

## Implementation Playbook: Chunking Bakeoff

```python
from dataclasses import dataclass


@dataclass
class Chunk:
    chunk_id: str
    doc_id: str
    text: str
    metadata: dict


def paragraph_chunks(doc_id: str, text: str, max_chars=1200):
    paragraphs = [p.strip() for p in text.split("\n\n") if p.strip()]
    chunks = []
    current = []
    current_len = 0

    for p in paragraphs:
        if current and current_len + len(p) > max_chars:
            idx = len(chunks)
            chunks.append(Chunk(f"{doc_id}#c{idx}", doc_id, "\n\n".join(current), {}))
            current = []
            current_len = 0
        current.append(p)
        current_len += len(p)

    if current:
        idx = len(chunks)
        chunks.append(Chunk(f"{doc_id}#c{idx}", doc_id, "\n\n".join(current), {}))
    return chunks
```

1. Chunk có self-contained không?
2. Có giữ heading/section context không?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Chunking theo chars mà không biết token budget">
**Triệu chứng:** Chunk 1000 chars, nhưng LLM context tính bằng tokens. 1000 chars có thể là 200-400 tokens, không đủ.

**Fix:** LLM context tính bằng tokens, không phải characters. Char-based splitter cần token-aware guard.
</Callout>

<Callout type="warning" title="Pitfall 2: Mất metadata source/page/section">
**Triệu chứng:** Citation yếu, không thể debug retrieval, không biết chunk thuộc section nào.

**Fix:** Không có metadata, citation yếu và debugging retrieval gần như không thể. Lưu metadata với mỗi chunk.
</Callout>

<Callout type="danger" title="Pitfall 3: Chunking bỏ qua ACL/version">
**Triệu chứng:** Production RAG retrieve dữ liệu cấm hoặc stale, không có version metadata.

**Fix:** Production RAG cần biết chunk thuộc tenant/doc version/permission nào. Nếu không, bạn có thể retrieve dữ liệu cấm hoặc stale.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Fixed-size chunking | đơn giản, dễ implement | phá structure, mất heading |
| Structure-aware chunking | giữ heading, section | phức tạp hơn |
| Semantic chunking | topic-coherent | tốn embeddings, khó debug |
| Parent-child chunking | retrieve small, context large | storage, complexity |
| Chunk size cao | nhiều context | nhiễu, tốn token |
| Overlap cao | giữ context boundary | duplicate top-k, cost |
| Table-aware parsing | table meaning | parsing complexity |

Chunking là information architecture của RAG. Mục tiêu là retrieve smallest sufficient evidence unit. Đừng xem chunking là preprocessing phụ.

## Answer key

1. Đủ context nhưng đủ specific, citeable và token-efficient.
2. Mất điều kiện/heading/context cần để answer đúng.
3. Nhiễu retrieval, tốn context, giảm precision.
4. Thử boundary lớn/tự nhiên trước, fallback dần xuống boundary nhỏ hơn.
5. Topic-coherent nhưng tốn compute, khó debug/tune.
6. Retrieve small chunk chính xác nhưng cung cấp parent context rộng hơn cho LLM.
7. doc_id, chunk_id, source, page/section, offsets, version, strategy, tenant/ACL.
