Production RAG là distributed data system cộng với probabilistic generation system. Vấn đề khó không phải trả lời đúng một câu hỏi demo; vấn đề là cập nhật tài liệu, xóa dữ liệu, enforce ACL, isolate tenants, monitor retrieval quality, kiểm soát cost và chứng minh answer grounded ở quy mô lớn.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Xóa tài liệu nhưng vector vẫn còn, 3,000 user thấy tài liệu đã xóa">
**Bối cảnh:** Một team build enterprise RAG. Họ có 10,000 documents. Khách hàng yêu cầu xóa tài liệu (GDPR right-to-erasure). Team xóa document khỏi source, nhưng **không xóa** vector khỏi index. Mục tiêu: deleted content không được retrieve.

**Sự cố:** Document deleted khỏi source, nhưng vector index vẫn còn. RAG retrieve vector, LLM trả lời bằng deleted document. User thấy tài liệu đã xóa, khiếu nại. GDPR violation: deleted content không được retrieve.

**Hậu quả:** 3,000 user thấy tài liệu đã xóa, 1 GDPR violation, 1 incident report, 2 tuần fix.

1. RAG index là derived data, không phải source of truth. Source documents là nguồn sự thật. Vector index là dữ liệu dẫn xuất phải rebuild được, versioned, auditable và delete được khi source thay đổi.
2. Deleted docs vẫn retrieve được là security/compliance incident. Delete semantics phải được test như feature production.
3. Production RAG cần biết chunk thuộc tenant/doc version/permission nào. Nếu không, bạn có thể retrieve dữ liệu cấm hoặc stale.
</Callout>

## 3. Core Intuition: RAG Index Is a Derived Data Product

```text
Source documents = source of truth
Vector index     = derived state
```

Derived state phải có tính chất:

- rebuildable,
- versioned,
- auditable,
- delete-aware,
- permission-aware,
- monitored.

<Callout type="danger" title="RAG index là derived data, không phải source of truth">
Source documents là nguồn sự thật. Vector index là dữ liệu dẫn xuất phải rebuild được, versioned, auditable và delete được khi source thay đổi.
</Callout>

## 4. Incremental Indexing

Không reindex toàn bộ corpus mỗi lần.

Use content hash:

```text
content_hash = sha256(normalized_document_text)
```

Flow:

```text
fetch doc
  ↓
compute content_hash
  ↓
if unchanged: skip
if changed: parse → chunk → embed → upsert new chunks
if deleted: tombstone/delete chunks
```

Chunk-level hash giúp re-embed chỉ chunks thay đổi.

## 6. Delete Semantics

Delete khó hơn upsert.

Options:

| Strategy | Ý nghĩa | Risk |
|---|---|---|
| Hard delete | xóa vectors/metadata ngay | hard to audit if no tombstone |
| Soft delete | mark deleted, filter out | storage grows |
| Tombstone | record deletion event | needs compaction |
| Rebuild index | recreate from source truth | expensive but clean |

Production requirement:

```text
deleted source must not be retrievable
```

Test delete propagation định kỳ.

---

## 7. Multi-Tenant Isolation

Options:

| Approach | Pros | Cons |
|---|---|---|
| Separate indexes | strong isolation | more ops/cost |
| Namespaces/collections | good balance | DB-specific behavior |
| Metadata filters | flexible | filter bugs dangerous |
| Shared index no filter | never for private tenants | data leak risk |

Tenant ID must be hard filter.

<Callout type="danger" title="Prompt không phải access control">
Không retrieve cross-tenant docs rồi nhờ LLM không tiết lộ. ACL phải enforce trước chunks vào prompt.
</Callout>

---

## 8. ACL-Aware Retrieval

Metadata:

```json
{
  "tenant_id": "tenant_a",
  "acl_groups": ["legal", "managers"],
  "classification": "confidential"
}
```

Retrieval filter:

```text
tenant_id = user.tenant_id
AND acl_groups intersects user.groups
AND status = active
```

Prefer pre-filtering at vector DB/index layer. Post-filtering phải cực kỳ cẩn thận.

---

## 9. Caching in Production RAG

Cache layers:

| Cache | Key includes | Invalidate when |
|---|---|---|
| Query embedding | query text + model | embedding model changes |
| Retrieval results | query + filters + index version | docs/ACL/index changes |
| Answer cache | prompt + context + user/tenant + versions | docs/prompt/permissions change |
| Semantic cache | query embedding + tenant | risky; threshold/policy changes |

Never share personalized/private answer cache across tenants.

## 10. Monitoring Retrieval

Metrics:

- retrieval p95 latency,
- empty result rate,
- low-score result rate,
- top-k score distribution,
- reranker score distribution,
- average chunks per answer,
- context token usage,
- citation coverage,
- no-answer rate,
- retrieval error rate.

Drift signals:

- recall@k drops on nightly eval,
- empty result rate spikes,
- score distribution shifts,
- user feedback worsens,
- new docs not retrieved.

## 12. Fallback and "I Don't Know" Policy

No-answer triggers:

- no retrieved chunks,
- top score below threshold,
- reranker score below threshold,
- conflicting sources,
- citation validation fails,
- user lacks permission,
- source outdated.

Good behavior:

```text
I don't have enough information in the available sources to answer that.
```

Optional:

- suggest query refinement,
- escalate to human,
- trigger ingestion request.

## 13. Security Risks in Production RAG

| Risk | Example | Mitigation |
|---|---|---|
| Prompt injection from docs | doc says ignore instructions | treat retrieved content as data |
| Cross-tenant leak | wrong filter | tenant hard filters/tests |
| Source poisoning | malicious doc inserted | source validation/review |
| Sensitive logs | raw context logged | redaction/retention/access control |
| Citation spoofing | model cites wrong chunk | citation validation |
| Stale ACL | permission revoked but cache valid | permission version in cache key |

## Implementation Playbook: Enterprise RAG Skeleton

Design schemas:

```sql
-- conceptual schema
CREATE TABLE documents (
  doc_id TEXT PRIMARY KEY,
  tenant_id TEXT NOT NULL,
  source_uri TEXT NOT NULL,
  document_version TEXT NOT NULL,
  content_hash TEXT NOT NULL,
  status TEXT NOT NULL,
  source_updated_at TIMESTAMP,

CREATE TABLE chunks (
  chunk_id TEXT PRIMARY KEY,
  doc_id TEXT NOT NULL,
  tenant_id TEXT NOT NULL,
  section TEXT,
  page INT,
  content_hash TEXT NOT NULL,
  embedding_model TEXT NOT NULL,
  acl_groups TEXT[]
);
```

Lab steps:

1. Implement content hash change detection.
2. Upsert changed chunks.
3. Tombstone deleted docs.
4. Add tenant/ACL filter to retrieval.
5. Log retrieval trace.
6. Run 20-query eval nightly.
7. Build simple dashboard table.

Production readiness checklist:

1. Can rebuild index from source of truth?
2. Are document versions tracked?
3. Are deletes propagated and tested?
4. Is tenant filter mandatory?
5. Are ACL filters applied before retrieval?
6. Are retrieved chunk IDs logged?
7. Are citations validated?
8. Does the system abstain when evidence missing?
9. Are prompt injection docs tested?
10. Are evals run before deploy?
11. Are embedding/chunking versions tracked?
12. Are caches invalidated on doc/ACL changes?
13. Are logs redacted and access-controlled?

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Không version documents/chunks/index">
**Triệu chứng:** Answer sai hoặc stale, không biết dựa trên document version nào, embedding model nào, chunking strategy nào.

**Fix:** Khi answer sai hoặc stale, bạn cần biết nó dựa trên document version nào, embedding model nào và chunking strategy nào.
</Callout>

<Callout type="warning" title="Pitfall 2: Chỉ eval lúc launch">
**Triệu chứng:** RAG quality drift, answer sai, không ai biết.

**Fix:** RAG quality drift khi docs, queries, embeddings, chunking hoặc user behavior thay đổi. Cần eval liên tục.
</Callout>

<Callout type="danger" title="Pitfall 3: Delete không xóa khỏi vector index">
**Triệu chứng:** Deleted docs vẫn retrieve được, GDPR violation, security incident.

**Fix:** Deleted docs vẫn retrieve được là security/compliance incident. Delete semantics phải được test như feature production.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Incremental indexing | không reindex toàn bộ | content hash logic |
| Document versioning | audit, rollback | storage, metadata |
| Hard delete | clean | hard to audit |
| Soft delete + tombstone | auditable | storage grows |
| Separate indexes | strong isolation | more ops/cost |
| Metadata filters | flexible | filter bugs dangerous |
| Pre-filtering | security | recall/latency tùy DB |
| Caching | latency thấp | invalidation complexity |
| Continuous eval | phát hiện drift | eval infrastructure |

Production RAG là data system + retrieval system + generation system. Vector index là derived state cần versioning, rebuild, delete và audit. Multi-tenant RAG phải enforce tenant/ACL trước retrieval.

## Answer key

1. Vì source docs mới là source of truth; index được build từ docs và phải rebuild/version/delete được.
2. So hash mới với hash cũ; unchanged thì skip, changed thì reprocess/upsert.
3. Deleted content không được retrieve; nếu còn trong index là leak/compliance issue.
4. Separate indexes, namespaces/collections, metadata filters — với tenant hard filter.
5. Trước khi chunks vào prompt, lý tưởng trong vector DB/index pre-filter.
6. Khi docs, ACL, prompt, embedding model, chunking strategy hoặc data version đổi.
7. Recall@k nightly, empty result rate, score distributions, citation coverage, feedback.
8. Khi retrieval thiếu evidence, confidence thấp, sources conflict, citation validation fail hoặc permission thiếu.
