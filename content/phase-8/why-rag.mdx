RAG — Retrieval-Augmented Generation — không phải là "nhét document vào prompt". RAG là architecture pattern để nối một frozen language model với **external knowledge** dưới ràng buộc về recall, grounding, latency, cost, privacy và update frequency. Nếu fine-tuning thay đổi hành vi model, RAG thay đổi **nguồn bằng chứng tại inference time**.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Chatbot nội bộ hallucinate chính sách đã cũ, 3 tháng sau vẫn dùng">
**Bối cảnh:** Một team build chatbot nội bộ. Họ có 500 policy documents. Thay vì RAG, họ fine-tune LLM để "nhét" knowledge vào model. Mục tiêu: chatbot trả lời câu hỏi chính sách.

**Sự cố:** Policy documents thay đổi hàng tuần. Team fine-tuned LLM với 500 policies. 3 tháng sau, 200 policies đã thay đổi. Chatbot trả lời tự tin bằng chính sách cũ, nhưng sai. User hỏi: "Policy refund có 30 ngày hay 60 ngày?" Chatbot trả lời: "30 ngày." (Policy cũ.) Policy mới: "60 ngày." User khiếu nại, team không biết vì sao.

**Hậu quả:** 12 user khiếu nại, 3 support ticket, 1 user bị từ chối refund sai vì dựa vào chatbot.

**Bài học rút ra:** Fine-tuning không phải database. Bạn không thể "nhét" knowledge vào model và hy vọng nó cập nhật. Khi policy đổi, bạn cần retrain, không thể retrain mỗi tuần. RAG là cách: retrieve evidence tại inference time, không nhét knowledge vào model.

1. Fine-tuning không đảm bảo model nhớ facts chính xác, không dễ xóa dữ liệu, và không tốt cho knowledge thay đổi liên tục.
2. LLM có knowledge cutoff, không biết tài liệu private của công ty bạn, không biết dữ liệu mới cập nhật hôm nay.
3. RAG giải quyết bằng cách đưa evidence liên quan vào prompt đúng lúc — không phải retrain.
</Callout>

## 2. Architecture Blueprint (The RAG Loop)

```text
[ User query ]
        │
        ▼
[ Query understanding / normalization ]
        │
        ▼
[ Retrieve relevant chunks ]
        │
        ▼
[ Rerank / filter / pack context ]
        │
        ▼
[ Prompt LLM with evidence ]
        │
        ▼
[ Generate grounded answer ]
        │
        ▼
[ Citations + abstention + validation ]
```

Hai câu hỏi cốt lõi:
1. Retriever có lấy đúng evidence không?
2. Generator có dùng evidence đúng không?

RAG quality bị chặn bởi retrieval quality. Nếu evidence đúng không vào context, generator không thể trả lời grounded một cách đáng tin cậy.

---

## 3. LLM Memory vs External Memory

```text
Parametric memory = knowledge encoded trong model weights
External memory   = knowledge nằm trong documents/databases/tools
RAG               = retrieve external evidence at inference time
```

Model weights khó cập nhật liên tục. Documents/databases cập nhật dễ hơn. RAG dùng retrieval để chọn phần external memory phù hợp với query.

---

## 4. RAG vs Fine-Tuning

| Need | RAG | Fine-tuning |
|---|---|---|
| Fresh facts | mạnh | yếu, cần retrain |
| Private docs | mạnh | rủi ro/cost cao |
| Style/format | trung bình | mạnh |
| Domain behavior | tốt nếu docs tốt | mạnh nếu dataset tốt |
| Update frequency cao | mạnh | yếu |
| Explainability/citations | mạnh | yếu |
| Cost per query | retrieval + LLM | model call only sau train |

Rule of thumb:

```text
Need facts/evidence → RAG
Need behavior/style/format adaptation → fine-tuning
Need both → RAG + fine-tuned/instruct model
```

<Callout type="warning" title="Fine-tuning không phải database">
Fine-tuning không đảm bảo model nhớ facts chính xác, không dễ xóa dữ liệu, và không tốt cho knowledge thay đổi liên tục. Nếu bạn muốn "nhét" facts vào model, RAG là câu trả lời.
</Callout>

---

## 5. RAG vs Long-Context Prompting

Long-context model cho phép đưa nhiều text hơn, nhưng:

- token cost cao,
- latency cao,
- lost-in-the-middle,
- context nhiễu làm answer kém,
- không giải quyết access control hay document versioning.

RAG tốt giúp context ngắn hơn và tập trung hơn.

```text
Large corpus → retrieval first
Small known context → long-context may be enough
```

---

## 6. What RAG Actually Optimizes

RAG không làm LLM "biết" tài liệu. Nó tăng xác suất model trả lời đúng bằng cách đưa evidence liên quan vào conditional context.

```text
P(answer | question)                 # no RAG
P(answer | question, retrieved_docs) # RAG
```

Nếu retrieved_docs sai, model có thể trả lời sai tự tin hơn.

---

## 7. RAG Failure Taxonomy

| Failure | Mô tả | Symptom |
|---|---|---|
| Ingestion failure | parse thiếu/sai docs | tài liệu không searchable |
| Cleaning failure | headers/footer noise | retrieve chunks nhiễu |
| Chunking failure | cắt mất context | answer thiếu điều kiện |
| Embedding mismatch | model không hợp domain/ngôn ngữ | semantic miss |
| Retrieval miss | relevant chunk không vào top-k | model nói không biết/sai |
| Reranking failure | chunk đúng bị đẩy xuống | context sai |
| Context packing failure | quá nhiều chunks trùng/nhiễu | model bỏ sót evidence |
| Prompt grounding failure | model bịa ngoài context | hallucination |
| Citation fabrication | cite doc không support claim | citation sai |
| Stale docs | index cũ | answer outdated |
| Permission leak | retrieve doc user không được xem | security incident |

---

## 8. Grounding and Abstention

Grounding nghĩa là answer phải dựa trên evidence đã retrieve.

Good RAG prompt rule:

```text
Use only provided context. If the answer is not supported, say "I don't have enough information." Cite source IDs for each factual claim.
```

Nhưng prompt-only không đủ. Cần:

- retrieval eval,
- citation validation,
- answer faithfulness checks,
- no-answer test cases.

<Callout type="danger" title="Không có evidence thì phải biết im lặng">
Một RAG system production phải có behavior "I don't know". Nếu nó luôn trả lời, nó sẽ hallucinate khi retrieval fail.
</Callout>

## 9. Tiny RAG from Scratch

```python
import numpy as np


def cosine(a, b):
    a = a / np.linalg.norm(a)
    b = b / np.linalg.norm(b)
    return float(a @ b)

# Conceptual placeholders
chunks = [
    {"id": "doc1#c1", "text": "Refunds are available within 30 days."},
    {"id": "doc2#c1", "text": "Password reset is in account settings."},
]

# embed(text) could be sentence-transformers or provider API
# query_emb = embed("How do I get my money back?")
# chunk_embs = [embed(c["text"]) for c in chunks]
# scores = [cosine(query_emb, e) for e in chunk_embs]
```

Điều cần hiểu: RAG không bắt đầu ở LLM. Nó bắt đầu ở evidence selection.

## Implementation Playbook: Tiny RAG from Scratch

**Goal:** build RAG nhỏ không dùng framework để hiểu cơ chế.

Steps:

1. Tạo 10 markdown docs.
2. Chunk mỗi doc thành paragraphs.
3. Embed chunks bằng sentence-transformers hoặc embedding API.
4. Store vectors trong numpy list.
5. Embed query.
6. Retrieve top-3 bằng cosine.
7. Build prompt với `{chunk_id, text}`.
8. Yêu cầu answer có citations.
9. Test 10 queries có expected source IDs.

Record:

| Query | Expected doc | Retrieved top-3 | Answer correct | Citation correct |
|---|---|---|---|---|

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Gọi RAG nhưng không log retrieved chunks">
**Triệu chứng:** Answer sai, không biết chunks nào được retrieve, không phân biệt retrieval failure và generation failure.

**Fix:** Log retrieved chunk IDs. Nếu answer sai, bạn có thể phân biệt retrieval failure (chunks sai) và generation failure (chunks đúng, answer sai).
</Callout>

<Callout type="warning" title="Pitfall 2: Dùng top-k bất kỳ không eval">
**Triệu chứng:** `top_k=5` không thần kỳ, nhưng không có recall@k, không biết k có đủ không.

**Fix:** `top_k=5` không thần kỳ. Cần đo recall@k trên query set thật. Benchmark recall@k và answer faithfulness theo corpus/task.
</Callout>

<Callout type="danger" title="Pitfall 3: Bỏ qua access control trong retrieval">
**Triệu chứng:** User retrieve docs không có quyền, data leak.

**Fix:** Không được retrieve docs user không có quyền rồi nhờ LLM giữ bí mật. Permission filter phải xảy ra trước context construction.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| RAG thay vì fine-tuning | cập nhật dễ, không retrain | retrieval cost + LLM cost |
| RAG + fine-tuned model | cả behavior lẫn facts | cost cả train lẫn retrieval |
| Long-context thay RAG | không cần retrieval | token cost, latency, lost-in-the-middle |
| Abstention (I don't know) | không hallucinate | user UX cần fallback |
| Citation validation | không cite sai | extra LLM calls |
| Access control | không leak | filter complexity |

RAG không thay thế fine-tuning; hai kỹ thuật giải quyết vấn đề khác nhau. RAG phù hợp với private/fresh knowledge, fine-tuning phù hợp với style/format/domain adaptation.

## Answer key

1. Parametric memory nằm trong weights; external memory nằm trong docs/databases/tools.
2. Vì docs có thể cập nhật và truy xuất tại inference time.
3. RAG cho facts/evidence; fine-tuning cho behavior/style/format/domain adaptation.
4. Không hoàn toàn; long context đắt, nhiễu và vẫn có lost-in-the-middle.
5. Evidence không vào context, generator khó trả lời grounded.
6. Model có thể fabricate citation hoặc cite chunk không support claim.
7. Khi context thiếu evidence hoặc retrieval confidence thấp.

## Điểm cần nhớ

- RAG nối LLM với external knowledge tại inference time.
- RAG phù hợp với private/fresh knowledge và citation requirements.
- Retrieval quality giới hạn answer quality.
- RAG không thay thế fine-tuning; hai kỹ thuật giải quyết vấn đề khác nhau.
- Production RAG cần grounding, abstention, citations, eval và access control.
