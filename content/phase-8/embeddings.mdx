Embeddings là measurement layer của semantic retrieval. Trong RAG, embedding model quyết định câu hỏi và tài liệu nào được xem là "gần nhau". Nếu embedding space sai, toàn bộ pipeline phía sau — vector DB, reranker, prompt, LLM — chỉ đang tối ưu trên tín hiệu sai.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Đổi embedding model không re-embed, retrieval return 0% recall">
**Bối cảnh:** Một team build RAG. Họ có 1M chunks đã embed bằng `text-embedding-3-small`. Họ muốn chất lượng cao hơn, nên đổi embedding model sang `text-embedding-3-large` (3072 dims). Mục tiêu: cải thiện recall.

**Sự cố:** Team đổi embedding model cho query, nhưng **không re-embed** 1M chunks. Query vector (3072 dims, model mới) so với document vectors (1536 dims, model cũ) — vector space khác nhau. Retrieval trả về 0% recall. Mọi query trả về chunks không liên quan.

**Hậu quả:** 4 giờ downtime, RAG trả lời sai toàn bộ, 1.2M tokens LLM bị lãng phí (re-embed corpus + reindex).

1. Vector spaces khác nhau. Query vector model mới so với document vectors model cũ là bug nghiêm trọng.
2. Không lưu embedding model/version/dimension/chunking strategy khiến migration, rollback và debugging gần như bất khả thi.
3. Embedding model là architecture decision cần eval, không phải "chọn theo demo".
</Callout>

## 2. Architecture Blueprint (Embedding Pipeline)

```text
[ Source documents ]
        │
        ▼
[ Chunking ]
        │
        ▼
[ Embedding model ]  ←  versioned, model name recorded
        │
        ▼
[ Vector store ]  ←  vector + metadata + embedding_model + version
        │
        ▼
[ Query ]
        │
        ▼
[ Embed query ]  ←  same model, same version
        │
        ▼
[ Similarity search ]
        │
        ▼
[ Top-k evidence ]
```

Điều kiện tiên quyết: **embedding model** phải khớp giữa query và document. Nếu đổi model, phải re-embed corpus.

## 3. Meaning as Geometry

```text
text → embedding model → vector in R^d
similar meaning → nearby vectors
```

Ví dụ:

```text
"How can I get my money back?"
```

nên gần:

```text
"Refunds are available within 30 days."
```

Dù không share keyword `refund`, semantic vector nên kéo chúng lại gần.

## 4. Dense vs Sparse vs Hybrid

| Representation | Strong at | Weak at |
|---|---|---|
| Sparse/BM25 | exact keywords, IDs, error codes | paraphrase/synonyms |
| Dense embeddings | semantic similarity | exact numbers, rare tokens, code identifiers |
| Hybrid | combines both | tuning and score normalization |

Production RAG thường dùng hybrid:

```text
BM25 sparse retrieval + dense vector retrieval + reranker
```

<Callout type="tip" title="Dense không thay thế sparse">
Query `ORA-00942` hoặc `getUserById` thường cần exact match. Dense-only search có thể miss identifiers quan trọng.
</Callout>

## 5. Embedding Model Families

### Provider embeddings

- OpenAI `text-embedding-3-small`: rẻ/nhanh hơn, phù hợp nhiều workloads.
- OpenAI `text-embedding-3-large`: chất lượng/dimension cao hơn, cost/storage cao hơn.
- Cohere embed models: thường được tối ưu cho retrieval/RAG, có multilingual variants.

### Open-source embeddings

- `all-MiniLM-L6-v2`: baseline nhẹ, tốt để học/prototype.
- BGE family: mạnh cho retrieval, có reranker ecosystem.
- E5 family: query/passage style embeddings.
- GTE models: general text embeddings.
- Multilingual/domain models: cần cho tiếng Việt, legal, medical, code.

Professor-level point: tên model không đủ. Bạn cần biết model có được train cho retrieval không, ngôn ngữ nào, domain nào, query/document format nào.

## 6. Query vs Document Embedding Format

Một số embedding models khuyến nghị prefix:

```text
query: how do I reset password?
passage: You can reset your password in account settings.
```

Nếu model được train với prefix mà bạn bỏ qua, quality có thể giảm.

## 7. Dimensions and Tradeoffs

Dimension cao hơn thường chứa nhiều capacity hơn, nhưng:

- storage tăng,
- memory index tăng,
- network payload tăng,
- search latency có thể tăng,
- cost embedding có thể tăng,
- returns có thể diminishing.

Storage estimate:

```text
storage_bytes = num_vectors × dimensions × bytes_per_float
```

Ví dụ:

```text
10M chunks × 1536 dims × 4 bytes ≈ 61.4 GB raw vectors
```

Chưa tính metadata/index overhead.

## 8. Similarity Metrics

### Cosine similarity

Đo cùng hướng:

```python
import numpy as np


def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
```

### Dot product

Nếu vectors đã normalize norm = 1:

```text
dot product == cosine similarity
```

### Euclidean distance / L2

Đo khoảng cách hình học. Với normalized vectors, liên quan chặt tới cosine.

<Callout type="danger" title="Metric mismatch làm retrieval sai âm thầm">
Nếu bạn normalize vectors nhưng index dùng metric khác, hoặc model expect dot product nhưng DB config cosine/L2 sai, retrieval quality có thể giảm mà không báo lỗi.
</Callout>

## 9. Batch Embedding in Production

Batching giảm overhead nhưng phải xử lý rate limits/timeouts.

```python
from hashlib import sha256


def content_hash(text: str) -> str:
    return sha256(text.encode("utf-8")).hexdigest()


def batch_items(items, batch_size=64):
    for i in range(0, len(items), batch_size):
        yield items[i:i + batch_size]


def embed_corpus(chunks, embed_fn, model_name: str):
    records = []
    for batch in batch_items(chunks, batch_size=64):
        texts = [c["text"] for c in batch]
        vectors = embed_fn(texts)
        for chunk, vector in zip(batch, vectors):
            records.append({
                "chunk_id": chunk["chunk_id"],
                "doc_id": chunk["doc_id"],
                "content_hash": content_hash(chunk["text"]),
                "embedding_model": model_name,
                "embedding": vector,
            })
    return records
```

Production additions:

- retries with backoff,
- idempotent chunk IDs,
- failed batch retry queue,
- rate limit handling,
- store embedding model/version,
- do not re-embed unchanged content.

## 11. Evaluation: Retrieval Quality

Bạn cần query set có relevance labels:

```json
{
  "query": "How do refunds work?",
  "relevant_chunk_ids": ["refund_policy#c3", "refund_policy#c4"]
}
```

Metrics:

| Metric | Ý nghĩa |
|---|---|
| recall@k | relevant chunk có nằm trong top-k không |
| MRR | relevant result xuất hiện sớm không |
| nDCG | ranking tốt với nhiều mức relevance không |
| hit rate | có ít nhất một relevant result không |

## 12. Hard Negatives

Hard negatives là chunks nhìn gần giống nhưng không đúng.

Ví dụ query:

```text
refund policy for enterprise plan
```

Hard negative:

```text
refund policy for free plan
```

Embedding eval cần hard negatives để biết model có phân biệt chi tiết quan trọng không.

## 13. Common Embedding Failure Modes

| Failure | Example | Mitigation |
|---|---|---|
| Exact ID miss | `ERR-1042` | hybrid BM25 |
| Numeric confusion | 30 days vs 60 days | reranker, exact checks |
| Table weakness | pricing table | table-aware parsing |
| Code identifiers | `getUserById` | code-aware search/sparse |
| Domain jargon | medical/legal terms | domain embeddings |
| Multilingual mismatch | Vietnamese query English docs | multilingual model |
| Short query ambiguity | "billing" | query expansion/clarification |
| Stale index | old embeddings | versioning/reindex |

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Chọn embedding model không eval">
**Triệu chứng:** Model popular, demo tốt, nhưng recall@k thấp trên corpus thật.

**Fix:** Model popular chưa chắc tốt cho domain/ngôn ngữ của bạn. Luôn benchmark trên query-doc pairs thật.
</Callout>

<Callout type="warning" title="Pitfall 2: Đổi embedding model mà không re-embed">
**Triệu chứng:** Query vector model mới so với document vectors model cũ — vector space khác nhau, retrieval trả 0% recall.

**Fix:** Vector spaces khác nhau. Query vector model mới so với document vectors model cũ là bug nghiêm trọng. Re-embed corpus, update index metadata, rerun retrieval eval.
</Callout>

<Callout type="danger" title="Pitfall 3: Quên embedding metadata/version">
**Triệu chứng:** Đổi model, không biết vector nào từ model nào, không thể rollback, không thể debug.

**Fix:** Không lưu model/version/dimension/chunking strategy khiến migration, rollback và debugging gần như bất khả thi. Lưu metadata với mỗi vector.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Dense embedding | semantic similarity | exact IDs/numbers/code miss |
| Hybrid search | cả exact lẫn semantic | score normalization, tuning |
| Dimension cao | capacity cao | storage, memory, latency |
| Multilingual model | tiếng Việt | cost, latency |
| Hard negatives eval | phân biệt chi tiết | cần label effort |
| Re-embed khi đổi model | vector space consistent | compute cost, reindex |

Embedding model là architecture decision. Không chọn theo demo, không chọn theo hype. Chọn bằng retrieval eval trên data thật.

## Answer key

1. Nó định nghĩa vector space nơi query và chunks được so sánh.
2. Exact IDs, numbers, code identifiers, rare tokens, bảng phức tạp.
3. Sparse bắt exact match; dense bắt semantic similarity; hai loại bổ sung nhau.
4. Cách tính similarity trong DB không khớp assumption của vectors/model.
5. Re-embed corpus, update index metadata, rerun retrieval eval.
6. Kiểm tra model có phân biệt chunks gần giống nhưng sai không.
7. Tăng storage, memory, bandwidth, latency/cost; có thể diminishing returns.
