Bạn đã học Full Fine-Tuning, LoRA và QLoRA. Nhưng trong team production, thành công không đến từ việc “notebook chạy được một lần”. Thành công đến từ một workflow có thể lặp lại: cấu hình rõ, dataset version rõ, experiment tracking đầy đủ, checkpoint có thể audit và eval report đi kèm.

Fine-tuning tools như Unsloth, Axolotl, LLaMA-Factory, TRL, W&B và MLflow giúp AI Engineer chuyển từ thử nghiệm thủ công sang training operations. Chúng không thay thế hiểu biết nền tảng, nhưng giúp bạn giảm boilerplate, giảm VRAM, chuẩn hóa config và debug training job nhanh hơn.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Notebook fine-tune tốt nhưng không thể tái tạo">
**Bối cảnh:** Một engineer fine-tune QLoRA trên notebook cá nhân. Demo nội bộ rất tốt, model trả lời đúng format và tone. Product muốn ship bản này trong tuần.

**Sự cố:** Khi team ML platform cố chạy lại training, kết quả khác hoàn toàn. Không ai biết chính xác dataset split nào đã dùng, base model revision nào, seed bao nhiêu, `max_seq_length` là gì, và checkpoint tốt nhất nằm ở cell output nào.

1. Training config nằm rải rác trong notebook.
2. Dataset đã bị chỉnh tay nhưng không version.
3. Không log W&B/MLflow đầy đủ.
4. Không có eval script cố định để so sánh checkpoint.
5. Không có artifact manifest để deploy.

**Hậu quả:** Model không được release. Team mất thêm 2 tuần để viết lại pipeline config-first, tái tạo dataset và rebuild eval harness.
</Callout>

Bài học rút ra: fine-tuning production là một MLOps workflow. Nếu không tái tạo được training run, bạn không thể audit, rollback hoặc cải thiện nó một cách đáng tin.

## 2. Architecture Blueprint (Fine-Tuning Toolchain)

Một toolchain tốt tách rõ data, config, training, tracking, eval và registry:

```text
[ Data repository ]
  - raw data
  - processed jsonl
  - split manifests
        │
        ▼
[ Training config ]
  - base model id/revision
  - method: SFT/LoRA/QLoRA/DPO
  - hyperparameters
  - eval config
        │
        ▼
[ Training runner ]
  - TRL / Unsloth / Axolotl / LLaMA-Factory
  - single GPU or distributed
        │
        ▼
[ Experiment tracking ]
  - W&B / MLflow
  - loss, LR, grad norm
  - GPU utilization
  - config and artifacts
        │
        ▼
[ Evaluation job ]
  - task eval
  - regression eval
  - safety/format eval
        │
        ▼
[ Model / Adapter registry ]
  - checkpoint
  - tokenizer
  - manifest
  - eval report
```

Mục tiêu: bất kỳ engineer nào cũng có thể lấy config, dataset version và command để reproduce cùng một run.

## 3. API Fine-Tuning vs Self-Hosted OSS Stack

Không phải dự án nào cũng cần tự quản lý GPU. Có hai hướng chính:

| Hướng | Khi dùng | Ưu điểm | Nhược điểm |
|---|---|---|---|
| Provider/API fine-tuning | Dữ liệu có thể gửi provider, workload đơn giản, muốn vận hành nhanh | Ít MLOps, serving managed | Ít kiểm soát, cost theo provider, giới hạn model |
| Self-host OSS stack | Cần kiểm soát model/data/runtime, PEFT/QLoRA, privacy | Linh hoạt, tối ưu cost, nhiều model | Phải quản lý GPU, dependency, serving |

### Provider/API fine-tuning phù hợp khi

- Bạn cần fine-tune model thương mại để cải thiện format/tone trên workload rõ.
- Không muốn tự vận hành GPU hoặc inference server.
- Data governance cho phép gửi dữ liệu ra provider.
- Bạn chấp nhận giới hạn về hyperparameters và model family.

### Self-host phù hợp khi

- Dữ liệu nhạy cảm hoặc cần on-prem/VPC.
- Cần LoRA/QLoRA, adapter routing hoặc merge.
- Cần benchmark nhiều open-weight models.
- Cần tối ưu unit economics ở scale.

<Callout type="info" title="Tool chọn theo constraint, không theo hype">
Nếu team chưa có GPU/MLOps và task đơn giản, API fine-tuning có thể là con đường nhanh nhất. Nếu bạn cần privacy, adapter lifecycle hoặc tối ưu chi phí sâu, self-host stack đáng đầu tư hơn.
</Callout>

## 4. Tool Map: Unsloth, Axolotl, LLaMA-Factory, TRL

| Tool | Vai trò | Điểm mạnh | Khi dùng |
|---|---|---|---|
| TRL | Python library cho SFT/DPO/RLHF workflows | Linh hoạt, code-first | Muốn kiểm soát training loop |
| Unsloth | Optimized single-GPU fine-tuning | Nhanh, tiết kiệm VRAM | QLoRA/SFT trên 1 GPU |
| Axolotl | YAML config framework cho LLM fine-tuning | Reproducible, multi-model, production-friendly | Team muốn config-first |
| LLaMA-Factory | Training UI/CLI cho nhiều method | Dễ thử nghiệm, nhiều mode | Research/app team cần tốc độ thử |
| W&B/MLflow | Experiment tracking | Metrics, artifacts, comparison | Gần như mọi training project |

### Unsloth

Unsloth tối ưu memory và tốc độ bằng cách viết lại một số kernel/operation, thường hữu ích cho single GPU QLoRA.

Ưu điểm:

- Giảm VRAM.
- Tăng tốc training.
- API gần với HuggingFace/TRL.
- Phù hợp prototyping nhanh.

Giới hạn:

- Không phải mọi architecture/method đều support như nhau.
- Dependency/GPU compatibility cần kiểm tra.
- Vẫn phải tự quản lý eval, registry và release.

### Axolotl

Axolotl giúp đưa training vào YAML config:

```yaml
base_model: meta-llama/Meta-Llama-3-8B-Instruct
model_type: AutoModelForCausalLM
tokenizer_type: AutoTokenizer

load_in_4bit: true
adapter: qlora
lora_r: 16
lora_alpha: 32
lora_dropout: 0.05
lora_target_modules:
  - q_proj
  - k_proj
  - v_proj
  - o_proj
  - gate_proj
  - up_proj
  - down_proj

datasets:
  - path: ./data/customer_support_sft.jsonl
    type: chat_template

sequence_len: 4096
sample_packing: true
micro_batch_size: 2
gradient_accumulation_steps: 8
num_epochs: 2
learning_rate: 0.0002
warmup_ratio: 0.03
optimizer: paged_adamw_8bit
bf16: auto

output_dir: ./outputs/support-qlora-v1
wandb_project: phase10-finetuning
wandb_run_name: support-qlora-v1
```

Ưu điểm lớn: config có thể review trong PR, lưu cùng repo và chạy lại trên môi trường khác.

### LLaMA-Factory

LLaMA-Factory hữu ích khi cần UI/CLI nhanh cho nhiều method như SFT, LoRA, QLoRA, DPO.

Phù hợp cho:

- Team ứng dụng muốn thử nhiều model nhanh.
- Demo nội bộ.
- Tạo baseline trước khi chuyển sang pipeline production.

Nhưng production vẫn cần export config, pin versions và chạy eval tự động.

## 5. Config-First Training Operations

Một run production cần có tối thiểu:

```text
training_run/
  config.yaml
  data_manifest.json
  environment.txt
  command.sh
  metrics.json
  eval_report.json
  adapter_or_model_manifest.json
```

### Data manifest

```json
{
  "dataset_id": "support_sft_2026_07_01",
  "train_path": "data/processed/train.jsonl",
  "validation_path": "data/processed/validation.jsonl",
  "test_path": "data/processed/test.jsonl",
  "split_seed": 42,
  "num_train": 18000,
  "num_validation": 1000,
  "num_test": 1000,
  "pii_scan": "passed",
  "license_review": "approved",
  "data_hash": "sha256:..."
}
```

### Environment capture

```bash
python --version
pip freeze > environment.txt
nvidia-smi > gpu_info.txt
git rev-parse HEAD > code_revision.txt
```

### Naming convention

```text
{task}-{method}-{base_model_short}-{dataset_version}-{date}-{run_number}

support-qlora-llama3-8b-supportsft20260701-20260701-r001
```

<Callout type="tip" title="Config review giống code review">
Training config quyết định behavior của model. Hãy review nó như production code: base model, dataset, LR, epoch, template, eval gate và output path đều phải rõ.
</Callout>

## 6. Monitoring Training Runs

Training dashboard không chỉ để nhìn loss đẹp. Nó giúp phát hiện lỗi sớm.

| Metric | Ý nghĩa | Cảnh báo |
|---|---|---|
| train loss | Model fit training data | Giảm quá nhanh có thể overfit |
| eval loss | Generalization trên validation | Tăng trong khi train loss giảm là overfit |
| learning rate | Scheduler hoạt động đúng không | Warmup/schedule sai |
| gradient norm | Stability | Spike lớn có thể gây NaN |
| tokens/sec | Throughput | Data loader hoặc GPU bottleneck |
| GPU utilization | Hardware efficiency | Util thấp do I/O/packing kém |
| memory allocated | VRAM pressure | Gần max dễ OOM |

### Debug dashboard pattern

```text
If GPU util low + CPU high:
  -> data preprocessing bottleneck

If GPU util low + many short sequences:
  -> enable packing

If eval loss rises early:
  -> reduce LR/epoch, improve data split

If grad norm spikes:
  -> reduce LR, add clipping, inspect bad batches
```

## 7. Implementation Playbook: Reproducible Training Command

Một workflow tối thiểu với Axolotl-style config:

```bash
# 1. Validate dataset
python scripts/validate_dataset.py \
  --input data/processed/train.jsonl \
  --schema chat_sft \
  --tokenizer meta-llama/Meta-Llama-3-8B-Instruct \
  --max-seq-length 4096

# 2. Launch training
accelerate launch -m axolotl.cli.train configs/support_qlora.yaml

# 3. Run evaluation
python eval/run_eval.py \
  --model outputs/support-qlora-v1 \
  --base-model meta-llama/Meta-Llama-3-8B-Instruct \
  --eval-set data/eval/support_holdout.jsonl \
  --report outputs/support-qlora-v1/eval_report.json

# 4. Package artifact
python scripts/package_adapter.py \
  --adapter outputs/support-qlora-v1 \
  --manifest outputs/support-qlora-v1/adapter_manifest.json
```

Một script packaging nên fail nếu thiếu:

1. Config YAML.
2. Dataset manifest.
3. Eval report.
4. Base model revision.
5. Tokenizer files.
6. Adapter/model weights.

## 8. Project Guide: Custom Q&A Fine-Tune (Phase 10 Easy)

Official project Phase 10 easy:

```text
Fine-tune a small/API model on custom Q&A
- Prepare a clean question-answer dataset
- Train using provider fine-tuning API or small open model SFT
- Compare against prompt-only baseline
- Stack: OpenAI fine-tuning API or TRL/Unsloth, W&B/MLflow
```

MVP deliverables:

1. 200-1,000 clean Q&A examples in JSONL.
2. Data card: source, license, PII handling, split seed.
3. Training config or provider fine-tuning job config.
4. Baseline vs fine-tuned eval report:
   - exactness/helpfulness,
   - format adherence,
   - hallucination checks,
   - latency/cost estimate.
5. Short README: when this should use RAG instead.

Stretch goals:

- Add regression prompts outside the narrow Q&A domain.
- Track runs with W&B or MLflow.
- Package model/adapter with manifest and rollback instructions.

## 9. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Hidden notebook state">
**Triệu chứng:** Run không reproduce được dù code nhìn giống nhau.

**Nguyên nhân:** Biến trong notebook, dataset tạm, seed hoặc package version không được ghi lại.

**Cách khắc phục:** Chuyển sang script/config, log environment, pin dataset hash và commit hash.
</Callout>

<Callout type="warning" title="Pitfall 2: Không tracking experiment">
**Triệu chứng:** Không biết run nào tốt nhất hoặc config nào tạo ra checkpoint đang deploy.

**Nguyên nhân:** Không dùng W&B/MLflow/artifact registry.

**Cách khắc phục:** Log config, metrics, artifacts và eval report cho mọi run.
</Callout>

<Callout type="warning" title="Pitfall 3: Config default nguy hiểm">
**Triệu chứng:** Training chạy nhưng dùng LR, template hoặc packing không như kỳ vọng.

**Nguyên nhân:** Tool có defaults ẩn hoặc version thay đổi behavior.

**Cách khắc phục:** Explicit mọi config quan trọng, pin tool version, lưu rendered/final config.
</Callout>

<Callout type="warning" title="Pitfall 4: Dataset split không cố định">
**Triệu chứng:** Eval metric dao động mạnh giữa các run.

**Nguyên nhân:** Split random mỗi lần hoặc leakage giữa train/test.

**Cách khắc phục:** Lưu split manifest, seed cố định, hash từng example và kiểm tra leakage.
</Callout>

## 10. Cost & Tooling Unit Economics

Tooling tốt có chi phí học ban đầu nhưng giảm chi phí lặp lại.

| Quyết định | Lợi ích | Trade-off |
|---|---|---|
| Notebook nhanh | Prototype cực nhanh | Khó reproduce, khó review |
| Config-first | Reproducible, reviewable | Cần discipline và template |
| Unsloth single GPU | Giảm GPU cost | Compatibility/version constraints |
| Axolotl YAML | Team workflow tốt | Phải hiểu config sâu |
| W&B/MLflow | Traceability, comparison | Thêm setup/security review |
| API fine-tuning | Ít ops | Ít kiểm soát và phụ thuộc provider |

Production takeaway: nếu dự án chỉ là demo một lần, notebook có thể đủ. Nếu model sẽ ảnh hưởng user thật, hãy đầu tư config-first workflow ngay từ đầu.

## Answer key

1. Vì state, seed, dataset, dependency và config thường không được version hóa đầy đủ, làm run khó reproduce/audit.
2. Nó đưa training parameters vào config reviewable/reproducible, giảm boilerplate và chuẩn hóa workflow.
3. Single-GPU SFT/QLoRA cần tối ưu tốc độ và VRAM.
4. Config, metrics, dataset manifest/hash, checkpoint/adapter, eval report, environment và code revision.
5. Để eval nhất quán giữa các run và tránh leakage hoặc random split làm metric dao động.

## Điểm cần nhớ

- Fine-tuning production phải config-first, versioned và reproducible.
- Tooling không thay thế evaluation; nó giúp training workflow đáng tin hơn.
- Unsloth mạnh cho single-GPU efficiency, Axolotl mạnh cho config-first team workflow, LLaMA-Factory mạnh cho thử nghiệm nhanh.
- W&B/MLflow/artifact registry là bắt buộc nếu model sẽ được deploy thật.
- Dataset manifest, environment capture và adapter/model manifest là nền tảng cho audit và rollback.
- Hãy chọn API fine-tuning hay self-host stack dựa trên privacy, cost, control và ops maturity.
