Bạn đã đi qua decision framework, Full Fine-Tuning, PEFT/QLoRA, tooling và dataset preparation. Nhưng một model fine-tuned chỉ có giá trị khi bạn chứng minh được nó tốt hơn baseline trên đúng workload, không regression ở năng lực quan trọng và có thể release/rollback bằng số liệu.

Evaluation sau fine-tuning là lớp bảo vệ cuối cùng trước production. Nó kết hợp task metrics, regression tests, safety checks, LLM-as-judge, human review, cost/latency measurement và release gates. Nếu không có eval harness, fine-tuning chỉ là một cảm giác “model có vẻ tốt hơn”.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Demo tốt nhưng production regression">
**Bối cảnh:** Một team fine-tune support assistant bằng 25,000 ticket đã làm sạch. Demo nội bộ rất ấn tượng: model trả lời ngắn gọn, đúng tone, ít cần prompt dài. Product quyết định canary 20% traffic.

**Sự cố:** Sau 48 giờ, complaint rate tăng. Model xử lý các câu hỏi phổ biến tốt hơn nhưng fail ở edge cases: refund ngoại lệ, account security, policy mới và user tức giận. Nó cũng hallucinate citation vì eval offline chỉ dùng 100 câu dễ do team tự chọn.

1. Eval set không đại diện production distribution.
2. Không có regression tests cho edge cases và safety cases.
3. Không so sánh với prompt/RAG baseline bằng cùng rubric.
4. Không có rollback threshold tự động.

**Hậu quả:** Canary rollback thủ công, mất niềm tin từ support team và phải rebuild eval harness trước khi train vòng tiếp theo.
</Callout>

Bài học rút ra: fine-tuning phải được release như software. Mỗi checkpoint cần test suite, release gate, monitoring và rollback condition.

## 2. Architecture Blueprint (Evaluation & Release Gate)

Một eval system tốt nằm giữa training và deployment:

```text
[ Candidate checkpoint / adapter ]
        │
        ▼
[ Offline evaluation harness ]
  - task-specific metrics
  - product holdout set
  - general regression set
  - safety/red-team set
  - format/tool-call checks
        │
        ▼
[ Judge + human review ]
  - rubric-based LLM judge
  - pairwise comparison
  - expert review for high-risk cases
        │
        ▼
[ Eval report ]
  - baseline vs candidate
  - confidence intervals
  - failure analysis
  - cost/latency estimate
        │
        ▼
[ Release decision ]
  - promote
  - reject
  - retrain
  - canary only
        │
        ▼
[ Production monitoring ]
  - complaint rate
  - task success
  - safety incidents
  - drift
  - rollback triggers
```

Mục tiêu: model mới chỉ được promote nếu cải thiện đúng metric chính và không làm hỏng các guardrail metric.

## 3. Academic Benchmarks vs Product Evals vs Safety Evals

Không có một metric duy nhất cho mọi fine-tuning project.

| Eval type | Đo cái gì? | Ví dụ | Rủi ro nếu thiếu |
|---|---|---|---|
| Academic benchmark | Năng lực chuẩn hóa | MMLU subset, HumanEval, GSM8K | Không biết general capability regress |
| Product eval | Workload thật của sản phẩm | support holdout, SQL tasks, extraction cases | Demo tốt nhưng production fail |
| Safety eval | Harm/PII/refusal/security | jailbreak, PII leak, unsafe advice | Incident pháp lý/safety |
| Format eval | Output parse được không | JSON schema, tool args | Downstream crash |
| Cost/latency eval | Có serve được không | p95 latency, tokens/sec, VRAM | Model tốt nhưng quá chậm/đắt |

### Product eval quan trọng nhất

Academic benchmarks hữu ích để phát hiện regression, nhưng product eval mới trả lời câu hỏi: “Model có giúp user của chúng ta hơn không?”.

Một product holdout set nên có:

1. Frequent cases từ production logs.
2. Edge cases do domain expert viết.
3. Recent cases sau data cutoff.
4. Negative/safety cases.
5. Ambiguous cases cần clarification.
6. Examples mà baseline từng fail.

## 4. Metrics cho Fine-Tuned Models

Chọn metric theo task.

| Task | Metric phù hợp | Ghi chú |
|---|---|---|
| Classification | Accuracy, F1, precision/recall | F1 tốt hơn accuracy nếu class imbalance |
| Extraction | Exact match, token F1, field-level accuracy | Đo từng field quan trọng |
| Code generation | pass@1, pass@k, unit-test pass rate | Cần sandbox an toàn |
| SQL generation | execution accuracy, result match | Parse đúng chưa đủ, chạy đúng mới quan trọng |
| Summarization | human rubric, factuality, coverage | ROUGE chỉ là signal phụ |
| Chat/helpfulness | pairwise win rate, human/LLM judge score | Cần rubric và calibration |
| Tool calling | tool name accuracy, argument validity, task success | Đo cả schema và outcome |
| Structured output | JSON valid rate, schema pass rate | Metric production-critical |

### Pass@k intuition

Với code generation, `pass@k` đo xác suất ít nhất một trong `k` samples pass tests.

```text
pass@1: model phải pass ngay sample đầu tiên
pass@k: model có k cơ hội sinh lời giải
```

Trong production, `pass@1` thường quan trọng hơn nếu bạn chỉ trả một answer cho user. `pass@k` hữu ích trong coding assistant hoặc agent có thể thử nhiều candidates.

## 5. LLM-as-Judge và Rubric Calibration

LLM-as-judge giúp scale evaluation cho open-ended outputs, nhưng judge cũng có bias.

### Rubric tốt

```text
Score each answer from 1 to 5:
1. Correctness: answer is factually and technically correct.
2. Grounding: answer only uses provided evidence/context.
3. Helpfulness: answer resolves the user's task.
4. Format: answer follows required structure.
5. Safety: answer avoids harmful, private, or unauthorized content.

Return JSON with:
- score_correctness
- score_grounding
- score_helpfulness
- score_format
- score_safety
- explanation_short
- failure_tags
```

### Pairwise judging

Thay vì chấm tuyệt đối, judge chọn output tốt hơn giữa baseline và candidate:

```text
Prompt + context
Answer A: baseline output
Answer B: candidate output
Judge: A wins / B wins / tie, with reason
```

Pairwise thường ổn định hơn scoring tuyệt đối, nhưng cần randomize vị trí A/B để tránh position bias.

### Calibration checklist

```text
[ ] Judge rubric đã được domain expert review.
[ ] Có gold set nhỏ do human chấm.
[ ] Judge agreement với human đạt threshold.
[ ] A/B order được randomize.
[ ] Judge không biết answer nào là baseline/candidate.
[ ] Có failure tags để phân tích lỗi.
[ ] Có manual review cho high-risk disagreements.
```

<Callout type="warning" title="Judge không phải sự thật tuyệt đối">
LLM judge là một evaluator có lỗi. Đừng dùng nó một mình cho high-stakes release. Hãy calibrate với human labels và phân tích disagreement.
</Callout>

## 6. Regression Testing Across Checkpoints

Fine-tuning phải so sánh nhiều checkpoint và baseline.

```text
baseline_model
checkpoint_1000
checkpoint_2000
checkpoint_3000
final_checkpoint
```

Eval table nên nhìn như sau:

| Model | Task win rate | JSON valid | General regression | Safety fail | p95 latency | Decision |
|---|---:|---:|---:|---:|---:|---|
| Baseline | 0.50 | 0.91 | 0.00 | 1.2% | 900ms | Reference |
| CKPT-1000 | 0.56 | 0.96 | -0.01 | 1.1% | 920ms | Candidate |
| CKPT-2000 | 0.61 | 0.98 | -0.03 | 1.3% | 940ms | Best |
| CKPT-3000 | 0.63 | 0.98 | -0.09 | 2.8% | 950ms | Reject |

Checkpoint cuối không luôn là checkpoint tốt nhất. Overfitting thường xuất hiện sau khi task metric tiếp tục tăng nhẹ nhưng regression/safety giảm mạnh.

## 7. Implementation Playbook: Automated Eval CLI

Một eval harness tối thiểu cần chạy cùng một eval set trên baseline và candidate.

```python

import json
from pathlib import Path


def load_jsonl(path):
    with open(path, "r", encoding="utf-8") as f:
        for line in f:
            yield json.loads(line)


def exact_match(prediction, expected):
    return prediction.strip() == expected.strip()


def json_valid(prediction):
    try:
        json.loads(prediction)
        return True
    except Exception:
        return False


def run_model(model_id, prompt):
    # Replace with provider call, vLLM endpoint, or local generation.
    raise NotImplementedError


def evaluate(model_id, eval_path):
    results = []
    for example in load_jsonl(eval_path):
        prediction = run_model(model_id, example["prompt"])
        result = {
            "id": example["id"],
            "prediction": prediction,
            "exact_match": exact_match(prediction, example.get("expected", "")),
            "json_valid": json_valid(prediction) if example.get("requires_json") else None,
            "tags": example.get("tags", []),
        }
        results.append(result)
    return results


def summarize(results):
    exact = [r["exact_match"] for r in results]
    json_checks = [r["json_valid"] for r in results if r["json_valid"] is not None]
    return {
        "num_examples": len(results),
        "exact_match": sum(exact) / len(exact) if exact else None,
        "json_valid_rate": sum(json_checks) / len(json_checks) if json_checks else None,
    }


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--model", required=True)
    parser.add_argument("--eval-set", required=True)
    parser.add_argument("--out", required=True)
    args = parser.parse_args()

    results = evaluate(args.model, args.eval_set)
    report = {
        "model": args.model,
        "eval_set": args.eval_set,
        "summary": summarize(results),
        "results": results,
    }

    Path(args.out).parent.mkdir(parents=True, exist_ok=True)
    with open(args.out, "w", encoding="utf-8") as f:
        json.dump(report, f, ensure_ascii=False, indent=2)
```

## 8. Release Gates và Rollback Policy

Một release gate tốt có threshold trước khi train, không viết sau khi thấy kết quả.

```json
{
  "release_gate": {
    "task_win_rate_delta": ">= +0.07",
    "json_valid_rate": ">= 0.98",
    "general_regression_delta": ">= -0.03",
    "safety_failure_rate": "<= baseline",
    "p95_latency_ms": "<= 1200"
  },
  "canary": {
    "traffic_percent": 5,
    "duration_hours": 24,
    "rollback_if": {
      "complaint_rate_delta": "> +15%",
      "safety_incidents": "> 0 critical",
      "tool_error_rate": "> baseline + 10%"
    }
  }
}
```

Release process:

1. Run offline eval.
2. Review failure analysis.
3. Promote to small canary only if gates pass.
4. Monitor production metrics by cohort.
5. Rollback automatically on critical threshold.
6. Convert production failures into regression tests.

<Callout type="tip" title="Mỗi incident phải trở thành test">
Nếu model fail ở production và bạn sửa bằng data/prompt/training, hãy thêm case đó vào regression eval. Nếu không, lỗi có thể quay lại ở vòng fine-tuning tiếp theo.
</Callout>

## 10. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Eval set quá dễ">
**Triệu chứng:** Offline score cao nhưng production fail edge cases.

**Nguyên nhân:** Eval chỉ gồm examples sạch/dễ hoặc do team tự chọn thiên lệch.

**Cách khắc phục:** Lấy production logs, thêm hard cases, edge cases, recent cases và failures của baseline.
</Callout>

<Callout type="warning" title="Pitfall 2: Judge bias">
**Triệu chứng:** LLM judge luôn thích answer dài hơn hoặc style giống candidate.

**Nguyên nhân:** Rubric mơ hồ, không randomize A/B, không calibrate với human labels.

**Cách khắc phục:** Pairwise blind judging, position randomization, gold set do human chấm và disagreement review.
</Callout>

<Callout type="warning" title="Pitfall 3: Benchmark contamination">
**Triệu chứng:** Model đạt điểm benchmark bất thường nhưng không cải thiện production.

**Nguyên nhân:** Test examples bị lẫn vào train hoặc benchmark public đã nằm trong pretraining/fine-tuning data.

**Cách khắc phục:** Dùng private holdout, leakage scan, group split và custom production eval.
</Callout>

<Callout type="warning" title="Pitfall 4: Không có rollback threshold">
**Triệu chứng:** Canary xấu nhưng team tranh luận thủ công quá lâu.

**Nguyên nhân:** Không định nghĩa metric rollback trước release.

**Cách khắc phục:** Viết rollback policy trước khi canary và tự động alert khi vượt threshold.
</Callout>

## 11. Evaluation Unit Economics

Eval cũng tốn tiền, nhưng rẻ hơn incident.

| Quyết định | Lợi ích | Trade-off |
|---|---|---|
| Eval nhỏ, chạy nhanh | Feedback nhanh | Dễ miss edge cases |
| Eval lớn, nhiều judge calls | Coverage tốt | Tốn API/compute |
| Human expert review | Chất lượng cao | Chậm và đắt |
| LLM-as-judge | Scale tốt | Bias/calibration risk |
| Eval mỗi checkpoint | Chọn checkpoint tốt hơn | Tốn thời gian |
| Production monitoring | Phát hiện drift thật | Cần instrumentation |

Cost estimate:

```text
eval_cost = num_examples * num_models * judge_calls_per_example * price_per_judge_call
human_cost = num_reviewed_examples * minutes_per_example * reviewer_hourly_rate / 60
```

Production takeaway: hãy chia eval thành tầng. Cheap automated checks chạy thường xuyên; expensive human/judge review chạy cho candidate checkpoint hoặc high-risk cases.

2. Product eval khác academic benchmark như thế nào?
3. LLM-as-judge cần calibration gì?
4. Vì sao checkpoint cuối không luôn là checkpoint tốt nhất?

---

## Answer key

1. Vì loss chỉ đo fit trên training objective; nó không đảm bảo task quality, safety, format adherence, general capability hoặc production latency.
2. Academic benchmark đo năng lực chuẩn hóa; product eval đo workload thật, edge cases và success criteria của sản phẩm.
3. Cần rubric rõ, gold set human-labeled, randomize A/B, blind comparison và disagreement review.
4. Vì overfitting/regression có thể tăng ở các bước cuối dù task metric tiếp tục tăng nhẹ.
5. Trước khi release/canary, tốt nhất trước cả khi train, để tránh chỉnh threshold theo kết quả mong muốn.

Phase 10 biến fine-tuning từ một kỹ thuật training thành một engineering lifecycle: quyết định đúng, dữ liệu đúng, training đúng, eval đúng và release đúng.
