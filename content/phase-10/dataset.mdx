Ở bài trước, bạn đã có toolchain để chạy fine-tuning lặp lại. Nhưng fine-tuning không bắt đầu từ GPU; nó bắt đầu từ dữ liệu. Một model nhỏ fine-tuned trên 2,000 ví dụ sạch, đúng template, không leakage và có rubric nhất quán thường tốt hơn một model lớn train trên 200,000 ví dụ nhiễu.

Dataset preparation là nơi AI Engineer biến raw conversations, tickets, documents, tool traces và human feedback thành training artifacts có thể audit. Nếu dữ liệu chứa PII, license mơ hồ, duplicate, bad labels hoặc template sai, model sẽ học chính những lỗi đó với tốc độ rất cao.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Model memorized secrets từ dataset support">
**Bối cảnh:** Một công ty fine-tune customer-support assistant bằng ticket lịch sử. Dataset được export nhanh từ helpdesk, chuyển sang JSONL và train QLoRA trong một đêm.

**Sự cố:** Trong red-team test, model đôi khi sinh ra email khách hàng thật, order ID thật và một đoạn API token từng xuất hiện trong ticket debug. Ngoài ra, model lặp lại câu trả lời boilerplate vì nhiều ticket duplicate được đưa vào train.

1. Không chạy PII/secrets scan trước training.
2. Không dedup near-duplicate tickets.
3. Train/test split bị leakage theo cùng customer thread.
4. Không có dataset card hoặc data owner phê duyệt.

**Hậu quả:** Dự án bị dừng để điều tra privacy incident. Team phải xóa checkpoint, rebuild dataset pipeline và thêm data governance gate trước mọi training run.
</Callout>

Bài học rút ra: dữ liệu fine-tuning là production asset. Nếu bạn không thể giải thích nguồn, license, cleaning steps, split và rủi ro privacy của dataset, bạn chưa sẵn sàng train.

## 2. Architecture Blueprint (Fine-Tuning Data Factory)

Một data factory tốt biến raw data thành dataset artifact có version:

```text
[ Raw sources ]
  - support tickets
  - expert answers
  - tool traces
  - documents
  - preference labels
        │
        ▼
[ Ingestion + normalization ]
  - canonical schema
  - role mapping
  - language/domain tags
        │
        ▼
[ Quality filters ]
  - empty/low-quality removal
  - PII/secrets scan
  - toxicity/safety filters
  - license checks
        │
        ▼
[ Dedup + leakage checks ]
  - exact dedup
  - near-dedup with MinHash
  - group split by user/thread/doc
        │
        ▼
[ Template validation ]
  - chat template
  - tool-call schema
  - length distribution
  - stop tokens
        │
        ▼
[ Dataset splits ]
  - train
  - validation
  - test
  - red-team/regression
        │
        ▼
[ Dataset card + manifest ]
  - source/license/PII
  - stats
  - known limitations
  - owner approval
```

Mục tiêu: dataset không chỉ “train được”, mà còn có thể audit, reproduce và defend trước incident.

## 3. Common Dataset Formats

### Instruction format

Dùng cho SFT đơn giản:

```json
{
  "instruction": "Tóm tắt policy hoàn tiền trong 3 bullet points.",
  "input": "Khách hàng mua gói annual và yêu cầu refund sau 10 ngày...",
  "output": "- Khách hàng đủ điều kiện..."
}
```

Ưu điểm: dễ tạo, dễ đọc. Nhược điểm: không biểu diễn nhiều turn hoặc tool calls tốt.

### Chat format

Dùng cho instruct/chat models:

```json
{
  "messages": [
    {"role": "system", "content": "You are a support assistant."},
    {"role": "user", "content": "Tôi muốn hoàn tiền."},
    {"role": "assistant", "content": "Mình cần kiểm tra ngày mua và gói dịch vụ của bạn."}
  ]
}
```

Đây là format an toàn nhất cho SFT chat nếu bạn dùng đúng tokenizer `apply_chat_template`.

### Tool-calling format

Dùng khi model cần học gọi function/API:

```json
{
  "messages": [
    {"role": "user", "content": "Kiểm tra đơn hàng ORD-123"},
    {
      "role": "assistant",
      "tool_calls": [
        {
          "name": "lookup_order",
          "arguments": {"order_id": "ORD-123"}
        }
      ]
    },
    {"role": "tool", "name": "lookup_order", "content": "{\"status\":\"delivered\"}"},
    {"role": "assistant", "content": "Đơn hàng ORD-123 đã được giao."}
  ]
}
```

Tool data cần schema validation nghiêm ngặt vì downstream tool có thể gây side effects.

### Preference format cho DPO/RLHF-style training

```json
{
  "prompt": "Giải thích lỗi này cho user không chuyên.",
  "chosen": "Lỗi này nghĩa là ứng dụng không kết nối được máy chủ...",
  "rejected": "Bạn cấu hình sai rồi. Tự kiểm tra lại đi."
}
```

Preference data không chỉ cần “chosen tốt hơn rejected”; nó cần rubric rõ: helpfulness, correctness, safety, tone, conciseness, evidence.

## 4. Data Quality Gates

### Schema validation

Trước khi tokenization, validate từng record:

| Gate | Fail nếu |
|---|---|
| Required fields | Thiếu `messages`, `instruction`, `output` |
| Role order | Assistant trả lời trước user hoặc role không hợp lệ |
| Empty content | Message rỗng hoặc chỉ whitespace |
| Tool schema | Arguments không parse được JSON/schema |
| Label consistency | `chosen` và `rejected` giống nhau |

### Length distribution

Một vài outliers dài có thể làm training chậm hoặc OOM.

```python
import numpy as np

lengths = [count_tokens(example) for example in dataset]
print({
    "p50": int(np.percentile(lengths, 50)),
    "p90": int(np.percentile(lengths, 90)),
    "p95": int(np.percentile(lengths, 95)),
    "p99": int(np.percentile(lengths, 99)),
    "max": max(lengths),
})
```

Rule of thumb:

- Nếu p99 quá dài nhưng phần lớn sample ngắn, cân nhắc truncate hoặc tách riêng long-context training.
- Đừng chọn `max_seq_length` chỉ vì 1% outliers.
- Luôn log truncation rate.

### PII and secrets filtering

Scan tối thiểu:

- Email, phone, address.
- API keys, tokens, private keys.
- Customer IDs nếu không được phép train.
- Health/financial/legal sensitive data.
- Internal URLs hoặc credentials trong logs.

<Callout type="danger" title="Checkpoint cũng là dữ liệu nhạy cảm">
Nếu model đã train trên secrets/PII, việc xóa raw dataset là chưa đủ. Checkpoint hoặc adapter có thể đã memorized thông tin đó. Hãy xử lý checkpoint như một artifact nhạy cảm và có policy xóa/rotate rõ.
</Callout>

## 5. Deduplication và Leakage Prevention

Duplicate làm model overfit và làm eval giả cao. Leakage xảy ra khi cùng một thông tin xuất hiện trong train và test.

### Exact dedup

```python
import hashlib

seen = set()
unique = []

for ex in examples:
    text = normalize_for_dedup(ex["text"])
    digest = hashlib.sha256(text.encode("utf-8")).hexdigest()
    if digest not in seen:
        seen.add(digest)
        unique.append(ex)
```

### Near-dedup với MinHash intuition

MinHash ước lượng Jaccard similarity giữa các tập shingles:

```text
text -> shingles -> minhash signature -> LSH buckets -> candidate duplicates
```

Ví dụ:

```python
from datasketch import MinHash, MinHashLSH

def shingles(text, n=5):
    tokens = text.lower().split()
    return {" ".join(tokens[i:i+n]) for i in range(max(0, len(tokens)-n+1))}

def make_minhash(text, num_perm=128):
    m = MinHash(num_perm=num_perm)
    for shingle in shingles(text):
        m.update(shingle.encode("utf-8"))
    return m

lsh = MinHashLSH(threshold=0.85, num_perm=128)

for idx, ex in enumerate(examples):
    mh = make_minhash(ex["text"])
    duplicates = lsh.query(mh)
    if not duplicates:
        lsh.insert(str(idx), mh)
        keep(ex)
```

### Split theo group, không split random mù

Nếu một customer thread có 10 messages liên quan, split random có thể đưa 8 messages vào train và 2 messages vào test. Eval sẽ bị leakage.

Split tốt hơn:

```text
Group key examples:
- customer_id
- ticket_id
- document_id
- repository_id
- conversation_thread_id
```

## 6. Dataset Mixing và Curriculum

Fine-tuning trên một domain hẹp dễ gây forgetting. Dataset mixing giúp giữ năng lực nền.

| Mix component | Vai trò |
|---|---|
| Domain task data | Tăng chất lượng task chính |
| General instruction data | Giữ khả năng follow instruction chung |
| Safety/refusal data | Giữ policy behavior |
| Format/schema data | Tăng output validity |
| Hard negatives | Giảm lỗi phổ biến |

Ví dụ mix:

```text
70% domain support examples
15% general instruction examples
10% safety/refusal examples
5% format/tool-call edge cases
```

Curriculum đơn giản:

1. Bắt đầu với examples sạch, ngắn, rõ.
2. Thêm cases dài/phức tạp.
3. Thêm edge cases và adversarial prompts.
4. Eval từng checkpoint để phát hiện regression.

<Callout type="tip" title="Data mix là một hyperparameter">
Tỷ lệ dataset mix ảnh hưởng behavior mạnh như learning rate. Hãy version nó trong config và sweep khi cần.
</Callout>

## 7. Synthetic Data và Self-Instruct

Synthetic data hữu ích khi thiếu examples, nhưng nó dễ khuếch đại bias hoặc tạo câu trả lời nghe hay mà sai.

Pipeline an toàn hơn:

```text
[ Seed examples by experts ]
        │
        ▼
[ Generate synthetic variations ]
        │
        ▼
[ Automatic filters ]
  - schema
  - toxicity/PII
  - length
  - duplicate
        │
        ▼
[ Strong model critique / judge ]
        │
        ▼
[ Human spot-check ]
        │
        ▼
[ Add to training mix with weight cap ]
```

Rules:

- Không để synthetic data chiếm 100% dataset nếu task có rủi ro cao.
- Luôn giữ seed set do expert tạo.
- Gắn tag `synthetic=true` để phân tích eval theo nguồn dữ liệu.
- Dùng human review cho high-impact samples.

## 8. Implementation Playbook: Dataset Validator

Một validator tối thiểu nên chạy trước mọi training job:

```python
import json
import re
from collections import Counter

EMAIL_RE = re.compile(r"[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}")
SECRET_RE = re.compile(r"(sk-[A-Za-z0-9]{20,}|AKIA[0-9A-Z]{16})")

VALID_ROLES = {"system", "user", "assistant", "tool"}

def validate_chat_record(record):
    errors = []
    messages = record.get("messages")

    if not isinstance(messages, list) or not messages:
        return ["missing_or_empty_messages"]

    for i, msg in enumerate(messages):
        role = msg.get("role")
        content = msg.get("content", "")

        if role not in VALID_ROLES:
            errors.append(f"invalid_role_at_{i}")
        if role != "assistant" and not str(content).strip():
            errors.append(f"empty_content_at_{i}")
        if EMAIL_RE.search(str(content)):
            errors.append(f"email_pii_at_{i}")
        if SECRET_RE.search(str(content)):
            errors.append(f"secret_at_{i}")

    if messages[0].get("role") == "assistant":
        errors.append("assistant_starts_conversation")

    return errors

stats = Counter()
valid = []

with open("data/raw/chat.jsonl", "r", encoding="utf-8") as f:
    for line_no, line in enumerate(f, start=1):
        record = json.loads(line)
        errors = validate_chat_record(record)
        if errors:
            for err in errors:
                stats[err] += 1
        else:
            valid.append(record)

print("valid_records", len(valid))
print("errors", dict(stats))
```

## 9. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Train/test leakage">
**Triệu chứng:** Eval rất cao offline nhưng production kém.

**Nguyên nhân:** Duplicate hoặc cùng thread/document xuất hiện ở train và test.

**Cách khắc phục:** Dedup trước split, split theo group key, chạy similarity check giữa train/test.
</Callout>

<Callout type="warning" title="Pitfall 2: Chat template sai">
**Triệu chứng:** Model sinh role tokens, không dừng hoặc trả lời lẫn nhiều turn.

**Nguyên nhân:** Dataset format không khớp tokenizer/chat template của base model.

**Cách khắc phục:** Dùng `apply_chat_template`, inspect tokenized samples và test generation trước training dài.
</Callout>

<Callout type="warning" title="Pitfall 3: Synthetic data chất lượng thấp">
**Triệu chứng:** Model trả lời nghe mượt nhưng sai facts hoặc lặp style máy móc.

**Nguyên nhân:** Synthetic data không được critique/human review, tỷ lệ quá cao.

**Cách khắc phục:** Giới hạn synthetic ratio, dùng judge/rubric, human spot-check và eval theo data source.
</Callout>

<Callout type="warning" title="Pitfall 4: Preference labels không nhất quán">
**Triệu chứng:** DPO làm model xấu hơn SFT hoặc học tone sai.

**Nguyên nhân:** Annotators chọn `chosen` theo tiêu chí khác nhau.

**Cách khắc phục:** Viết rubric, đo inter-annotator agreement, audit sample disagreement và retrain labelers.
</Callout>

## 10. Data Governance & Dataset Unit Economics

Dữ liệu tốt tốn tiền, nhưng dữ liệu xấu làm hỏng model và tạo incident.

| Quyết định | Lợi ích | Trade-off |
|---|---|---|
| Human-written expert data | Chất lượng cao | Đắt, chậm |
| Synthetic augmentation | Nhanh, đa dạng | Cần filter/review |
| Dedup nghiêm ngặt | Giảm overfit/leakage | Có thể xóa nhầm near-duplicates hữu ích |
| PII review | Giảm privacy risk | Tốn thời gian/security process |
| Group split | Eval trung thực hơn | Ít random, cần metadata |
| Dataset card | Audit tốt | Cần discipline |

Production takeaway: dataset prep thường chiếm phần lớn thời gian fine-tuning. Đó không phải overhead; đó là chất lượng model.

## Answer key

1. Vì các message cùng thread/customer/document có thể bị chia giữa train và test, làm eval thấy các thông tin gần giống training.
2. Exact dedup xóa bản giống hệt; near-dedup phát hiện các mẫu rất giống nhau dù khác một vài từ hoặc format.
3. Vì model/checkpoint có thể memorize và sinh lại thông tin nhạy cảm; checkpoint sau training cũng trở thành artifact nhạy cảm.
4. Cần rubric rõ, consistency giữa annotators, metadata nguồn và validation để tránh model học preference sai.
5. Gắn tag nguồn, filter tự động, judge/critique, human spot-check và giới hạn tỷ lệ trong data mix.
