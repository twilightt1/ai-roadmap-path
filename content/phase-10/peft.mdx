Ở bài trước, bạn đã thấy Full Fine-Tuning có thể mạnh nhưng đắt, khó vận hành và dễ gây regression. Trong thực tế, phần lớn dự án fine-tuning hiện đại không cần cập nhật 100% trọng số. Chúng ta chỉ cần thêm một lớp thích ứng nhỏ để điều chỉnh hành vi model theo domain hoặc task.

Parameter-Efficient Fine-Tuning (PEFT) là nhóm kỹ thuật làm điều đó: đóng băng base model, chỉ train một số tham số nhỏ như adapter. Với AI Engineer, LoRA và QLoRA là lựa chọn mặc định khi cần thử nghiệm nhanh, tiết kiệm VRAM, version nhiều biến thể model và deploy theo tenant/task mà không nhân bản toàn bộ model.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Adapter train xong nhưng production trả lời như base model">
**Bối cảnh:** Một team fine-tune LoRA cho coding assistant nội bộ. Training loss giảm, notebook demo sinh code đúng style công ty. Team copy adapter lên inference server và bật traffic canary.

**Sự cố:** Production gần như không cải thiện so với base model. Một số request còn lỗi vì adapter không load được hoặc load nhầm revision của base model.

1. Adapter được train trên base model revision A nhưng deploy với revision B.
2. Target modules trong config training khác với inference loader.
3. Chat template training khác prompt template production.
4. Không có eval gate so sánh base vs adapter trước khi canary.

**Hậu quả:** Canary bị rollback. Team phải thêm adapter manifest, pin base model hash, chuẩn hóa chat template và chạy eval tự động trước deploy.
</Callout>

Bài học rút ra: PEFT giảm chi phí training, nhưng tăng yêu cầu quản lý artifact. Adapter chỉ có ý nghĩa khi base model, tokenizer, template, target modules và inference runtime khớp chính xác.

## 2. Architecture Blueprint (Adapter Lifecycle)

PEFT production không chỉ là một file `adapter_model.safetensors`. Nó là một lifecycle:

```text
[ Base model registry ]
  - model id
  - revision/hash
  - tokenizer
        │
        ▼
[ PEFT training config ]
  - r, alpha, dropout
  - target_modules
  - quantization config
        │
        ▼
[ Train adapter ]
  - frozen base weights
  - train LoRA matrices
  - log metrics/config
        │
        ▼
[ Adapter registry ]
  - adapter weights
  - base model hash
  - dataset version
  - chat template
        │
        ├──► [ Serve dynamically ]
        │       - load base once
        │       - route adapters by task/tenant
        │
        └──► [ Merge for deployment ]
                - merge adapter into base
                - export standalone checkpoint
                - run eval again
```

Điểm cần nhớ: adapter là delta so với một base model cụ thể, không phải model độc lập.

## 3. LoRA: Low-Rank Adaptation

LoRA dựa trên giả thuyết rằng thay đổi cần thiết khi fine-tune có intrinsic rank thấp. Thay vì update trực tiếp ma trận lớn `W`, ta học một delta có rank thấp:

```text
Original linear layer:
y = W0 x

LoRA adapted layer:
y = W0 x + scale * B A x

where:
A shape = r x input_dim
B shape = output_dim x r
scale = alpha / r
r = LoRA rank
```

Base matrix `W0` bị freeze. Chỉ `A` và `B` được train.

### Vì sao tiết kiệm tham số?

Nếu một projection có kích thước `4096 x 4096`:

```text
Full update params = 4096 * 4096 = 16,777,216
LoRA r=16 params   = 16 * 4096 + 4096 * 16 = 131,072
```

Chỉ khoảng 0.78% số tham số của layer đó được train.

### Target modules

Trong Transformer, LoRA thường gắn vào các linear projections:

| Module | Ý nghĩa | Khi target |
|---|---|---|
| `q_proj`, `k_proj`, `v_proj`, `o_proj` | Attention projections | Gần như mặc định |
| `gate_proj`, `up_proj`, `down_proj` | MLP layers | Khi cần capacity cao hơn |
| `lm_head` | Output vocabulary head | Cẩn thận, dễ tăng memory/risk |
| Embeddings | Token representations | Chỉ khi thêm special tokens/domain vocab |

Rule of thumb:

- Bắt đầu với attention projections.
- Nếu underfit, thêm MLP projections.
- Nếu vẫn underfit, tăng rank hoặc cân nhắc Full FT.

## 4. LoRA Hyperparameters

| Hyperparameter | Ý nghĩa | Rule of thumb |
|---|---|---|
| `r` | Rank của adapter | 8, 16, 32 là điểm bắt đầu phổ biến |
| `lora_alpha` | Scaling factor | thường 2x đến 4x rank |
| `lora_dropout` | Regularization | 0.0-0.1 tùy dataset |
| `target_modules` | Layer được gắn adapter | attention trước, MLP sau |
| `bias` | Có train bias không | thường `none` |

### Rank quá thấp vs quá cao

| Triệu chứng | Có thể do | Hướng xử lý |
|---|---|---|
| Train loss không giảm, task không cải thiện | Rank quá thấp hoặc target modules thiếu | Tăng rank, thêm MLP modules |
| Train tốt nhưng val xấu | Rank quá cao, data nhỏ, overfit | Giảm rank/dropout, thêm data |
| Adapter lớn, serve chậm | Rank/target quá rộng | Giảm target modules hoặc merge |
| Output lệch style quá mạnh | LR/epoch quá cao | Giảm LR, giảm epoch |

<Callout type="tip" title="Đừng tune rank bằng cảm giác">
Hãy chạy sweep nhỏ: `r=8,16,32` với cùng dataset split và eval harness. Chọn adapter theo validation metric + regression metric, không chọn theo train loss.
</Callout>

## 5. QLoRA: Fine-Tuning trên Quantized Base Model

QLoRA giúp train adapter trên base model đã quantize 4-bit. Base weights được nén để tiết kiệm VRAM, còn LoRA adapter vẫn train ở precision phù hợp.

```text
[ 4-bit quantized base model ]  frozen
          +
[ LoRA adapter weights ]        trainable
          +
[ paged optimizer ]             reduce memory spikes
```

Các thành phần chính:

| Thành phần | Vai trò |
|---|---|
| NF4 | 4-bit format tối ưu cho phân phối normal của weights |
| Double quantization | Quantize thêm quantization constants để tiết kiệm memory |
| Paged optimizer | Giảm memory spike bằng unified memory/paging |
| BF16 compute | Tính toán ổn định hơn FP16 trên GPU hỗ trợ |

QLoRA phù hợp khi:

- Bạn chỉ có 16GB-24GB VRAM.
- Muốn fine-tune 7B/8B hoặc đôi khi 13B trên single GPU.
- Chấp nhận inference/deployment phải xử lý quantized base hoặc merge/export cẩn thận.

## 6. Implementation Playbook: PEFT/QLoRA Config Skeleton

Một cấu hình PEFT tốt phải explicit về quantization, LoRA config và training config.

```python
from transformers import AutoModelForCausalLM, AutoTokenizer, BitsAndBytesConfig
from peft import LoraConfig
from trl import SFTTrainer, SFTConfig

BASE_MODEL = "meta-llama/Meta-Llama-3-8B-Instruct"

bnb_config = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_use_double_quant=True,
    bnb_4bit_compute_dtype="bfloat16",
)

model = AutoModelForCausalLM.from_pretrained(
    BASE_MODEL,
    quantization_config=bnb_config,
    device_map="auto",
)

tokenizer = AutoTokenizer.from_pretrained(BASE_MODEL)
tokenizer.pad_token = tokenizer.eos_token

peft_config = LoraConfig(
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    bias="none",
    task_type="CAUSAL_LM",
    target_modules=[
        "q_proj", "k_proj", "v_proj", "o_proj",
        "gate_proj", "up_proj", "down_proj",
    ],
)

training_args = SFTConfig(
    output_dir="./adapters/code-assistant-lora-v1",
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    learning_rate=2e-4,
    num_train_epochs=2,
    max_seq_length=4096,
    packing=True,
    bf16=True,
    logging_steps=10,
    save_steps=250,
    report_to="wandb",
)

trainer = SFTTrainer(
    model=model,
    tokenizer=tokenizer,
    args=training_args,
    train_dataset=train_dataset,
    eval_dataset=eval_dataset,
    peft_config=peft_config,
    formatting_func=format_chat,
)

trainer.train()
trainer.model.save_pretrained("./adapters/code-assistant-lora-v1")
tokenizer.save_pretrained("./adapters/code-assistant-lora-v1")
```

### Adapter manifest

Luôn lưu manifest cạnh adapter:

```json
{
  "adapter_id": "code-assistant-lora-v1",
  "base_model": "meta-llama/Meta-Llama-3-8B-Instruct",

  "tokenizer_revision": "same-as-base",
  "chat_template": "llama3-instruct-v1",
  "dataset_version": "code_sft_2026_07_01",
  "peft": {
    "r": 16,
    "lora_alpha": 32,
    "target_modules": ["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"]
  },
  "eval_report": "reports/code-assistant-lora-v1.json"
}
```

## 7. Serving: Dynamic Adapter vs Merge

| Strategy | Cách hoạt động | Ưu điểm | Nhược điểm |
|---|---|---|---|
| Dynamic adapter | Load base model một lần, swap adapter theo request/task | Multi-tenant rẻ, nhiều biến thể | Runtime phức tạp, adapter load latency |
| Merged model | Merge LoRA vào base và export checkpoint | Inference đơn giản, latency ổn định | Mỗi model là một artifact lớn |
| Quantized adapter serving | Serve base quantized + adapter | VRAM thấp | Toolchain/runtime compatibility |

Merge cần eval lại vì numerical behavior có thể thay đổi nhẹ.

```python
from peft import PeftModel
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained(BASE_MODEL, torch_dtype="bfloat16")
model = PeftModel.from_pretrained(base, "./adapters/code-assistant-lora-v1")
merged = model.merge_and_unload()
merged.save_pretrained("./models/code-assistant-merged-v1")
```

<Callout type="warning" title="Adapter routing là một production system">
Nếu bạn route adapter theo tenant/task, hãy log adapter id, base model revision, prompt template version và eval cohort cho từng request. Nếu không, incident debugging gần như bất khả thi.
</Callout>

## 9. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Base model mismatch">
**Triệu chứng:** Adapter load được nhưng quality rất kém hoặc output kỳ lạ.

**Nguyên nhân:** Adapter train trên base revision khác với inference base.

**Cách khắc phục:** Pin base model revision/hash trong manifest và validate khi load.
</Callout>

<Callout type="warning" title="Pitfall 2: Target modules sai">
**Triệu chứng:** Loss không giảm hoặc adapter gần như không tác động.

**Nguyên nhân:** Tên module không khớp architecture hoặc chỉ target quá ít layer.

**Cách khắc phục:** In `model.named_modules()`, chọn đúng `q_proj/v_proj/...`, thêm MLP modules nếu underfit.
</Callout>

<Callout type="warning" title="Pitfall 3: Quantization instability">
**Triệu chứng:** NaN loss, training không ổn định, output degrade.

**Nguyên nhân:** Precision config không phù hợp, GPU không hỗ trợ BF16 tốt, LR quá cao.

**Cách khắc phục:** Dùng BF16 nếu có, giảm LR, kiểm tra bitsandbytes version, thử 8-bit hoặc non-quantized PEFT nếu cần.
</Callout>

<Callout type="warning" title="Pitfall 4: Train/inference template mismatch">
**Triệu chứng:** Eval offline tốt nhưng production trả lời sai format.

**Nguyên nhân:** Training dùng chat template khác prompt builder production.

**Cách khắc phục:** Dùng cùng tokenizer `apply_chat_template`, version prompt template và test generated prompts trước deploy.
</Callout>

## 10. Cost & Adapter Unit Economics

PEFT thường thắng Full FT về chi phí, nhưng không miễn phí.

| Quyết định | Lợi ích | Trade-off |
|---|---|---|
| QLoRA single GPU | Rẻ, dễ thử nghiệm | Phụ thuộc quantization runtime |
| Rank cao | Capacity cao hơn | Adapter lớn, dễ overfit |
| Target MLP + attention | Chất lượng tốt hơn | Memory/time tăng |
| Dynamic adapter serving | Multi-tenant hiệu quả | Routing/logging phức tạp |
| Merge adapter | Inference đơn giản | Mất lợi thế adapter nhỏ |

Estimate đơn giản:

```text
adapter_storage_saving = full_model_size_per_variant - adapter_size_per_variant
training_saving = full_ft_gpu_cost - peft_gpu_cost
ops_cost = adapter_registry + routing + compatibility testing
```

Production takeaway: PEFT rẻ nhất khi bạn có nhiều biến thể model trên cùng một base, nhưng chỉ an toàn khi artifact management đủ nghiêm túc.

## Answer key

1. Vì LoRA freeze ma trận gốc và chỉ train hai ma trận low-rank nhỏ `A` và `B` để biểu diễn delta.
2. QLoRA quantize base model weights xuống 4-bit, trong khi adapter vẫn là phần trainable nhỏ.
3. Không. Adapter là delta phụ thuộc base model, tokenizer và template cụ thể.
4. Khi underfit: train/eval task không cải thiện dù data, LR và target modules đã hợp lý.
5. Vì merge có thể thay đổi numerical behavior, deployment artifact khác runtime adapter và cần kiểm tra regression/latency.

## Điểm cần nhớ

- PEFT là lựa chọn mặc định cho phần lớn fine-tuning LLM thực dụng.
