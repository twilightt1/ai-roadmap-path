Phase 12 đã giúp bạn đưa AI system vào production: data versioning, experiment tracking, training infrastructure, eval gates, serving, cloud, monitoring, CI/CD và security. Phase 13 chuyển sang năng lực senior hơn: **thiết kế hệ thống AI end-to-end** trước khi viết code.

AI System Design không chỉ là vẽ vài box “Frontend → Backend → LLM”. Bạn phải làm rõ yêu cầu sản phẩm, chọn pattern AI phù hợp, quyết định model placement, thiết kế data flow, kiểm soát latency/cost, có fallback khi provider lỗi, có eval/monitoring và có guardrails. Nếu thiếu framework, hệ thống AI có thể chạy demo tốt nhưng sập khi gặp traffic thật.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Viral launch làm app tóm tắt PDF đốt 12,000 USD trong một buổi sáng">
**Bối cảnh:** Một startup ra mắt tính năng “tóm tắt tài liệu pháp lý”. Kiến trúc ban đầu rất đơn giản: Frontend upload PDF, FastAPI parse PDF, gửi toàn bộ 30,000 tokens lên GPT-4o, chờ model trả lời rồi response lại cho user.

**Sự cố:** Một KOL chia sẻ sản phẩm. Trong một giờ, 5,000 người dùng upload cùng một bộ luật phổ biến.

1. Backend xử lý đồng bộ, mỗi request giữ worker 20-40 giây.
2. Không có queue, task id hoặc retry-safe workflow.
3. Không có exact/semantic cache nên cùng một PDF được xử lý lặp lại hàng nghìn lần.
4. Không có rate limit/token budget nên user reload liên tục tạo request mới.
5. Không có cost alert theo feature nên team chỉ biết khi hóa đơn đã tăng đột biến.

**Hậu quả:** API timeout, người dùng retry, chi phí token nhân lên theo cấp số nhân. Hệ thống phải tắt khẩn cấp để thiết kế lại.
</Callout>

Bài học rút ra: AI System Design bắt đầu từ **constraints**, không bắt đầu từ model. Nếu bạn không xác định latency, throughput, cost/request, dữ liệu nhạy cảm và failure mode, mọi lựa chọn kiến trúc sau đó đều là phỏng đoán.

## 2. Architecture Blueprint (AI System Design Decision Flow)

Một framework thiết kế AI system production nên đi theo luồng quyết định sau:

```text
[ 1. Clarify requirements ]
  - user types, RPS, concurrency
  - latency SLA, async tolerance
  - cost/request and monthly budget
  - privacy/compliance domain
        │
        ▼
[ 2. Identify AI components ]
  - LLM call, RAG, agent, router
  - embeddings, reranker, classifier
  - eval, guardrails, moderation
        │
        ▼
[ 3. Design data flow ]
  - request path
  - ingestion path
  - state and storage
  - API contracts and task lifecycle
        │
        ▼
[ 4. Choose placement and scaling ]
  - server / edge / client / async worker
  - model provider vs self-host
  - cache and queue boundaries
        │
        ▼
[ 5. Reliability, cost, security ]
  - fallback and circuit breaker
  - token budgets and rate limits
  - observability and eval gates
  - tenant isolation and guardrails
```

Output của system design không phải là một sơ đồ đẹp; output là một tập quyết định có thể review, triển khai, đo lường và thay đổi khi traffic/cost thay đổi.

## 3. Clarify Requirements: Câu hỏi trước khi vẽ kiến trúc

Một AI system design interview hoặc design doc nên bắt đầu bằng câu hỏi. Đừng nhảy ngay vào chọn GPT-4o hay vector DB.

| Nhóm câu hỏi | Ví dụ cần làm rõ | Ảnh hưởng kiến trúc |
|---|---|---|
| User & traffic | DAU, RPS, peak concurrency, tenant count | scale, cache, queue, DB partitioning |
| Latency | realtime dưới 1s hay async vài phút được? | sync API vs task queue |
| Quality | cần factuality, citation, format JSON, safety? | eval, RAG, structured output |
| Cost | cost/request tối đa, budget/tháng | model routing, caching, max tokens |
| Data | dữ liệu public hay PII/regulated? | region, encryption, guardrails |
| Freshness | data thay đổi mỗi phút hay mỗi ngày? | cache invalidation, ingestion pipeline |
| Failure tolerance | provider lỗi thì degrade hay fail closed? | fallback, circuit breaker |
| Audit | cần trace/citation/log retention không? | observability, storage design |

Một câu trả lời system design tốt thường nói rõ giả định:

```text
Assumption:
- 50,000 daily active users
- p95 latency target: 3 seconds for chat, 2 minutes for PDF ingestion
- max cost/request: 0.5 cents for normal chat, 5 cents for premium analysis
- documents may contain PII, region: EU only
```

Không có giả định, bạn không thể đánh đổi.

## 4. AI Component Map

Sau requirements, hãy xác định hệ thống cần những component AI nào.

| Component | Khi cần | Câu hỏi thiết kế |
|---|---|---|
| LLM generation | chat, summary, reasoning | model nào, context length, structured output? |
| RAG | cần knowledge riêng/cập nhật | chunking, embeddings, vector DB, reranker, ACL? |
| Agent/tool use | cần hành động/dữ liệu động | tool permissions, max iterations, audit? |
| Classifier/router | cần phân luồng/cost optimization | threshold, fallback, latency? |
| Embedding service | semantic search/cache | provider vs self-host, dimensions, versioning? |
| Reranker | improve retrieval precision | latency/cost trade-off? |
| Guardrails | safety, PII, prompt injection | input/output/action policies? |
| Eval pipeline | release quality | golden set, judge, regression gates? |
| Observability | debug/cost/quality | traces, token usage, feedback? |

Ví dụ: “AI customer support” có thể cần intent classifier, RAG, LLM generator, escalation router, safety guardrails và cost monitoring. “Autocomplete” có thể chỉ cần small model + debounce + cache.

## 5. Data Flow and API Contracts

AI systems thường có nhiều luồng dữ liệu song song:

```text
User path:
client -> gateway -> auth/rate limit -> cache -> router -> RAG/LLM -> response

Ingestion path:
upload -> storage -> queue -> parser/OCR -> chunk -> embed -> vector DB -> index ready

Evaluation path:
candidate prompt/model -> golden set -> judge -> report -> release gate

Monitoring path:
request spans -> metrics/traces -> eval samples -> alerts
```

Mỗi luồng cần API contract rõ ràng. Ví dụ async PDF analysis:

```http
POST /documents/analyze
Response 202 Accepted
{
  "task_id": "task_123",
  "status_url": "/tasks/task_123"
}

GET /tasks/task_123
{
  "status": "running",
  "progress": 0.45,
  "result": null,
  "error": null
}
```

Nếu contract không rõ trạng thái, frontend sẽ poll sai, user retry sai và backend xử lý trùng task.

## 6. Non-Functional Requirements for AI

NFR truyền thống gồm availability, latency, scalability. Với AI, thêm các NFR đặc thù:

| NFR | Metric ví dụ | Thiết kế liên quan |
|---|---|---|
| Quality | golden eval score, thumbs-down rate | eval gates, feedback loop |
| Groundedness | hallucination rate | RAG, citation, judge eval |
| Cost | cost/request, cost/user/month | token budgets, caching, routing |
| Latency | TTFT, p95 E2E | streaming, async, placement |
| Safety | unsafe output rate | moderation, guardrails |
| Privacy | PII leakage incidents | redaction, region, logging policy |
| Resilience | fallback success rate | multi-provider routing |
| Auditability | trace coverage, citation coverage | observability, provenance |

Production AI system design phải nói được bạn đo các NFR này ở đâu và dùng metric nào để rollback.

## Implementation Playbook: AI Design Doc + ADR Template

### Bước 1: Viết one-page problem framing

```md
# AI Feature Design: Contract Summarizer

## Goal
Allow legal users to upload contracts and receive a structured summary with citations.

## Users
- Internal legal analysts
- 1,000 monthly active users
- Peak: 50 concurrent document jobs

## Constraints
- Documents may contain PII and confidential clauses
- Region: EU only
- Target: result within 2 minutes
- Budget: under $0.08 per document
```

### Bước 2: Vẽ architecture diagram có luồng sync/async

```text
[ Web app ]
    │ POST /documents
    ▼
[ API Gateway ] -> [ Auth / Quota ] -> [ Object Storage ]
    │                                  │
    │ 202 task_id                       ▼ event
    ▼                              [ Queue ]
[ Task Status API ]                    │
    ▲                                  ▼
    │                            [ Workers ]
    │                              - OCR
    │                              - chunk
    │                              - RAG summary
    │                                  │
    └──────────── [ Result DB + Evidence/Citations ]
```

### Bước 3: Ghi model/data decisions

```md
## Model Decision
- Use GPT-4o-mini for extraction and citation drafting.
- Escalate to GPT-4o only when confidence score is low or document risk is high.
- Use structured output JSON schema for result validation.

## Data Decision
- Store raw PDF in encrypted EU bucket.
- Store extracted text and chunks with document_id and tenant_id.
- Do not log raw contract text to SaaS observability.
```

### Bước 4: Viết ADR cho quyết định quan trọng

```md
# ADR-003: Use Async Task Queue for Document Analysis

## Status
Accepted

## Context
Document analysis can take 30-120 seconds. Synchronous HTTP requests timeout and block API workers.

## Decision
Use POST /documents to create a task, enqueue work in Redis/RabbitMQ, process in workers, and expose GET /tasks/{id} plus optional webhook.

## Consequences
- Better reliability and scaling.
- Requires task state management and idempotency keys.
- Frontend must support progress UI.
```

### Bước 5: Thêm risk table

```md
| Risk | Mitigation | Owner |
|---|---|---|
| Provider outage | fallback to secondary provider, retry budget | platform |
| PII in logs | redaction before tracing, retention 7 days | security |
| Cost spike | per-tenant token budget, alerts | FinOps |
| Hallucination | citations + groundedness eval | AI team |
```

## Project Guide: Design Doc for AI Feature (Phase 13 Easy)

Mục tiêu project: viết một design doc 5 trang cho một AI feature thực tế trước khi code.

### Chọn một feature

Ví dụ:

- Contract summarizer,
- customer support chatbot,
- PDF Q&A for internal docs,
- meeting transcript summarizer,
- AI email assistant.

### Deliverables

Design doc phải có:

1. Problem statement và success metrics.
2. Assumptions: users, RPS, latency, budget, risk domain.
3. Architecture diagram.
4. Data flow cho request path và background path.
5. Model selection và fallback strategy.
6. Cost estimate theo request/user/month.
7. Security/privacy considerations.
8. Monitoring/eval plan.
9. 2-3 ADRs cho quyết định lớn.

### Acceptance criteria

- Có ít nhất một async hoặc caching decision nếu feature có latency/cost cao.
- Có cost/request estimate, không chỉ “dùng model tốt nhất”.
- Có fallback khi provider lỗi hoặc model confidence thấp.
- Có kế hoạch eval/monitoring trước production.
- Có rủi ro và mitigation rõ ràng.

Stretch goal: vẽ diagram bằng Mermaid hoặc Excalidraw và thêm sequence diagram cho request chính.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Thiết kế bắt đầu bằng model thay vì requirement">
**Triệu chứng:** Team chọn GPT-4o cho mọi use case, sau đó mới phát hiện cost/request vượt biên lợi nhuận.

**Fix:** Luôn bắt đầu bằng SLA, quality target, cost budget và risk domain. Model là quyết định sau, không phải điểm xuất phát.
</Callout>

<Callout type="warning" title="Pitfall 2: Sơ đồ không có failure path">
**Triệu chứng:** Provider lỗi 503 làm toàn bộ product trả 500; không có fallback hoặc degrade mode.

**Fix:** Vẽ explicit failure path: timeout, circuit breaker, fallback provider, cached response, hoặc “try again later” có kiểm soát.
</Callout>

<Callout type="danger" title="Pitfall 3: Không thiết kế cost guardrails từ đầu">
**Triệu chứng:** Traffic tăng là bill tăng tuyến tính không giới hạn; attacker có thể wallet-drain bằng prompt dài.

**Fix:** Gateway phải có token estimate, per-user budget, max tokens, rate limits và alerts ngay từ bản đầu.
</Callout>

## Cost / Latency / Reliability Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Dùng frontier model cho mọi request | quality cao ban đầu | gross margin xấu, khó scale |
| Router model nhỏ trước | giảm cost | risk route sai, cần eval |
| Async processing | tránh timeout, scale tốt | UX phức tạp, cần task state |
| Semantic cache | giảm LLM calls | embedding cost, stale/security risk |
| Multi-provider fallback | resilience tốt | integration và eval phức tạp |
| Log raw prompts | debug dễ | privacy/compliance risk |

AI System Design là nghệ thuật chuyển requirements thành trade-offs có thể đo được.
