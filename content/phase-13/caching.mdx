Ở bài trước, bạn đã chọn nơi inference chạy: server, edge, client hoặc async worker. Nhưng placement tốt vẫn không giải quyết một sự thật: nhiều request AI không cần tính lại. Nếu câu hỏi giống nhau, tài liệu giống nhau hoặc prompt prefix giống nhau, cách tối ưu nhất là **không gọi model**.

**Caching Strategies** trong AI phức tạp hơn web caching truyền thống vì hai câu khác chữ có thể cùng nghĩa, answer có thể chứa dữ liệu cá nhân, context RAG có version, prompt thay đổi theo thời gian và provider có prompt cache riêng. Cache tốt có thể giảm 50-80% cost; cache sai có thể leak dữ liệu hoặc trả chính sách cũ.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Semantic cache leak link bảng lương của HR director">
**Bối cảnh:** Một chatbot HR dùng semantic cache để giảm chi phí. Cache key chỉ dựa trên embedding của câu hỏi. Value là answer cuối của LLM.

**Sự cố:** HR director hỏi: “Tôi xem bảng lương cá nhân ở đâu?” LLM trả link Google Drive riêng của director. Một nhân viên khác hỏi: “Làm sao kiểm tra bảng lương của tôi?” Semantic similarity đạt 0.98, hệ thống trả lại answer cached của director cho nhân viên.

1. Cache key không chứa `tenant_id`, `user_id`, role hoặc ACL context.
2. Cache value chứa dữ liệu personalized nhưng được dùng như public answer.
3. Không phân loại cacheable vs non-cacheable response.
4. Không có DLP/safety check trước khi write cache.

**Hậu quả:** Rò rỉ PII nghiêm trọng. Semantic cache bị tắt toàn hệ thống và team phải thiết kế lại tenant isolation.
</Callout>

Bài học rút ra: AI caching là bài toán **performance + security + freshness**. Cache hit không đáng giá nếu nó trả dữ liệu sai người hoặc sai phiên bản tài liệu.

## 2. Architecture Blueprint (Multi-Tier AI Cache Gateway)

Một cache gateway an toàn nên chạy theo tầng:

```text
[ Request ]
  - tenant_id
  - user_id / role
  - prompt_version
  - content_version
  - model_version
  - query
        │
        ▼
[ Cache policy check ]
  - cacheable task?
  - personalized output?
  - sensitive domain?
        │
        ▼
[ L1: Exact cache - Redis ]
  key = hash(scope + normalized query + versions)
        │ miss
        ▼
[ L2: Semantic cache - Vector DB ]
  filter by tenant/user/ACL/scope
  similarity threshold + intent/action check
        │ miss
        ▼
[ L3: Prompt/prefix cache / provider cache ]
  stable prompt prefix, reusable context
        │ miss
        ▼
[ LLM / RAG / Inference ]
        │
        ▼
[ Safety + cache write policy ]
  - no PII/private link unless user-scoped
  - TTL and invalidation metadata
```

Mục tiêu: exact cache trước vì rẻ và nhanh, semantic cache sau vì tốn embedding/vector search, rồi mới gọi model.

## 3. Exact Cache

Exact cache dùng hash của input đã normalize. Đây là tầng nên có trong hầu hết AI gateway.

```text
key = sha256(tenant_id + user_scope + prompt_version + model_version + content_version + normalized_query)
```

Nên include:

- tenant/user scope,
- prompt version,
- model version nếu output thay đổi theo model,
- RAG content/index version,
- locale/language,
- tool/config flags,
- normalized query.

Không nên include timestamp hiện tại nếu không cần, vì nó làm cache miss mọi lúc.

Exact cache phù hợp cho:

- FAQ public,
- deterministic classification,
- template generation,
- repeated eval cases,
- expensive but stable summaries.

---

## 4. Semantic Cache

Semantic cache dùng embeddings để tìm request tương đương về nghĩa.

```text
query -> embedding -> vector search within allowed scope -> similar cached answer
```

Nhưng semantic similarity không hiểu business action đầy đủ. Ví dụ:

- “kích hoạt tài khoản” vs “hủy tài khoản” có thể gần nhau,
- “có được hoàn tiền không?” vs “làm sao từ chối hoàn tiền?” gần domain nhưng khác intent,
- “bảng lương của tôi” là personalized.

Vì vậy semantic cache cần guardrails:

| Guardrail | Lý do |
|---|---|
| tenant/user/ACL filter | tránh leak chéo |
| high threshold | giảm false hit |
| intent/action check | tránh trả answer opposite action |
| response cacheability label | không cache private output chung |
| TTL ngắn cho policy data | tránh stale answer |
| eval sample | đo false hit rate |

Ngưỡng similarity phụ thuộc domain. FAQ public có thể `0.92-0.95`; tài chính/pháp lý/HR nên cao hơn hoặc tắt semantic cache cho output personalized.

---

## 5. Prompt / Prefix Cache

Prompt caching có hai loại:

1. **Application prompt cache:** bạn lưu result hoặc context ở Redis/DB.
2. **Provider prefix cache:** OpenAI/Anthropic/vLLM cache token prefix giống hệt ở tầng model/KV cache.

Provider prompt cache chỉ hoạt động tốt khi phần đầu prompt ổn định:

```text
GOOD:
[ Long system instructions ]
[ Stable policy document/context ]
[ Few-shot examples ]
[ User-specific question at the end ]

BAD:
[ Current timestamp / random request id ]
[ User question ]
[ Long stable context after dynamic text ]
```

Nếu bạn đặt timestamp hoặc user-specific data lên đầu prompt, prefix cache bị phá.

---

## 6. RAG Cache and Invalidation

RAG có nhiều điểm cache:

| Cache layer | Cache gì | Invalidation |
|---|---|---|
| Query rewrite cache | query normalized/rewrite | prompt version change |
| Embedding cache | query embedding | embedding model version change |
| Retrieval cache | top-k chunk IDs | index/content version change |
| Rerank cache | ordered chunk IDs | reranker version change |
| Answer cache | final answer | prompt/model/content/user scope change |
| Citation cache | source snippets | document version change |

RAG cache key nên include `index_version` hoặc `content_version_hash`. Khi tài liệu thay đổi, cache cũ tự miss.

```text
cache_key = hash(tenant_id, query_norm, prompt_v, model_v, index_v, acl_scope)
```

Không có invalidation thì cache sẽ trả chính sách cũ dù vector DB đã cập nhật.

## Implementation Playbook: Safe Exact + Semantic Cache

### Bước 1: Cache scope và key builder

```python
import hashlib
import json
from dataclasses import dataclass

@dataclass(frozen=True)
class CacheScope:
    tenant_id: str
    user_id: str | None
    role: str
    prompt_version: str
    model_version: str
    content_version: str
    locale: str = "vi"


def normalize_query(query: str) -> str:
    return " ".join(query.lower().strip().split())


def exact_cache_key(scope: CacheScope, query: str) -> str:
    payload = {
        "tenant_id": scope.tenant_id,
        "user_id": scope.user_id,
        "role": scope.role,
        "prompt_version": scope.prompt_version,
        "model_version": scope.model_version,
        "content_version": scope.content_version,
        "locale": scope.locale,
        "query": normalize_query(query),
    }
    raw = json.dumps(payload, sort_keys=True, ensure_ascii=False)
    return hashlib.sha256(raw.encode("utf-8")).hexdigest()
```

### Bước 2: Cacheability policy

```python
def is_cacheable_response(answer: str, metadata: dict) -> bool:
    if metadata.get("contains_pii"):
        return False
    if metadata.get("personalized") and metadata.get("scope") != "user":
        return False
    if metadata.get("risk_domain") in {"medical", "legal", "financial"}:
        return metadata.get("grounded") is True
    return True
```

### Bước 3: Exact cache lookup/write

```python
def get_exact(redis_client, scope: CacheScope, query: str) -> str | None:
    key = exact_cache_key(scope, query)
    value = redis_client.get(f"ai:exact:{key}")
    return value.decode("utf-8") if value else None


def set_exact(redis_client, scope: CacheScope, query: str, answer: str, ttl_seconds: int = 3600):
    key = exact_cache_key(scope, query)
    redis_client.set(f"ai:exact:{key}", answer, ex=ttl_seconds)
```

### Bước 4: Semantic cache rules

```python
def semantic_cache_allowed(task_type: str, risk_domain: str, personalized: bool) -> bool:
    if personalized:
        return False
    if risk_domain in {"hr", "medical", "legal", "financial"}:
        return task_type in {"public_faq", "policy_summary"}
    return True


def semantic_threshold(risk_domain: str) -> float:
    if risk_domain in {"hr", "medical", "legal", "financial"}:
        return 0.97
    return 0.93
```

Production semantic cache cần vector DB filter theo tenant/ACL:

```text
vector_search(
  embedding=query_embedding,
  filter={tenant_id, content_version, visibility="public"},
  threshold=0.97
)
```

## 7. Cache Observability

Bạn cần đo cache như một subsystem riêng:

| Metric | Ý nghĩa |
|---|---|
| exact hit rate | phần trăm request được Redis trả ngay |
| semantic hit rate | phần trăm request tương đương nghĩa |
| false hit rate | cache hit nhưng user/judge đánh giá sai |
| cost saved | tokens/API cost tránh được |
| stale response incidents | trả dữ liệu cũ |
| cache latency | overhead của embedding/vector search |
| cache memory usage | Redis/vector DB capacity |

Semantic cache hit rate cao nhưng false hit rate cao là thảm họa. Hãy sample eval cached responses.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Stale cache sau khi chính sách thay đổi">
**Triệu chứng:** Người dùng vẫn nhận chính sách hoàn tiền 14 ngày dù tài liệu đã đổi thành 30 ngày.

**Fix:** Include `content_version_hash` trong cache key, TTL hợp lý, và purge theo doc/version khi ingest cập nhật.
</Callout>

<Callout type="warning" title="Pitfall 2: Semantic threshold quá thấp trả nhầm intent">
**Triệu chứng:** Hỏi “hủy tài khoản” nhưng cache trả hướng dẫn “kích hoạt tài khoản”.

**Fix:** Tăng threshold, thêm intent/action check, hoặc tắt semantic cache cho intent có hành động đối nghịch/high-risk.
</Callout>

<Callout type="danger" title="Pitfall 3: Cache poisoning">
**Triệu chứng:** Nhiều user nhận answer độc hại/sai từ cache chung.

**Fix:** Chỉ cache shared cho public/approved content, validate output trước write cache, isolate tenant/user scope và có purge tool khẩn cấp.
</Callout>

## Cost / Latency / Safety Unit Economics

| Cache strategy | Lợi ích | Chi phí/rủi ro | Best practice |
|---|---|---|---|
| Exact cache | Dưới 1ms, rất rẻ, an toàn hơn | hit rate thấp nếu query thay đổi | luôn đặt trước semantic cache |
| Semantic cache | tiết kiệm lớn cho FAQ tương tự | embedding latency/cost, false hit | filter scope + threshold cao |
| Prompt cache | giảm giá/latency token prefix | cần prompt prefix ổn định | đặt static prompt/context lên đầu |
| Retrieval cache | giảm vector/rerank latency | stale chunks nếu index đổi | include index/content version |
| Answer cache | giảm LLM calls mạnh nhất | privacy/staleness cao nhất | cacheability policy + TTL |

Caching tốt thường theo waterfall: exact trước, semantic có điều kiện, prompt cache qua prompt design, và LLM call cuối cùng.

2. Exact cache dùng hash chuỗi giống hệt; semantic cache dùng embedding để tìm câu tương đương nghĩa.

## Điểm cần nhớ

- AI cache phải cân bằng latency, cost, privacy và freshness.
- Exact cache rẻ và nên đứng trước semantic cache.
- Semantic cache phải filter theo tenant/user/ACL và dùng threshold phù hợp domain.
- RAG cache cần content/index version để tránh stale answer.
- Không cache shared cho output personalized hoặc chứa PII.
