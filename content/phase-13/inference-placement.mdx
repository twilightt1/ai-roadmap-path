Ở bài trước, bạn đã thấy các classic designs như chatbot memory, RAG, PDF Q&A và intent router. Nhưng mỗi design đều cần một quyết định nền tảng: **inference chạy ở đâu?** Backend server, GPU inference service riêng, edge node, browser/mobile client hay async worker?

**Inference Placement Strategy** quyết định latency, cost, security, data residency, offline capability và complexity vận hành. Chọn sai placement có thể làm lộ model weights, làm user chờ 30 giây, đốt GPU 24/7 hoặc vi phạm compliance vì dữ liệu đi sai region.

---

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Đưa model eKYC xuống client làm nóng máy và lộ trọng số độc quyền">
**Bối cảnh:** Một ngân hàng muốn giảm latency và tiết kiệm GPU server cho eKYC. Team nén model nhận diện khuôn mặt thành `.tflite` rồi bundle vào mobile app để chạy client-side.

**Sự cố:** Sau launch, điện thoại người dùng nóng, pin tụt nhanh và app bị OS kill trên thiết bị yếu. Hai ngày sau, hacker reverse engineer APK, trích xuất model weights, phân tích decision boundary và tạo ảnh giả vượt qua eKYC.

1. Model quá nặng cho thiết bị client phổ thông.
2. Model weights là IP/security asset nhưng bị phân phối công khai.
3. Không có server-side verification fallback.
4. Không phân tầng: client nên chỉ làm pre-check nhẹ, quyết định cuối nên server-side.

**Hậu quả:** App bị gỡ khẩn cấp, model độc quyền bị leak và security team yêu cầu redesign hybrid inference.
</Callout>

Bài học rút ra: client-side inference không miễn phí; bạn đổi cloud cost lấy battery cost, UX risk và model extraction risk. Placement là trade-off, không phải tối ưu một chiều.

## 2. Architecture Blueprint (Inference Placement Decision Matrix)

Một hệ thống AI có thể đặt inference ở nhiều vị trí:

```text
[ Client device ]
  - mobile NPU / WebGPU / browser
  - offline, privacy, low server cost
        │
        ▼
[ Edge runtime ]
  - Cloudflare Workers AI / regional functions
  - low network latency, stateless tasks
        │
        ▼
[ Backend API ]
  - secure orchestration, business logic
  - calls provider/self-host endpoint
        │
        ▼
[ Dedicated inference service ]
  - vLLM/TGI/TensorRT on GPU
  - batching, KV cache, autoscaling
        │
        ▼
[ Async worker / batch job ]
  - long-running analysis/generation
  - queue, task id, webhooks
```

Không phải chọn một. Production thường dùng hybrid placement: client prefilter, edge classify, backend route, GPU service generate, async worker xử lý tác vụ dài.

## 3. Server-Side Inference

Server-side inference là lựa chọn mặc định cho LLM/RAG/agent production.

```text
client -> API gateway -> backend/router -> inference endpoint/provider -> response
```

Ưu điểm:

- bảo vệ model weights và prompts,
- kiểm soát auth, rate limit, logging,
- dễ dùng model lớn,
- tích hợp RAG/database/tools an toàn hơn,
- dễ cập nhật model mà không ép user update app.

Nhược điểm:

- cloud/API cost cao,
- network latency,
- cần autoscaling và reliability,
- dữ liệu user đi qua server/provider.

Server-side phù hợp khi:

- model lớn hoặc proprietary,
- cần RAG/tools/private data,
- domain regulated,
- cần audit và centralized guardrails,
- output ảnh hưởng quyết định quan trọng.

---

## 4. Edge Inference

Edge inference chạy gần user hơn: CDN edge, regional functions, lightweight model endpoints.

```text
user in Vietnam -> edge node Singapore/HCMC -> small model classify/moderate -> backend if needed
```

Phù hợp cho tác vụ:

- intent classification,
- language detection,
- toxicity prefilter,
- lightweight rewrite,
- personalization đơn giản,
- low-latency global routing.

Không phù hợp khi:

- cần state/database ở một region xa,
- cần model lớn/GPU mạnh,
- cần job dài vượt timeout edge runtime,
- cần transaction phức tạp.

<Callout type="warning" title="Edge không thắng nếu dữ liệu nằm ở origin xa">
Nếu edge model xử lý 20ms nhưng vẫn phải gọi database ở `us-east-1` mất 250ms, tổng latency không giảm. Data gravity quyết định placement không kém model latency.
</Callout>

---

## 5. Client-Side / On-Device Inference

Client-side inference chạy trên browser/mobile/desktop:

- TensorFlow Lite,
- ONNX Runtime Mobile/Web,
- Core ML,
- WebGPU/WebNN,
- WebLLM cho mô hình nhỏ/quantized.

Phù hợp cho:

| Use case | Lý do |
|---|---|
| wake word / local STT nhẹ | cần phản hồi tức thì, privacy |
| toxicity/profanity prefilter | giảm request rác trước server |
| OCR/vision precheck nhẹ | giảm upload dữ liệu không cần thiết |
| autocomplete nhỏ/offline | latency thấp |
| accessibility features | offline và privacy |

Không phù hợp cho:

- model IP rất nhạy cảm,
- model nhiều GB trên web,
- thiết bị user yếu,
- output high-risk cần audit server-side,
- logic cần secrets/private data.

Client-side guardrail không phải security control cuối cùng. User có thể sửa JS, bypass mobile hook hoặc gọi API trực tiếp.

---

## 6. Async Worker Inference

Một placement hay bị quên: inference không nằm trong request path mà nằm trong worker.

```text
POST /generate-video -> task_id
queue -> GPU worker -> storage -> webhook/status
```

Phù hợp cho:

- image/video generation,
- PDF/contract analysis,
- long agent workflows,
- batch summarization,
- data ingestion/embedding,
- evaluations.

Lợi ích:

- tránh HTTP timeout,
- batch GPU tốt hơn,
- scale workers theo queue,
- retry/resume dễ hơn,
- UX có progress/status.

Trade-off: hệ thống cần task state, idempotency, cancellation, retries và notification.

## 7. Placement Decision Matrix

| Tiêu chí | Server-side | Edge | Client-side | Async worker |
|---|---|---|---|---|
| Model size lớn | tốt | kém | kém | tốt |
| Model IP cần bảo vệ | tốt | trung bình | kém | tốt |
| Latency global thấp | trung bình | tốt | tốt | kém cho realtime |
| Offline support | kém | kém | tốt | kém |
| Private data/server secrets | tốt | tùy region | kém | tốt |
| Long-running tasks | trung bình | kém | kém | tốt |
| Cloud cost thấp | kém | trung bình | tốt | trung bình |
| Observability/audit | tốt | trung bình | kém | tốt |
| Device/battery risk | không | không | cao | không |

Quy tắc nhanh:

- Realtime + model nhỏ + privacy: client/edge.
- Model lớn + private data: server-side/dedicated inference.
- Job > 3-5 giây: async worker.
- Global low-latency stateless: edge.
- Security-critical decision: server-side verification.

## Implementation Playbook: Hybrid Placement ADR

### Scenario

Chatbot public có nhiều prompt độc hại. Team muốn giảm cost moderation nhưng vẫn giữ security.

### Architecture

```text
[ Browser ]
  - lightweight toxicity prefilter
  - debounce input
        │ safe enough
        ▼
[ API Gateway ]
  - auth, rate limit, server-side moderation sample
  - token budget
        │
        ▼
[ Backend Router ]
  - easy FAQ -> small model
  - complex RAG -> server-side LLM
  - long document -> async task
```

### ADR example

```md
# ADR-004: Use Hybrid Client + Server Inference for Chat Safety

## Status
Accepted

## Context
A large percentage of requests are obvious toxic/spam prompts. Sending all of them to paid moderation and LLM APIs increases cost. However, client-side checks can be bypassed.

## Decision
Run a lightweight toxicity classifier in the browser for UX/cost optimization. Keep server-side rate limits, anomaly detection, and sampled moderation as the security boundary. All high-risk actions remain server-side.

## Consequences
- Reduces avoidable API calls.
- Improves perceived latency for blocked content.
- Does not eliminate server-side guardrails.
- Requires model size under 50MB and fallback if browser lacks WebGPU.
```

### Browser prefilter sketch

```javascript
async function submitMessage(message) {
  const localVerdict = await runLocalToxicityClassifier(message);

  if (localVerdict.label === "toxic" && localVerdict.score > 0.9) {
    showWarning("Vui lòng chỉnh lại nội dung trước khi gửi.");
    return;
  }

  const response = await fetch("/api/chat", {
    method: "POST",
    headers: {"Content-Type": "application/json"},
    body: JSON.stringify({message}),
  });

  render(await response.json());
}
```

### Server-side enforcement remains mandatory

```python
def server_chat_handler(user, message):
    enforce_rate_limit(user.id)
    enforce_token_budget(user.id, estimate_tokens(message))

    if looks_like_abuse(message):
        return safe_rejection()

    return route_to_model(message)
```

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Edge inference vẫn gọi DB ở region xa">
**Triệu chứng:** Edge model chạy nhanh nhưng p95 latency không giảm.

**Fix:** Chỉ đưa stateless tasks ra edge, hoặc replicate state/config gần edge nếu compliance cho phép.
</Callout>

<Callout type="warning" title="Pitfall 2: Model client-side quá lớn làm UX tải lần đầu tệ">
**Triệu chứng:** User web phải tải hàng trăm MB/GB trước khi dùng feature.

**Fix:** Chỉ dùng model nhỏ/quantized, lazy load, cache model, hoặc chuyển sang native app/server-side.
</Callout>

<Callout type="danger" title="Pitfall 3: Tin tưởng client-side guardrail như security boundary">
**Triệu chứng:** Attacker bypass JS classifier và gọi API trực tiếp.

**Fix:** Client-side chỉ là optimization. Server vẫn phải có auth, quota, moderation/anomaly detection và policy validation.
</Callout>

## Cost / Latency / Security Unit Economics

| Placement | Cost profile | Latency profile | Security profile |
|---|---|---|---|
| Server-side provider | pay per token/API | network + provider latency | tốt nếu data policy đúng |
| Self-host GPU | fixed GPU cost | tốt khi batched | kiểm soát cao, vận hành khó |
| Edge | pay per request/compute | network thấp | state/secrets hạn chế |
| Client-side | server cost thấp | tốt sau khi tải model | model weights lộ, bypass được |
| Async worker | optimize batching | không realtime | audit tốt nếu queue/state chuẩn |

Placement tốt thường là hybrid: đặt tác vụ nhẹ gần user, đặt logic nhạy cảm ở server, đặt job dài vào worker.

## Answer key

1. User có thể sửa client, bypass JS/mobile checks hoặc gọi API trực tiếp; server vẫn phải enforce policy.
2. Nếu dữ liệu/state nằm xa edge, edge phải gọi về origin làm latency tổng không giảm.
3. Khi tác vụ dài, dễ timeout, cần queue, batching, retry/resume hoặc progress UI.
4. Model weights không bị phân phối cho user, dễ kiểm soát access, logging và update.
5. Dùng nhiều vị trí inference cho các phần khác nhau: client/edge cho nhẹ, server cho nhạy cảm, worker cho dài.

---

## Điểm cần nhớ

- Inference placement quyết định latency, cost, privacy, security và vận hành.
- Server-side là mặc định cho model lớn, private data và audit.
- Edge chỉ hiệu quả cho stateless/low-latency tasks; data gravity rất quan trọng.
- Client-side tốt cho offline/privacy/cost nhưng có rủi ro model extraction và bypass.
- Job dài nên chuyển sang async worker thay vì giữ HTTP request mở.
