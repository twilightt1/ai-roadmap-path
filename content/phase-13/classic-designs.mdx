Ở bài trước, bạn đã có framework thiết kế AI system: clarify requirements, xác định AI components, vẽ data flow, chọn placement, rồi kiểm soát reliability/cost/security. Bài này chuyển framework đó thành các **classic designs** thường gặp trong sản phẩm thật.

Kỹ sư giỏi hiếm khi thiết kế từ trang trắng. Họ nhận ra pattern: chatbot cần memory, RAG cần ingestion pipeline, PDF Q&A cần async processing, customer support cần intent router, và platform lớn cần multi-LLM routing. Biết các blueprint này giúp bạn trả lời system design interview và triển khai production nhanh hơn.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Chatbot memory lưu thẳng vào Postgres làm nghẽn toàn hệ thống">
**Bối cảnh:** Một startup sales assistant xây chatbot giống ChatGPT. Để bot “nhớ” cuộc hội thoại, backend lưu từng message vào PostgreSQL. Mỗi request sẽ query 30 messages gần nhất, ghép vào prompt rồi gọi LLM.

**Sự cố:** Sau chiến dịch quảng cáo, 2,000 sales reps online cùng lúc. PostgreSQL bị cạn connection pool vì mỗi lượt chat vừa đọc history vừa ghi message mới. Khi DB chậm, chatbot mất memory, API timeout và user retry càng làm DB quá tải.

1. Lưu session hot-path vào database transactional thay vì Redis/session cache.
2. Backend không stateless vì phụ thuộc local conversation buffer ở một vài instance.
3. Không có summary memory nên prompt dài dần theo thời gian.
4. Không tách write-ahead session memory khỏi long-term analytics storage.

**Hậu quả:** 6 giờ downtime, team phải redesign memory layer bằng Redis + async persistence + conversation summarization.
</Callout>

Bài học rút ra: mỗi classic AI design có bottleneck riêng. Nếu dùng sai storage, sai queue hoặc sai router, hệ thống sẽ sập ở điểm không nằm trong model.

## 2. Architecture Blueprint (Classic AI Product Patterns)

Năm blueprint phổ biến trong AI products:

```text
[ 1. Chatbot with Memory ]
  user -> gateway -> stateless chat API -> Redis memory -> LLM -> stream response

[ 2. Enterprise RAG ]
  docs -> ingestion queue -> parser/chunker/embedder -> vector DB -> retriever/reranker -> LLM

[ 3. PDF Q&A at Scale ]
  upload -> object storage -> async workers -> task status -> query over indexed document

[ 4. Customer Support Router ]
  message -> intent classifier -> FAQ/RAG/human escalation/model routing

[ 5. Multi-LLM Gateway ]
  request -> policy/router -> provider A/B/C -> fallback/cost tracking/observability
```

Các pattern này không loại trừ nhau. Một enterprise support bot có thể dùng cả memory, RAG, intent router, async ingestion và multi-LLM gateway trong cùng một hệ thống.

## 3. Pattern 1: AI Chatbot with Memory

Chatbot memory có hai loại:

| Memory | Dùng cho | Storage phù hợp |
|---|---|---|
| Short-term session memory | vài lượt chat gần nhất | Redis/list/cache |
| Summary memory | tóm tắt hội thoại dài | Redis + DB |
| Long-term user memory | preference, facts, profile | Postgres/document DB/vector DB |
| Audit history | compliance, analytics | Postgres/S3/event log |

Blueprint production:

```text
[ Client ]
   │ SSE/WebSocket
   ▼
[ Stateless Chat API ]
   ├── read last N messages from Redis
   ├── read durable user preferences from DB
   ├── build prompt with budget
   ├── stream LLM response
   ├── append messages to Redis with TTL
   └── async persist transcript to Postgres/S3
```

Key decisions:

- Redis key phải include `tenant_id:user_id:session_id`.
- Set TTL để tránh memory leak.
- Khi history dài, summarize thay vì nhét toàn bộ vào prompt.
- Long-term memory cần user consent và delete policy.
- Streaming disconnect không được làm mất message state.

Ví dụ memory prompt budget:

```text
System prompt: 500 tokens
User profile: 300 tokens
Summary memory: 700 tokens
Recent messages: 1,500 tokens
Retrieved docs: 2,000 tokens
User query: 300 tokens
Output max: 700 tokens
```

Nếu không có budget, context sẽ phình không kiểm soát.

## 4. Pattern 2: Enterprise RAG Knowledge Base

RAG production có hai path riêng biệt: ingestion path và query path.

```text
Ingestion path:
[ Sources: PDF, DOCX, HTML, DB ]
      │
      ▼
[ Connector + ACL extractor ]
      │
      ▼
[ Queue ] -> [ Parser/OCR ] -> [ Chunker ] -> [ Embedder ] -> [ Vector DB ]
                                      │
                                      └── metadata: tenant_id, doc_id, ACL, version

Query path:
[ User query ] -> [ Auth ] -> [ Query rewrite ] -> [ Retriever with ACL filter ]
              -> [ Reranker ] -> [ Context builder ] -> [ LLM ] -> [ citations ]
```

Điểm production quan trọng:

- ACL/tenant filter phải được áp dụng **trước** khi context vào prompt.
- Document update/delete phải sync sang vector DB.
- Chunk metadata cần `doc_id`, `chunk_id`, `source_url`, `version`, `tenant_id`.
- Reranker giúp giảm hallucination bằng cách tăng precision.
- Citations phải trỏ đúng chunk/version.

<Callout type="warning" title="RAG không chỉ là vector search">
Vector search trả “đoạn có vẻ liên quan”. Production RAG cần ACL, freshness, reranking, citation, eval, cache invalidation và deletion sync.
</Callout>

## 5. Pattern 3: PDF Q&A at Scale

PDF Q&A là RAG nhưng có ingestion nặng hơn: upload file, OCR, layout parsing, table extraction và task state.

```text
[ Upload PDF ]
   │
   ▼
[ Object storage ]
   │ event
   ▼
[ Queue ]
   │
   ▼
[ Workers ]
  - virus scan
  - OCR / text extraction
  - table parsing
  - chunking
  - embedding
  - index update
   │
   ▼
[ Task status: ready / failed ]
```

User không nên chờ ingestion đồng bộ. API nên trả `task_id`, frontend hiển thị progress, và chỉ bật Q&A khi index ready.

Failure modes cần thiết kế:

| Failure | Cách xử lý |
|---|---|
| PDF quá lớn | reject trước upload hoặc async quota |
| OCR thất bại | mark failed + user-friendly error |
| Duplicate file | hash file và reuse existing index |
| User xóa PDF | tombstone chunks trong vector DB |
| Ingestion partial | status failed, cleanup chunks orphan |

## 6. Pattern 4: AI Customer Support Intent Router

Không nên gửi mọi ticket vào model đắt nhất. Customer support thường có intent rõ ràng:

```text
[ Customer message ]
        │
        ▼
[ Fast intent classifier ]
  - refund
  - password reset
  - shipping
  - angry customer
  - complex/legal
        │
        ├── FAQ template / deterministic workflow
        ├── RAG answer over policy docs
        ├── LLM draft response
        └── Human escalation
```

Router giúp:

- giảm cost bằng cách dùng template/model nhỏ cho request đơn giản,
- tăng safety bằng cách escalate high-risk cases,
- giảm latency cho FAQ,
- đo analytics theo intent.

Router decision không nên chỉ dựa vào model confidence. Nên có hard rules:

```text
if message contains threat/self-harm/legal escalation -> human
if payment/refund over threshold -> human approval
if classifier confidence < 0.7 -> fallback to safe RAG or human
if tenant is enterprise VIP -> stronger model / human faster
```

## 7. Pattern 5: Multi-LLM Gateway and Recommendation Router

Khi sản phẩm lớn, bạn cần gateway giữa app và providers.

```text
[ Product services ]
        │ OpenAI-compatible request
        ▼
[ LLM Gateway ]
  - auth and tenant budgets
  - prompt/version registry
  - model routing policy
  - fallback provider
  - retries with budget
  - cost/latency logging
        │
        ├── OpenAI
        ├── Anthropic
        ├── Gemini
        └── Self-host vLLM
```

Gateway cho phép đổi provider/model bằng config thay vì sửa code ở mọi service.

Routing criteria:

| Criteria | Ví dụ |
|---|---|
| Task type | summarization -> cheap model, legal reasoning -> strong model |
| User tier | free -> small model, enterprise -> stronger/faster |
| Latency SLA | realtime -> low latency provider |
| Cost budget | monthly budget nearing limit -> downgrade/degrade |
| Provider health | 5xx spike -> fallback |
| Data policy | EU data -> Azure EU endpoint |

## Implementation Playbook: Chatbot Memory + RAG + Router Skeleton

Đây là skeleton tích hợp ba pattern: memory, intent router và RAG.

```python
import json
import redis

redis_client = redis.Redis(host="redis", port=6379, decode_responses=True)


def memory_key(tenant_id: str, user_id: str, session_id: str) -> str:
    return f"chat:{tenant_id}:{user_id}:{session_id}"


def get_recent_messages(tenant_id: str, user_id: str, session_id: str, limit: int = 10):
    raw = redis_client.lrange(memory_key(tenant_id, user_id, session_id), -limit, -1)
    return [json.loads(x) for x in raw]


def append_message(tenant_id: str, user_id: str, session_id: str, role: str, content: str):
    key = memory_key(tenant_id, user_id, session_id)
    redis_client.rpush(key, json.dumps({"role": role, "content": content}))
    redis_client.expire(key, 3600)


def classify_intent(message: str) -> dict:
    # Replace with small ONNX classifier or cheap LLM.
    if "hoàn tiền" in message.lower() or "refund" in message.lower():
        return {"intent": "refund", "confidence": 0.92}
    return {"intent": "general", "confidence": 0.75}


def handle_chat(req):
    intent = classify_intent(req.message)

    if intent["confidence"] < 0.7:
        return escalate_to_human(req)

    history = get_recent_messages(req.tenant_id, req.user_id, req.session_id)
    context = retrieve_policy_docs(req.tenant_id, req.message, intent["intent"])

    messages = build_prompt(history=history, context=context, user_message=req.message)
    answer = call_model_router(messages=messages, intent=intent)

    append_message(req.tenant_id, req.user_id, req.session_id, "user", req.message)
    append_message(req.tenant_id, req.user_id, req.session_id, "assistant", answer)

    return {"answer": answer, "intent": intent["intent"]}
```

Production checklist:

- Redis TTL và max memory policy.
- Tenant/user isolation trong keys.
- RAG retriever có ACL filters.
- Router confidence logging.
- Human escalation path.
- Trace toàn bộ request: intent, chunks, model, tokens, cost.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Redis memory leak vì quên TTL">
**Triệu chứng:** Redis OOM sau vài tuần, chatbot mất memory toàn hệ thống.

**Fix:** Set TTL cho session keys, giới hạn số messages lưu, bật `maxmemory-policy allkeys-lru` và theo dõi memory usage.
</Callout>

<Callout type="warning" title="Pitfall 2: Dangling vectors sau khi xóa tài liệu">
**Triệu chứng:** RAG vẫn trả nội dung từ PDF đã xóa.

**Fix:** Xóa tài liệu phải phát event để worker xóa/tombstone chunks trong vector DB theo `doc_id` và `version`.
</Callout>

<Callout type="danger" title="Pitfall 3: Intent router thành bottleneck">
**Triệu chứng:** API chậm dù model chính chưa được gọi.

**Nguyên nhân:** Classifier chạy bằng Python `transformers.pipeline()` đồng bộ trên CPU trong web process.

**Fix:** Export classifier sang ONNX/TensorRT, batch requests hoặc chạy router thành service riêng có autoscaling.
</Callout>

## Cost / Scalability / Reliability Unit Economics

| Pattern | Chi phí chính | Tối ưu |
|---|---|---|
| Chatbot memory | input tokens tăng theo history | summary memory, context budget |
| Enterprise RAG | embedding + vector DB + reranker | incremental ingestion, self-host embeddings |
| PDF Q&A | OCR/parse/embedding jobs | file hash dedup, async workers |
| Intent router | router latency + false route risk | ONNX small model, confidence thresholds |
| Multi-LLM gateway | integration/observability overhead | shared client, config-based routing |

Classic design tốt nhất là design biết giới hạn: khi nào dùng memory, khi nào cache, khi nào async và khi nào escalate cho người thật.

## Answer key

1. Redis có latency thấp và phù hợp session hot-path; Postgres phù hợp durable storage/analytics.
2. Ingestion xử lý tài liệu thành chunks/embeddings; query path retrieve/rerank/generate cho user query.
3. Nó route request đơn giản sang template/model rẻ và chỉ dùng model mạnh cho case khó.
4. Embeddings/chunks nằm trong vector DB riêng, cần delete/tombstone theo `doc_id`.
5. Tập trung routing, fallback, budgets, logging và provider abstraction ở một lớp chung.

## Điểm cần nhớ

- Classic AI systems là tổ hợp pattern: memory, RAG, async, router, gateway.
- Backend nên stateless; session memory nóng đặt ở Redis và durable history lưu async.
- Enterprise RAG cần ACL, metadata, deletion sync và eval, không chỉ vector search.
- Intent router là đòn bẩy cost lớn nhưng cần confidence/fallback/human escalation.
- Multi-LLM gateway giúp kiểm soát provider, cost, fallback và observability.
