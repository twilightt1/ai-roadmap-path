Bạn đã học framework, classic designs, inference placement, caching và async architecture. Tất cả những quyết định đó cuối cùng hội tụ vào một câu hỏi rất thực tế: **mỗi request AI có lãi hay lỗ?** Với AI products, cost không chỉ là server cố định theo tháng; token, embeddings, reranker, GPU time, retries và evals tăng theo từng hành động của user.

**Cost-Aware Architecture** là năng lực thiết kế hệ thống AI có biên lợi nhuận dương. Bạn không dùng model mạnh nhất cho mọi việc; bạn route theo task, cache khi an toàn, nén context, giới hạn output, đặt hard spend limits và đo cost theo feature/user/tenant.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Autocomplete miễn phí đốt 3,000 USD vì gọi frontier model mỗi lần user dừng gõ">
**Bối cảnh:** Một SaaS editor thêm tính năng autocomplete. Mỗi khi user dừng gõ 500ms, frontend gửi 2,000 tokens context lên GPT-4o để sinh 10 tokens gợi ý. Feature miễn phí cho mọi user.

**Sự cố:** Power users gõ cả ngày. Mỗi user tạo hàng nghìn request nhỏ. Không có debounce đủ dài, không có per-user token budget, không cache, không model routing, và không có cost attribution theo feature.

1. Task autocomplete đơn giản nhưng dùng model đắt nhất.
2. Input tokens quá dài so với output cần 10 tokens.
3. Không có `max_tokens` cứng và stop sequences.
4. Không có budget alert theo feature.
5. Free tier không có hard spend limit.

**Hậu quả:** Hóa đơn tăng 3,000 USD/tháng cho một feature không tạo doanh thu trực tiếp. Team phải tắt feature và thiết kế lại bằng small model + debounce + cache + budget guard.
</Callout>

Bài học rút ra: cost-aware không phải tối ưu sau cùng; nó là requirement kiến trúc. Nếu feature không có cost model, bạn chưa biết nó có thể scale hay không.

## 2. Architecture Blueprint (Cost-Aware AI Gateway)

Một kiến trúc cost-aware đặt LLM gateway ở giữa product và model providers:

```text
[ Product feature request ]
        │
        ▼
[ Cost-aware gateway ]
  - estimate input/output tokens
  - check tenant/user budget
  - debounce/dedup/idempotency
  - cache lookup
  - classify task complexity
  - choose model/provider
  - enforce max tokens
        │
        ├── cheap/small model
        ├── frontier model
        ├── self-host vLLM
        └── fallback provider
        │
        ▼
[ Observability + billing ]
  - cost per request
  - cost per feature
  - cost per user/tenant
  - quality and latency metrics
```

Mục tiêu: mỗi request đi qua policy layer trước khi đụng model đắt tiền.

## 3. Unit Economics: Cost per Feature, User and Outcome

Công thức đơn giản:

```text
request_cost = input_tokens * input_price + output_tokens * output_price + fixed_overhead
feature_margin = feature_revenue - feature_cost
```

Nhưng production cần đo theo nhiều chiều:

| Dimension | Câu hỏi |
|---|---|
| Feature | tính năng nào đốt tiền nhất? |
| User/tenant | ai đang vượt budget? |
| Model | model nào cost cao nhưng quality không hơn? |
| Prompt version | prompt mới có làm cost tăng? |
| Cache status | cache hit đã tiết kiệm bao nhiêu? |
| Outcome | cost per successful task là bao nhiêu? |

Ví dụ: chatbot support có cost/request thấp nhưng giải quyết nhiều ticket; legal analysis có cost/request cao nhưng user trả phí premium. Không thể tối ưu cost tách khỏi giá trị business.

## 4. Model Routing by Task Value

Không có “best model” cho mọi request. Có model phù hợp với từng task.

| Task | Model phù hợp | Lý do |
|---|---|---|
| autocomplete | small/fast model, local/edge nếu được | output ngắn, latency thấp |
| rewrite/format | cheap LLM | ít reasoning |
| FAQ public | cache + small model/RAG | lặp lại nhiều |
| legal/medical reasoning | frontier model + citations | high risk |
| extraction JSON | cheap model nếu schema đơn giản | structured, eval được |
| escalation/high-value customer | stronger model/human | value cao |

Router có thể dùng:

- rules/heuristics,
- small classifier,
- embedding similarity to task clusters,
- LLM judge/router nhẹ,
- user tier/business value,
- budget remaining.

Router phải được eval. Nếu route sai câu khó sang model yếu, cost giảm nhưng quality giảm âm thầm.

## 5. Token Budgets and Prompt Compression

Token là đơn vị tiền tệ của AI system.

Các chiến lược giảm token:

| Strategy | Cách làm | Rủi ro |
|---|---|---|
| Max input length | truncate theo policy | mất context quan trọng |
| RAG top-k nhỏ hơn | lấy ít chunks hơn | recall giảm |
| Reranking | giữ chunks chất lượng hơn | thêm latency/cost |
| Contextual compression | nén context trước LLM lớn | mất chi tiết nếu nén sai |
| Summary memory | tóm tắt history dài | mất nuance |
| Structured output | JSON ngắn thay vì văn xuôi | cần schema validation |
| `max_tokens` | giới hạn output | answer có thể cụt nếu quá thấp |

<Callout type="warning" title="Nén prompt trong domain số liệu cần cực kỳ cẩn thận">
Văn bản tài chính/pháp lý/y tế thường phụ thuộc vào từ phủ định, con số nhỏ và điều kiện ngoại lệ. Compression mù quáng có thể xóa đúng phần quan trọng nhất.
</Callout>

## 6. Budgets, Quotas and Hard Spend Limits

Cost-aware architecture cần guardrails cứng:

```text
free user:
  max 100 requests/day
  max 50,000 tokens/day
  cheap models only

pro user:
  max 2,000 requests/day
  max 2,000,000 tokens/month
  frontier model allowed for premium workflows

enterprise tenant:
  monthly budget cap
  per-feature budgets
  alert at 50/80/100 percent
  allow overage only with contract setting
```

Budget checks nên nằm trước LLM call:

```text
estimate tokens -> check remaining budget -> choose model -> enforce max_tokens -> log actual cost
```

Nếu chỉ log sau khi gọi model, bạn đã tiêu tiền rồi.

## 7. Cost Observability and Alerts

Metrics bắt buộc:

| Metric | Alert ví dụ |
|---|---|
| cost/hour | tăng hơn 3 lần baseline |
| cost/feature/day | vượt budget feature |
| cost/user/day | user bất thường hoặc abuse |
| input/output tokens/request | prompt regression làm dài context |
| cache hit savings | hit rate giảm bất thường |
| model mix | frontier model share tăng đột biến |
| retry cost | provider lỗi gây retry storm |
| cost per successful task | feature không còn profitable |

Cost dashboard nên gắn với release version. Khi prompt v14 làm output dài hơn 2 lần, bạn cần biết ngay.

## Implementation Playbook: Cost Router + Budget Guard

### Bước 1: Pricing config theo provider/model

```python
PRICING_PER_1M = {
    "gpt-4o": {"input": 5.00, "output": 15.00},
    "gpt-4o-mini": {"input": 0.15, "output": 0.60},
    "claude-sonnet": {"input": 3.00, "output": 15.00},
    "local-llama-8b": {"input": 0.05, "output": 0.10},
}


def estimate_cost_usd(model: str, input_tokens: int, output_tokens: int) -> float:
    price = PRICING_PER_1M[model]
    return (input_tokens / 1_000_000) * price["input"] + (output_tokens / 1_000_000) * price["output"]
```

### Bước 2: Model router theo task và budget

```python
def choose_model(task_type: str, risk_level: str, user_tier: str, remaining_budget_usd: float) -> str:
    if remaining_budget_usd <= 0:
        raise PermissionError("Budget exhausted")

    if task_type in {"autocomplete", "format", "rewrite"}:
        return "gpt-4o-mini"

    if risk_level == "high" and user_tier in {"pro", "enterprise"}:
        return "gpt-4o"

    if task_type == "faq" and remaining_budget_usd < 1.0:
        return "local-llama-8b"

    return "gpt-4o-mini"
```

### Bước 3: Gateway policy trước khi gọi model

```python
def handle_ai_request(user, task_type: str, prompt: str, requested_max_tokens: int):
    input_tokens = estimate_tokens(prompt)
    max_tokens = min(requested_max_tokens, user.plan.max_output_tokens)

    model = choose_model(
        task_type=task_type,
        risk_level=classify_risk(prompt),
        user_tier=user.plan.name,
        remaining_budget_usd=user.remaining_budget_usd,
    )

    estimated = estimate_cost_usd(model, input_tokens, max_tokens)
    if estimated > user.remaining_budget_usd:
        return {"error": "budget_exceeded", "estimated_cost": estimated}

    cached = lookup_cache(user, task_type, prompt)
    if cached:
        log_cost_saved(user.id, model, estimated)
        return cached

    response = call_llm(model=model, prompt=prompt, max_tokens=max_tokens)
    actual_cost = estimate_cost_usd(model, response.input_tokens, response.output_tokens)
    record_usage(user.id, task_type, model, actual_cost)
    return response.text
```

### Bước 4: Alert rules

```text
Alert: feature_cost_spike
Condition: cost_per_hour(feature) > 3x rolling_7d_baseline for 15 minutes
Action: page feature owner, downgrade model route if auto-policy enabled

Alert: tenant_budget_80_percent
Condition: tenant_monthly_spend > 0.8 * tenant_budget
Action: notify customer success and tenant admin

Alert: frontier_model_share_spike
Condition: frontier_model_requests / total_requests > expected + 20 percent
Action: inspect router regression
```

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Router tiết kiệm tiền nhưng quality drift âm thầm">
**Triệu chứng:** Bill giảm 50% nhưng thumbs-down rate tăng mạnh.

**Fix:** Shadow eval một phần request cheap-model bằng strong model, theo dõi quality delta và tune router thresholds.
</Callout>

<Callout type="warning" title="Pitfall 2: Prompt compression xóa mất phủ định hoặc số liệu quan trọng">
**Triệu chứng:** Model trả sai điều khoản tài chính/pháp lý sau khi bật compression.

**Fix:** Không nén mù quáng domain số liệu/high-risk; dùng extraction/reranking thay vì summarization lossy; eval trước rollout.
</Callout>

<Callout type="danger" title="Pitfall 3: Wallet drain attack qua prompt dài và retry">
**Triệu chứng:** Một user mới tạo hàng nghìn request dài, bill tăng trước khi rate limit phát hiện.

**Fix:** Hard spend limit trước LLM call, max input/output tokens, per-user/day budget, anomaly detection và captcha/verification cho free tier.
</Callout>

## Cost / Quality / Margin Unit Economics

| Lever | Savings potential | Quality risk | Guardrail |
|---|---|---|---|
| Model routing | cao | route sai task khó | golden eval per route |
| Exact cache | trung bình-cao | stale data | versioned keys + TTL |
| Semantic cache | cao | false hit/privacy | scope filter + threshold |
| Prompt compression | trung bình | mất context | domain-aware eval |
| Max output tokens | trung bình | answer cụt | task-specific limits |
| Self-host small model | cao ở scale | ops complexity | compare TCO with APIs |
| Async batching | trung bình | wait time | SLA and queue metrics |
| Budget caps | bảo vệ tài chính | user bị chặn | transparent UX and upgrade path |

Cost-aware architecture không đơn giản là “rẻ hơn”. Nó tối ưu cost dưới ràng buộc quality, latency, safety và business value.

## Knowledge check

1. Vì sao cost-aware architecture phải nằm ở gateway/policy layer?
2. Model routing có rủi ro gì?
3. Vì sao output tokens thường cần giới hạn chặt?
4. Hard spend limit khác gì budget alert?
5. Cost per successful task hữu ích hơn cost per request khi nào?

---

## Answer key

1. Vì cần estimate, budget check, cache, route và enforce limits trước khi gọi model đắt tiền.
2. Router có thể gửi task khó sang model yếu, làm quality giảm hoặc safety fail.
3. Output tokens thường đắt hơn input và model có thể verbose nếu không giới hạn.
4. Hard limit chặn request khi vượt ngân sách; alert chỉ thông báo sau/ngay khi tiến gần ngưỡng.
5. Khi request có thể fail/retry hoặc feature tạo business outcome khác nhau; cost cần gắn với giá trị đạt được.

## Điểm cần nhớ

- AI cost tăng theo từng request/token; phải thiết kế guardrails từ đầu.
- Không có model tốt nhất cho mọi task; model routing là đòn bẩy margin lớn.
- Token budgets, cache, compression và max tokens cần đi cùng eval để không giảm quality âm thầm.
- Cost observability phải theo feature/user/tenant/model/prompt version.
- Phase 13 Medium là cost calculator; Phase 13 Hard là triển khai một AI system design production-grade với ADRs.
