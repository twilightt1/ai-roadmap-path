Ở bài trước, bạn đã dùng cache để tránh gọi model khi không cần. Nhưng với các tác vụ AI thật sự nặng — phân tích PDF dài, sinh ảnh/video, chạy agent nhiều bước, ingestion embeddings, batch evaluation — cache không giúp được request đầu tiên. Nếu bạn giữ HTTP connection mở trong 30-120 giây, hệ thống sẽ timeout và user sẽ retry.

**Async AI Architecture** tách luồng nhận yêu cầu khỏi luồng xử lý AI. Thay vì `POST /analyze` rồi chờ, client nhận `task_id`, worker xử lý nền, frontend theo dõi progress qua polling/webhook/SSE. Đây là pattern bắt buộc khi latency vượt quá vài giây hoặc workload cần retry/resume.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: API gateway sập vì 200 request PDF giữ thread 45 giây">
**Bối cảnh:** Một công ty bảo hiểm xây API trích xuất hợp đồng: `POST /extract -> parse PDF -> call GPT-4o -> return JSON`. Mỗi request mất 30-60 giây.

**Sự cố:** 9h sáng, 200 đại lý upload PDF cùng lúc. Mỗi request giữ một worker/thread của API gateway. Thread pool cạn, mọi endpoint khác cũng timeout. User thấy 504 và bấm gửi lại, tạo duplicate jobs. Một số request vẫn chạy ở backend nên công ty bị tính tiền LLM nhiều lần cho cùng một file.

1. Tác vụ dài được xử lý đồng bộ trong web request.
2. Không có task id, idempotency key hoặc dedup theo file hash.
3. Không có queue length monitoring hoặc worker autoscaling.
4. Retry từ client tạo duplicate execution.

**Hậu quả:** 3 giờ downtime, chi phí token tăng mạnh và kết quả bị trùng trong database.
</Callout>

Bài học rút ra: request AI dài phải trở thành **job**. Web API nhận việc nhanh, queue giữ backlog, worker xử lý có timeout/retry, và client theo dõi trạng thái.

## 2. Architecture Blueprint (Async Task Queue + Status API)

Pattern chuẩn:

```text
[ Client ]
   │ POST /tasks
   ▼
[ API Gateway ]
  - auth
  - quota
  - idempotency key
  - create task row
   │
   ├── 202 Accepted {task_id}
   │
   ▼
[ Message Queue ]
  - Redis/RabbitMQ/SQS/Kafka
   │
   ▼
[ Worker Pool ]
  - OCR/parser
  - embeddings
  - LLM calls
  - result validation
   │
   ▼
[ Task Store ]
  - queued/running/succeeded/failed/cancelled
  - progress
  - result URL
  - error code
   │
   ├── GET /tasks/{id} polling
   ├── webhook callback
   └── SSE/WebSocket progress
```

Mục tiêu: web layer không bị block, worker layer scale theo queue, task state giúp retry/resume và user không cần gửi lại request mù quáng.

## 3. When Sync Is OK vs When Async Is Required

| Tác vụ | Sync được? | Lý do |
|---|---|---|
| intent classification < 100ms | có | nhanh, deterministic |
| chat streaming 1-5s | có, nếu stream | user thấy token sớm |
| RAG answer 2-8s | tùy | có thể sync với timeout/streaming |
| PDF ingestion 30s-10m | async | parse/OCR/embed dài |
| image/video generation | async | latency cao, GPU queue |
| multi-step agent | async hoặc streaming task | không biết thời gian hoàn thành |
| batch eval/training | async/job | workload nền |

Quy tắc thực dụng:

- dưới 1 giây: sync thường ổn,
- 1-5 giây: sync + streaming hoặc async tùy UX,
- trên 5-10 giây: async,
- tác vụ có retry/resume/cancel: async,
- tác vụ tốn tiền cao: async với idempotency và budget check.

## 4. Task State Machine

Task phải có state rõ ràng:

```text
created -> queued -> running -> succeeded
                    ├-> failed_retryable -> queued
                    ├-> failed_terminal
                    └-> cancelled
```

Task record tối thiểu:

```json
{
  "task_id": "task_123",
  "tenant_id": "tenant_a",
  "user_id": "user_42",
  "type": "pdf_extract",
  "status": "running",
  "progress": 0.45,
  "input_hash": "sha256:...",
  "idempotency_key": "tenant_a:file_hash:extract_v2",
  "result_url": null,
  "error_code": null,
  "created_at": "2026-07-02T10:00:00Z",
  "updated_at": "2026-07-02T10:01:12Z"
}
```

Nếu state machine không chuẩn, frontend không biết hiển thị gì, support không debug được và retry dễ tạo duplicate.

## 5. Queue Semantics, Retries and Idempotency

Queue thường có delivery semantics “at least once”: job có thể chạy lại. Vì vậy worker phải idempotent.

### Idempotency key

```text
idempotency_key = hash(tenant_id + user_id + file_hash + task_type + config_version)
```

Nếu user gửi lại cùng file/cùng task, API trả task cũ thay vì tạo task mới.

### Retry policy

| Error | Retry? | Ghi chú |
|---|---|---|
| network timeout tải file | có | exponential backoff |
| provider 429 | có nhưng respect retry-after | tránh retry storm |
| invalid PDF | không | terminal failure |
| prompt output invalid JSON | có giới hạn | có thể repair/retry 1-2 lần |
| user cancelled | không | stop if possible |
| quota exceeded | không | user action needed |

### Dead letter queue

Job fail nhiều lần nên vào DLQ để điều tra, không retry vô hạn.

```text
queue -> worker -> fail 3 times -> dead-letter-queue -> alert/runbook
```

## 6. Polling vs Webhook vs SSE/WebSocket

| Notification pattern | Khi dùng | Trade-off |
|---|---|---|
| Polling | web/mobile đơn giản | dễ DDoS nếu poll quá dày |
| Exponential polling | task vài giây/phút | giảm tải DB/API |
| Webhook | B2B/API integrations | cần signature, retry, endpoint của khách |
| SSE | browser nhận progress một chiều | đơn giản hơn WebSocket |
| WebSocket | realtime 2 chiều | phức tạp hơn, connection state |
| Email/in-app notification | task rất dài | UX chậm nhưng rẻ |

Polling endpoint nên cache/status store bằng Redis nếu QPS cao. Frontend không nên poll mỗi 100ms.

## Implementation Playbook: Queue + Task Status API

Ví dụ production skeleton dùng FastAPI + Redis Queue concept. Trong thực tế có thể dùng Celery/RQ/BullMQ/SQS.

### Bước 1: API tạo task có idempotency

```python
import hashlib
import uuid
from fastapi import FastAPI, Header
from pydantic import BaseModel

app = FastAPI()
task_store = {}
queue = []

class ExtractRequest(BaseModel):
    tenant_id: str
    user_id: str
    document_url: str
    document_sha256: str


def make_idempotency_key(req: ExtractRequest, config_version: str = "extract-v2") -> str:
    raw = f"{req.tenant_id}:{req.user_id}:{req.document_sha256}:{config_version}"
    return hashlib.sha256(raw.encode()).hexdigest()

@app.post("/tasks/extract", status_code=202)
def create_extract_task(req: ExtractRequest, idempotency_key: str | None = Header(default=None)):
    idem = idempotency_key or make_idempotency_key(req)

    for task in task_store.values():
        if task["idempotency_key"] == idem:
            return {"task_id": task["task_id"], "status": task["status"], "deduped": True}

    task_id = f"task_{uuid.uuid4().hex}"
    task = {
        "task_id": task_id,
        "status": "queued",
        "progress": 0.0,
        "idempotency_key": idem,
        "result": None,
        "error": None,
        "request": req.model_dump(),
    }
    task_store[task_id] = task
    queue.append(task_id)
    return {"task_id": task_id, "status_url": f"/tasks/{task_id}"}
```

### Bước 2: Worker xử lý task

```python
def worker_loop():
    while True:
        if not queue:
            continue

        task_id = queue.pop(0)
        task = task_store[task_id]
        task["status"] = "running"

        try:
            task["progress"] = 0.2
            text = parse_pdf(task["request"]["document_url"])
            task["progress"] = 0.5
            extracted = call_llm_extract(text)
            task["progress"] = 0.9
            validate_schema(extracted)

            task["status"] = "succeeded"
            task["progress"] = 1.0
            task["result"] = extracted
        except RetryableError as e:
            task["status"] = "failed_retryable"
            task["error"] = str(e)
            # real queue would requeue with backoff and retry count
        except Exception as e:
            task["status"] = "failed_terminal"
            task["error"] = str(e)
```

### Bước 3: Status endpoint

```python
from fastapi import HTTPException

@app.get("/tasks/{task_id}")
def get_task(task_id: str):
    task = task_store.get(task_id)
    if not task:
        raise HTTPException(status_code=404, detail="Task not found")
    return {
        "task_id": task_id,
        "status": task["status"],
        "progress": task["progress"],
        "result": task["result"] if task["status"] == "succeeded" else None,
        "error": task["error"],
    }
```

### Bước 4: Production replacement

Thay in-memory store/queue bằng:

- Postgres for task records,
- Redis for hot status/progress,
- S3/GCS for large input/result artifacts,
- Celery/RQ/BullMQ/SQS for queue,
- Prometheus metrics for queue length and worker duration.

## 7. Worker Scaling and Backpressure

Autoscale theo queue metrics:

| Metric | Ý nghĩa | Action |
|---|---|---|
| queue length | backlog hiện tại | scale workers up |
| oldest job age | user chờ lâu chưa | alert nếu vượt SLA |
| worker duration p95 | task đang chậm | kiểm tra provider/OCR |
| failure rate | nhiều job fail | pause queue / rollback |
| retry count | retry storm | circuit breaker provider |
| GPU utilization | worker GPU có bận không | tune batch/replicas |

Backpressure cần ở API gateway:

- reject nếu queue quá dài,
- per-tenant concurrency limits,
- quota theo task type,
- `Retry-After` cho 429/503,
- pricing/credits trước khi enqueue job đắt.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: Queue backlog nhưng không ai biết">
**Triệu chứng:** User task chờ hàng giờ, API vẫn nhận thêm job.

**Fix:** Alert theo oldest job age, queue length, p95 task duration; hiển thị ETA/degraded mode và chặn enqueue khi quá tải.
</Callout>

<Callout type="warning" title="Pitfall 2: Polling endpoint tự DDoS database">
**Triệu chứng:** Postgres read QPS tăng mạnh chỉ vì frontend poll status.

**Fix:** Exponential backoff, Redis status cache, SSE cho progress, và giới hạn poll frequency.
</Callout>

<Callout type="danger" title="Pitfall 3: Duplicate execution tính tiền hai lần">
**Triệu chứng:** Cùng file được xử lý 2-3 lần do user retry hoặc queue redelivery.

**Fix:** Idempotency key, task dedup, output write idempotent, và billing chỉ charge theo task succeeded duy nhất.
</Callout>

## Cost / Reliability / Throughput Unit Economics

| Quyết định | Lợi ích | Chi phí/rủi ro |
|---|---|---|
| Async queue | tránh timeout, scale worker | state management phức tạp |
| More workers | giảm queue wait | tăng GPU/API cost |
| Batch workers | tăng throughput | tăng wait time từng job |
| Aggressive retries | tăng success transient errors | duplicate/cost spike |
| Webhooks | tốt cho B2B | cần signature/retry endpoint |
| Polling | dễ làm frontend | tải lớn nếu không backoff |
| Scale to zero | tiết kiệm idle | cold start worker |

Async architecture tốt không chỉ “dùng queue”; nó kiểm soát task lifecycle, idempotency, retry budget, worker scaling và user communication.

---

## Answer key

1. Nó giữ worker/thread lâu, dễ timeout, làm nghẽn endpoint khác và tạo retry storm.
2. Worker phải idempotent vì cùng job có thể được deliver/chạy lại.
3. Lưu job fail nhiều lần để điều tra thay vì retry vô hạn.
4. Tạo tải lớn lên API/DB chỉ để hỏi trạng thái, có thể tự DDoS hệ thống.
5. Cơ chế giới hạn/reject/degrade khi queue quá tải để bảo vệ hệ thống và ngân sách.

## Điểm cần nhớ

- Tác vụ AI dài nên trở thành task async với `task_id` và status API.
- Queue delivery thường at-least-once, nên idempotency là bắt buộc.
- Retry cần phân loại lỗi và có giới hạn; không retry vô hạn với LLM calls đắt tiền.
- Polling cần backoff hoặc thay bằng SSE/webhook cho progress.
- Worker autoscaling nên dựa trên queue length, oldest job age, duration và failure rate.
