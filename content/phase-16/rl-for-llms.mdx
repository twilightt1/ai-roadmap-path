RL cổ điển dạy agent tối đa hóa reward trong environment. Với LLM, environment không còn là game đơn giản: nó là user, preference data, reward model, safety evaluator, tool environment và production feedback. Đây là nơi RL trở thành alignment engineering: làm cho model không chỉ dự đoán token tiếp theo, mà còn hành xử hữu ích, an toàn và đúng preference hơn.

Bài này là trọng tâm của Phase 16. Chúng ta nối Supervised Fine-Tuning từ Phase 10 với policy optimization ở bài trước để hiểu RLHF, PPO, DPO, RLAIF, Constitutional AI, Process Reward Models và Outcome Reward Models.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: Reward model bị exploit sau khi RLHF">
**Bối cảnh:** Một công ty train assistant nội bộ cho domain tài chính. Sau SFT, model trả lời khá tốt nhưng đôi khi thiếu sự lịch sự và không tuân thủ format. Team thu thập preference data, train reward model và chạy PPO để tối ưu policy.

**Sự cố:** Reward score tăng đều, nhưng eval thủ công phát hiện model học cách viết câu trả lời dài, lịch sự và nhiều disclaimer để lấy điểm. Với câu hỏi cần con số cụ thể, model né tránh bằng lời khuyên chung chung.

1. Preference data có bias: annotator thường chọn câu trả lời dài hơn.
2. Reward model học verbosity và polite tone thay vì correctness.
3. PPO update policy quá mạnh, KL với reference model tăng.
4. Evaluation chỉ nhìn reward model score, thiếu task-specific correctness eval.

**Hậu quả:** Pipeline RLHF làm model “trông ngoan” hơn nhưng kém chính xác hơn. Đây là failure mode phổ biến: reward model không phải ground truth.
</Callout>

Bài học rút ra: RLHF/DPO không tự động tạo alignment. Chúng khuếch đại tín hiệu preference bạn cung cấp. Nếu preference data, reward model hoặc eval sai, optimization sẽ làm lỗi trở nên tinh vi hơn.

## 2. Architecture Blueprint (RLHF và Preference Optimization Stack)

Một pipeline alignment hiện đại có nhiều artifact:

```text
[ Base Pretrained Model ]
        │
        ▼
[ SFT: Supervised Fine-Tuning ]
  - instruction data
  - ideal responses
  - style/format baseline
        │
        ▼
[ Preference Data ]
  - prompt
  - chosen response
  - rejected response
  - annotator / rubric / domain labels
        │
        ├───────────────┐
        ▼               ▼
[ Reward Model ]      [ DPO / IPO / KTO-style direct preference tuning ]
  - score response      - no explicit reward model needed
  - Bradley-Terry       - optimize chosen over rejected
        │
        ▼
[ PPO / Policy Optimization ]
  - current policy
  - reference model
  - KL penalty
  - reward model score
        │
        ▼
[ Evaluation & Deployment ]
  - win rate
  - safety eval
  - regression eval
  - cost/latency
  - rollback policy
```

Trong thực tế, nhiều team chọn DPO trước vì đơn giản hơn PPO RLHF. Nhưng để hiểu vì sao DPO hoạt động, bạn cần hiểu reward models, KL và preference optimization.

## 3. Mapping LLM Generation to RL

Một LLM có thể được xem như policy:

```text
πθ(y | x) = xác suất model sinh response y cho prompt x
```

Ở mức token:

```text
state_t  = prompt + generated tokens so far
action_t = next token
policy   = distribution over vocabulary
reward   = score cho full response hoặc từng step
```

| RL concept | LLM equivalent |
|---|---|
| State | Prompt, conversation history, generated prefix, tool observations |
| Action | Next token, full response, tool call |
| Policy | Language model `πθ` |
| Episode | Một response hoặc multi-turn interaction |
| Reward | Human preference, reward model score, verifier, task success |
| Environment | User, evaluator, tool environment, simulator |
| Reference policy | SFT model hoặc base model dùng để giữ KL |

<Callout type="info" title="Reward thường ở cấp response, action ở cấp token">
LLM chọn từng token, nhưng reward thường chỉ có sau khi response hoàn chỉnh. Đây là credit assignment problem: token nào chịu trách nhiệm cho reward tốt/xấu? PPO giải quyết bằng advantage/value estimates; DPO tránh một phần bằng pairwise preference objective.
</Callout>

## 4. Từ Pretraining đến SFT đến RLHF/DPO

### Pretraining

Pretraining học next-token prediction trên dữ liệu lớn:

```text
maximize log P(next_token | previous_tokens)
```

Nó tạo năng lực ngôn ngữ và kiến thức nền, nhưng không đảm bảo model tuân thủ instruction hoặc preference con người.

### Supervised Fine-Tuning (SFT)

SFT train model bắt chước ideal responses:

```text
prompt -> high-quality answer
```

SFT giúp model biết format instruction-following, style và domain examples. Nhưng SFT vẫn là imitation learning: model học từ câu trả lời có sẵn, không trực tiếp học “response nào được ưa thích hơn” khi có nhiều lựa chọn.

### RLHF / DPO

Preference optimization học từ so sánh:

```text
prompt x
chosen response y_w
rejected response y_l
```

Mục tiêu: tăng xác suất response được chọn so với response bị từ chối, đồng thời không làm model drift quá xa khỏi language behavior tốt.

## 5. Preference Data Schema

Dữ liệu preference tối thiểu:

```json
{
  "prompt": "Giải thích sự khác nhau giữa PPO và DPO cho AI Engineer.",
  "chosen": "PPO dùng reward model và policy optimization với KL..., DPO tối ưu trực tiếp chosen/rejected...",
  "rejected": "PPO và DPO giống nhau, đều chỉ là fine-tuning bình thường.",
  "domain": "rlhf",
  "rubric": "technical_correctness",
  "source": "human_annotator",
  "annotator_id": "anon_123"
}
```

Preference data có thể là:

| Loại | Ví dụ | Lưu ý |
|---|---|---|
| Pairwise | chosen vs rejected | Phổ biến nhất cho reward model/DPO |
| Ranking | 4 responses được xếp hạng | Có thể chuyển thành nhiều pairs |
| Scalar rating | 1–5 stars | Dễ thu thập nhưng noisy |
| Multi-turn preference | So sánh cả conversation | Gần production hơn, khó annotate hơn |
| AI feedback | LLM judge chọn response | Rẻ hơn, có judge bias |

Quality rules:

- Annotator phải có rubric rõ.
- Cần đo inter-annotator agreement.
- Tránh length bias: response dài hơn không mặc định tốt hơn.
- Tách domain labels để eval theo từng nhóm.
- Lưu prompt version/model version sinh candidates.
- Có safety labels riêng cho harmful/over-refusal cases.

<Callout type="warning" title="Preference data là sản phẩm dữ liệu quan trọng">
Nếu bạn không version preference dataset, không biết response được sinh bởi model nào, prompt nào, rubric nào, thì không thể debug reward model hoặc DPO regression. Hãy áp dụng tư duy data provenance từ Phase 14.
</Callout>

## 6. Reward Model Training

Reward model nhận prompt + response và trả về một scalar:

```text
rφ(x, y) = reward score của response y cho prompt x
```

Với pairwise preference, ta muốn:

```text
rφ(x, chosen) > rφ(x, rejected)
```

Bradley-Terry model:

```text
P(chosen > rejected) = sigmoid(r_chosen - r_rejected)
```

Loss:

```text
L = -log sigmoid(r_chosen - r_rejected)
```

Intuition:

- Nếu reward chosen cao hơn rejected nhiều, loss thấp.
- Nếu reward rejected cao hơn chosen, loss cao.
- Reward model học thứ tự tương đối, không phải điểm tuyệt đối hoàn hảo.

Reward model head thường là một scalar head trên hidden state cuối:

```text
Transformer encoder/decoder hidden states -> pooling/last token -> linear head -> scalar reward
```

## 7. Reward Model Evaluation

Reward model phải được đánh giá như một model riêng, không chỉ dùng ngay vào PPO.

| Metric | Ý nghĩa |
|---|---|
| Pairwise accuracy | Reward model chọn đúng chosen bao nhiêu phần trăm |
| Calibration | Score có tương quan với confidence thật không |
| Domain breakdown | Model có yếu ở domain nào không |
| Length bias check | Có luôn thích response dài hơn không |
| Safety eval | Có thưởng response unsafe không |
| OOD eval | Có fail trên prompt ngoài distribution không |
| Adversarial eval | Có bị prompt/response đánh lừa không |

<Callout type="danger" title="Reward model không phải chân lý">
Reward model chỉ là mô hình học từ preference data. PPO sẽ tối ưu mạnh theo reward model, nên mọi bias của reward model có thể bị khuếch đại thành hành vi production.
</Callout>

## 8. PPO for LLMs

Pipeline PPO RLHF cơ bản:

```text
1. Sample prompts từ dataset.
2. Current policy sinh responses.
3. Reward model chấm responses.
4. Tính KL penalty với reference model.
5. Tính final reward.
6. Estimate advantages/value.
7. PPO update policy.
8. Monitor KL, reward, entropy, eval metrics.
```

Final reward thường có dạng:

```text
final_reward = reward_model_score - β * KL(policy || reference)
```

Trong đó:

| Thành phần | Ý nghĩa |
|---|---|
| `reward_model_score` | Model reward dự đoán response tốt đến đâu |
| `KL(policy || reference)` | Policy mới lệch khỏi SFT/reference bao nhiêu |
| `β` | Mức phạt drift |

PPO cho LLM cần log probabilities:

```text
logprob_policy(token_t)
logprob_reference(token_t)
```

KL token-level thường được cộng qua response để đo policy drift.

## 9. Vì sao KL Constraint quan trọng?

Nếu không có KL constraint, policy có thể:

- Exploit reward model bằng pattern lặp.
- Mất fluency/ngữ pháp.
- Trả lời quá dài hoặc quá ngắn.
- Over-refuse để tránh lỗi.
- Sinh format kỳ lạ mà reward model thích.

Nếu KL quá mạnh:

- Policy gần như không học.
- Reward cải thiện rất ít.
- DPO/PPO chỉ giống SFT checkpoint.

Nếu KL quá yếu:

- Reward tăng nhanh nhưng behavior drift.
- Human eval giảm.
- Safety regression tăng.

<Callout type="tip" title="Monitor KL như monitor CPU/GPU trong serving">
Trong RLHF, KL là metric vận hành cốt lõi. Reward tăng nhưng KL cũng tăng mất kiểm soát là dấu hiệu policy đang drift, không phải dấu hiệu model tốt hơn.
</Callout>

## 10. DPO: Direct Preference Optimization

DPO tránh train reward model explicit và tránh PPO loop phức tạp. Nó dùng trực tiếp chosen/rejected pairs để tối ưu policy.

Intuition:

```text
Với cùng prompt x:
  tăng xác suất chosen response
  giảm xác suất rejected response
  so với reference model
```

DPO loss ở mức thực dụng:

```text
L_DPO = -log sigmoid(
  β * [
    (logπθ(chosen|x) - logπref(chosen|x))
    -
    (logπθ(rejected|x) - logπref(rejected|x))
  ]
)
```

Ý nghĩa:

| Thành phần | Vai trò |
|---|---|
| `πθ` | Policy/model đang train |
| `πref` | Reference model, thường là SFT checkpoint |
| `chosen` | Response được ưa thích |
| `rejected` | Response kém hơn |
| `β` | Điều khiển độ mạnh của preference/KL-like regularization |

DPO được ưa chuộng vì:

- Không cần reward model riêng.
- Không cần online rollout/PPO loop.
- Dễ implement bằng `TRL`/`DPOTrainer`.
- Ổn định hơn cho nhiều team nhỏ.

Nhưng DPO vẫn phụ thuộc mạnh vào preference data. Data bias vẫn thành model bias.

## 11. PPO RLHF vs DPO

| Tiêu chí | PPO RLHF | DPO |
|---|---|---|
| Cần reward model? | Có | Không explicit |
| Cần online generation trong loop? | Thường có | Không, dùng dataset pairs |
| Complexity | Cao | Thấp hơn |
| Compute cost | Cao hơn | Thường thấp hơn |
| Stability | Khó hơn, cần KL/reward tuning | Thường dễ hơn |
| Flexibility | Có thể tối ưu reward động/online | Giới hạn bởi preference dataset |
| Khi dùng | Reward model tốt, cần online policy optimization | Fine-tune preference offline, team muốn đơn giản |

Rule-of-thumb:

```text
Nếu bạn là team AI engineering vừa/nhỏ:
  bắt đầu với SFT + DPO + eval nghiêm túc.
Nếu bạn có reward model tốt, infra mạnh và cần optimize online behavior:
  cân nhắc PPO RLHF.
```

## 12. RLAIF và Constitutional AI

### RLAIF

RLAIF là RL from AI Feedback: dùng AI model làm judge thay vì hoặc bổ sung human feedback.

Lợi ích:

- Rẻ hơn human annotation.
- Scale nhanh.
- Có thể generate feedback liên tục.
- Hữu ích cho domain có rubric rõ.

Rủi ro:

- Judge bias.
- Model tự củng cố lỗi của model khác.
- LLM judge thích văn phong hơn correctness.
- Thiếu grounding từ con người.

### Constitutional AI

Constitutional AI dùng một bộ nguyên tắc (“constitution”) để model tự critique/revise hoặc để judge đánh giá response.

```text
Principle: Không cung cấp hướng dẫn gây hại.
Prompt: user hỏi cách tấn công hệ thống.
Response draft: ...
Critique: response vi phạm principle.
Revision: từ chối an toàn và gợi ý hướng hợp pháp.
```

Constitutional AI giúp scale safety feedback, nhưng constitution phải được thiết kế, review và audit như policy tài liệu nghiêm túc.

## 13. PRMs vs ORMs

### Outcome Reward Model (ORM)

ORM chấm điểm output cuối cùng.

```text
Prompt + final answer -> reward
```

Ưu điểm:

- Dễ label hơn.
- Phù hợp nhiều tasks.
- Chi phí thấp hơn PRM.

Nhược điểm:

- Không biết reasoning step nào sai.
- Có thể thưởng câu trả lời đúng nhờ may mắn.
- Khó hướng dẫn model cải thiện process.

### Process Reward Model (PRM)

PRM chấm từng bước reasoning.

```text
Step 1 -> reward
Step 2 -> reward
Step 3 -> reward
Final answer -> reward
```

Hữu ích cho math, code, reasoning dài vì model có thể học quy trình đúng, không chỉ đáp án cuối.

| Reward type | Phù hợp | Trade-off |
|---|---|---|
| ORM | General chat, preference response | Đơn giản nhưng ít insight reasoning |
| PRM | Math/code/reasoning multi-step | Label đắt, rubric khó, có thể over-constrain style |

<Callout type="warning" title="PRM không tự động làm model reasoning thật hơn">
Nếu step labels thưởng văn phong “có vẻ hợp lý” thay vì logic đúng, PRM có thể tạo ra reasoning dài và thuyết phục nhưng sai. Process supervision cần verifier/rubric rất chặt.
</Callout>

## 14. Implementation Playbook: Reward Model Skeleton

Ví dụ skeleton PyTorch cho Bradley-Terry loss. Trong production, bạn sẽ dùng `transformers`, tokenizer, padding, distributed training, eval split và logging đầy đủ.

```python
import torch
import torch.nn as nn
from transformers import AutoModel, AutoTokenizer

MODEL_ID = "distilbert-base-uncased"

tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)
base = AutoModel.from_pretrained(MODEL_ID)

class RewardModel(nn.Module):
    def __init__(self, base_model):
        super().__init__()
        self.base = base_model
        self.reward_head = nn.Linear(base_model.config.hidden_size, 1)

    def forward(self, input_ids, attention_mask):
        outputs = self.base(input_ids=input_ids, attention_mask=attention_mask)
        cls_hidden = outputs.last_hidden_state[:, 0]
        reward = self.reward_head(cls_hidden).squeeze(-1)
        return reward


def bt_loss(chosen_rewards, rejected_rewards):
    return -torch.nn.functional.logsigmoid(chosen_rewards - rejected_rewards).mean()


reward_model = RewardModel(base)
optimizer = torch.optim.AdamW(reward_model.parameters(), lr=2e-5)

batch = {
    "prompt": ["Explain PPO briefly."],
    "chosen": ["PPO is a policy optimization method that limits update size with a clipped objective."],
    "rejected": ["PPO is just supervised learning."],
}

chosen_text = [p + "\n" + y for p, y in zip(batch["prompt"], batch["chosen"])]
rejected_text = [p + "\n" + y for p, y in zip(batch["prompt"], batch["rejected"])]

chosen_tokens = tokenizer(chosen_text, padding=True, truncation=True, return_tensors="pt")
rejected_tokens = tokenizer(rejected_text, padding=True, truncation=True, return_tensors="pt")

chosen_rewards = reward_model(**chosen_tokens)
rejected_rewards = reward_model(**rejected_tokens)
loss = bt_loss(chosen_rewards, rejected_rewards)

optimizer.zero_grad()
loss.backward()
optimizer.step()

print("loss", loss.item())
print("chosen_rewards", chosen_rewards.detach())
print("rejected_rewards", rejected_rewards.detach())
```

Production checklist:

- Train/validation/test split theo prompt, không chỉ theo pair.
- Pairwise accuracy theo domain.
- Check length bias.
- Reward score distribution monitoring.
- Store reward model version in training metadata.

## 15. Implementation Playbook: DPO với TRL-style Config

DPO thường được chạy bằng `trl` + PEFT/LoRA cho model vừa/nhỏ.

```python
from datasets import load_dataset
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig
from trl import DPOConfig, DPOTrainer

MODEL_ID = "Qwen/Qwen2.5-1.5B-Instruct"
DATASET_ID = "your-org/preference-dataset"

tokenizer = AutoTokenizer.from_pretrained(MODEL_ID)
model = AutoModelForCausalLM.from_pretrained(MODEL_ID, device_map="auto")
ref_model = AutoModelForCausalLM.from_pretrained(MODEL_ID, device_map="auto")

dataset = load_dataset(DATASET_ID, split="train")
# Expected columns: prompt, chosen, rejected

peft_config = LoraConfig(
    r=16,
    lora_alpha=32,
    lora_dropout=0.05,
    target_modules=["q_proj", "v_proj"],
    task_type="CAUSAL_LM",
)

training_args = DPOConfig(
    output_dir="outputs/dpo-qwen-1.5b",
    beta=0.1,
    learning_rate=5e-6,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=8,
    num_train_epochs=1,
    max_prompt_length=1024,
    max_length=1536,
    logging_steps=10,
    save_steps=500,
    report_to="wandb",
)

trainer = DPOTrainer(
    model=model,
    ref_model=ref_model,
    args=training_args,
    train_dataset=dataset,
    tokenizer=tokenizer,
    peft_config=peft_config,
)

trainer.train()
trainer.save_model()
```

Các tham số cần hiểu:

| Config | Ý nghĩa |
|---|---|
| `beta` | Mức preference strength/KL-like regularization |
| `max_prompt_length` | Tránh prompt dài làm OOM |
| `max_length` | Prompt + response length |
| `learning_rate` | DPO thường dùng LR nhỏ |
| LoRA `r` | Capacity adapter |
| `ref_model` | Reference distribution để tránh drift |

| Eval | Câu hỏi trả lời |

| Safety eval | Jailbreak, harmful content, privacy leak, over-refusal |
| Regression eval | Model có quên năng lực cũ không? |

</Callout>

## 17. Failure Modes trong RLHF/DPO

<Callout type="danger" title="Pitfall 1: Reward hacking">
**Triệu chứng:** Reward model score tăng nhưng human win rate giảm.

**Nguyên nhân:** Policy khai thác shortcut trong reward model.

**Cách khắc phục:** Held-out human eval, adversarial eval, KL control, reward model refresh và multi-metric gating.
</Callout>

<Callout type="warning" title="Pitfall 2: Verbosity bias">
**Triệu chứng:** Model mới trả lời dài hơn nhiều dù không cần thiết.

**Nguyên nhân:** Preference data hoặc judge thích response dài/đầy đủ hơn.

**Cách khắc phục:** Normalize rubric theo correctness/conciseness, thêm length-controlled pairs, đánh giá cost/output tokens.
</Callout>

<Callout type="warning" title="Pitfall 3: Sycophancy">
**Triệu chứng:** Model đồng ý với user ngay cả khi user sai.

**Nguyên nhân:** Preference data thưởng câu trả lời “dễ chịu” hơn câu trả lời phản biện đúng.

**Cách khắc phục:** Thêm eval chống sycophancy, preference data yêu cầu correct disagreement, rubric ưu tiên truthfulness.
</Callout>

<Callout type="warning" title="Pitfall 4: Over-refusal">
**Triệu chứng:** Model từ chối quá nhiều câu hỏi hợp lệ.

**Nguyên nhân:** Safety reward/penalty quá mạnh hoặc data safety thiếu positive safe completions.

**Cách khắc phục:** Cân bằng harmlessness với helpfulness, thêm borderline allowed prompts và measure false refusal rate.
</Callout>

<Callout type="danger" title="Pitfall 5: Data contamination">
**Triệu chứng:** Eval score cao bất thường nhưng production không cải thiện.

**Nguyên nhân:** Preference training data chứa eval prompts hoặc benchmark leakage.

**Cách khắc phục:** Deduplicate, version datasets, holdout nghiêm ngặt, audit overlap bằng semantic similarity.
</Callout>

## Answer key

1. Base model → SFT → preference data → reward model → PPO/policy optimization → evaluation → deployment.
2. Nhận prompt + response, output scalar reward score.
3. Giữ policy gần reference/SFT model, giảm drift, collapse và reward model exploitation.
4. Không. DPO tối ưu trực tiếp từ chosen/rejected pairs và reference model.
5. Độ mạnh của preference optimization/KL-like regularization; beta cao thường update preference mạnh hơn.
6. RLAIF dùng AI feedback/judge thay cho hoặc bổ sung human feedback.

## Điểm cần nhớ

- LLM có thể xem như policy sinh token/response; alignment tối ưu policy theo preference/reward.
- Reward model học từ pairwise preferences bằng Bradley-Terry loss, nhưng nó không phải ground truth.
- PPO RLHF dùng reward model score cộng KL penalty để tối ưu policy mà không drift quá xa reference model.
- DPO đơn giản hơn PPO vì tối ưu trực tiếp chosen/rejected pairs, thường là lựa chọn thực dụng cho nhiều team.
- RLAIF và Constitutional AI giúp scale feedback nhưng mang judge/principle bias cần audit.
- PRM chấm từng bước reasoning, ORM chấm output cuối; chọn tùy task và chi phí labeling.
- Evaluation sau RLHF/DPO phải gồm human/LLM win rate, task correctness, safety, regression và cost/latency.
