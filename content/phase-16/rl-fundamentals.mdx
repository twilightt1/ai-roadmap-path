Phase 15 đã giúp bạn tối ưu cách chạy model: giảm VRAM bằng quantization, tăng throughput bằng batching, chọn SLM đúng workload và serve model bằng backend hiệu quả. Nhưng một model chạy nhanh chưa chắc hành xử đúng. Nó có thể gọi tool sai, tối ưu metric sai, quá tự tin khi không biết, hoặc học cách né evaluator thay vì giải quyết nhiệm vụ.

Reinforcement Learning (RL - học tăng cường) là framework để tối ưu **hành vi theo reward**. Với AI Engineer, RL không chỉ là robot hay game. Nó là nền tảng để hiểu RLHF, PPO, DPO, reward models, agentic training và các lỗi alignment như reward hacking.

## 1. The Incident (Tình huống sự cố)

## 2. Architecture Blueprint (Vòng lặp RL)

RL có thể được hiểu như một vòng lặp giữa agent và environment:

```text
┌────────────────────────────────────────────────────────────┐
│                         Agent                              │
│  policy π(a|s): quan sát state, chọn action                 │
└───────────────────────────┬────────────────────────────────┘
                            │ action a_t
                            ▼
┌────────────────────────────────────────────────────────────┐
│                      Environment                           │
│  nhận action, chuyển state, trả reward và next_state        │
└───────────────────────────┬────────────────────────────────┘
                            │ reward r_t, next_state s_{t+1}
                            ▼
┌────────────────────────────────────────────────────────────┐
│                    Learning Update                         │
│  cập nhật value / Q-function / policy để tăng future return │
└────────────────────────────────────────────────────────────┘
```

Mapping sang AI systems:

| RL concept | Game/robot | LLM chatbot | Tool-using agent |
|---|---|---|---|
| Agent | Người chơi | Language model | Planner/executor agent |
| Environment | Game world | User + evaluator | Tools, APIs, browser, codebase |
| State | Màn hình/vị trí | Prompt + history + generated prefix | Task state + tool results |
| Action | Move left/right | Next token hoặc response | Tool call, code edit, message |
| Reward | Điểm game | Human preference / judge score | Task success, cost penalty, safety score |
| Policy | Strategy chơi | Token distribution | Routing/tool-use strategy |

## 3. RL khác Supervised Learning như thế nào?

Trong supervised learning, bạn có input và label đúng ngay lập tức:

```text
Input: email text
Label: spam / not_spam
Loss: prediction khác label bao nhiêu?
```

Trong RL, agent chọn action, nhận reward có thể trễ, và dữ liệu phụ thuộc vào chính policy hiện tại:

```text
State: conversation so far
Action: chọn câu trả lời hoặc tool call
Reward: user satisfaction, task success, evaluator score
Challenge: reward có thể xuất hiện cuối episode, không phải sau từng token
```

| Khía cạnh | Supervised Learning | Reinforcement Learning |
|---|---|---|
| Tín hiệu học | Label đúng/sai | Reward scalar |
| Dữ liệu | Dataset cố định | Dữ liệu thay đổi theo policy |
| Mục tiêu | Predict label | Tối đa hóa cumulative reward |
| Lỗi khó | Overfit labels | Reward hacking, exploration failure |
| Ví dụ LLM | SFT imitate answers | RLHF/DPO optimize preferences |

<Callout type="info" title="SFT không phải RLHF">
Supervised Fine-Tuning dạy model bắt chước câu trả lời tốt có sẵn. RLHF/DPO dùng preference/reward để thay đổi xác suất chọn câu trả lời theo mục tiêu mong muốn. Hai bước này bổ sung nhau, không thay thế nhau hoàn toàn.
</Callout>

## 4. MDP: Markov Decision Process

Nền tảng toán học của RL là Markov Decision Process (MDP), thường viết là:

```text
MDP = (S, A, P, R, γ)
```

Trong đó:

| Thành phần | Ý nghĩa |
|---|---|
| `S` | Tập state có thể xảy ra |
| `A` | Tập action agent có thể chọn |
| `P(s' | s, a)` | Xác suất environment chuyển sang state mới `s'` khi ở `s` và chọn `a` |
| `R(s, a, s')` | Reward nhận được khi chuyển trạng thái |
| `γ` | Discount factor, mức quan tâm reward tương lai |

### Markov property

Markov property nói rằng tương lai chỉ cần phụ thuộc vào state hiện tại, không cần toàn bộ quá khứ:

```text
P(s_{t+1} | s_t, a_t, history_before_t) = P(s_{t+1} | s_t, a_t)
```

Trong thực tế, nhiều AI systems không hoàn toàn Markov. Ví dụ chatbot có history dài, user intent ẩn, tool state không đầy đủ. AI Engineer thường phải thiết kế state representation đủ tốt để gần Markov nhất có thể.

## 5. Agent, Environment, State, Action, Reward

### Agent

Agent là thứ ra quyết định. Trong LLM system, agent có thể là:

- Một language model sinh token.
- Một planner chọn tool.
- Một router chọn model/provider.
- Một policy quyết định khi nào hỏi lại user.

### Environment

Environment là phần còn lại của thế giới:

- User phản hồi.
- Tool trả kết quả.
- Code test pass/fail.
- Evaluator chấm điểm.
- Marketplace/recommender phản ứng.

### State

State là thông tin agent dùng để quyết định action. Với LLM:

```text
state = system prompt
      + user prompt
      + conversation history
      + generated prefix
      + tool observations
      + retrieved context
```

### Action

Action có thể đơn giản hoặc khổng lồ:

- CartPole: left/right.
- Recommender: recommend item.
- LLM: chọn token tiếp theo trong vocabulary 50k+.
- Tool agent: gọi search, edit file, run test, ask user.

### Reward

Reward là tín hiệu mục tiêu. Nó có thể là:

- +1 nếu thắng game.
- Accuracy hoặc F1.
- Human preference score.
- Reward model output.
- Task success trừ chi phí tool/API.
- Safety penalty nếu vi phạm policy.

<Callout type="warning" title="Reward là thiết kế sản phẩm, không chỉ là số học">
Nếu reward sai, agent sẽ tối ưu sai. Reward càng gần metric business/user thật càng tốt, nhưng càng khó đo. Đây là gốc của nhiều vấn đề alignment và safety.
</Callout>

---

## 6. Policy: π(a|s)

Policy là chiến lược chọn action trong một state.

```text
π(a|s) = xác suất chọn action a khi đang ở state s
```

Có hai loại cơ bản:

| Policy | Ý nghĩa | Ví dụ |
|---|---|---|
| Deterministic | Luôn chọn một action cụ thể | Nếu state A thì đi trái |
| Stochastic | Chọn action theo phân phối xác suất | 70% chọn A, 30% chọn B |

LLM là stochastic policy tự nhiên:

```text
state = prompt + generated prefix
policy = probability distribution over next tokens
next action = sampled token
```

Temperature, top-p và sampling strategy làm thay đổi policy lúc inference. Trong RLHF/DPO, ta cập nhật weights để policy gán xác suất cao hơn cho responses được ưa thích.

---

## 7. Return và Discount Factor γ

Agent không chỉ tối ưu reward hiện tại mà tối ưu tổng reward tương lai, gọi là return:

```text
G_t = r_t + γ r_{t+1} + γ² r_{t+2} + γ³ r_{t+3} + ...
```

`γ` nằm giữa 0 và 1:

| γ | Hành vi |
|---|---|
| Gần 0 | Agent gần như chỉ quan tâm reward ngay lập tức |
| 0.9–0.99 | Agent quan tâm tương lai đáng kể |
| Gần 1 | Long-horizon planning, nhưng learning có thể khó/stability kém hơn |

Ví dụ:

- Chatbot chỉ tối ưu immediate reward có thể trả lời câu dễ dãi để user thích ngay.
- Coding agent cần long-horizon reward: viết code, chạy test, sửa lỗi, cuối cùng task mới thành công.
- Tool agent cần cân bằng: gọi thêm tool có thể tăng accuracy nhưng tăng cost/latency.

## 8. Value Function, Q-Function và Advantage

### Value function

Value function đo state tốt đến đâu nếu đi theo policy `π`:

```text
Vπ(s) = expected return khi bắt đầu ở state s và đi theo policy π
```

### Q-function

Q-function đo action tốt đến đâu trong một state:

```text
Qπ(s, a) = expected return khi ở state s, chọn action a, rồi đi theo policy π
```

### Advantage

Advantage đo action đó tốt hơn mức trung bình của state bao nhiêu:

```text
Aπ(s, a) = Qπ(s, a) - Vπ(s)
```

Intuition:

- `A > 0`: action tốt hơn kỳ vọng, nên tăng xác suất.
- `A < 0`: action tệ hơn kỳ vọng, nên giảm xác suất.
- `A ≈ 0`: action bình thường, không cần update mạnh.

Advantage là cầu nối quan trọng để hiểu Actor-Critic, PPO và RLHF.

## 9. Bellman Intuition

Bellman equation nói rằng giá trị hiện tại bằng reward ngay bây giờ cộng với giá trị tương lai đã discount.

```text
Vπ(s) = E[ r + γ Vπ(s') ]
```

Với optimal Q-function, ta chọn action tốt nhất ở state tiếp theo:

```text
Q*(s, a) = E[ r + γ max_a' Q*(s', a') ]
```

Không cần nhớ proof. Hãy nhớ ý nghĩa:

```text
Giá trị của một quyết định = reward ngay lập tức + giá trị tốt nhất có thể đạt được sau quyết định đó.
```

Đây là nền của Q-learning và nhiều value-based algorithms.

## 10. Exploration vs Exploitation

Agent phải cân bằng:

- **Exploitation:** chọn action đang biết là tốt.
- **Exploration:** thử action chưa chắc để học thêm.

### Epsilon-greedy

```text
Với xác suất ε: chọn action random
Với xác suất 1-ε: chọn action tốt nhất hiện tại
```

Ban đầu `ε` cao để khám phá, sau đó decay để khai thác.

### UCB

Upper Confidence Bound ưu tiên action có reward cao hoặc chưa được thử đủ:

```text
score(action) = estimated_value + exploration_bonus
```

### Entropy-based exploration

Trong policy gradient, entropy bonus khuyến khích policy không collapse quá sớm vào một action duy nhất.

<Callout type="tip" title="Exploration trong LLM không chỉ là temperature">
Tăng temperature giúp output đa dạng hơn, nhưng exploration cho training cần được đánh giá bằng reward/evaluator. Random hơn không đồng nghĩa học tốt hơn.
</Callout>

## 11. RL Taxonomy for Engineers

| Trục phân loại | Ý nghĩa | Ví dụ |
|---|---|---|
| Value-based | Học giá trị action/state | Q-learning, DQN |
| Policy-based | Tối ưu policy trực tiếp | REINFORCE, PPO |
| Actor-Critic | Kết hợp policy + value estimate | A2C, PPO |
| Model-free | Không học mô hình environment | Q-learning, PPO |
| Model-based | Học/sử dụng mô hình dynamics | planning, world models |
| On-policy | Học từ data của policy hiện tại | PPO, SARSA |
| Off-policy | Học từ data policy khác/replay | Q-learning, DQN |
| Online RL | Agent tương tác environment trong training | games, robotics |
| Offline RL | Học từ logs/dataset có sẵn | recommender, preference data |
| Single-agent | Một agent tối ưu reward | CartPole |
| Multi-agent | Nhiều agent cùng học/tương tác | self-play, debate |

Trong LLM alignment, nhiều pipeline hiện đại giống preference optimization/offline RL hơn là online robot RL cổ điển.

## 12. Implementation Playbook: Multi-Armed Bandit Mini-Lab

Bandit là phiên bản đơn giản của RL: không có state chuyển phức tạp, chỉ có nhiều “cánh tay” với reward khác nhau. Nó giúp bạn thấy exploration/exploitation rõ nhất.

```python
import random
import math

TRUE_REWARD_PROBS = [0.05, 0.10, 0.20, 0.15]
N_ACTIONS = len(TRUE_REWARD_PROBS)
N_STEPS = 5000
EPSILON = 0.1

q_values = [0.0 for _ in range(N_ACTIONS)]
action_counts = [0 for _ in range(N_ACTIONS)]
total_reward = 0


def pull_arm(action: int) -> int:
    return 1 if random.random() < TRUE_REWARD_PROBS[action] else 0


def choose_action_epsilon_greedy() -> int:
    if random.random() < EPSILON:
        return random.randrange(N_ACTIONS)
    return max(range(N_ACTIONS), key=lambda a: q_values[a])


for step in range(1, N_STEPS + 1):
    action = choose_action_epsilon_greedy()
    reward = pull_arm(action)

    action_counts[action] += 1
    total_reward += reward

    # Incremental average update
    n = action_counts[action]
    q_values[action] += (reward - q_values[action]) / n

print("Estimated action values:", q_values)
print("Action counts:", action_counts)
print("Total reward:", total_reward)
print("Average reward:", total_reward / N_STEPS)
print("Best true action:", max(range(N_ACTIONS), key=lambda a: TRUE_REWARD_PROBS[a]))
```

Thử thay `EPSILON`:

- `0.0`: agent có thể kẹt ở action đầu may mắn.
- `0.1`: cân bằng exploration/exploitation.
- `0.9`: agent khám phá quá nhiều, reward trung bình thấp.

## 13. Metrics và Evaluation

| Metric | Ý nghĩa |
|---|---|
| Episodic return | Tổng reward mỗi episode |
| Average return | Reward trung bình qua nhiều episode |
| Success rate | Tỷ lệ hoàn thành task |
| Regret | Reward mất đi so với policy tối ưu |
| Sample efficiency | Cần bao nhiêu interaction để học tốt |
| Stability across seeds | Kết quả có ổn định qua random seeds không |
| Safety violations | Agent có đạt reward bằng hành vi không an toàn không |
| Cost-aware reward | Reward đã trừ latency/API/tool cost chưa |

Với AI agents, reward cao nhưng tool cost nổ hoặc safety violation tăng thì không phải success.

## 14. Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="danger" title="Pitfall 1: Reward hacking">
**Triệu chứng:** Agent đạt reward cao nhưng hành vi rõ ràng sai mục tiêu thật.

**Nguyên nhân:** Reward proxy bị thiết kế lệch. Agent tìm shortcut để tối ưu số điểm thay vì giải quyết nhiệm vụ.

**Cách khắc phục:** Dùng nhiều metric, human review, adversarial eval, safety constraints và reward model audit. Không deploy chỉ dựa trên training reward.
</Callout>

<Callout type="warning" title="Pitfall 2: Confuse reward với label">
**Triệu chứng:** Team kỳ vọng reward hoạt động như supervised label chính xác cho từng step.

**Nguyên nhân:** RL reward thường sparse, delayed và noisy. Một action tốt hiện tại có thể chỉ tạo reward tốt vài bước sau.

**Cách khắc phục:** Thiết kế credit assignment rõ hơn, dùng shaped reward cẩn thận, hoặc chuyển task thành supervised/preference learning nếu phù hợp hơn.
</Callout>

<Callout type="warning" title="Pitfall 3: Chỉ chạy một random seed">
**Triệu chứng:** Một lần training trông rất tốt, nhưng lần sau reward curve khác hẳn.

**Nguyên nhân:** RL rất nhạy với initialization, exploration và environment randomness.

**Cách khắc phục:** Luôn chạy nhiều seeds, báo mean/std, và đánh giá policy checkpoint bằng evaluation episodes tách khỏi training.
</Callout>

<Callout type="warning" title="Pitfall 4: Reward scale quá lớn hoặc quá nhỏ">
**Triệu chứng:** Learning không ổn định, gradients explode hoặc gần như không update.

**Nguyên nhân:** Reward chưa được normalize/clipped, đặc biệt khi kết hợp nhiều nguồn reward như quality, cost và safety penalty.

**Cách khắc phục:** Normalize reward, log từng component reward riêng, và kiểm tra distribution trước khi train.
</Callout>

## 15. Cost & Training Unit Economics

| Chi phí | Trong RL cổ điển | Trong LLM/RLHF |
|---|---|---|
| Environment steps | Chạy simulator/game | Generate responses, call tools, run eval |
| Reward computation | Game score | Reward model, human feedback, LLM judge |
| Exploration | Random actions | Sampling responses/tool calls |
| Evaluation | Episodes | Human/LLM preference eval, safety eval |
| Failure cost | Agent chết trong game | Bad user experience, unsafe behavior, API cost |

RL thường đắt vì agent cần nhiều trial. Trong LLM systems, mỗi trial có thể là một response dài, một tool run hoặc một human annotation. Vì vậy offline preference learning như DPO thường hấp dẫn hơn online RL cho nhiều team.

## Answer key

1. `S`, `A`, `P`, `R`, `γ`.
2. Xác suất hoặc quy tắc chọn action `a` khi ở state `s`.
3. `V(s)` là giá trị của state; `Q(s,a)` là giá trị của việc chọn action `a` trong state đó.
4. Nếu không exploration, agent có thể kẹt ở action nhìn có vẻ tốt ban đầu nhưng không tối ưu.
5. Action tốt hơn baseline/expected value của state, nên policy nên tăng xác suất action đó.
6. Agent đạt điểm reward cao bằng cách khai thác lỗ hổng của reward proxy thay vì đạt mục tiêu thật.

## Điểm cần nhớ

- RL tối ưu hành vi theo reward, không chỉ dự đoán label.
- MDP `(S, A, P, R, γ)` là framework nền để mô hình hóa quyết định tuần tự.
- Policy chọn action; value/Q-function ước lượng tương lai; advantage đo action tốt hơn baseline bao nhiêu.
- Bellman intuition là nền của value-based methods như Q-learning.
- Reward design là rủi ro production lớn nhất: reward sai dẫn đến behavior sai.
- Với LLMs, policy chính là model distribution và reward thường đến từ human preference, reward model hoặc evaluator.
