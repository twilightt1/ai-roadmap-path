## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: gọi frontier LLM cho ticket classification 1M tickets/ngày, hóa đơn $50,000/tháng">
**Bối cảnh:** Một team build ticket classification. Họ dùng frontier LLM API cho từng ticket (1M tickets/ngày). Họ không có baseline, không có model routing, không có eval set nội bộ.

**Sự cố:** Hóa đơn $50,000/tháng. p95 latency 2s. 5,000 user nhận classification chậm. Họ không biết model có thật sự tốt hơn baseline.

1. Dùng frontier LLM cho classification nhãn cố định — quá nặng, tốn cost.
2. Không có baseline TF-IDF + Logistic Regression — không biết model có đáng cost không.
3. Không có model routing — easy queries dùng frontier LLM, hard queries dùng frontier LLM.
4. Không có eval set nội bộ — không biết model quality.
5. Không log latency/cost — không biết pipeline chậm.

**Hậu quả:** 5,000 user nhận classification chậm. Họ phải tắt chatbot, thêm baseline TF-IDF + Logistic Regression, model routing, và eval set. Hóa đơn incident response 2 tuần.
</Callout>

Bài học rút ra: model tốt nhất không chỉ là model thông minh nhất. Nó là model đạt quality đủ tốt với cost, latency, privacy, context, tooling và vận hành phù hợp. Luôn có baseline nhỏ/rẻ, eval trên data thật, và model routing.

## 3. Core intuition: architecture shapes behavior

Model family thường gắn với kiến trúc/objective:

```text
Encoder-only     → understand input
Decoder-only     → generate continuation
Encoder-decoder  → transform input text to output text
```

Không phải ranh giới tuyệt đối, nhưng là rule of thumb tốt.

| Architecture | Training style | Strength |
|---|---|---|
| Encoder-only | masked/bidirectional | classification, NER, retrieval |
| Decoder-only | next-token/causal | chat, generation, code |
| Encoder-decoder | seq2seq/text-to-text | translation, summarization |

Ví dụ:

```text
Task: classify support tickets into 20 labels
Bad default: gọi frontier chat LLM cho từng ticket
Better: fine-tuned small encoder hoặc logistic regression/TF-IDF baseline
```

```text
Task: chatbot có tool calling + long context + multimodal screenshots
Bad default: BERT
Better: decoder/chat LLM API hoặc self-host instruct model hỗ trợ tool format
```

<Callout type="tip" title="Best model = best fit under constraints">
Model tốt nhất không chỉ là model thông minh nhất. Nó là model đạt quality đủ tốt với cost, latency, privacy, context, tooling và vận hành phù hợp.
</Callout>

## 4. Encoder-only: BERT family

BERT dùng transformer encoder với bidirectional attention. Mỗi token có thể nhìn trái và phải.

Mạnh ở: text classification, sentiment/intent detection, NER, extractive QA, reranking, sentence embeddings qua variants như Sentence-BERT.

```python
from transformers import AutoTokenizer, AutoModelForSequenceClassification

checkpoint = "bert-base-uncased"
tok = AutoTokenizer.from_pretrained(checkpoint)
model = AutoModelForSequenceClassification.from_pretrained(checkpoint)

batch = tok(["I love this roadmap!"], padding=True, truncation=True, return_tensors="pt")
out = model(**batch)
print(out.logits.shape)
```

BERT không tối ưu cho free-form generation vì nó không được train để generate token tiếp theo liên tục.

<Callout type="info" title="BERT vẫn rất hữu ích trong production">
Dù decoder LLM nổi bật hơn, BERT-style models vẫn mạnh cho classification/reranking/NER khi cần latency thấp, cost rẻ và output có nhãn rõ ràng.
</Callout>

## 5. BERT variants

| Model | Ý tưởng chính |
|---|---|
| RoBERTa | train BERT mạnh hơn với recipe/data tốt hơn |
| DistilBERT | nhỏ/nhanh hơn nhờ distillation |
| DeBERTa | cải tiến attention/position representations |
| Sentence-BERT | fine-tune để tạo sentence embeddings |
| Modern embedding/reranker models | thường dùng encoder/bi-encoder/cross-encoder variants |

Khi cần embedding retrieval, bạn thường không dùng `bert-base` raw pooling tùy tiện; hãy dùng model được train cho sentence embeddings.

## 6. Decoder-only: GPT, Claude, LLaMA, Mistral, Qwen

Decoder-only autoregressive models dự đoán next token:

```text
P(next_token | previous_tokens)
```

Mạnh ở: chat/generation, instruction following, summarization flexible, code generation, reasoning tasks, tool/function calling nếu model/API hỗ trợ, agent workflows.

| Family | Deployment thường gặp | Ghi chú |
|---|---|---|
| GPT/OpenAI | API | mạnh, tooling tốt, structured outputs/tool calling |
| Claude/Anthropic | API | mạnh về writing, long context, safety style |
| Gemini/Google | API | multimodal, long context, Google ecosystem |
| LLaMA/Meta | open-weight | self-host/fine-tune phổ biến |
| Mistral/Mixtral | open/API | efficient, MoE variants |
| Qwen/Alibaba | open/API | mạnh multilingual/code ở nhiều size |
| DeepSeek-style models | open/API tùy model | reasoning/code tùy phiên bản |

<Callout type="warning" title="Tên family không đủ để chọn model">
Cùng một family có nhiều size, instruct/base variants, context lengths và quantization levels. Luôn kiểm tra model card, license, benchmark phù hợp task và eval nội bộ.
</Callout>

## 7. Base vs instruct/chat models

Base model học next-token prediction trên corpus lớn. Nó hoàn thành text, không nhất thiết follow instruction tốt.

```text
Prompt: "Explain transformers"
Base model: có thể tiếp tục như một article ngẫu nhiên
```

Instruct/chat model được instruction-tuned/RLHF/DPO để trả lời theo yêu cầu.

```text
Prompt: "Explain transformers in 5 bullets"
Chat model: trả lời đúng format hơn
```

Rule of thumb:

- Dùng **instruct/chat model** cho assistant, RAG, tool calling.
- Dùng **base model** khi nghiên cứu pretraining/fine-tuning sâu hoặc cần continuation behavior đặc biệt.

## 8. Encoder-Decoder: T5 and BART family

Encoder-decoder có encoder đọc input và decoder generate output.

T5 biến mọi task thành text-to-text:

```text
translate English to German: Hello → Hallo
summarize: long text → summary
classify sentiment: text → positive
```

Mạnh ở: translation, summarization, structured seq2seq, data-to-text, text normalization.

```python
from transformers import AutoTokenizer, AutoModelForSeq2SeqLM

checkpoint = "t5-small"
tok = AutoTokenizer.from_pretrained(checkpoint)
model = AutoModelForSeq2SeqLM.from_pretrained(checkpoint)

inputs = tok("translate English to German: Hello world", return_tensors="pt")
ids = model.generate(**inputs, max_new_tokens=20)
print(tok.decode(ids[0], skip_special_tokens=True))
```

Hiện nay decoder-only chat LLM rất linh hoạt, nhưng encoder-decoder vẫn quan trọng cho seq2seq pipelines và research.

## 9. Multimodal models

Multimodal models nhận nhiều loại input:

```text
text + image
text + audio
text + video
text + document pages
```

Ví dụ capability: image QA, screenshot understanding, chart/table extraction, scanned document understanding, video/audio summarization, UI automation with screenshots.

Các model/API phổ biến có thể gồm GPT-4o/GPT vision, Claude Vision, Gemini multimodal và open multimodal models tùy thời điểm.

<Callout type="danger" title="Multimodal không thay thế pipeline tài liệu tốt">
Gửi PDF scan 100 trang thẳng vào vision model có thể rất đắt/chậm. Production thường cần OCR, layout parsing, chunking, caching và RAG trước khi gọi model lớn.
</Callout>

## 10. API models vs open-weight/self-host models

### API models

Ưu điểm: triển khai nhanh, chất lượng cao, không cần vận hành GPU, tooling tốt (structured outputs, tool calling, batch API, safety), model updates từ provider.

Nhược điểm: dữ liệu đi qua bên thứ ba, cost tăng theo usage, ít control internals, vendor lock-in, rate limits/region constraints.

### Open-weight/self-host models

Ưu điểm: control privacy/data locality, fine-tuning sâu hơn, cost control ở scale nếu vận hành tốt, có thể custom serving/quantization, tránh phụ thuộc hoàn toàn vào một provider.

Nhược điểm: cần GPU/infra, serving phức tạp (vLLM/TGI/SGLang, batching, KV cache), quality có thể thấp hơn frontier API tùy task, phải tự lo monitoring/safety/updates.

<Callout type="tip" title="Hybrid deployment thường thực tế nhất">
Nhiều team dùng API frontier model cho task khó/low volume, và self-host small/open model cho task lặp lại/high volume/nhạy dữ liệu.
</Callout>

## 11. Model size, quantization and serving

Model lớn thường chất lượng hơn nhưng: latency cao hơn, VRAM nhiều hơn, cost/token cao hơn, throughput thấp hơn.

Quantization giảm precision trọng số:

```text
FP16/BF16 → INT8 → INT4
```

Ưu điểm: giảm VRAM, tăng khả năng chạy local, rẻ hơn.

Tradeoff: có thể giảm quality, một số task nhạy hơn task khác, cần benchmark nội bộ.

Serving engines phổ biến cho open models: vLLM, Text Generation Inference (TGI), SGLang, llama.cpp cho local/edge CPU/GPU nhẹ.

---

## 12. Context window and long-context models

Context window là số tokens model có thể nhận/generate trong một request.

Long context hữu ích cho: phân tích tài liệu dài, codebase context, legal/financial reports, multi-turn chat history.

Nhưng long context không miễn phí: latency tăng, cost tăng, attention/memory tăng, model có thể bỏ sót thông tin giữa context, retrieval vẫn thường cần thiết.

<Callout type="warning" title="Long context không tự động tốt hơn RAG">
Nếu bạn có 500 trang tài liệu, nhồi tất cả vào prompt thường tệ hơn retrieval/reranking tốt. Long context nên dùng có chọn lọc.
</Callout>

## 13. Task-based model selection

| Task | Model type gợi ý | Notes |
|---|---|---|
| Classification nhanh/rẻ | Logistic regression, BERT/DistilBERT | fine-tune nếu có labels |
| NER/POS/parsing | spaCy hoặc BERT token classifier | spaCy nếu cần nhẹ/nhanh |
| Semantic search | Sentence embedding model | chọn multilingual/domain nếu cần |
| Reranking | Cross-encoder/reranker | tốt sau retrieval top-k |
| Chat assistant | GPT/Claude/Gemini hoặc instruct open model | cần eval instruction following |
| RAG QA | Chat LLM + embedding + reranker | model phải grounded tốt |
| Code generation | code-capable decoder LLM | test bằng unit tests |
| Translation/summarization | T5/BART hoặc chat LLM | tùy format/cost |
| Vision QA/document images | Multimodal LLM + OCR/layout | tối ưu input image |
| High-volume extraction | small fine-tuned model hoặc structured LLM | schema validation |
| Sensitive/private data | self-host/open model hoặc compliant API | review policy/security |

## 14. Decision framework

Khi chọn model, hỏi theo thứ tự:

1. Task là classification, extraction, retrieval, generation hay multimodal?
2. Output có cần tự do hay schema cố định?
3. Quality target là gì và đo bằng metric nào?
4. Latency budget bao nhiêu?
5. Cost/request tối đa bao nhiêu?
6. Data có ràng buộc privacy/compliance không?
7. Có cần fine-tune không?
8. Có cần tool calling/structured output không?
9. Context length cần bao nhiêu tokens?
10. Team có khả năng vận hành GPU không?

Nếu câu trả lời cho output là "nhãn cố định", đừng mặc định dùng frontier LLM. Nếu câu trả lời là "reasoning/generation phức tạp", small classifier có thể không đủ.

## 15. Tokenization pitfalls

<Callout type="warning" title="Mistake 1: dùng LLM lớn cho mọi thứ">
Nếu task là classification nhãn cố định hoặc rule extraction đơn giản, model nhỏ/fine-tuned có thể rẻ, nhanh và ổn định hơn nhiều.
</Callout>

<Callout type="warning" title="Mistake 2: chọn model theo benchmark chung">
Benchmark tổng quát không đảm bảo model tốt trên data, ngôn ngữ, format và latency của bạn. Luôn eval bằng test set nội bộ.
</Callout>

<Callout type="danger" title="Mistake 3: quên license/privacy/compliance">
Open-weight không có nghĩa dùng thương mại thoải mái. API không có nghĩa dữ liệu được phép gửi. Kiểm tra license, data policy, retention, region và compliance.
</Callout>

## Implementation Playbook: Model Selection Matrix

```python
from dataclasses import dataclass


@dataclass
class ModelOption:
    name: str
    quality: int      # 1-5
    latency: int      # 1-5, higher is faster
    cost: int         # 1-5, higher is cheaper
    privacy: int      # 1-5
    ops: int          # 1-5, higher is easier to operate

    def score(self, weights):
        return (
            self.quality * weights["quality"] +
            self.latency * weights["latency"] +
            self.cost * weights["cost"] +
            self.privacy * weights["privacy"] +
            self.ops * weights["ops"]
        )


options = [
    ModelOption("frontier-api", quality=5, latency=3, cost=2, privacy=3, ops=5),
    ModelOption("self-host-llama", quality=4, latency=3, cost=4, privacy=5, ops=2),
    ModelOption("fine-tuned-distilbert", quality=3, latency=5, cost=5, privacy=5, ops=4),
]

weights = {
    "quality": 0.40,
    "latency": 0.20,
    "cost": 0.15,
    "privacy": 0.15,
    "ops": 0.10,
}

ranked = sorted(options, key=lambda m: m.score(weights), reverse=True)
for model in ranked:
    print(model.name, round(model.score(weights), 2))
```

Đây không phải công thức tuyệt đối. Nó ép team nói rõ tradeoffs thay vì chọn theo hype.

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: dùng LLM lớn cho mọi thứ">
**Triệu chứng:** Task là classification nhãn cố định hoặc rule extraction đơn giản. Model nhỏ/fine-tuned có thể rẻ, nhanh và ổn định hơn nhiều.

**Fix:** Dùng baseline TF-IDF + Logistic Regression. Dùng small encoder hoặc BERT. Model routing: easy queries dùng small model, hard queries dùng frontier LLM.
</Callout>

<Callout type="warning" title="Pitfall 2: chọn model theo benchmark chung">
**Triệu chứng:** Benchmark tổng quát không đảm bảo model tốt trên data, ngôn ngữ, format và latency của bạn. Model fail trên production.

**Fix:** Eval bằng test set nội bộ. Benchmark trên data thật của bạn. Chia eval theo slices.
</Callout>

<Callout type="danger" title="Pitfall 3: quên license/privacy/compliance">
**Triệu chứng:** Open-weight không có nghĩa dùng thương mại thoải mái. API không có nghĩa dữ liệu được phép gửi. Model fail trên production.

**Fix:** Kiểm tra license, data policy, retention, region và compliance. Dùng self-host model cho sensitive data. Review policy/security.
</Callout>
