Một model tối ưu chỉ tạo ra giá trị khi nó được serve đúng cách. Notebook `model.generate()` có thể đủ cho demo, nhưng production inference cần API ổn định, scheduler thông minh, monitoring rõ ràng, autoscaling, model registry, rollback và benchmark dưới tải thật. Đây là nơi các serving backend như vLLM, TGI, Ollama, llama.cpp, ONNX Runtime và TensorRT-LLM trở thành lựa chọn kiến trúc.

Model Serving Efficiency là bài toán cuối của Phase 15: gom quantization, KV cache, batching, SLMs và distillation vào một hệ thống inference có thể vận hành, đo lường và tối ưu chi phí.

## 1. The Incident (Tình huống sự cố)

<Callout type="danger" title="Báo cáo sự cố: FastAPI + Transformers làm sập GPU launch day">
**Bối cảnh:** Một startup deploy CodeLLM nội bộ để hỗ trợ developer. Backend FastAPI gọi trực tiếp `AutoModelForCausalLM.from_pretrained()` rồi chạy `model.generate()` cho từng request.

**Sự cố:** Trong ngày rollout, 300 developer bắt đầu dùng tính năng autocomplete và code explanation. GPU utilization nhìn có vẻ cao, nhưng throughput thấp, latency p95 vượt 20 giây và server thỉnh thoảng OOM.

1. Mỗi request chạy riêng, không có continuous batching.
2. KV cache bị cấp phát thiếu hiệu quả.
3. Model switching giữa code model và chat model gây cold start.
4. Không có metric TTFT/TPOT, chỉ log tổng thời gian request.
5. Không có fallback pool khi một inference worker chết.

**Hậu quả:** Team có model tốt nhưng serving stack không đủ production-grade. Người dùng mất niềm tin vì trải nghiệm chậm hơn cả gọi API bên ngoài.
</Callout>

Bài học rút ra: Serving backend quyết định model có trở thành sản phẩm hay chỉ là demo. Production AI cần inference engine chuyên dụng, không phải vòng lặp generate thủ công.

## 2. Architecture Blueprint (Production Inference Stack)

Một inference stack thực tế thường có nhiều lớp:

```text
[ Client / Product API ]
          │
          ▼
[ API Gateway / Auth / Rate Limit ]
          │
          ▼
[ Model Router ]
  - chọn SLM / LLM lớn
  - chọn quantized variant
  - fallback provider
          │
          ▼
[ Inference Server Pool ]
  ├── vLLM replicas: high-throughput GPU serving
  ├── TGI replicas: HuggingFace model serving
  ├── Ollama/llama.cpp: local/dev/edge models
  └── TensorRT-LLM: NVIDIA-optimized path
          │
          ▼
[ Observability ]
  - Prometheus metrics
  - Grafana dashboards
  - structured logs/traces
  - cost per request
          │
          ▼
[ Model Registry / Config Store ]
  - model version
  - quantization format
  - max context
  - rollout/rollback policy
```

Mục tiêu không chỉ là “trả lời được”. Mục tiêu là trả lời nhanh, ổn định, đo được, rollback được và có cost/request hợp lý.

## 3. Serving Backend Landscape

| Backend | Điểm mạnh | Khi dùng | Cẩn trọng |
|---|---|---|---|
| vLLM | PagedAttention, continuous batching, OpenAI-compatible API | High-throughput LLM serving trên GPU | Cần cấu hình VRAM/context/concurrency kỹ |
| TGI | Hugging Face ecosystem, production server, streaming tốt | Deploy HF models, tích hợp HF stack | Feature/support có thể khác vLLM theo model |
| Ollama | Local/dev workflow cực dễ, quản lý model đơn giản | PoC, desktop, internal tools, local chat | Không tối ưu concurrency như vLLM |
| llama.cpp | GGUF, CPU/Mac/edge mạnh | Local/offline/edge, quantized GGUF | Throughput GPU server không phải mục tiêu chính |
| ONNX Runtime | Cross-platform optimized inference | Encoder/classifier, smaller models, edge/server | LLM generation stack có thể cần nhiều glue code |
| TensorRT-LLM | Tối ưu NVIDIA sâu, throughput cao | Large-scale NVIDIA production | Build/compile/config phức tạp |

Không có backend duy nhất cho mọi workload. Một công ty có thể dùng Ollama cho developer local, vLLM cho production chat, ONNX Runtime cho classifier nhỏ và TensorRT-LLM cho workload GPU quy mô lớn.

## 4. vLLM: PagedAttention và Continuous Batching

vLLM phổ biến vì nó giải quyết hai bottleneck lớn của LLM serving:

1. **PagedAttention:** quản lý KV cache theo block/page, giảm lãng phí VRAM và fragmentation.
2. **Continuous batching:** thêm request mới vào batch khi các sequence cũ hoàn tất, tăng GPU utilization.

```text
Naive serving:
  request A -> generate riêng
  request B -> generate riêng
  request C -> chờ hoặc chạy cạnh tranh kém hiệu quả

vLLM serving:
  scheduler gom A/B/C theo từng iteration
  sequence nào xong thì giải phóng KV blocks
  request mới được chèn vào batch liên tục
```

vLLM cũng cung cấp OpenAI-compatible API, giúp backend đang dùng OpenAI SDK chuyển sang self-hosted model bằng cách đổi `base_url`.

```bash
docker run --runtime nvidia --gpus all \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  -p 8000:8000 \
  --ipc=host \
  vllm/vllm-openai:latest \
  --model meta-llama/Meta-Llama-3-8B-Instruct \
  --dtype auto \
  --max-model-len 4096 \
  --gpu-memory-utilization 0.85 \
  --api-key local-secret
```

<Callout type="tip" title="OpenAI-compatible API là một lợi thế vận hành">
Nếu app của bạn gọi qua OpenAI SDK hoặc LLM Gateway, bạn có thể chuyển traffic giữa OpenAI, vLLM self-hosted và fallback provider mà ít sửa code hơn. Đây là pattern đã học ở Phase 7 và Phase 13.
</Callout>

## 5. TGI: Text Generation Inference

TGI (Text Generation Inference) là serving stack từ Hugging Face, thường phù hợp khi model, tokenizer và deployment workflow đã nằm trong Hugging Face ecosystem.

Điểm mạnh:

- Streaming response.
- Tensor parallel support cho một số model.
- Tích hợp tốt với Hugging Face Hub.

Ví dụ chạy TGI:

```bash
docker run --gpus all --shm-size 1g -p 8080:80 \
```

## 7. ONNX Runtime và TensorRT-LLM

### ONNX Runtime

ONNX Runtime phù hợp khi bạn cần inference tối ưu cross-platform cho các model nhỏ hơn như encoder, classifier, reranker hoặc extractor.

Use cases:

- Intent classifier.
- Embedding/reranking model.
- Safety classifier.
- Edge/server hybrid deployment.
- CPU/GPU execution provider linh hoạt.

Nó không phải luôn là lựa chọn đơn giản nhất cho LLM generation lớn, nhưng rất đáng dùng cho các component xung quanh LLM.

### TensorRT-LLM

TensorRT-LLM là hướng tối ưu sâu cho NVIDIA GPUs.

Phù hợp khi:

- Bạn có team platform đủ mạnh.
- Workload đủ lớn để justify effort.
- Cần vắt tối đa throughput/latency trên NVIDIA fleet.
- Model/config tương đối ổn định để compile/optimize.

Trade-off:

- Build pipeline phức tạp.
- Debug khó hơn runtime cấp cao.
- Cần hiểu CUDA/NVIDIA stack tốt hơn.

## 8. Production Architecture Patterns

### Pattern 1: Router + multi-backend pool

```text
[ Request ]
    │
    ▼
[ Router ]
    ├── intent/extraction ──> ONNX/SLM pool
    ├── chat general ──────> vLLM 8B/14B pool
    ├── local/private ─────> Ollama/llama.cpp edge
    └── hard reasoning ────> frontier API fallback
```

Pattern này kết hợp bài học SLM, distillation và model routing.

### Pattern 2: Separate pools by workload

```text
Interactive Chat Pool:
  - ưu tiên TTFT thấp
  - max batch tokens vừa phải
  - p95/p99 latency nghiêm ngặt

Batch Summarization Pool:
  - ưu tiên throughput
  - batch lớn hơn
  - latency mềm hơn
```

Đừng để batch job dài làm nghẽn interactive chat.

### Pattern 3: Model registry + rollout

```text
model_name: support-router
version: v12
base_model: qwen2.5-3b-instruct
quantization: awq-int4
runtime: vllm
max_model_len: 4096
traffic_split:
  stable: 90%
  candidate: 10%
rollback_if:
  p95_latency_ms > 1200
  error_rate > 1%
  quality_score_drop > 3%
```

Model serving cần rollout/rollback giống software deployment.

## Implementation Playbook: Concurrent vLLM Benchmark

Ví dụ dưới đây gọi một vLLM OpenAI-compatible server và chạy nhiều request đồng thời để đo latency cơ bản. Đây là skeleton cho project **High-Throughput Inference Server**.

```python
import asyncio
import statistics
import time
from openai import AsyncOpenAI

client = AsyncOpenAI(
    api_key="local-secret",
    base_url="http://localhost:8000/v1",
)

MODEL = "meta-llama/Meta-Llama-3-8B-Instruct"
PROMPT = "Giải thích continuous batching trong LLM serving bằng 5 gạch đầu dòng."


async def one_request(i: int):
    start = time.perf_counter()
    first_token = None
    chunks = 0

    stream = await client.chat.completions.create(
        model=MODEL,
        messages=[{"role": "user", "content": PROMPT}],
        temperature=0.0,
        max_tokens=256,
        stream=True,
    )

    async for chunk in stream:
        delta = chunk.choices[0].delta.content
        if delta:
            chunks += 1
            if first_token is None:
                first_token = time.perf_counter()

    end = time.perf_counter()
    return {
        "request_id": i,
        "ttft": (first_token - start) if first_token else None,
        "total": end - start,
        "chunks": chunks,
    }


async def run_benchmark(concurrency: int):
    tasks = [one_request(i) for i in range(concurrency)]
    results = await asyncio.gather(*tasks)

    totals = [r["total"] for r in results]
    ttfts = [r["ttft"] for r in results if r["ttft"] is not None]

    def p95(values):
        return statistics.quantiles(values, n=20)[18] if len(values) >= 20 else max(values)

    print({
        "concurrency": concurrency,
        "total_p50": round(statistics.median(totals), 3),
        "total_p95": round(p95(totals), 3),
        "ttft_p50": round(statistics.median(ttfts), 3),
        "ttft_p95": round(p95(ttfts), 3),
    })


for concurrency in [1, 10, 50, 100]:
    asyncio.run(run_benchmark(concurrency))
```

Trong production, hãy thêm:

- prompt length variations
- output token estimates
- error/retry tracking
- GPU metrics scrape
- per-model version tags
- cost/request calculation

## Post-Mortem & Troubleshooting (SRE/Gỡ lỗi)

<Callout type="warning" title="Pitfall 1: gpu-memory-utilization đặt quá cao">
**Triệu chứng:** Server khởi động được nhưng OOM khi traffic tăng nhẹ.

**Fix:** Giảm `--gpu-memory-utilization`, đặt `max_model_len` thực tế, benchmark concurrency và theo dõi peak VRAM dưới tải.
</Callout>

<Callout type="warning" title="Pitfall 2: Max context quá dài cho mọi user">
**Triệu chứng:** GPU memory bị chiếm lớn dù phần lớn request chỉ cần context ngắn.

**Fix:** Tách pool theo context length: chat ngắn, RAG medium, long-context premium. Dùng router và quota để tránh long-context làm nghẽn mọi traffic.
</Callout>

<Callout type="danger" title="Pitfall 3: Model switching gây cold start">
**Triệu chứng:** Request đầu tiên sau khi đổi model mất hàng chục giây hoặc timeout.

**Fix:** Tránh dynamic model switching trong cùng process cho hot path. Dùng pool riêng cho từng model quan trọng, warm replicas trước rollout, và route traffic bằng gateway.
</Callout>

<Callout type="warning" title="Pitfall 4: Benchmark thiếu concurrency">
**Triệu chứng:** Demo single-user rất nhanh, production multi-user rất chậm.

**Fix:** Benchmark theo concurrency ladder: 1, 10, 50, 100, 500 users. Đo p95/p99, error rate và GPU metrics.
</Callout>

<Callout type="danger" title="Pitfall 5: Không tách interactive và batch workloads">
**Triệu chứng:** Người dùng chat realtime bị chậm mỗi khi batch summarization chạy.

**Fix:** Tách pool hoặc queue priority. Batch jobs dùng pool throughput-optimized; chat dùng pool latency-optimized với SLA riêng.
</Callout>

## Cost / Performance / Serving Unit Economics

| Quyết định | Tác động latency | Tác động chi phí | Trade-off |
|---|---|---|---|
| Dùng vLLM/TGI thay naive generate | Giảm queue/TPOT dưới concurrency | Tăng tokens/sec/GPU | Cần vận hành server riêng |
| Quantized model serving | Có thể giảm TPOT/VRAM | GPU nhỏ hơn hoặc concurrency cao hơn | Quality/compatibility risk |
| SLM pool | Latency thấp cho task hẹp | Giảm request sang model lớn | Cần router/fallback |
| Tách workload pools | Bảo vệ p95/p99 | Có thể tăng idle capacity | Vận hành nhiều pool hơn |
| TensorRT-LLM | Tối ưu sâu | Tốt ở scale lớn | Engineering cost cao |
| Autoscaling GPU | Bảo vệ burst traffic | Tránh overprovision lâu dài | Cold start và capacity availability |

Công thức cuối cùng của Phase 15:

```text
Sustainable AI serving = quality đủ tốt
                         + latency đạt SLA
                         + throughput cao
                         + cost/request thấp
                         + observability rõ
                         + rollback được
```

Nếu thiếu một trong các yếu tố này, hệ thống có thể demo tốt nhưng production sẽ đau.

## Answer key

1. `model.generate()` không có continuous batching, KV cache management, scheduler và streaming; khi concurrency cao, GPU utilization thấp, queue time tăng và server OOM.
2. PagedAttention giảm KV cache fragmentation và lãng phí VRAM; continuous batching thêm request mới vào batch khi sequence cũ xong, tăng GPU utilization.
3. Ollama/llama.cpp phù hợp cho local/dev/edge, CPU/Mac/privacy/offline; vLLM/TGI phù hợp cho high-concurrency GPU server.
4. Batch job dài làm nghẽn interactive chat vì scheduler ưu tiên throughput; tách pool bảo vệ SLA latency cho interactive và cho phép batch pool tối ưu throughput riêng.
5. Model registry giúp version, rollback và canary deployment; khi model mới gây regression, rollback về version cũ bằng config thay vì redeploy code.

- Ollama/llama.cpp/GGUF rất mạnh cho local/dev/edge, nhưng cần benchmark trước khi dùng cho high-concurrency server.

- Serving efficiency phải đo TTFT, TPOT, queue time, p95/p99 latency, GPU utilization, throughput và cost/request.
- Tách pool theo workload/model/context length giúp bảo vệ SLA và giảm blast radius.

## Tổng kết Phase 15 (Phase 15 Wrap-up)

Bạn vừa hoàn thành chuyên đề **Quantization, Optimization & Efficiency**:

1. **Model Quantization:** giảm precision từ FP16/BF16 xuống INT8/INT4, hiểu GPTQ, AWQ, GGUF và bitsandbytes để giảm VRAM/cost.
2. **Inference Optimization:** tối ưu prefill, decode, KV cache, continuous batching, speculative decoding, Flash Attention và parallelism.
3. **Small Language Models (SLMs):** chọn model nhỏ đúng workload để thắng về latency, privacy và unit economics.
4. **Knowledge Distillation:** chuyển hành vi từ teacher model lớn sang student model nhỏ để giảm chi phí production.
5. **Model Serving Efficiency:** đóng gói model tối ưu vào serving backend, benchmark, observe, autoscale và rollback như một hệ thống production.

Phase 15 là cầu nối giữa “model thông minh” và “dịch vụ AI bền vững về kinh tế”. Bạn không chỉ biết model nào tốt, mà còn biết làm sao chạy nó nhanh, rẻ và đáng tin cậy.
